> Consolidated actionable principles from six reference skills
> Date: 2026-03-22
> Source skills: skill-creator, skill-judge, skill-review, brainstorming, subagent-driven-development, dispatching-parallel-agents

---

## 1. Frontmatter Design (Description, Name, Triggers)

### Principles

**Description is the PRIMARY triggering mechanism.** It must answer three questions simultaneously: WHAT the skill does, WHEN to activate it, and WHAT KEYWORDS signal need. A description that requires reading the body to understand activation criteria has already failed.

**Name follows function, not aspiration.** The name field should mirror the verb-object pattern of what the skill actually executes. `hivemind-delegation-protocol` communicates both scope (hivemind) and operation (delegation protocol) without requiring context.

**Trigger specificity determines skill precision.** Vague triggers ("use when working on code") create false activations. Concrete triggers ("use when splitting work across 2+ agents, managing task dependencies, or monitoring team progress") reduce noise. Every trigger phrase should map to a real workflow moment where a human would naturally think "I need help with this."

**Required frontmatter fields:**
```yaml
name: skill-identifier
description: |
  [WHAT] - concrete action the skill performs
  [WHEN] - specific scenarios, 3-5 trigger phrases minimum
  [KEYWORDS] - technical terms that signal need
```

**Anti-pattern: "When to use" buried in body.** If activation criteria live in the SKILL.md body instead of the frontmatter description, the skill will never trigger reliably. The description IS the gate.

## 2. SKILL.md Structure (Anatomy, Layering, Progressive Disclosure)

### Anatomy

Every SKILL.md follows a fixed contract:

1. **YAML frontmatter** — name, description (required); bundled resources declared here
2. **Markdown instructions** — the executable knowledge; imperative form; under 500 lines
3. **Bundled resources** — `scripts/`, `references/`, `assets/` declared in frontmatter, loaded on demand

### 3-Layer Progressive Disclosure

| Layer | Content | Loaded When |
|-------|---------|-------------|
| **Layer 1: Metadata** | name, description, trigger phrases | Always — every session turn |
| **Layer 2: Body** | Core instructions, decision trees, patterns | Trigger matched — skill activated |
| **Layer 3: Resources** | Scripts, reference files, heavy examples | On demand — when instruction references them |

**The 500-line ceiling is not a suggestion.** SKILL.md exceeding 500 lines signals that heavy content has not been extracted to references. The body should contain decision logic and patterns, not exhaustive documentation. References carry the weight.

**Resource declaration pattern:**
```yaml
resources:
  - path: scripts/delegate.sh
    description: "Dispatch subagent with context packet"
  - path: references/decision-tree.md
    description: "Sequential vs parallel routing logic"
```

**"Do NOT Load" guard:** Resources that exist in the skill directory but are NOT declared in frontmatter must be marked with a comment header `<!-- NOT-ACTIVE: do not load without explicit instruction -->` to prevent accidental ingestion during context assembly.

### Structural Anti-Patterns

- **Over-Engineered**: Multiple nested subdirectories, custom build steps, configuration files for the skill itself
- **Wrong Location**: Skill files outside the declared skill directory, scattered across the repo
- **Invisible Skill**: Valid SKILL.md exists but is never referenced or triggered by any workflow

## 3. Writing Patterns (Imperative, Examples, Anti-Patterns, Decision Trees)

### Imperative Form with WHY

Every instruction uses imperative voice: "Dispatch the subagent with a self-contained prompt" not "You should dispatch the subagent." But imperative without WHY produces brittle instructions that break when context shifts.

**Pattern: Action + Rationale in two sentences.**
```
Dispatch each subagent with full context upfront.
Subagents should never need to read the plan file itself — they receive everything they need in the prompt.
```

### Theory of Mind

Write as if the reader has expert knowledge of their domain but no knowledge of this skill's specific procedures. Assume they understand delegation conceptually. Do not explain what delegation IS. Explain how THIS skill's delegation WORKS.

### Examples as Input/Output Pairs

Every non-trivial instruction should include a concrete example showing input and expected output:

**Input:** "Three unrelated test failures in three different modules"
**Action:** Route to parallel dispatch — three subagents, one per module
**Output:** Three independent fix summaries, reviewed sequentially after return

**Input:** "Database migration that changes a shared schema used by API and UI"
**Action:** Route to sequential dispatch — schema change first, then API, then UI
**Output:** Single chain of dependent fixes with integration verification at each gate

### Decision Trees as Tables

Binary routing decisions should use tables, not prose:

| Condition | Route To | Reason |
|-----------|----------|--------|
| Failures share no code paths | Parallel dispatch | Independence confirmed |
| Failures import same module | Sequential dispatch | Shared state risk |
| Unknown dependency | Sequential dispatch (default) | Safe assumption |

## 4. Delegation Workflow Design (Sequential/Parallel Gates, Packet Structure, Return Contracts)

### Gate Architecture

Every delegation workflow has three mandatory gates:

1. **Dispatch Gate** — Validate task independence, scope, and context completeness BEFORE sending
2. **Return Gate** — Two-stage review (spec compliance THEN code quality) BEFORE marking complete
3. **Integration Gate** — Conflict check across all returned work, full suite execution

**Gate skipping is a stop condition.** If any gate is bypassed, the workflow halts and reports the violation. No exceptions for "simple" tasks.

### Task Packet Structure

Every dispatched task must include:

```markdown
## Task Packet

### Scope
[Single problem domain — one sentence]

### Context
[All files, types, interfaces, contracts needed — subagent should not need to search]

### Constraints
[What must NOT change, what must be preserved, hard boundaries]

### Expected Output
[Exact format of return: file list, test results, summary structure]

### Stop Conditions
[When to halt and report instead of proceeding: unknown dependency, test failure, ambiguity]
```

### Return Contract

Subagents return exactly this structure:

```markdown
## Return Report

### Changes Made
- File: line range — what changed and why

### Spec Compliance
- [requirement]: met/not-met — evidence

### Tests
- Ran: [command]
- Result: pass/fail — count

### Blockers Encountered
- [any stop conditions hit, ambiguities found, assumptions made]

### Follow-up Needed
- [anything outside scope that was discovered]
```

### Sequential vs Parallel Decision Framework

**Default to sequential.** Parallel dispatch requires proving independence:

1. Do the tasks share any imports, types, or interfaces? → Sequential
2. Do the tasks modify the same directory? → Sequential
3. Could one task's output be another task's input? → Sequential
4. Are the tasks in completely separate codebases/modules with no shared state? → Parallel candidate

**Parallel candidate must pass one more check:** Can the full integration suite run against all parallel results simultaneously without ordering dependencies? If uncertain, sequential.

## 5. Subagent Dispatch Patterns (Fresh Context, Focused Scope, Self-Contained Prompts)

### Fresh Context Per Task

Each subagent receives a clean context window. Never chain subagents through shared conversation state. Context pollution from prior tasks degrades output quality measurably.

**Why this matters:** A subagent that processed a database migration will carry schema assumptions into a UI task if context is shared. Fresh context eliminates cross-contamination.

### Focused Scope: One Problem Domain Per Agent

A single subagent handles ONE of:
- One module's test failures
- One interface's type changes
- One file's refactor
- One integration point's contract update

**Never combine unrelated domains in one subagent.** "Fix the database migration AND update the UI components AND write tests" violates focused scope. Three agents, three tasks.

### Self-Contained Prompt Requirements

Every subagent prompt must contain:

| Component | Required | Example |
|-----------|----------|---------|
| **Goal** | Yes | "Fix the type error in src/core/trajectory.ts line 47" |
| **Context** | Yes | Full file content, relevant interfaces, error message |
| **Constraints** | Yes | "Do not modify the public API surface" |
| **Expected Output** | Yes | "Return the changed file content and test results" |
| **Stop Conditions** | Yes | "If the fix requires changing shared/types.ts, stop and report" |
| **Scene-Setting** | Yes | "You are working on a TypeScript plugin that uses the OpenCode SDK" |

### Common Dispatch Mistakes

| Mistake | Consequence | Fix |
|---------|-------------|-----|
| Too broad scope | Subagent produces unfocused changes | Split into atomic tasks |
| No context provided | Subagent searches for files, wastes turns | Include all needed content upfront |
| No constraints | Subagent refactors unrelated code | Explicit boundaries |
| Vague output format | Returns narrative instead of structured report | Specify exact return template |
| Missing stop conditions | Subagent proceeds through failures | Define halt criteria |

### When NOT to Dispatch in Parallel

- Failures that import the same module
- Tasks where output of one feeds input of another
- Tasks that touch the same directory tree
- Unknown dependency relationships (assume related until proven otherwise)
- Tasks requiring full-context understanding of a shared domain

## 6. Review and Gating (Two-Stage Review, Stop Conditions, Recovery)

### Two-Stage Review Protocol

**Stage 1: Spec Compliance Review**
- Does the output match the task packet's expected output format?
- Were all constraints respected?
- Were stop conditions honored (did it halt when it should have)?
- Is every requirement in the return contract addressed?

**Stage 2: Code Quality Review**
- Does the code follow project conventions?
- Are types correct? Do imports resolve?
- Is the change minimal (addresses only the scoped task)?
- Are there side effects outside the stated scope?

**Order is non-negotiable.** Spec compliance first catches scope violations before investing in quality review. A perfectly coded solution to the wrong problem is still wrong.

### Review Loops

```
Dispatch → Subagent Executes → Returns
  → Stage 1 Review → FAIL → Re-dispatch with specific feedback
  → Stage 1 Review → PASS → Stage 2 Review
    → Stage 2 Review → FAIL → Re-dispatch with specific feedback
    → Stage 2 Review → PASS → Mark complete
```

**Never skip re-review.** If a subagent was re-dispatched, the returned work goes through BOTH stages again. Partial approval ("the first fix was good, only the second needs work") creates tracking debt.

### Stop Conditions (Workflow Level)

The orchestrator halts the entire delegation chain when:
- A subagent hits a stop condition and reports it
- Two consecutive re-dispatches produce the same failure
- Stage 1 review finds a scope violation (task was incorrectly scoped)
- Integration gate detects conflicts between parallel returns

### Recovery Procedures

| Failure Type | Recovery Action |
|--------------|-----------------|
| Subagent hit stop condition | Analyze blocker, re-scope task, re-dispatch |
| Spec compliance failure | Provide specific feedback on which requirement failed, re-dispatch |
| Quality failure | Provide specific code feedback with line references, re-dispatch |
| Integration conflict | Dispatch a resolution agent with both conflicting outputs as context |
| Repeated failure (2x) | Escalate to human — task may be incorrectly specified |

### Nine-Phase Audit Adaptation (from skill-review)

For comprehensive delegation audits, adapt the 9-phase structure:

1. **Pre-Review** — Verify all task packets are complete before dispatch
2. **Standards** — Check outputs against project conventions
3. **Contract Verification** — Validate against SDK/API interfaces
4. **Code Examples** — Verify any generated code against reference patterns
5. **Cross-Task Consistency** — Check parallel returns don't conflict
6. **Dependencies** — Verify import chains and type compatibility
7. **Categorize** — Classify each finding as Critical/High/Medium/Low
8. **Fix** — Auto-fix Low/Medium; ask user for Critical/High
9. **Verify** — Re-run integration suite after fixes

## 7. Progressive Disclosure and Resource Loading (3-Layer System, Triggers, "Do NOT Load")

### 3-Layer Loading Strategy Applied to Delegation

| Layer | Delegation Content | Trigger |
|-------|-------------------|---------|
| **Always Loaded** | Dispatch decision framework (parallel vs sequential), stop conditions, return contract template | Every session — these are the delegation primitives |
| **On Activation** | Full workflow gates, review protocols, recovery procedures, packet templates | When delegation task is identified |
| **On Demand** | Audit checklists, integration test commands, specific SDK references | When reviewing, debugging, or integrating |

### Resource Loading Rules

1. **Declared resources load on skill activation.** If `scripts/delegate.sh` is declared in frontmatter, it loads when the skill triggers.
2. **Undeclared resources require explicit instruction.** A subagent should never auto-load a file not referenced in the active instructions.
3. **Heavy content goes to references, not body.** If a decision tree has more than 30 lines, extract it to `references/decision-tree.md`.
4. **"Do NOT Load" markers prevent accidental ingestion.** Any resource that could confuse or mislead if loaded out of context gets:
   ```markdown
   <!-- NOT-ACTIVE: do not load without explicit instruction -->
   ```

### Trigger Refinement

Progressive disclosure only works if triggers are precise. Apply the brainstorming principle of incremental validation:
- Start with broad trigger description
- Test against real workflow scenarios
- Narrow based on false activation rate
- Validate each refinement against 2-3 alternatives before settling

### YAGNI Applied to Resource Declaration

- If a resource is declared but never referenced in the body → remove the declaration
- If a reference file exists but no instruction points to it → mark as NOT-ACTIVE or delete
- If a script automates something that takes <30 seconds manually → the script is overhead, remove it

## 8. Anti-Patterns to Avoid (from skill-judge AND Delegation-Specific)

### Universal Skill Anti-Patterns

| Anti-Pattern | Symptom | Fix |
|--------------|---------|-----|
| **Tutorial** | Step-by-step walkthrough that assumes no expertise | Replace with decision logic and patterns; reader is an expert who needs procedures, not education |
| **Dump** | Exhaustive reference material crammed into SKILL.md | Extract to references/; body should be under 500 lines |
| **Orphan References** | Referenced files that don't exist or aren't declared | Audit all file references against actual directory contents |
| **Checkbox Procedure** | Rigid checklist with no branching or judgment | Replace with decision trees that accommodate variation |
| **Vague Warning** | "Be careful with X" without specifying what goes wrong | State the specific failure mode and why it happens |
| **Invisible Skill** | SKILL.md exists but triggers never match | Rewrite description with concrete trigger phrases |
| **Wrong Location** | Files outside declared skill structure | Consolidate into single declared directory |
| **Over-Engineered** | Nested configs, build steps, meta-files | Flatten; SKILL.md + resources is the entire structure |
| **Freedom Mismatch** | High-freedom instructions for fragile operations OR low-freedom for creative tasks | Calibrate instruction specificity to operation risk |

### Delegation-Specific Anti-Patterns

| Anti-Pattern | Description | Consequence | Fix |
|--------------|-------------|-------------|-----|
| **Context Hoarding** | Sending minimal context with "figure it out" instructions | Subagent wastes turns searching, makes wrong assumptions | Full context packet upfront |
| **Scope Creep by Design** | Task packets that bundle unrelated work | Cross-domain contamination, integration conflicts | One domain per packet |
| **Review Skip** | Proceeding without completing both review stages | Defects propagate to integration | Gate enforcement — halt on skip |
| **Parallel Assumption** | Defaulting to parallel without proving independence | Integration conflicts, wasted rework | Default sequential; prove parallel |
| **Feedback Vagueness** | Re-dispatch with "fix the issues" instead of specific failures | Subagent repeats same mistakes | Line-level feedback with requirement references |
| **Silent Recovery** | Handling subagent failures internally without logging | Invisible debt accumulation | Always log stop conditions and recovery actions |
| **Freedom Overload** | Giving subagents open-ended refactoring authority | Scope violation, unrelated changes | Tight constraints + explicit "do not" list |

### Expert Anti-Pattern Requirement

Every anti-pattern entry must include WHY it fails, not just WHAT fails. "Scope creep causes bugs" is a vague warning. "Scope creep causes cross-domain type conflicts that break integration because two subagents modified the same interface differently" is an expert anti-pattern.

## 9. Operational Precision (Testable Instructions, Freedom Calibration, Real-Workload Readiness)

### Testable Instructions

Every instruction in the delegation protocol must pass this test: **Can a subagent execute it without ambiguity, and can the orchestrator verify the result objectively?**

| Instruction | Testable? | Why |
|-------------|-----------|-----|
| "Fix the bug" | No | No specific bug, no verification criteria |
| "Fix the type error at line 47 in src/core/trajectory.ts by changing `string` to `string | undefined`" | Yes | Specific location, specific change, verifiable |
| "Review the code quality" | No | No quality criteria specified |
| "Review for: unused imports, missing JSDoc on public methods, any function exceeding 300 LOC" | Yes | Specific checks, countable, verifiable |

### Freedom Calibration Matrix

| Operation Type | Freedom Level | Instruction Style |
|----------------|---------------|-------------------|
| **Creative problem-solving** (refactor approach, architecture choice) | High | "Explore 2-3 approaches, recommend one with rationale" |
| **Implementation** (write code, fix specific error) | Medium | "Implement using pattern X, constraints Y, stop if Z" |
| **Fragile operations** (schema migration, public API change, shared types) | Low | "Change exactly this, verify against this contract, do not touch anything else" |
| **Recovery** (fix failed dispatch, resolve conflict) | Low | "Apply this specific fix, re-run this specific test, report this specific result" |

**Mismatch consequence:** High freedom on fragile operations → scope violation. Low freedom on creative tasks → suboptimal solutions that miss better approaches.

### Real-Workload Readiness Checklist

Before deploying a delegation protocol against real work:

1. **Packet completeness** — Can a subagent execute without asking for more context?
2. **Return contract clarity** — Can the orchestrator verify compliance mechanically?
3. **Stop condition coverage** — Are all foreseeable failure modes captured?
4. **Review stage coverage** — Do both stages have clear pass/fail criteria?
5. **Integration test** — Can all parallel returns be validated together?
6. **Recovery path** — Is there a defined action for each failure type?
7. **Feedback specificity** — On failure, can the orchestrator provide line-level feedback?

If any answer is "no," the protocol is not ready for real workload.

### One-at-a-Time Validation (from brainstorming)

When designing delegation workflows:
- Present one section (200-300 words) at a time
- Validate each section before moving to the next
- Explore 2-3 alternatives for each design decision
- Apply YAGNI ruthlessly — if a feature doesn't serve the core delegation loop, cut it
- Go back when something doesn't make sense — flexibility over momentum

## 10. Reusable Template Structures (Packets, Briefs, Checkpoints, Loops)

### Task Packet Template

```markdown
# Task Packet: [ID]

## Scope
[One sentence — single problem domain]

## Context
### Files
- `[path]` — [why needed]
### Types/Interfaces
- `[name]` — [definition or reference]
### Contracts
- [SDK/API surface that must be preserved]

## Constraints
- MUST: [required actions]
- MUST NOT: [prohibited actions]
- STOP IF: [halt conditions]

## Expected Output
### Format
- Changes: file:line — description
- Tests: command — result
- Compliance: requirement — status

### Return Contract
Use the standard Return Report template.
```

### Dispatch Brief Template

```markdown
# Dispatch Brief: [Workflow Name]

## Workflow Type
[Sequential | Parallel | Hybrid]

## Task Chain
1. Packet [ID] → Agent [type] → Gate [review stage]
2. Packet [ID] → Agent [type] → Gate [review stage]
3. Integration → Full suite → Complete

## Independence Proof (if parallel)
- Task A: modifies `src/module-a/`, imports `shared/types-a.ts`
- Task B: modifies `src/module-b/`, imports `shared/types-b.ts`
- Shared state: none confirmed
- Integration test: `npm test` covers both

## Recovery Points
- After Packet 1: [recovery action if failed]
- After Packet 2: [recovery action if failed]
- Integration: [recovery action if conflicts found]
```

### Checkpoint Template

```markdown
# Checkpoint: [Stage Name]

## Completed
- [x] Packet [ID]: [status summary]

## In Progress
- [ ] Packet [ID]: [current state]

## Blocked
- [ ] Packet [ID]: [blocker description]

## Gate Status
- Stage 1 Review: [pass/fail/pending]
- Stage 2 Review: [pass/fail/pending]
- Integration: [pass/fail/pending]

## Decision
- [ ] Proceed to next stage
- [ ] Re-dispatch with feedback: [specifics]
- [ ] Escalate to human: [reason]
```

### Review Loop Template

```markdown
# Review Loop: Packet [ID] — Attempt [N]

## Stage 1: Spec Compliance
| Requirement | Status | Evidence |
|-------------|--------|----------|
| [req from packet] | met/not-met | [file:line or test output] |

## Stage 2: Code Quality
| Check | Status | Evidence |
|-------|--------|----------|
| [convention/rule] | pass/fail | [file:line] |

## Feedback (if re-dispatch)
### Specific Failures
1. [Requirement]: [what was expected vs what was received]
2. [Quality check]: [specific issue with line reference]

### Re-dispatch Instructions
[Exact changes needed — no ambiguity]
```

### Integration Gate Template

```markdown
# Integration Gate: [Workflow Name]

## Returns Received
- [x] Packet [ID]: Agent [type] — [summary]
- [x] Packet [ID]: Agent [type] — [summary]

## Conflict Check
| Conflict Type | Detected? | Resolution |
|---------------|-----------|------------|
| Type/interface collision | yes/no | [action] |
| Import chain break | yes/no | [action] |
| Test interference | yes/no | [action] |
| File write collision | yes/no | [action] |

## Full Suite Result
- Command: `[test command]`
- Result: pass/fail
- Failures: [list if any]

## Verdict
- [ ] All clear — mark workflow complete
- [ ] Conflicts found — dispatch resolution agent
- [ ] Suite failed — re-dispatch affected packets
```

## Cross-Cutting Principles

These principles apply across ALL sections:

1. **Specificity over generality.** Every instruction, template, and decision rule must reference concrete artifacts (files, lines, commands, types).
2. **Default to safety.** Sequential over parallel, low freedom over high, narrow scope over broad. Prove the exception before taking it.
3. **Evidence before assertion.** Never claim a task is complete without running the verification command. Never claim independence without showing the proof.
4. **YAGNI is law.** If a template field is never populated in practice, remove it. If a gate never catches anything, simplify it. Ruthless minimalism in structure.
5. **Feedback is specific or it's noise.** "Fix the issues" is not feedback. "Line 47: `string` should be `string | undefined` because the SDK returns undefined for missing session IDs" is feedback.
