# Plan 13-02: Wire Result Capture into Observer, Process Runner, and Notifications

## Goal

Wire the `result-capture.ts` module (from Plan 13-01) into the two async code paths (`lifecycle-background-observer.ts` for subsessions, `lifecycle-process-runner.ts` for processes), persist captured results to continuity, enrich `TaskNotification` with actual content, and populate the long-empty `DelegationPacket.artifacts[]` and `.commits[]`.

## Requirements

- **PH13-06**: Async subsession completion captures result via `captureSubsessionResult()` before notification
- **PH13-07**: Async process completion captures result via `captureProcessResult()` before notification
- **PH13-08**: Captured result persisted to continuity via `patchSessionContinuity({ resultCapture })`
- **PH13-09**: `DelegationPacket.artifacts[]` and `.commits[]` populated from captured result
- **PH13-10**: `TaskNotification` enriched with `resultPreview`, `artifacts`, and `commits` fields
- **PH13-11**: `buildTaskNotificationFromContinuity()` uses captured result when available

## <threat_model>

### STRIDE Analysis

| Threat | Category | Risk | Mitigation |
|--------|----------|------|------------|
| Capture failure blocks completion notification | Denial of Service | HIGH | Wrap all capture calls in try/catch; notification proceeds even if capture fails |
| Stale captured result from previous session overwrite | Tampering | LOW | `capturedAt` timestamp enables staleness detection; consumer checks recency |
| Parent receives notification with incomplete capture data | Information Disclosure | LOW | `partial: true` flag on CapturedResult signals incomplete data |
| Notification text includes sensitive file paths | Information Disclosure | MEDIUM | resultPreview truncated to 500 chars; full paths only in continuity store (parent-controlled) |

</threat_model>

**At completion branch (lines 271-293):** After `args.patchLifecycle(...)` succeeds (line 283) and before `notifyParentWithFallback(...)` (line 286), insert result capture and persistence:

```typescript
if (advanced) {
  try {
    const captured = await captureSubsessionResult(args.client, args.sessionID)
    args.patchSessionContinuity(args.sessionID, { resultCapture: captured })
    if (captured.artifactPaths.length > 0 || captured.gitCommits.length > 0) {
      args.patchSessionContinuity(args.sessionID, {
        delegationPacket: {
          ...(args.getSessionContinuity(args.sessionID)?.metadata.delegationPacket ?? {}),
          artifacts: captured.artifactPaths,
          commits: captured.gitCommits,
        },
      })
    }
  } catch {
    // Capture failure must not block notification
  }
  const completedContinuity = args.getSessionContinuity(args.sessionID)
  if (completedContinuity?.metadata.parentSessionID) {
    void notifyParentWithFallback(
      args.client,
      completedContinuity.metadata.parentSessionID,
      buildTaskNotificationFromContinuity(completedContinuity, "completed"),
    )
  }
}
```

```typescript
try {

  args.patchSessionContinuity(args.sessionID, { resultCapture: { ...captured, partial: true } })
} catch {
  // Partial capture best-effort
}
```

**At failed branches** (retry-exhausted at ~246-268, poll-failed at ~323-346): Same partial capture pattern wrapped in try/catch.

All capture calls MUST be wrapped in try/catch so that capture failure never prevents the completion/failure notification from being sent.
