Restructure `observeBackgroundCompletion` to use the new poll strategy with exponential backoff. Wire into `HarnessLifecycleManager` for parent coordination via the existing `patchLifecycle` contract.

### Pattern 1: Start Gate Verification (D-10)
**What:** Before a delegation is considered "started", verify the child session has produced real work evidence. Current OpenCode `session.messages()` payloads expose reasoning output as `ReasoningPart` with `type: "reasoning"` [CITED: https://raw.githubusercontent.com/anomalyco/opencode/dev/packages/sdk/js/src/gen/types.gen.ts]. Upstream Anthropic/OpenCode error surfaces still refer to `thinking` / `redacted_thinking` blocks [CITED: https://github.com/anomalyco/opencode/issues/2599], so the detector should treat `reasoning` as canonical and accept `thinking` / `redacted_thinking` as compatibility aliases.
**When to use:** Every time `observeBackgroundCompletion` begins polling.
**Current state:** `observeBackgroundCompletion` already has `seenBusy` and `startupWindowElapsed` guards, but no formal start gate with specific evidence thresholds.

## Common Pitfalls

### Pitfall 1: Start Gate Too Aggressive
**What goes wrong:** Requiring ≥2 tool calls before "started" may never pass for simple tasks (e.g., research tasks that only read files and produce text).
**Why it happens:** Not all delegation tasks produce tool calls — some are pure reasoning tasks.
**How to avoid:** Start gate should have a timeout fallback — if no tool calls after N seconds, check if there's substantial assistant output (thinking blocks + text). Consider "OR" logic: (≥1 thinking + ≥2 tool calls) OR (≥3 assistant messages + idle).
**Warning signs:** Start gate never passes, delegation hangs at "starting" status.

### Pitfall 2: Exponential Backoff Missing Activity Reset
**What goes wrong:** Poll interval keeps growing even when the session is actively producing output.
**Why it happens:** Backoff is applied unconditionally regardless of whether new evidence was found.
**How to avoid:** Reset poll interval to initial (15s) whenever new evidence is detected (new messages or tool calls). Only backoff when no new evidence is found.
**Warning signs:** 60-second gaps between polls during active work.

### Pitfall 3: Two-Poll Stability Race with Session Deletion
**What goes wrong:** First poll shows idle → waiting for second poll → session gets deleted between polls.
**Why it happens:** Session lifecycle is asynchronous; OpenCode may clean up idle sessions.
**How to avoid:** On second poll, if session doesn't exist, check if it was previously idle with evidence. If yes, treat as completed (session was cleaned up after finishing). If no evidence, treat as failed.
**Warning signs:** False failures when sessions are cleaned up quickly after completion.

### Pitfall 4: Resume-First with Stale Session
**What goes wrong:** Retrying with the same session ID but the session has gone stale (context window full, corrupted state).
**Why it happens:** Session reuse is preferred per D-13, but stale sessions may not accept new prompts.
**How to avoid:** On retry, verify session is responsive (can receive a prompt) before reusing. If session.get() fails or returns error state, create a new session.
**Warning signs:** Retries fail immediately because stale session can't process new prompts.

### Pitfall 5: Parent Coordination Deadlock
**What goes wrong:** Main session waits for all delegations, but a delegation's notification never arrives (parent session offline, notification dropped).
**Why it happens:** Notification delivery is best-effort; `notifyParentSession` can fail silently.
**How to avoid:** Parent coordination must also poll continuity store for delegation status, not just rely on notifications. Use a timeout: if delegation hasn't reported after X minutes, check continuity directly.
**Warning signs:** Main session hangs indefinitely waiting for a delegation that already completed.

## Code Examples

### Start Gate Verification (D-10)
```typescript
// Source: Adapted from existing getCombinedEvidenceCount in lifecycle-background-observer.ts
// Counts reasoning/thinking blocks and tool calls separately for start gate evidence

export type StartGateEvidence = {
  thinkingBlocks: number
  toolCalls: number
  assistantMessages: number
  passed: boolean
}

function countThinkingBlocks(message: unknown): number {
  if (!message || typeof message !== "object") return 0
  const parts = (message as { parts?: unknown }).parts
  if (!Array.isArray(parts)) return 0
  return parts.reduce((count, part) => {
    if (!part || typeof part !== "object") return count
    const type = (part as { type?: unknown }).type
    return type === "reasoning" || type === "thinking" || type === "redacted_thinking"
      ? count + 1
      : count
  }, 0)
}

export async function verifyStartGate(
  client: OpenCodeClient,
  sessionID: string,
): Promise<StartGateEvidence> {
  const messages = await getSessionMessages(client, sessionID)
  const assistantMsgs = messages.filter((msg) => {
    if (!msg || typeof msg !== "object") return false
    return (msg as { role?: string }).role === "assistant"
  })

  const thinkingBlocks = assistantMsgs.reduce(
    (count, msg) => count + countThinkingBlocks(msg), 0
  )
  const toolCalls = assistantMsgs.reduce(
    (count, msg) => count + countToolCallParts(msg), 0  // existing function
  )

  return {
    thinkingBlocks,
    toolCalls,
    assistantMessages: assistantMsgs.length,
    passed: thinkingBlocks >= 1 && toolCalls >= 2,
  }
}
```

### Exponential Backoff Poll Strategy (D-11)
```typescript
// New module: src/lib/tasking/completion/poll-strategy.ts

export type PollConfig = {
  readonly initialIntervalMs: number    // 15000 (D-11)
  readonly backoffIncrementMs: number   // 5000 (D-11)
  readonly maxIntervalMs: number        // 60000 (D-11)
}

export const DEFAULT_POLL_CONFIG: PollConfig = {
  initialIntervalMs: 15_000,
  backoffIncrementMs: 5_000,
  maxIntervalMs: 60_000,
}

export class PollStrategy {
  private currentIntervalMs: number
  constructor(private readonly config: PollConfig = DEFAULT_POLL_CONFIG) {
    this.currentIntervalMs = config.initialIntervalMs
  }

  /** Get current interval, then advance to next (with backoff). */
  nextInterval(): number {
    const interval = this.currentIntervalMs
    this.currentIntervalMs = Math.min(
      this.currentIntervalMs + this.config.backoffIncrementMs,
      this.config.maxIntervalMs,
    )
    return interval
  }

  /** Reset to initial interval — call when new evidence is found. */
  reset(): void {
    this.currentIntervalMs = this.config.initialIntervalMs
  }
}
```

### Failure Handler State Machine (D-13)
```typescript
// New module: src/lib/tasking/completion/failure-handler.ts

export type FailureState = {
  readonly maxRetries: number
  retryCount: number
  lastSessionID: string
  idleSinceMs: number | null
  readonly idleTimeoutMs: number  // 180_000 (D-13)
}

export function createFailureState(sessionID: string): FailureState {
  return {
    maxRetries: 2,
    retryCount: 0,
    lastSessionID: sessionID,
    idleSinceMs: null,
    idleTimeoutMs: 180_000,
  }
}

export function shouldRetry(state: FailureState): boolean {
  return state.retryCount < state.maxRetries
}

export function markIdleStart(state: FailureState, now: number): FailureState {
  return { ...state, idleSinceMs: now }
}

export function markActivity(state: FailureState): FailureState {
  return { ...state, idleSinceMs: null }
}

export function checkIdleTimeout(state: FailureState, now: number): {
  timedOut: boolean
  idleDurationMs: number
} {
  if (state.idleSinceMs === null) return { timedOut: false, idleDurationMs: 0 }
  const duration = now - state.idleSinceMs
  return { timedOut: duration >= state.idleTimeoutMs, idleDurationMs: duration }
}
```

## Open Questions (Resolved)

None — the three previously open implementation blockers are now resolved with code/documentation evidence.

1. **Thinking block type detection (resolved)**
   - **Resolved answer:** Current OpenCode `session.messages()` payloads expose model reasoning as `ReasoningPart` with `type: "reasoning"` [CITED: https://raw.githubusercontent.com/anomalyco/opencode/dev/packages/sdk/js/src/gen/types.gen.ts]. Provider-facing / Anthropic error surfaces still refer to `thinking` and `redacted_thinking` [CITED: https://github.com/anomalyco/opencode/issues/2599].
   - **Implementation guidance:** Keep the decision language as “thinking block”, but implement the detector as `reasoning || thinking || redacted_thinking` plus the existing tool-call count [VERIFIED: `src/lib/lifecycle-background-observer.ts:34-64`].

2. **Resume-first retry feasibility with same session ID (resolved)**
   - **Resolved answer:** Same-session retry is feasible and already used as a live harness pattern. `HarnessLifecycleManager.requestAutoLoopRetry()` re-prompts an existing session by ID via `sendPrompt()` [VERIFIED: `src/lib/lifecycle-manager.ts:239-243`], and the idle-session hook tests assert that an idle session is retried through that exact path instead of creating a new session [VERIFIED: `tests/hooks/create-session-hooks.test.ts:298-331`]. The SDK documentation also defines `session.prompt({ path: { id } })` as “Send prompt message” to an existing session [CITED: https://opencode.ai/docs/sdk/].
   - **Implementation guidance:** D-13 should use same-session re-prompt as the default retry mechanism. Do **not** create a new child session on first failure. Only fall back to terminal failure if re-prompting the same session ID throws or the retry budget is exhausted. If the background retry path uses `sendPromptAsync()`, add explicit regression coverage for that resumed-child path.

3. **Source of truth for “all user tasks complete” (resolved)**
   - **Resolved answer:** There is no separate persisted “user task queue” in the current architecture. Durable truth for delegated child progress lives in continuity-backed lifecycle state (`patchSessionContinuity()` / `metadata.lifecycle.phase`) [VERIFIED: `src/lib/lifecycle-manager.ts:666-714`, `src/lib/continuity.ts:216-254`]. The parent session’s own outstanding user-facing work is tracked by its session-output/auto-loop state, because the existing hook only stops retrying when the latest assistant text includes the completion signal [VERIFIED: `src/hooks/create-session-hooks.ts:226-264`].
   - **Implementation guidance:** For D-14, define `canCloseMainSession()` as: **all registered child sessions are terminal in continuity (`completed` or `failed`) AND the parent session no longer needs another auto-loop retry (ie. it has already produced its completion/synthesis response).** Do **not** scan unrelated continuity records for “user tasks”; scope this gate to the current parent session plus its registered child sessions [VERIFIED: `src/lib/state.ts:141-152`].
