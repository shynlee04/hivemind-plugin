## Summary

Phase 9.1 fixes 5 runtime bugs that make delegation completely non-functional, then rewrites ~200 mock-heavy tests to validate real behavior. The bugs have been precisely identified with line numbers from forensic analysis: Bug 1 (notification text, L104-113), Bug 2 (default "busy", L73-92), Bug 3 (prompt as evidence, L54-58), Bug 4 (cached idle bypass in CompletionDetector), Bug 5 (architectural — fixed by 1-4). The test rewrites target 4 specific mock-heavy test files that currently pass 12 tests while the actual runtime fails 5/5 delegation attempts.

The primary risk is Bug 3 (evidence counting) and Bug 4 (cached idle bypass) — both change core completion detection logic. The test rewrites must establish in-memory SDK adapters as the project's canonical testing pattern for future phases (9.2, 9.3, 11).

**Primary recommendation:** Fix bugs atomically (one commit per bug, each with its own regression test), then rewrite test files using in-memory `OpenCodeClient` adapters that mock only SDK transport — never internal business logic.

### Pattern 1: In-Memory SDK Client
**What:** Create a test helper that implements the same call signatures as `OpenCodeClient` using in-memory Maps instead of HTTP calls.
**When to use:** Every test that currently mocks `session-api.js` module with `vi.mock`.
**Example:**
```typescript
// tests/lib/helpers/in-memory-client.ts
// WHY: Clean Architecture testing — mock the boundary (SDK transport),
// not the internals (business logic, state machines, concurrency).

export function createInMemoryClient() {
  const sessions = new Map<string, { status: { type: string }; parentID?: string }>()
  const messages = new Map<string, unknown[]>()

  return {
    session: {
      get: vi.fn(async ({ path }) => {
        const s = sessions.get(path.id)
        if (!s) throw new Error("not found")
        return { data: s }
      }),
      create: vi.fn(async ({ body }) => {
        const id = `ses_${Math.random().toString(36).slice(2)}`
        sessions.set(id, { status: { type: "idle" }, ...body })
        return { data: { id, ...body } }
      }),
      messages: vi.fn(async ({ path }) => {
        return { data: messages.get(path.id) ?? [] }
      }),
      status: vi.fn(async () => {
        const map: Record<string, { type: string }> = {}
        for (const [id, s] of sessions) map[id] = s.status
        return { data: map }
      }),
      prompt: vi.fn(async () => ({ data: {} })),
      promptAsync: vi.fn(async () => ({ status: 204 })),
      abort: vi.fn(async () => ({ data: {} })),
    },
    tui: { showToast: vi.fn() },
    // Test helpers for programmatic state manipulation
    _sessions: sessions,
    _messages: messages,
    _setStatus(sessionID: string, type: string) {
      const s = sessions.get(sessionID)
      if (s) s.status.type = type
    },
    _addMessage(sessionID: string, msg: unknown) {
      const arr = messages.get(sessionID) ?? []
      arr.push(msg)
      messages.set(sessionID, arr)
    },
  } as unknown as OpenCodeClient & { _sessions: typeof sessions; _messages: typeof messages; _setStatus: typeof _setStatus; _addMessage: typeof _addMessage }
}
```

### Anti-Patterns to Avoid
- **Mocking internal modules with `vi.mock("../../src/lib/session-api.js")`:** This is the root cause of 200 passing tests that don't validate runtime behavior. Use in-memory client instead.
- **Instant sleep functions:** The current observer test uses `instantSleep = vi.fn().mockResolvedValue(undefined)` which collapses all polling into synchronous execution. This hides timing bugs. Use `vi.useFakeTimers()` with real `setTimeout` instead.
- **Mocking `buildTaskNotificationFromContinuity`:** The observer test mocks this function, meaning it tests whether the mock returns what we tell it to return — not whether the real function produces correct output from real continuity records.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SDK client mocking | Custom vi.mock per test file | Shared `createInMemoryClient()` helper | Consistency, reusability, avoids per-test mock drift |
| Time control in tests | Manual `mockNow` + `instantSleep` | `vi.useFakeTimers()` + `vi.advanceTimersByTime()` | Built-in, handles setTimeout/setInterval correctly, no edge cases |
| Test state management | Global mutable state between tests | `beforeEach` factory pattern | Test isolation, no cross-contamination |

**Key insight:** The project already has the right test infrastructure (`completion-detector.test.ts` uses `vi.useFakeTimers` correctly). The problem is that the delegation tests chose `vi.mock` over in-memory adapters.

## Common Pitfalls

### Pitfall 1: Bug 4 Fix Breaks CompletionDetector's Internal Timer
**What goes wrong:** Fixing Bug 4 (don't cache external idle events) could break the stability timer's own internal caching (L109 in completion-detector.ts).
**Why it happens:** The `cachedResults` map serves two purposes: (1) caching external events that arrive before `watch()`, and (2) caching internal stability timer results. Blindly removing external caching also removes internal timer caching.
**How to avoid:** The fix must distinguish between external `feed()` calls (from `session.idle` events) and internal timer completions. The internal timer at L107-109 also sets `cachedResults`. Solution: add a `source` field to `CompletionResult` — `"internal"` vs `"external"` — and only bypass cached results from external sources. OR: simply don't call `feed()` for `session.idle` events in the observer (the observer already polls directly, so `feed()` is redundant for idle events).
**Warning signs:** `completion-detector.test.ts` test "resolves with idle after stable count" fails after Bug 4 fix.

### Pitfall 2: Evidence Count Filter is Too Aggressive
**What goes wrong:** Filtering to `role === "assistant"` might exclude tool results that come as separate messages in some SDK versions.
**Why it happens:** The OpenCode SDK message format isn't fully documented; tool results may appear as messages with `role: "tool"` or embedded in assistant message parts.
**How to avoid:** Count (1) messages where `role === "assistant"` AND (2) tool-call parts within ANY message. The current `countToolCallParts` already handles parts correctly. The fix should filter the `messages.length` part to only assistant messages, but keep the tool-call counting across all messages (or just assistant messages).
**Warning signs:** `combinedEvidenceCount` is 0 for sessions that clearly have tool calls.

### Pitfall 3: checkSessionExists "unknown" Status Not Handled
**What goes wrong:** Returning `{ type: "unknown" }` from `checkSessionExists()` could cause the observer's status switch to fall through without action, leading to an infinite loop.
**Why it happens:** The observer loop at L167-253 only has explicit branches for "busy", "idle", and "retry". An "unknown" status would fall through to the `await doSleep(pollIntervalMs)` at L253 (the default "busy means still working" branch). This is actually correct behavior — unknown status should be treated as "wait and poll again."
**How to avoid:** Ensure the "busy" branch (L169-170) does NOT set `seenBusy = true` for "unknown" status. The `seenBusy` flag should only be set for genuine "busy" status.
**Warning signs:** Session transitions from "unknown" → "idle" without ever being seen as "busy", which could bypass the `seenBusy` guard at L181.

### Pitfall 4: Test Rewrites Miss Edge Cases
**What goes wrong:** Rewriting tests to use in-memory adapters might miss edge cases that the mock-heavy tests accidentally covered (like empty status maps, null sessions, etc.).
**Why it happens:** In-memory adapters return well-structured data by default, while real SDK calls return messy, sometimes partial data.
**How to avoid:** The in-memory adapter should support error injection (e.g., `_setGetSessionError(error)`) to simulate SDK failures. Port all existing test scenarios to the new pattern.
**Warning signs:** Test count drops significantly after rewrite without clear rationale for each removed test.

### Pitfall 5: delegate-task.test.ts File Not Found
**What goes wrong:** The CONTEXT.md lists `delegate-task.test.ts` and `delegate-task-overflow.test.ts` as files to rewrite, but these files DON'T EXIST in `tests/lib/`.
**Why it happens:** These tests may have been deleted during Phase 01 baseline cleanup or never existed in this worktree.
**How to avoid:** Verify actual file existence before planning rewrite tasks. The 4 files that actually exist are: `lifecycle-background-observer.test.ts` (576 LOC, 12 tests), `lifecycle-process-runner.test.ts` (58 LOC, 1 test). The `delegate-task.test.ts` and `delegate-task-overflow.test.ts` files need to be CREATED, not rewritten.
**Warning signs:** Planner creates tasks to "rewrite" files that don't exist.

```typescript
async function getCombinedEvidenceCount(client, sessionID) {
  const messages = await getSessionMessages(client, sessionID)
```

### Bug 4 Fix: Don't cache external idle events (completion-detector.ts)

```typescript
// CURRENT (buggy in context of observer):
// The observer at lifecycle-background-observer.ts calls:
//   completionDetector.feed("session.idle", sessionID) — this caches the result
// Then later calls:
//   completionDetector.watch(sessionID, timeoutMs) — this returns the cached result immediately
// This BYPASSES the stability timer.

// FIX OPTION A (minimal): Don't feed external idle events to the detector.
// The observer already polls directly and handles idle detection itself.
// The detector should ONLY receive events from its own internal stability timer.
// In lifecycle-background-observer.ts, the `feed()` call for idle should be removed
// or the observer should not call `feed()` — it should only use `feedMessageCount()` + `watch()`.

// FIX OPTION B (robust): Add source tracking to CompletionDetector.
// In completion-detector.ts feed() method:
feed(eventType: string, sessionID: string | undefined, error?: string, source?: "internal" | "external"): void {
  // ... existing logic ...
  // Only cache results from internal sources (stability timer)
  if (source !== "internal") {
    // External events: if watcher exists, resolve it; otherwise DON'T cache
    const watcher = this.watchers.get(sessionID)
    if (watcher) {
      clearTimeout(watcher.timeoutId)
      this.watchers.delete(sessionID)
      watcher.resolve(result)
    }
    // Don't cache — let the poll-based stability check handle it
    return
  }
  // Internal source: existing cache behavior
  this.cachedResults.set(sessionID, result)
}

// RECOMMENDED: Option A — simpler, observer doesn't feed idle events at all.
// The observer already handles idle detection through its own polling loop
// and evidence counting. The completionDetector is only needed for its
// stability timer via feedMessageCount() + watch().
```
