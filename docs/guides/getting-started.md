# Getting Started

## Prerequisites

Before you begin, ensure your development environment meets these requirements:

| Dependency | Version | Verified via |
|------------|---------|-------------|
| Node.js | `>= 20.0.0` | `package.json` â†’ `engines.node` |
| OpenCode | `>= 1.14.28` | `package.json` â†’ `engines.opencode` (peer dependency) |
| npm | `>= 10.0.0` (bundled with Node 20) | Standard Node.js distribution |

The harness is a **TypeScript npm package** (`opencode-harness`). It compiles to ES2022 with NodeNext module resolution. No database, Docker, or system-level services are required.

> **Note:** This guide covers both scenarios: contributing to the harness source code, and consuming the published package in your own OpenCode project. The "Installation" and "First Run" sections address both.

## Installation

### Contributing to the harness

Clone the repository and install all dependencies (production + development):

```bash
git clone https://github.com/hivemind-plugin/opencode-harness.git
cd opencode-harness
npm install
```

### Using the harness in your OpenCode project

If you are consuming the published package, install it as a project dependency:

```bash
npm install opencode-harness
```

The package provides three entrypoints:

| Entrypoint | Import path | Purpose |
|-----------|-------------|---------|
| Core | `opencode-harness` | Public API â€” re-exports all `src/lib/` modules |
| Plugin | `opencode-harness/plugin` | Composition root â€” the `HarnessControlPlane` export OpenCode loads |
| CLI | `opencode-harness/cli` | `runCli(argv, io?)` for programmatic CLI dispatch |

## First Run

### 1. Build the harness (contributor scenario)

```bash
npm run build
```

This compiles `src/` â†’ `dist/` with TypeScript declarations and source maps. The build output is what OpenCode loads at runtime.

### 2. Verify with a type-check and test run

```bash
npm run typecheck   # Type-check without emitting â€” fast sanity check
npm test            # Run all tests (vitest, ~1163 tests)
```

All tests should pass. The test suite uses vitest with globals enabled â€” you do not need to import `describe`, `it`, or `expect` in test files.

### 3. Load the harness into OpenCode (consumer scenario)

Create or edit `opencode.json` in your project root. Add the plugin entry:

```json
{
  "plugin": ["./dist/plugin.js"]
}
```

If you installed the published package, the plugin path is resolved via Node module resolution â€” OpenCode will find `opencode-harness/plugin` automatically.

On the next OpenCode startup, the harness will:
- Register 16 custom tools (`delegate-task`, `delegation-status`, `prompt-skim`, etc.)
- Wire event hooks (session lifecycle, tool guards, shell environment)
- Initialize durable state persistence to `.hivemind/state/session-continuity.json`
- Begin enforcing concurrency limits, category gates, and runtime policy

For detailed configuration options (environment variables, runtime policy, concurrency limits), see [Configuration](../configuration/settings.md).

## Common Setup Issues

### Build fails with "Cannot find module @opencode-ai/plugin"

The `@opencode-ai/plugin` package is a **peer dependency**. If you are contributing to the harness source, it is also listed as a devDependency and will be installed by `npm install`. If you see this error, run:

```bash
npm install
```

The package is also listed in `devDependencies` at `^1.14.28` for development and testing.

### Missing environment variables for API providers

The `.env` file at the project root contains API keys for MCP providers (Tavily, Brave, GitHub, etc.) that the harness's configured agents use. If you see `Missing API key` errors in agent tool calls, check that your `.env` file contains the required keys. A reference `.env.example` file is not published â€” consult the [Configuration](../configuration/settings.md) doc or the `.env` file in the repository for the expected key names.

> **Security:** The `.env` file is gitignored and never committed. Never commit API keys or secrets. The `opencode.json` references `${OSIRIS_API_KEY}` rather than a hardcoded key â€” the actual key lives in `.env`.

### Plugin fails to load â€” "dist/plugin.js not found"

If OpenCode cannot find the compiled plugin, ensure you have run `npm run build` first. The plugin entrypoint is `./dist/plugin.js`, which is generated by the TypeScript compiler from `src/plugin.ts`. The `dist/` directory is not committed to the repository.

### Node.js version is too old

The harness requires `Node.js >= 20.0.0` (ES2022 target, `NodeNext` module resolution). Run `node --version` to verify. If your Node.js version is older, use `nvm` or your platform's package manager to upgrade:

```bash
nvm install 20
nvm use 20
```

### TypeScript type errors after pulling latest changes

The harness uses `verbatimModuleSyntax: true` â€” type-only imports must use `import type` syntax. If you see type errors after pulling, run:

```bash
npm run typecheck
```

This performs a full type check without emitting compiled output.
