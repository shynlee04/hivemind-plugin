# Phase 55 Context: Doc Intelligence Engine

**Phase:** 55 — Doc Intelligence Engine  
**Status:** research-locked / pre-planning

**Post-read action:** derive bounded implementation scope and evidence-first planning for migrating the document intelligence engine from product-detox into harness-experiment.

## Intent

Phase 55 exists to port the doc intelligence engine from product-detox (`src/intelligence/doc/`) into a read-only document intelligence surface within the harness-experiment hard harness. The capability gap is clear: Phase 52 E2E workflows and agents need to read `.planning/`, `docs/`, and `.hivemind/` documents programmatically but lack structured markdown AST parsing, outline extraction, token-aware chunking, regex search, and sandboxed path safety. This phase closes that gap through concept extraction (not code copy) under the migration gates established by Phase 16.4 and the integration runway defined by Phase 54.

## Scope

- `src/lib/doc-intelligence/parser.ts` — markdown AST parsing via `remark` and `unist-util-visit`, extracting headings, frontmatter, sections, and link references into a typed AST structure.
- `src/lib/doc-intelligence/surface-router.ts` — path-safe document surface routing with sandboxed boundaries that restrict reads to allowed directories (e.g., `.planning/`, `docs/`, `.hivemind/`) and reject traversal outside those roots.
- `src/lib/doc-intelligence/chunker.ts` — token-aware chunking with configurable size and overlap; must handle code blocks, tables, and frontmatter without breaking syntax boundaries.
- `src/tools/hivemind_doc.ts` — tool exposing 5 read-only actions: `skim` (outline/toc), `skim_directory` (directory listing with metadata), `read` (full or ranged content), `chunk` (token-aware segments), and `search` (regex over sandboxed paths).
- Stack dependencies: `remark`, `unist-util-visit`, `fast-glob`, `ignore` — with version constraints and harness compatibility notes documented before any implementation begins.
- Integration points with Phase 54 sidecar read-model contracts: doc intelligence must surface structured documents as read models consumable by the sidecar render/view layer.

## Non-Goals

- Do not implement full-text search indexing (e.g., FTS, inverted index, vector search) — that is a separate concern downstream of structured reading.
- Do not port product-detox source code wholesale — every file must pass through concept-extraction gates where contracts (not implementations) are the migration artifact.
- Do not modify Phase 54 sidecar implementation — Phase 54 is a runway (planning frame), not an implementation target; doc intelligence produces the read models the sidecar will later consume.
- Do not create document-authoring or mutation tools — the `hivemind_doc` surface is read-only and does not cross the CQRS write boundary.

## Assumptions

- **The `remark` ecosystem is the correct foundation for markdown AST parsing in the harness context.** If this is wrong, the parse surface would need a full stack re-evaluation before any implementation.
- **Document intelligence tools are read-only and do not cross the CQRS write boundary.** If this is wrong, `hivemind_doc` would need write-surface guards equivalent to existing CQRS enforcement in `src/hooks/hook-cqrs-boundary.ts`.
- **Phase 54 provides sufficient read-model contracts for doc intelligence to surface structured documents.** If this is wrong, the Phase 55 tool surface may need to define its own read-model contracts independently.
- **The sandboxed path safety model from product-detox is directly applicable to harness-experiment.** If this is wrong, the surface-router must be redesigned to accommodate harness-experiment's distinct directory conventions and security boundaries.

## Success Criteria

1. A planner can map `src/lib/doc-intelligence/` as a read-only surface with clear parser, router, and chunker module boundaries, each with defined input/output contracts.
2. The `hivemind_doc` tool's 5 actions (skim, skim_directory, read, chunk, search) are defined with typed input schemas, output envelopes, and sandboxed path validation before any implementation.
3. Stack dependencies (`remark`, `unist-util-visit`, `fast-glob`, `ignore`) are documented with version constraints, compatibility notes, and any divergence from product-detox's dependency set is explained.
4. Integration points with Phase 54 sidecar read-model contracts and Phase 52 E2E workflow document-reading needs are explicitly stated so downstream phases can consume doc intelligence without reverse-engineering.
5. Migration gates from product-detox are documented — every file maps to a product-detox concept source, a migration contract (not implementation), and a harness-experiment target location.

## Downstream Research Questions

1. What is the exact `remark` plugin chain needed to replicate product-detox's AST parsing behavior (heading extraction, frontmatter parsing, link resolution, section nesting)?
2. How should token-aware chunking handle code blocks, tables, and YAML frontmatter in markdown documents without breaking syntax boundaries or producing unusable fragments?
3. What are the precise read-model contracts Phase 54 must provide (or that doc intelligence must define itself) for the sidecar to consume structured document surfaces?
4. How does sandboxed path safety interact with the harness's existing CQRS write-surface guards — does `hivemind_doc` need additional boundary enforcement beyond path allowlisting?
5. Which specific Phase 52 E2E workflow scenarios (delegation transcripts, journal export, lineage reports) are blocked until doc intelligence can programmatically read `.planning/` and `.hivemind/` documents?
