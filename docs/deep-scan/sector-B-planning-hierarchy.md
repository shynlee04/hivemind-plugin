## Hierarchy Tree Mechanics

### Core Architecture

The hierarchy system uses a **tree-based structure** (v2.0) rather than a flat `trajectory в†’ tactic в†’ action` model. Key types:

```typescript
interface HierarchyNode {
  id: string           // {level_prefix}_{MiMiHrHrDDMMYYYY}[-n]
  level: HierarchyLevel  // "trajectory" | "tactic" | "action"
  content: string      // Human-readable description
  status: ContextStatus // "active" | "pending" | "complete" | "blocked"
  created: number      // Epoch ms
  stamp: string        // MiMiHrHrDDMMYYYY вЂ” cross-session grep traceability
  completed?: number   // Epoch ms when status в†’ "complete"
  summary?: string     // Set by janitor when branch is pruned
  children: HierarchyNode[]
}

interface HierarchyTree {
  version: 1 | 2
  root: HierarchyNode | null
  cursor: string | null  // ID of current working node
  branches: BranchCursor[]  // v2+ multi-branch support
  primary_branch: string  // defaults to "main"
}
```

### Hierarchy Chain: Grand в†’ Phase в†’ Sub в†’ Atomic

**This is NOT the same as trajectory в†’ tactic в†’ action.** The planning hierarchy uses a different model:

| Plan Type | Description | Parent | Children |
|-----------|-------------|--------|----------|
| `root` | Grand plan / META01 | none | sub plans |
| `sub` | Phase-level plan / META01-SUB01 | root | sub or atomic |
| `atomic` | Leaf task / META01-SUB01-ATOMIC01 | sub | none |

**The trajectory в†’ tactic в†’ action chain** is a **session hierarchy** stored in `hierarchy.json`:
- `trajectory` = root-level focus (declared via `declare_intent`)
- `tactic` = phase/sub-level focus (declared via `map_context`)
- `action` = atomic-level focus (declared via `map_context`)

### Level Transition Rules

```typescript
function isValidLevelTransition(parentLevel, childLevel): boolean {
  if (parentLevel === "trajectory") return childLevel === "tactic"
  if (parentLevel === "tactic") return childLevel === "action"
  return false  // action cannot have children
}
```

### Timestamp Stamps (Cross-Session Traceability)

The `MiMiHrHrDDMMYYYY` format enables grep-traceable timestamps:
- Example: `301411022026` = 14:30 on Feb 11, 2026
- Generated by `generateStamp(date: Date): string`
- Parsed by `parseStamp(stamp: string)` в†’ `{ minutes, hours, day, month, year }`

### Multi-Branch Support (v2)

Branches allow parallel workstreams:
```typescript
interface BranchCursor {
  name: string          // "main", "auth", "testing"
  cursor_id: string | null  // Node ID this branch points to
  created: number
  last_active: number
  status: "active" | "paused" | "completed"
}
```

Operations:
- `createBranch(tree, name, nodeId)` вЂ” create new branch at node
- `switchBranch(tree, name)` вЂ” switch primary branch
- `pauseBranch(tree, name)` вЂ” pause non-primary branch
- `completeBranch(tree, name)` вЂ” mark branch complete

## Validation Logic

### Validation Artifacts

Each plan can have a `VALIDATION-{PREFIX}.md` file:

```typescript
interface ValidationArtifact {
  id: string
  type: "validation"
  plan_id: string          // FK to plan
  status: "pending" | "pass" | "fail" | "partial"
  evidence: ValidationEvidence[]
  criteria: PlanCompletionCriteria[]  // Machine-checkable
}
```

### Completion Criteria Engine

`checkPlanCompletionCriteria()` evaluates 6 dimensions:

```typescript
interface PlanCompletionCriteria {
  all_nodes_complete: boolean      // Every node in tree has status: "complete"
  all_links_resolve: boolean       // parent_id and root_id point to existing plans
  all_validations_pass: boolean    // validation_state === "validated" or "skipped"
  yaml_headers_valid: boolean      // Plan files parseable with id + type
  navigation_intact: boolean       // Plan files exist at manifest paths
  graph_sync: boolean              // Manifest entries match graph/plans.json
}
```

### Validation Flow

1. `createValidationArtifact(prefix, planId, criteria)` вЂ” creates VALIDATION file
2. `updateValidationStatus(validationPath, status, evidence)` вЂ” updates frontmatter
3. `syncPlanValidationState(projectRoot, planId, state)` вЂ” syncs to manifest

## Status State Machine

### Plan Status Transitions


**Transition triggers:**
- `pending в†’ active`: Work started on plan

## Summary

The Planning & Hierarchy sector implements a sophisticated dual-hierarchy system:

1. **Session Hierarchy** (`hierarchy.json`): trajectory в†’ tactic в†’ action with timestamp traceability and multi-branch support
2. **Plan Hierarchy** (`plans/`): root в†’ sub в†’ atomic with FK relationships and validation artifacts

The system is designed for:
- **Cross-session traceability** via timestamp stamps
- **Relational integrity** via manifest FK references
- **Graceful migration** from v1.x flat structure
- **Clean onboarding** for fresh projects
- **Staleness detection** for context drift awareness

Key architectural decisions:
- All paths resolved through single `getEffectivePaths()` function
- YAML+Markdown for human-readable planning files
- JSON for machine-readable runtime state
- Validation artifacts as separate files with machine-checkable criteria
