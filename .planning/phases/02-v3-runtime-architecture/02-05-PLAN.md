   ```typescript
   export type StalenessCheck = {
     sessionID: string;
     lastActive: number;
     stalenessMinutes: number;
     isStale: boolean;
   };
   export type RiskAssessment = {
     sessionID: string;
     lastActive: number;
     stalenessMinutes: number;
     activeDelegations: number;
     pendingTasks: number;
     governanceRules: number;
     riskLevel: "low" | "medium" | "high";
   };
   export type RecoveryLog = {
     sessionID: string;
     timestamp: number;
     restoredSessions: string[];
     activeDelegations: string[];
     skippedSessions: string[];
     riskAssessments: RiskAssessment[];
   };
   export type RecoveryConfig = {
     stalenessThresholdMs: number;  // D-18: configurable (discretion on default)
   };
   ```

2. Create `src/lib/session-recovery.ts`:
   - `SessionRecovery` class with `RecoveryConfig` (default stalenessThresholdMs = 30 * 60 * 1000 = 30 minutes)
   - `checkStaleness(record: SessionContinuityRecord): StalenessCheck` — compare `updatedAt` vs now
   - `assessRisk(record: SessionContinuityRecord, manifest: DelegationManifest): RiskAssessment`:
     - Count active delegations for this session
     - Count pending tasks (status = "pending" or "running")
     - Calculate riskLevel: low (<15min, 0 pending), medium (15-60min or 1-2 pending), high (>60min or 3+ pending)
   - `recoverSessions(records: SessionContinuityRecord[], config?: RecoveryConfig): RecoveryLog`:
     - For each record: check staleness → if not stale, restore state via `hydrateDelegationState()`
     - If stale: include in riskAssessments but don't auto-restore (D-18: present for user confirmation)
     - Write recovery log to `.hivemind/sessions/recovery-${timestamp}.json`
   - `presentRiskAssessments(assessments: RiskAssessment[]): string` — format as human-readable summary for `client.tui.showToast()` or compaction injection
   - All errors prefixed `[Harness]`

   - Risk assessment: high staleness + 3 pending tasks → high risk
   - Recovery log written with correct structure
   - Simulate crash mid-task: write continuity with status="running", restart, verify recovery restores to "running" state

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Continuity file → Recovery | Corrupted continuity.json could cause recovery to restore invalid state |
| Staleness threshold → Auto-restore | Too-short threshold could skip valid sessions; too-long could restore stale data |
| Recovery log → Session directory | Recovery log could grow unbounded |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-15 | Tampering | SessionRecovery.recoverSessions | mitigate | Validate continuity records before hydration: check version, required fields, timestamp sanity (>2020, <future+1hr) |
| T-02-16 | Denial of Service | Recovery log growth | mitigate | Limit recovery log to 50 most recent entries per session; older entries archived or pruned |
| T-02-17 | Spoofing | Stale session auto-restore | mitigate | Stale sessions NEVER auto-restored (D-18); only presented for user confirmation; default threshold 30 min |
| T-02-18 | Repudiation | Recovery audit | mitigate | Every recovery action logged to .hivemind/sessions/recovery-{timestamp}.json with session IDs, timestamps, and decisions |
</threat_model>
