1. Add types to `src/lib/types.ts` (D-06, RUN-3f AC#1):
   ```typescript
   export type InjectionRule = {
     id: string;
     conditions: {
       agent?: string;            // Match agent name (or "*" for all)
       category?: string;         // Match delegation category
       trajectoryPhase?: string;  // Match lifecycle phase
       hasDelegation?: boolean;   // Match if session has delegation
     };
     inject: {
       rules?: string[];          // Governance rule IDs to inject
       commands?: string[];       // Command names to inject
       skills?: string[];         // Skill names to inject
       tools?: string[];          // Tool names to inject
     };
     description?: string;
   };
   export type InjectionResult = {
     ruleId: string;
     injected: {
       rules?: string[];
       commands?: string[];
       skills?: string[];
       tools?: string[];
     };
     skipped: string[];           // Items skipped due to governance blocking
   };
   export type InjectionLog = {
     sessionID: string;
     timestamp: number;
     results: InjectionResult[];
   };
   ```

2. Create `src/lib/injection-engine.ts`:
   - `InjectionEngine` class with constructor(governanceEngine?: GovernanceEngine):
     - `rules: InjectionRule[]` — in-memory rule store
     - `logs: Map<string, InjectionLog[]>` — per-session injection logs
     - `loadRules(path?: string): InjectionRule[]` — reads `.hivemind/state/injection-rules.json`
     - `addRule(rule: InjectionRule): void` — runtime rule addition
     - `removeRule(ruleId: string): boolean` — runtime rule removal
     - `evaluateInjections(sessionID: string, context: { agent?: string, category?: string, trajectoryPhase?: string, hasDelegation?: boolean }): InjectionResult[]`:
       - For each rule: check if conditions match current context
       - If match: collect inject items
       - If governanceEngine provided: filter out items blocked by governance rules (RUN-3f AC#3)
       - Track skipped items (governance-blocked) in result
       - Write InjectionLog to `.hivemind/state/injection-log.json` (RUN-3f AC#4)
     - `getSessionLog(sessionID: string): InjectionLog[]` — retrieve audit trail
   - Scoped ruleset (D-06): only inject rules, commands, skills, tools — not full policy engine
   - All errors prefixed `[Harness]`

3. Tests in `tests/lib/injection-engine.test.ts`:
   - Matching session (agent="researcher", category="research") → receives injection
   - Non-matching session → receives zero injections
   - Governance-blocked rule → injection item skipped, logged
   - Injection log written with correct structure
   - Add/remove rule at runtime → evaluation updated

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Injection rule file → Engine | Malicious rules could inject harmful content into session context |
| Compaction hook → Session prompt | Injected content becomes part of system prompt — could manipulate AI behavior |
| Governance bypass | Injection could include tools/commands blocked by governance |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-26 | Injection | InjectionEngine.evaluateInjections | mitigate | All injected content is pre-defined identifiers (rule/command/skill/tool names), not arbitrary text; only names are injected, not content |
| T-02-27 | Spoofing | Governance bypass via injection | mitigate | Every injection item checked against GovernanceEngine before inclusion; blocked items explicitly skipped and logged |
| T-02-28 | Tampering | Injection rule config | mitigate | Validate rule shape on load: required fields (id, conditions, inject); inject items must be non-empty arrays; reject rules with no injectable content |
| T-02-29 | Information Disclosure | Injection log | accept | Log contains rule IDs and item names — no PII or secrets; audit trail purpose; stored in .hivemind/state/ (local only) |
</threat_model>
