  <name>Task 2: Create tests/scripts/sync-fork.bats with 3 integration scenarios</name>
  <files>tests/scripts/sync-fork.bats</files>
  <behavior>
    - **Scenario (a) — Clean fast-forward:** Test creates a local bare repo (simulating upstream fork), pushes 2 new commits to it (non-pinned files only), creates a vendored clone from the initial state with no pinned-file conflicts. Invokes `sync-fork.sh`. Expects: exit 0, new commits present in working tree.
    - **Scenario (b) — Non-conflicting 3-way merge:** Test creates a local bare repo, makes a local change in the vendored copy (non-pinned file), pushes a different non-pinned change to upstream. Invokes `sync-fork.sh`. Expects: exit 0, both changes present in working tree.
    - **Scenario (c) — Conflicting pinned file abort:** Test creates a local bare repo, then pushes a commit that modifies a pinned file to upstream. Invokes `sync-fork.sh`. Expects: exit 1, pinned file's base name in stderr, pinned file content unchanged.
  </behavior>
  <action>
    Create `tests/scripts/sync-fork.bats` with the following structure and behavior:

    **Test helper loading:**
    - Use `setup()` to load `bats-assert` and `bats-support` from npm's `node_modules` via `BATS_LIB_PATH` or direct path.
    - Each test creates its own test fixture directory using `BATS_TEST_TMPDIR` (auto-cleaned by bats per RESEARCH §Don't Hand-Roll).

    **Common setup pattern (in each test):**
    ```
    setup() { ... }
    ```
    - `PROJECT_ROOT`: resolved from `BATS_TEST_FILENAME` location (`../../` from `tests/scripts/`)
    - `PATH` includes `$PROJECT_ROOT/scripts` so `sync-fork.sh` is callable directly

    **Test fixture creation (in each test or shared `setup()`):**
    1. Create temp fork: `git init --bare "$BATS_TEST_TMPDIR/fork.git"`
    2. Clone fork worktree: `git clone "$BATS_TEST_TMPDIR/fork.git" "$BATS_TEST_TMPDIR/fork-wt"`
    3. In fork-wt: create `src/tmux.ts` (non-pinned), `src/config.ts` (non-pinned), `src/session-manager.ts` (PINNED), commit, push to `main`
    4. Clone vendored dir: `git clone "$BATS_TEST_TMPDIR/fork.git" "$BATS_TEST_TMPDIR/vendored"`
    5. Remove origin remote from vendored dir (simulating history-less vendored copy)
    6. Export `SYNC_FORK_REMOTE_URL` pointing at the local fork.git
    7. `cd "$PROJECT_ROOT"` for script invocation

    **Scenario (a): Clean fast-forward**
    - After fixture setup, push 2 new commits to fork that modify non-pinned files (e.g., `src/tmux.ts`)
    - `run sync-fork.sh`
    - `assert_success`
    - Verify: `git -C "$PROJECT_ROOT/opencode-tmux" log --oneline | head -3` shows the new commits
    - (Or use the vendored dir directly — the script commits to the monorepo working tree)

    **Scenario (b): Non-conflicting 3-way merge**
    - After fixture setup:
      - Make a local change in vendored copy: modify `src/config.ts` (non-pinned), commit locally
      - Push a change to fork: modify `src/tmux.ts` (non-pinned, different file), push to fork
    - `run sync-fork.sh`
    - `assert_success`
    - Verify vendored dir now has both changes

    **Scenario (c): Conflicting pinned file abort**
    - After fixture setup:
      - Push a commit to fork that modifies `src/session-manager.ts` (a pinned file)
    - `run sync-fork.sh`
    - `assert_failure 1`
    - `assert_output --partial "session-manager.ts"`
    - `refute_output --partial "Sync complete"` (merge never happened)
    - Verify: pinned file `src/session-manager.ts` content is unchanged from original

    ## Testing approach for BATs

    Since the bats tests use a local git repo as fake upstream, they do NOT require network access (per Assumption 6 resolution — env var `SYNC_FORK_REMOTE_URL` is set to local path). The test fixture creates:
    - A bare git repo (the "fork")
    - A vendored clone with origin removed (simulating `opencode-tmux/`)

    The test then:
    - Pushes new commits to the bare repo (simulating upstream changes)
    - Runs `sync-fork.sh` against the local bare repo via env var override
    - Asserts expected exit code and working tree state

    **Important design choices:**
    - Tests must use `$BATS_TEST_TMPDIR` for isolation (auto-cleanup by bats)
    - Tests use `run` helper to capture sync-fork.sh output
    - Tests use `SYNC_FORK_REMOTE_URL` env var to override the real GitHub URL
    - Tests do NOT run the real script against GitHub — no network dependency
    - The 4 pinned files are created in the test fixture at the correct paths
    - Test files are minimal — just enough content to be valid TypeScript for git tracking

    ## What to AVOID:
    - ❌ Do NOT test against the real GitHub fork — use local bare repos
    - ❌ Do NOT hardcode REMOTE_URL in tests — use env var override
    - ❌ Do NOT leave test fixture repos in the filesystem — bats auto-cleans `BATS_TEST_TMPDIR`
    - ❌ Do NOT use `cd` to change directory permanently — use `cd` in a subshell or run commands with `-C` flag
    - ❌ Do NOT test content of merge-tree output directly — test observable behavior (exit code, file existence, content)
