# Phase 16.4 Decision Validation Register

Reader: future GSD planners/executors/verifiers landing cold on Phase 16.4. Post-read action: decide whether an inherited architecture claim is safe to reuse before planning runtime, state, lifecycle, migration, Hiveminder/Hivefiver, or sidecar work.

This register implements D-01 through D-05. The controlling rule is: docs are evidence inputs, not runtime truth. Every inherited claim below must be interpreted through current code/platform evidence, replacement rules, and reuse safety.

## Verdict Vocabulary

- `valid`: Current code/platform evidence supports the claim as a safe rule or baseline.
- `partial`: A portion remains useful, but future agents must apply the replacement rule before reuse.
- `stale`: The claim was once useful or appears in older plans, but current code/platform evidence supersedes it.
- `rejected`: Current evidence or locked Phase 16.4 decisions contradict the claim; do not reuse.
- `needs-research`: The claim is not safe to reuse until a later phase gathers current evidence.

## Reuse Rules for Future Agents

1. Reuse only rows with `Reuse Safe? = Yes` or `Yes, with replacement rule`, and apply the replacement rule verbatim.
2. Do not reuse `stale` or `rejected` rows as architecture truth; use them as warnings and anti-patterns.
3. Treat docs as evidence inputs, not runtime truth, unless the row cites current code/platform evidence and a Phase 16.4 decision trace.
4. Re-run `guards/architecture-facts.mjs --check-register` after editing this register.
5. Platform-sensitive claims marked `needs-research` require fresh OpenCode/package evidence before implementation.
