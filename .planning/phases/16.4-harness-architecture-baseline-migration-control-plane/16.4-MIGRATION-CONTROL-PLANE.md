# Phase 16.4 Migration Control Plane

Reader: future migration planners deciding whether a product-detox concept or state-root change may enter the harness. Post-read action: apply gates before moving code, changing writers, or selecting the next architecture-aligned build.

## Control Plane Rules

- No product-detox code is copied as-is.
- Concepts and contracts migrate before code.
- No second runtime authority without an explicit architecture gate.
- Current .opencode/state/opencode-harness persistence is migration input, not desired long-term canonical root.
- `.hivemind/` must be structured by purpose, role, owner, schema/index, and recovery behavior before it receives canonical writers.

## .opencode/state to .hivemind Transition

| Transition Concern | Required Behavior | Decision Trace |
|---|---|---|
| compatibility | Existing env overrides and current `.opencode/state/opencode-harness/` reads continue until a later writer-cutover plan passes tests. | D-12, D-13 |
| migration | A future bridge reads current continuity/delegation records, writes validated `.hivemind/` categories, and records source hashes/versions. | D-11, D-13 |
| fallback | If `.hivemind/` migration output is invalid or missing, current `.opencode/state/opencode-harness/` remains readable as migration input. | D-12, D-13 |
| recovery | Atomic writes, version checks, and rebuildable projections are mandatory before changing canonical writer paths. | D-13, D-16 |
| structured categories | Each `.hivemind/` category declares owner, role, schema, index, retention, and quick-read surface. | D-14 |
| canonical-vs-projection markers | Every file/category declares `canonical runtime state`, `derived projection`, `audit trail`, `archived`, or `future/optional`. | D-15 |
| sync direction | During migration, sync direction is current implementation input → future `.hivemind/` target until an explicit cutover gate changes authority. | D-16 |

## Product-Detox Concept Gate

Every candidate must pass this gate before implementation: source paths are inspected, purpose is restated, target runtime category is assigned, state owner is named, conflict risk is recorded, concept-migration plan is defined, code-copy exception is explicit, tests/guards are listed, and first-big-win scoring is referenced.

## Rejected / Blocked Inputs

- Wholesale product-detox source ports are blocked.
- Draft “READY” labels are advisory inputs, not execution authorization.
- Any migration that writes canonical `.hivemind/` state before bridge/cutover tests is blocked.
- Delegation manifest/notification mediation remains paused until architecture baseline gates exist.
