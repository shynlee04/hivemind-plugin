## Domain Context

This phase was born from a strategic reassessment of the BOOT workstream after architecture changes (SR restructuring) and late tech stack version bumps (Phase 11 reconciliation) potentially left BOOT implementations stale.

**Key insight driving this phase:** The project has suffered from feature creep — multiple workstreams started in parallel, none truly completed end-to-end. The safest path forward is a thin vertical slice that proves the entire config-to-behavior pipeline works before expanding scope.

---

## Scope: Config Schema MVP + Entry Init Verification

### What This Phase Covers

Lock the **4-field MVP config schema** and trace **one field end-to-end** from schema definition through runtime behavior:

| Field | Type | Purpose |
|-------|------|---------|
| `mode` | `expert-advisor \| hivemind-powered \| free-style` | Primary mode dispatch |
| `user_expert_level` | `clumsy-vibecoder \| beginner-friendly \| ... \| absolute-expert` | Response depth tuning |
| `conversation_language` | `en \| vi \| zh \| fr \| ...` | UI/conversation language |
| `documents_and_artifacts_language` | `en \| vi \| zh \| fr \| ...` | Document generation language |

### Why These Four Fields

These fields were chosen because they represent the **smallest complete dependency chain**:

```
Schema (Zod definition)
  → configs.json (defaults file written by hivemind init)
    → Runtime subscriber (config reader)
      → Behavioral profile (mode dispatch)
        → Visible behavior change (observable proof)
```

Each link in this chain must be verified working. Once proven, all remaining config fields (25+ additional fields) become mechanical additions — no architectural risk.

### What "End-to-End" Means

**L3 evidence required:** Not just "schema exists" or "tests pass" — but a demonstrable trace:

1. `npx hivemind init --yes --root <temp>` produces `.hivemind/configs.json` with the 4 MVP fields populated from defaults
2. Runtime reads configs.json and the `mode` field dispatches the correct behavioral profile
3. Changing `mode` in configs.json produces a visible behavior change on next session
4. `hivemind doctor` reports clean validation for the 4 MVP fields
