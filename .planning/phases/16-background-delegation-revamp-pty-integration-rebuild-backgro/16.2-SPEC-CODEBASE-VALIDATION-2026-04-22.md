## Section 2: Terminal Lifecycle (R-LC-01 through R-LC-05)

### R-LC-01: handleSessionDeleted() defers cleanup by configurable grace period
**Verdict: ⚠️ CONFLICT — SPEC assumes new feature needed, but code doesn't have it**

Current `handleSessionDeleted()` in `delegation-manager.ts` (lines 154-167):
```typescript
handleSessionDeleted(sessionId: string): void {
  const delegationId = this.delegationsBySession.get(sessionId)
  if (!delegationId) return
  const delegation = this.delegations.get(delegationId)
  if (!delegation) {
    this.cleanupTracking(delegationId, sessionId)
    return
  }
  delegation.status = "error"
  delegation.error = "Delegated session deleted before completion"
  delegation.completedAt = Date.now()
  this.persistAllDelegations()
  this.cleanupTracking(delegationId, sessionId)
}
```

**Facts:**
- `cleanupTracking()` (line 272) clears all timers AND removes from `delegationsBySession` — but does NOT remove from `delegations` Map
- `TASK_CLEANUP_DELAY_MS` — does NOT exist anywhere in the codebase
- No grace period timer is set
- The SPEC says "default: 10 minutes, controlled by constant `TASK_CLEANUP_DELAY_MS`" — this constant needs to be CREATED

**SPEC accuracy:** The SPEC correctly identifies the gap (no grace period). The SPEC's proposed solution (add `TASK_CLEANUP_DELAY_MS` timer) is accurate. But the SPEC says the gap is in `handleSessionDeleted()` — actually, the pruning was PARTIALLY added (the `pruneCompletedDelegations` method exists) but with broken references.

## SPEC Issues Requiring Revision

### Issue 1: Grace Period Model Mismatch (R-LC-01, R-LC-03)
- **SPEC says:** Per-delegation timer (`TASK_CLEANUP_DELAY_MS`) fires after 10 minutes
- **Code has:** Threshold-based batch pruning (`pruneCompletedDelegations()` when Map > 50)
- **Resolution:** These are complementary. SPEC should acknowledge both mechanisms or choose one. The pruning method already exists (broken), so it may be simpler to add constants and fix imports rather than implement per-delegation timers.

### Issue 2: R-NOTIF-04 Platform Constraint
- **SPEC says:** Emit event through OpenCode hook pipeline
- **Reality:** OpenCode's plugin API allows SUBSCRIBING to hooks but does not appear to support EMITTING custom events. The deprecated `notification-handler.ts` uses direct `client.session.prompt()` which is the only viable mechanism.
- **Resolution:** Downgrade R-NOTIF-04 from "hook pipeline" to "direct SDK call with fire-and-forget". Update priority from P2 to reflect reality.

### Issue 3: R-OBS-02 Field Name
- **SPEC line 516 says:** "Add `state` filter parameter"
- **Code has:** `status` parameter (already implemented)
- **Resolution:** SPEC should reference `status` not `state`.

### Issue 4: R-LC-05 File References
- **SPEC says:** `transitionToTerminal()` goes in `delegation-manager.ts`
- **Reality:** This is correct, but the SPEC should note that terminal paths in `command-delegation.ts` and `sdk-delegation.ts` would need to call back to DelegationManager instead of handling terminal transitions locally. This is a significant architectural change — the handlers would need a new callback for terminal transitions.

### Issue 5: LOC Estimates
- **SPEC says delegation-manager.ts is 309 LOC**
- **Actual:** 348 LOC (includes the broken pruning code)
- **SPEC says notification-handler.ts is 169 LOC**
- **Actual:** 179 LOC
- Minor discrepancies but worth updating.
