# Phase 25.2: Trajectory Immutability Guard — SPEC

**Locked:** 2026-05-29
**Type:** spec
**Status:** LOCKED — implementation must satisfy all requirements below

---

## Problem Statement

`closeTrajectory()` sets `status: "closed"` but no mutation operation checks status before writing. `upsertTrajectory()` at `store-operations.ts:136` updates existing trajectories without checking if they are closed. This means:

- `attachTrajectoryEvidence` on a closed trajectory **succeeds and mutates evidenceRefs**
- `checkpointTrajectory` on a closed trajectory **succeeds and appends checkpoints**
- `eventTrajectory` on a closed trajectory **succeeds and appends events**
- `closeTrajectory` on an already-closed trajectory **succeeds and overwrites closeSummary**

Evidence integrity is compromised. Closed trajectories must be immutable.

## Requirements

### REQ-IMM-01: eventTrajectory() guard

**Current behavior:** Succeeds on closed trajectories — events can be added after closure.

**Target behavior:** Throws when trajectory status is "closed".

**Acceptance criteria:**
- Calling `eventTrajectory()` on a closed trajectory throws `Error` with message `[Harness] trajectory is closed: {trajectoryId}`
- Calling `eventTrajectory()` on an active trajectory succeeds (no regression)

**Verification:** Test case in `store-operations.test.ts` — create trajectory, close it, call `eventTrajectory()`, assert throws with correct message.

## Implementation Guidance

### Guard placement

Add guard check at the **top** of each mutation function, before `readTrajectoryLedger()` or `upsertTrajectory()`:

```typescript
// At top of attachTrajectoryEvidence, checkpointTrajectory, eventTrajectory:
const existing = ledger.trajectories[input.trajectoryId]
if (existing?.status === "closed") {
  throw new Error(`[Harness] trajectory is closed: ${input.trajectoryId}`)
}
```

### closeTrajectory() guard

Add idempotency check after trajectory lookup:

```typescript
if (trajectory.status === "closed") {
  throw new Error(`[Harness] trajectory is already closed: ${input.trajectoryId}`)
}
```

### Helper option (agent discretion)

Whether to extract an `assertTrajectoryNotClosed(trajectory)` helper or inline the check is left to the implementer. Either approach satisfies the spec.

### Read operations

`inspectTrajectoryLedger` and `traverseTrajectory` must NOT have status guards — they are read-only and should return closed trajectories for inspection.

## Falsification Criteria

This spec is falsified if any of the following are true after implementation:

1. `eventTrajectory()` on a closed trajectory does NOT throw
2. `attachTrajectoryEvidence()` on a closed trajectory does NOT throw
3. `checkpointTrajectory()` on a closed trajectory does NOT throw
4. `closeTrajectory()` on an already-closed trajectory does NOT throw
5. `inspectTrajectoryLedger()` or `traverseTrajectory()` throws on closed trajectories
6. Any mutation operation on an active trajectory throws (regression)
7. Error messages do not match `[Harness] trajectory is closed: {id}` format
