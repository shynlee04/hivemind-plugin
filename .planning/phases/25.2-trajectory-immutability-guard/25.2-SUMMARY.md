## What Was Built

Trajectory immutability guard — all four mutation operations now throw when called on closed trajectories, preventing evidence integrity violations.

---

## Changes

### `src/task-management/trajectory/store-operations.ts`

Added status guard at top of each mutation function:

| Function | Guard | Error Message |
|----------|-------|---------------|
| `attachTrajectoryEvidence()` | `existing?.status === "closed"` | `[Harness] trajectory is closed: {id}` |
| `checkpointTrajectory()` | `existing?.status === "closed"` | `[Harness] trajectory is closed: {id}` |
| `eventTrajectory()` | `existing?.status === "closed"` | `[Harness] trajectory is closed: {id}` |
| `closeTrajectory()` | `trajectory.status === "closed"` | `[Harness] trajectory is already closed: {id}` |

### `tests/task-management/trajectory/store-operations.test.ts`

Added `seedClosedTrajectory()` helper and 7 new tests:
- T-01: `eventTrajectory` throws on closed
- T-02: `attachTrajectoryEvidence` throws on closed
- T-03: `checkpointTrajectory` throws on closed
- T-04: `closeTrajectory` throws on already-closed
- T-05: Read operations succeed on closed trajectories
- T-06: Active trajectory regression (all mutations still work)

## Decisions Made

- **Inline guards** over shared `assertTrajectoryOpen()` helper — only 4 call sites, `closeTrajectory` has different error message
- **Guard before `readTrajectoryLedger`** is unnecessary — the ledger read is cheap and needed to check status
- **No guard on read operations** — `inspectTrajectoryLedger` and `traverseTrajectory` are read-only
