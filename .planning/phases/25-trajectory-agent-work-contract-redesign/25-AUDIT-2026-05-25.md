## 1. Design Errors

### 1.1 Trajectory–Agent-Work-Contract Evidence Cross-Linking Is Unidirectional

**Files:** `src/features/agent-work-contracts/operations.ts`, `src/task-management/trajectory/store-operations.ts`

**What happens:** When `createAgentWorkContract()` succeeds, it calls `attachTrajectoryEvidence()` with a ref formatted as `agent-work-contract:<contractId>`. This writes an evidence reference into the trajectory ledger — the trajectory knows about the contract.

**What is missing:** The agent-work-contract record does **NOT** store a reverse reference back to the trajectory. The `AgentWorkContract` type has no `trajectoryId` or `trajectoryRef` field. This means:

- Given a contract, you cannot navigate to the trajectory entry that recorded its creation.
- If a trajectory is pruned, the contract becomes orphaned with no indication of which trajectory it belonged to.
- Cross-module queries ("find all contracts for trajectory X") require scanning all contracts and matching ref patterns.

**Fix approach:** Add an optional `trajectoryRef: string` field to `AgentWorkContract` (in `src/features/agent-work-contracts/types.ts`). Populate it during contract creation in `src/features/agent-work-contracts/operations.ts`.

### 1.2 Pressure-Based Contract Creation Gating Uses Wrong Tool Name

**Files:** `src/features/agent-work-contracts/operations.ts`, `src/features/runtime-pressure/authority-matrix.ts`

**What happens:** `createAgentWorkContract()` calls `detectRuntimePressure({ score, tier, toolName: "createAgentWorkContract" })` — but the authority matrix registers the tool as `"hivemind-agent-work-create"`.

**Why it's wrong:** The `detectRuntimePressure()` function does an exact match lookup via `getToolAuthority(toolName)`. Since `"createAgentWorkContract"` does not match `"hivemind-agent-work-create"`, `getToolAuthority()` returns `undefined`, and the code falls back to `unknownToolFallback()` — which is the same conservative default. **Behavior is identical** by coincidence, but the pressure evidence recorded in the trajectory ledger will say `tool "createAgentWorkContract"` (unknown tool) instead of `tool "hivemind-agent-work-create"` (properly registered). This degrades auditability.

**Fix approach:** Change the tool name in `detectRuntimePressure()` call inside `operations.ts` from `"createAgentWorkContract"` to `"hivemind-agent-work-create"`.

### 1.3 No Concurrent Contract Creation Lock

**Files:** `src/features/agent-work-contracts/store.ts`

**What happens:** The store uses atomic file rename for writes (`writeFileSync` with `{ flag: 'w' }` then `renameSync`). However, there is no in-process or cross-process lock. Two concurrent calls to `createAgentWorkContract()` could both read the store, both modify it, and the second `persist()` would overwrite the first's changes.

**Why it's wrong:** The path `.hivemind/state/agent-work-contracts.json` is a single JSON array. Concurrent writes from parallel subagents (which the delegation system supports) would lose one contract.

**Fix approach:** Add a file-level advisory lock (e.g., `lockfile` package or hand-rolled `.lock` file with `mkdir` atomicity) around the read-modify-write cycle. Alternatively, switch to an append-only journal with periodic compaction.

### 1.4 Trajectory Corruption Quarantine May Lose Evidence

**Files:** `src/task-management/trajectory/store-operations.ts`

**What happens:** When `readLedger()` encounters unparseable JSON, it calls `quarantineCorruptStore()` which moves the file to a `.corrupted` backup and returns an empty ledger. This is a survival mechanism — the session continues.

**Why it's wrong:** Evidence refs attached to the corrupted trajectory are **irrecoverably lost**. The quarantine path is timestamped but there is no automated recovery path to reattach orphaned evidence after a new empty ledger is created. Agent-work-contracts that were cross-referenced now point to a nonexistent trajectory ID.

**Fix approach:** Add a `maxCorruptionRecoveryAttempts` guard and surface a `recoverOrphanedEvidence()` operation that scans agent-work-contracts and other evidence sources for dangling trajectory refs after a corruption event. Document the quarantine path in `src/task-management/trajectory/store-operations.ts`.

### 1.5 Compaction Bounds Are Tight but Not Enforced at Schema Level

**Files:** `src/schema-kernel/agent-work-contract.schema.ts`, `src/features/agent-work-contracts/operations.ts`

**What happens:** The Zod schema defines compaction fields as `z.string().max(1000).optional()` for `briefing` and `summary`, and `z.string().max(2000).optional()` for `reinjectionPayload` and `z.array(z.string().max(100)).max(10).optional()` for `anchors`. But the actual runtime enforcement in `operations.ts` uses different limits: `BRIEFING_LIMIT=1200`, `SUMMARY_LIMIT=1200`, `REINJECTION_LIMIT=2400`, `ANCHOR_LIMIT=20`.

**Why it's wrong:** The schema constrains to different values than the runtime enforcement. A Zod-validation-only path (bypassing the operation function) would accept values the runtime would reject, and vice versa. The two sets of limits are inconsistent — the schema allows shorter briefing (1000) than runtime (1200), but fewer anchors (10 vs 20).

**Fix approach:** Derive the Zod schema limits from the same constants used in `operations.ts`. Export the constants from a shared location and reference them in both the schema and the operation function.

### 1.6 No Compaction Evidence for Contracts Created Via Pressure Bypass

**Files:** `src/features/agent-work-contracts/operations.ts`

**What happens:** When `detectRuntimePressure()` returns `"block"`, the function returns `AgentWorkCreateResult` with status `"error"` and message about pressure rejection. When it returns `"require_approval"` and `pressureApproved` is false, same error path. In both cases the contract is **not** created.

**Why it's wrong:** The trajectory ledger is only updated — via `attachTrajectoryEvidence()` — on the success path. A blocked contract creation attempt leaves **no trace** in the trajectory. Auditors cannot distinguish "this contract was never attempted" from "this contract was blocked by pressure."

**Fix approach:** On pressure-denied paths, call `attachTrajectoryEvidence()` with a ref like `agent-work-contract:<blocked-id>` or record a trajectory event with event type `contract_blocked`.

### 1.7 Store Version Hardcoded Without Migration Path

**Files:** `src/features/agent-work-contracts/store.ts`, `src/task-management/trajectory/store-operations.ts`

**What happens:** Both stores hardcode `store_version: 1` in their written JSON.

**Why it's wrong:** If the schema evolves (e.g., adding `trajectoryRef` per §1.1), there is no version migration logic. Version 1 files would be silently read into a v2 type, losing any new fields.

**Fix approach:** Implement a `migrateStore()` function that reads `store_version` and applies transformations. For now it is identity, but the scaffolding must exist before the schema changes.

## 2. Feature Integration Map

### 2.1 Integration: Runtime-Pressure → Agent-Work-Contracts

```
createAgentWorkContract(input)
  │
  ├─► detectRuntimePressure({ score, tier, toolName })
  │     ├─► classifyRuntimePressure()         → tier + band
  │     ├─► getToolAuthority(toolName)        → ToolAuthority or undefined
  │     ├─► resolveOutcome(classification, tool) → "allow" | "advise" | "require_approval" | "block"
  │     └─► returns PressureDecision
  │
  ├─► if block          → return { status: "error", message: "...rejected..." }
  ├─► if require_approval && !pressureApproved → return { status: "error", message: "...requires approval..." }
  └─► if allow/advise   → proceed to contract construction
        │
        ├─► build AgentWorkContract record
        ├─► upsert to store
        └─► attachTrajectoryEvidence({ ref: "agent-work-contract:<contractId>" })
```

**Gap:** Tool name mismatch (§1.2) means `getToolAuthority()` always returns `undefined` for contract creation calls.

### 2.2 Integration: Agent-Work-Contracts → Trajectory

```
createAgentWorkContract(success)
  └─► attachTrajectoryEvidence(trajectoryId, { ref: "agent-work-contract:<id>" })
        └─► store-operations.ts → reads ledger, appends evidence ref, writes

createAgentWorkContract(blocked)
  └─► NO trajectory update ← DESIGN ERROR (§1.6)
```

**Gap:** Unidirectional refs (§1.1) and missing blocked-attempt recording (§1.6).

### 2.3 Integration: Trajectory → Delegation (Indirect)

Trajectory records store evidence refs to delegation records (via format `delegation:<sessionId>`). The `hivemind-trajectory` tool is registered in the authority matrix with `HIVEMIND_STATE_WRITER` pressure behavior. Delegation-manager operations do **not** call trajectory functions directly — the trajectory is only updated through tool-mediated calls.

**Gap:** No direct code path connects delegation completion with trajectory evidence recordation. Delegation completion is detected by `completion/detector.ts` (dual-signal protocol) but does not automatically trigger trajectory updates. This means delegation completions are not recorded in the trajectory unless explicitly attached by tool call.

### 2.4 Integration: Runtime-Pressure → Authority Matrix

The authority matrix (`src/features/runtime-pressure/authority-matrix.ts`) registers **16 tools** with 4 pressure behavior profiles:

| Profile | Tools | steady | advisory | gated | blocking |
|---------|-------|--------|----------|-------|----------|
| hivemind-state | delegate-task, session-patch, hivemind-trajectory, hivemind-pressure, hivemind-agent-work-create (name mismatch), hivemind-sdk-supervisor | allow | advise | require_approval | block |
| opencode-primitive | configure-primitive | allow | advise | require_approval | block |
| read-only | delegation-status, prompt-skim, prompt-analyze, session-journal-export, hivemind-doc, hivemind-command-engine, hivemind-agent-work-export, validate-restart | allow | allow | advise | defer |
| external-command | run-background-command | allow | advise | require_approval | block |

**Gap:** `hivemind-sdk-supervisor` is registered with `hivemind-state` behavior but its `stateSurface` is `"read-only"` in the matrix — this is a **data inconsistency** in the authority matrix row (line 166 says `stateSurface: "read-only"` but line 165 says `pressureBehavior: HIVEMIND_STATE_WRITER`). The pressure behavior and state surface should be consistent: a read-only tool should use `READ_ONLY_INSPECTOR` behavior.

## 3. Mechanism Deep-Dive

### 3.1 Agent-Work-Contract Lifecycle

**Lifecycle state machine:**

```
[created] → [running] → [completed]
                │            │
                ├→ [blocked] ├→ [cancelled]
                │
                └→ [cancelled]
```

**Transitions:**
- `created`: After pressure gate passes, contract is written to store
- `running`: Started by a delegation dispatch (external to the contract module)
- `blocked`: Set when a blocking condition is detected during execution
- `completed`: Final state — all required proof collected
- `cancelled`: Manual termination

**Evidence levels (from Zod schema):**
- `L1`: Live runtime proof (running app, open browser, etc.)
- `L2`: Fresh artifact (build output, test report, generated file)
- `L3`: Fresh terminal output (CLI command result)
- `L4`: Verified code change (diff, commit, PR)
- `L5`: Documentation/design artifact

**Compaction limits (runtime-enforced):**
- `BRIEFING_LIMIT = 1200` chars
- `SUMMARY_LIMIT = 1200` chars
- `REINJECTION_LIMIT = 2400` chars
- `ANCHOR_LIMIT = 20` entries

**Compaction preservation packet structure (v1.0.0):**
```json
{
  "packet_version": "1.0.0",
  "session_id": "ses_<id>",
  "purpose_category": "discovery | ...",
  "delegation_depth": 0,
  "lifecycle_phase": "unknown"
}
```

### 3.2 Trajectory Ledger

**Storage:** `.hivemind/state/trajectory-ledger.json` — single JSON file with store version 1.

**Ledger structure:**
```typescript
interface TrajectoryLedger {
  store_version: number
  records: Map<string, TrajectoryRecord>  // keyed by trajectoryId
}
```

**Record structure:**
```typescript
interface TrajectoryRecord {
  trajectoryId: string
  rootSessionId: string
  sessionId: string
  parentTrajectoryId?: string
  checkpoints: TrajectoryCheckpoint[]
  events: TrajectoryEvent[]
  evidenceRefs: string[]  // e.g., "agent-work-contract:<id>", "delegation:<sessionId>"
  closedAt?: string
}
```

**Operations:**
- `inspect(trajectoryId)` — read trajectory by ID
- `attachEvidence(trajectoryId, ref)` — append evidence ref
- `recordCheckpoint(trajectoryId, checkpointId, summary)` — write checkpoint
- `recordEvent(trajectoryId, eventId, eventType, summary, evidenceRefs)` — write event
- `traverse(trajectoryId)` — read + lineage-lookup (parent chain traversal)

**Corruption handling:**
- On parse failure, quarantine file to `.corrupted.<timestamp>`
- Return empty ledger — session continues, but evidence refs are orphaned

### 3.3 Runtime Pressure Decision Engine

**Entry point:** `detectRuntimePressure(input)` in `src/features/runtime-pressure/control-plane.ts`

**Algorithm:**
1. `classifyRuntimePressure(input)` clamps score/tier to 0-9, maps to band:
   - 0-1 → steady
   - 2-4 → advisory
   - 5-7 → gated
   - 8-9 → blocking
2. `getToolAuthority(toolName)` looks up tool in `TOOL_AUTHORITY_MATRIX`
3. `resolveOutcome(classification, tool)` returns:
   - With tool: `tool.pressureBehavior[band]`
   - Without tool: fallback (steady→allow, advisory→advise, gated→require_approval, blocking→block)
4. `deriveSeverity(band, outcome)` — e.g., block→error, allow→info
5. `buildReason()` — human-readable string for audits

**Decision outcomes:** `"allow" | "advise" | "require_approval" | "defer" | "block"`

**Divergent outcomes:** When outcome is not `allow` or `advise`, a `blockingRationale` is added.

### 3.4 Context Injection

**What it is:** The delegation notifier periodically (~30 seconds) injects status context into the orchestrator agent's prompt. This is the **only** runtime status signal available to the orchestrator (no TUI toast is visible).

**Injection packet structure (from prior verification):**
```
[Context Injection] — Session status summary
- Session <sessionId> — status: running, duration: ~Xs, tool calls: N
- Delegation <delegationId> — status: running, output wait count: N
```
