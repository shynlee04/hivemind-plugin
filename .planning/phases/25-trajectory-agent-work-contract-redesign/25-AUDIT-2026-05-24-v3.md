## 1. Design Errors

### CRITICAL

#### DE-01: Trajectory `upsertTrajectory` silently overwrites `rootSessionId` on existing records
- **Files:** `src/task-management/trajectory/store-operations.ts:136-142`
- **Code:**
  ```typescript
  const existing = ledger.trajectories[input.trajectoryId]
  if (existing) {
    existing.rootSessionId = input.rootSessionId ?? existing.rootSessionId
    existing.sessionId = input.sessionId ?? existing.sessionId
    existing.parentTrajectoryId = input.parentTrajectoryId ?? existing.parentTrajectoryId
    return existing
  }
  ```
- **Problem:** The `??` operator means if `input.rootSessionId` is `undefined`, the existing value is preserved. But if `input.rootSessionId` is a **different** value (e.g., a second caller provides a different root session ID), the trajectory record's `rootSessionId` is **silently changed**. Since `TrajectoryMutationInput.rootSessionId` is optional, any caller providing a mismatched value corrupts the lineage.
- **Impact:** A trajectory record can end up associated with the wrong root session.
- **Fix:** Either: (a) make `rootSessionId`/`sessionId` immutable after creation, or (b) validate that the caller's values match the existing record, throwing on mismatch.

#### DE-02: `closeTrajectory` has no active-state prerequisite check
- **File:** `src/task-management/trajectory/store-operations.ts:99-112`
- **Code:**
  ```typescript
  export function closeTrajectory(input: { ... }): { ledger: TrajectoryLedger; trajectory: TrajectoryRecord } {
    const ledger = readTrajectoryLedger(input.projectRoot)
    const trajectory = ledger.trajectories[input.trajectoryId]
    if (!trajectory) { throw new Error(`[Harness] Trajectory "${input.trajectoryId}" not found`) }
    const now = Date.now()
    trajectory.status = "closed"  // No check that status === "active" first
  }
  ```
- **Problem:** No check that `trajectory.status === "active"` before closing. Double-close silently succeeds.
- **Impact:** Callers cannot detect double-close errors.
- **Fix:** Add `if (trajectory.status !== "active") throw new Error(...)` before transitioning to closed.

#### DE-03: `upsertAgentWorkContract` has no lifecycle enforcement
- **File:** `src/features/agent-work-contracts/store.ts:61-70`
- **Code:**
  ```typescript
  export function upsertAgentWorkContract(projectRoot: string, contract: AgentWorkContract): AgentWorkContract {
    const store = readAgentWorkContracts(projectRoot)
    const nextContract = cloneContract({ ...contract, updatedAt: Date.now() })
    persistAgentWorkContracts(projectRoot, { ... })
    return cloneContract(nextContract)
  }
  ```
- **Problem:** Any caller can overwrite any contract at any status. No validation prevents overwriting a `completed` or `running` contract. `AgentWorkStatusSchema` defines `["created", "running", "blocked", "completed", "cancelled"]` but `upsertAgentWorkContract` accepts all without checking the existing status.
- **Impact:** A buggy caller or race condition can silently overwrite a running contract.
- **Fix:** Add status-transition validation: only allow `created‚Üírunning‚Üíblocked‚Üícompleted|cancelled` transitions. Throw on invalid transitions.

#### DE-04 [NEW]: `hivemind-session-view.ts` parses trajectory ledger as array but format is object
- **File:** `src/tools/hivemind/hivemind-session-view.ts:76-88`
- **Code:**
  ```typescript
  async function readTrajectoryForSession(projectRoot: string, sessionId: string): Promise<Record<string, unknown> | null> {
    try {
      const ledgerPath = resolve(projectRoot, ".hivemind", "state", "trajectory-ledger.json")
      const raw = await readFile(ledgerPath, "utf-8")
      const allRecords = JSON.parse(raw) as Array<Record<string, unknown>>  // ‚Üê BUG: cast to Array
      const sessionRecords = allRecords.filter((r) =>                                    // ‚Üê .filter() on non-array throws
        (r as Record<string, unknown>).rootSessionId === sessionId ||
        (r as Record<string, unknown>).sessionId === sessionId,
      )
      return sessionRecords.length > 0 ? { total: sessionRecords.length, entries: sessionRecords.slice(0, 50) } : null
    } catch { return null }  // ‚Üê Error silently caught, always returns null
  }
  ```
- **Problem:** The trajectory ledger file format is `{ version: number, updatedAt: number, trajectories: Record<string, TrajectoryRecord> }` (defined in `src/task-management/trajectory/types.ts:81-88`), but `readTrajectoryForSession` casts `JSON.parse(raw)` as `Array<Record<string, unknown>>`. At runtime, `JSON.parse` returns an object `{}`, and calling `.filter()` on a non-array throws a TypeError. This error is caught by the bare `catch { return null }` on line 87, so the trajectory section of `hivemind-session-view` **always silently returns null** for ALL sessions.
- **Impact:** The `hivemind-session-view` tool, which is supposed to provide a unified view from 3 data roots (session, delegations, trajectory), silently drops ALL trajectory data. This defect was introduced in Phase 16 (Plan 05) and has been present since the tool was created.
- **Fix:** Read the `trajectories` key from the ledger object instead of treating the entire file as an array:
  ```typescript
  const ledger = JSON.parse(raw) as { trajectories: Record<string, TrajectoryRecord> }
  const sessionRecords = Object.values(ledger.trajectories).filter((r) =>
    r.rootSessionId === sessionId || r.sessionId === sessionId,
  )
  ```
  Or better, use the existing `traverseTrajectory()` API from the trajectory module rather than reading the file directly.

### HIGH

#### DE-05: Hand-rolled deep clone is fragile to schema extension
- **File:** `src/features/agent-work-contracts/store.ts:122-146`
- **Problem:** The hand-rolled `cloneContract` must be updated every time `AgentWorkContract` schema changes. If a new nested field is added inside `scope`, `evidence`, `compaction`, or `owner`, the cloner silently drops it.
- **Fix:** Use `JSON.parse(JSON.stringify(contract))` round-trip or `structuredClone()`.

#### DE-06: `writeTrajectoryLedger` lacks atomic write pattern
- **File:** `src/task-management/trajectory/ledger.ts:67-72`
- **Problem:** Uses direct `writeFileSync` without temp file + rename. A crash during write corrupts the ledger file. The agent-work-contract store (`store.ts:46-52`) correctly uses the atomic pattern, creating an inconsistency.
- **Fix:** Adopt the same atomic pattern as `store.ts:46-52`.

#### DE-07: Missing Zod schema for trajectory record types
- **File:** `src/schema-kernel/trajectory.schema.ts` (entire file, 49 lines)
- **Comparison:** `src/schema-kernel/agent-work-contract.schema.ts` (148 lines) has full Zod schemas for all types
- **Problem:** No Zod schema for `TrajectoryRecord`, `TrajectoryLedger`, `TrajectoryEvent`, or `TrajectoryCheckpoint`. Runtime validation in `ledger.ts:80-85` is an incomplete hand-rolled check that does NOT verify nested structures.
- **Fix:** Create `TrajectoryLedgerSchema`, `TrajectoryRecordSchema`, `TrajectoryEventSchema`, `TrajectoryCheckpointSchema`.

#### DE-08: `trajectoryId` attachment creates inconsistency window
- **File:** `src/features/agent-work-contracts/operations.ts:56-64`
- **Problem:** Contract is persisted FIRST, then trajectory evidence is attached. If `attachTrajectoryEvidence` throws, the contract is already on disk with `trajectoryId` set but no corresponding trajectory entry.
- **Fix:** Either: (a) attach trajectory evidence before persisting the contract, or (b) wrap both in a try-catch with rollback.

#### DE-09 [NEW]: `hivemind-session-view.ts` hardcodes trajectory ledger path instead of using API
- **File:** `src/tools/hivemind/hivemind-session-view.ts:79`
- **Code:** `const ledgerPath = resolve(projectRoot, ".hivemind", "state", "trajectory-ledger.json")`
- **Problem:** This duplicates `getTrajectoryLedgerPath()` from `src/task-management/trajectory/ledger.ts:19-22`. If the ledger path or filename changes (e.g., version 2 migration), `hivemind-session-view.ts` silently breaks while the rest of the system works. Additionally, this path hardcoding bypasses `assertPathWithinRoot` security validation.
- **Impact:** Silent data mismatch between the real trajectory ledger and the session-view display. Path changes require manual synchronization.
- **Fix:** Import `getTrajectoryLedgerPath` from the trajectory module instead of hardcoding the path. Or better, use `traverseTrajectory()` to read trajectory data through the proper API.

#### DE-10: Cross-feature pressure dependency in agent-work-contracts
- **File:** `src/features/agent-work-contracts/operations.ts:3,27-31`

  import { detectRuntimePressure } from "../runtime-pressure/index.js"

- **Problem:** `src/features/agent-work-contracts/` imports from `src/features/runtime-pressure/`. Per ARCHITECTURE.md, features should be standalone modules.

## 2. Feature Integration

### 2.1 Trajectory ‚Üî Session-View Integration Bug [NEW ‚Äî CRITICAL]

| Aspect | Expected | Actual |

| Trajectory ledger format | `{ version, updatedAt, trajectories: {...} }` | Same (correct) |

**Finding (INT-01):** `hivemind-session-view.ts:81` parses the trajectory ledger as `JSON.parse(raw) as Array<Record<string, unknown>>`. The `.filter()` call on a non-array throws TypeError, caught by the bare `catch { return null }`. **Every `hivemind-session-view` call silently drops trajectory data.** This was introduced in Phase 16 Plan 05 (`hivemind-session-view.ts` creation, 124 LOC) and has been broken since creation.

**Recommendation:** Fix immediately ‚Äî either use the trajectory module's API or parse the correct object shape.

### 2.2 Trajectory ‚Üî Session-Tracker Lineage Redundancy

| Aspect | Session-Tracker | Trajectory |
|--------|----------------|------------|
| Storage | `.hivemind/session-tracker/` (directory tree) | `.hivemind/state/trajectory-ledger.json` (flat file) |
| Lineage tracking | Hierarchy manifest with parent-child edges | `TrajectoryRecord` with `parentTrajectoryId` |
| Status | `status` field per session | `TrajectoryStatus` (`active`/`closed`) |
| Events | Full message/tool-call history in `.json` | `TrajectoryEvent` with `eventType` + `summary` |
| Test coverage | 418 tests in session-tracker alone | **5 tests** |
| Consumers | `hivemind-session-view` (3-root consolidation) | `hivemind-trajectory` tool (standalone) |

**Finding (INT-02):** Both systems track session lineage. The trajectory's value is "resumability evidence" ‚Äî events and checkpoints that survive compaction. The lineage fields (`rootSessionId`, `sessionId`, `parentTrajectoryId`) duplicate session-tracker. But the session-view integration is broken (DE-04), so this redundancy currently serves no purpose.

### 2.3 Trajectory ‚Üî Agent-Work-Contract Circular Evidence References

**Finding (INT-03):** 
1. Agent-work-contracts stores `trajectoryEvidenceRef: "agent-work-contract:{id}"` (line 42)
2. Agent-work-contracts calls `attachTrajectoryEvidence` with that ref (line 58-63)
3. The trajectory stores the evidence ref on the trajectory record

This creates a circular reference: contract ‚Üí trajectory ‚Üí contract. Cleanup is non-trivial.

### 2.4 Trajectory ‚Üî Agent-Work-Contract ‚Üî Pressure Three-Way Coupling

**Finding (INT-04):** Three systems are coupled:
- agent-work-contracts calls `detectRuntimePressure()` (DE-10)
- agent-work-contracts calls `attachTrajectoryEvidence()` (DE-08)
- hivemind-pressure tool writes to trajectory via `attach_event`

### 2.5 Tool Registration and Function Coverage

**Finding (INT-05):** 4 tools interact with trajectory data:
1. `hivemind-trajectory` ‚Äî 6 actions (inspect, traverse, attach, checkpoint, event, close)
2. `hivemind-agent-work-create` ‚Äî creates contracts with optional trajectory attachment
3. `hivemind-agent-work-export` ‚Äî exports contracts (read-only)
4. `hivemind-pressure` ‚Äî attach_event writes to trajectory

The registration density (4 tools for 2 subsystems) is appropriate, but `hivemind-session-view` (tool #5) should use the trajectory API instead of raw file reads.

## 4. Overlap Analysis

### 4.1 Trajectory ‚Üî Session-Tracker

Both systems track session lineage. The session-tracker has 418 tests, directory-based storage, and full message/tool-call history. Trajectory has 5 tests, single-file storage, and evidence/event focus. The trajectory lineage fields (`rootSessionId`, `sessionId`, `parentTrajectoryId`) duplicate session-tracker data.

**Verdict:** Legitimate but redundant. Phase 25 should remove lineage fields from `TrajectoryRecord` and make trajectory purely an event/checkpoint evidence store.

### 4.2 Trajectory ‚Üî Continuity Store

**No overlap.** Both write to `.hivemind/state/` but own different file names. The `sessionId` field on trajectory records is a reference (not a duplicate).

### 4.3 Agent-Work-Contracts ‚Üî Delegation Records

| Aspect | Agent-Work-Contracts | Delegation Records |
|--------|---------------------|-------------------|
| Store file | `agent-work-contracts.json` | `delegations.json` |
| Status lifecycle | created‚Üírunning‚Üíblocked‚Üícompleted‚Üícancelled | dispatched‚Üírunning‚Üícompleted‚Üíerror‚Üítimeout |
| Purpose | Work *authorization* (what, boundaries) | Work *execution* (dispatched, result) |

**Verdict:** Legitimate separation. The `trajectoryId` reference bridges them correctly.

### 4.4 Session-View Direct File Read Bypass

**Finding (OVL-01):** `hivemind-session-view.ts:79` reads `trajectory-ledger.json` directly instead of using `traverseTrajectory()` from the trajectory module. This creates a hidden coupling point ‚Äî the session-view assumes the trajectory file format rather than using the module API. Additionally, it bypasses `assertPathWithinRoot` security validation.

### 4.5 Pressure Authority Matrix

**Finding (OVL-02):** The pressure authority matrix (`src/features/runtime-pressure/authority-matrix.ts:147,157,187`) lists `evidenceAttachment: "trajectory-ledger"` for 3 tools (`hivemind-pressure`, `hivemind-pressure-detect`, `hivemind-agent-work-create`). This confirms trajectory is the intentional evidence store for pressure decisions, but the integration is one-directional ‚Äî pressure writes to trajectory, but trajectory never reads from pressure.
