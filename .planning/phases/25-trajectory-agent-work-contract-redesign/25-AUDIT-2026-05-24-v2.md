## 1. Design Errors

### CRITICAL

#### DE-01: Trajectory `upsertTrajectory` silently overwrites `rootSessionId` on existing records
- **Files:** `src/task-management/trajectory/store-operations.ts:136-142`
- **Code:**
  ```typescript
  const existing = ledger.trajectories[input.trajectoryId]
  if (existing) {
    existing.rootSessionId = input.rootSessionId ?? existing.rootSessionId
    existing.sessionId = input.sessionId ?? existing.sessionId
    existing.parentTrajectoryId = input.parentTrajectoryId ?? existing.parentTrajectoryId
    return existing
  }
  ```
- **Problem:** The `??` (nullish coalescing) operator means if `input.rootSessionId` is `undefined`, the existing value is preserved. But if `input.rootSessionId` is a **different** value (e.g., a second caller provides a different root session ID), the trajectory record's `rootSessionId` is **silently changed**. Since `TrajectoryMutationInput.rootSessionId` is optional (`rootSessionId?: string`), any caller that omits it preserves the existing value — but any caller that provides a mismatched value corrupts the lineage association.
- **Impact:** A trajectory record can end up associated with the wrong root session. Multi-session workflows where multiple agents attach evidence to the same trajectory will silently corrupt the lineage tree.
- **Fix:** Either: (a) remove `rootSessionId`/`sessionId` from mutation after creation (immutable once set), or (b) validate that the caller's values match the existing record, throwing on mismatch.

#### DE-02: `closeTrajectory` has no active-state prerequisite check
- **File:** `src/task-management/trajectory/store-operations.ts:99-112`
- **Code:**
  ```typescript
  export function closeTrajectory(input: { ... }): { ledger: TrajectoryLedger; trajectory: TrajectoryRecord } {
    const ledger = readTrajectoryLedger(input.projectRoot)
    const trajectory = ledger.trajectories[input.trajectoryId]
    if (!trajectory) { throw new Error(`[Harness] Trajectory "${input.trajectoryId}" not found`) }
    const now = Date.now()
    trajectory.status = "closed"
    // ...
  }
  ```
- **Problem:** No check that `trajectory.status === "active"` before closing. Calling `closeTrajectory` on an already-closed trajectory silently succeeds. The `status` field is the only lifecycle state, yet there's no state machine enforcement anywhere.
- **Impact:** Callers cannot detect double-close errors. Downstream consumers (e.g., `hivemind-session-view`) see a "closed" trajectory but cannot distinguish "closed after completion" from "closed twice."
- **Fix:** Add `if (trajectory.status !== "active") throw new Error(...)` before transitioning to closed.

#### DE-03: `upsertAgentWorkContract` has no lifecycle enforcement
- **File:** `src/features/agent-work-contracts/store.ts:61-70`
- **Code:**
  ```typescript
  export function upsertAgentWorkContract(projectRoot: string, contract: AgentWorkContract): AgentWorkContract {
    const store = readAgentWorkContracts(projectRoot)
    const nextContract = cloneContract({ ...contract, updatedAt: Date.now() })
    persistAgentWorkContracts(projectRoot, { ... })
    return cloneContract(nextContract)
  }
  ```
- **Problem:** Any caller can overwrite any contract at any status. There is no validation that prevents overwriting a `completed` or `running` contract. The `AgentWorkStatusSchema` defines `["created", "running", "blocked", "completed", "cancelled"]` but `upsertAgentWorkContract` accepts all of them without checking the existing status.
- **Impact:** A buggy caller or race condition can silently overwrite a running contract. The `status` field becomes meaningless.
- **Fix:** Add status-transition validation: only allow `created→running→blocked→completed|cancelled` transitions. Throw on invalid transitions.

### HIGH

#### DE-04: Hand-rolled deep clone is fragile to schema extension
- **File:** `src/features/agent-work-contracts/store.ts:122-146`
- **Code:**
  ```typescript
  function cloneContract(contract: AgentWorkContract): AgentWorkContract {
    return {
      ...contract,
      owner: { ...contract.owner },
      scope: { taskBoundary: ..., allowedSurfaces: [...], dependencies: [...], nonGoals: [...] },
      evidence: { requiredProof: [...], minimumEvidenceLevel: ..., verificationCommands: [...], blockedStateRules: [...] },
      compaction: { briefing: ..., summary: ..., anchors: [...], reinjectionPayload: ..., sourceRefs: [...] },
    }
  }
  ```
- **Problem:** This hand-rolled cloner must be updated every time `AgentWorkContract` schema changes. If a new top-level field is added (e.g., `metadata`, `tags`, `priority`), the spread operator on line 124 will include it — but if a new nested field is added inside `scope`, `evidence`, `compaction`, or `owner`, the cloner silently drops it because it only spreads known properties.
- **Impact:** Silent data loss on schema extension. Future developers adding fields to the schema will likely miss updating this cloner.
- **Fix:** Use `JSON.parse(JSON.stringify(contract))` round-trip (consistent with the deep-clone-on-read pattern from `continuity.ts`), or use `structuredClone()` (available in Node.js 20+).

#### DE-05: `writeTrajectoryLedger` lacks atomic write pattern
- **File:** `src/task-management/trajectory/ledger.ts:67-72`
- **Code:**
  ```typescript
  export function writeTrajectoryLedger(projectRoot: string, ledger: TrajectoryLedger): string {
    const ledgerPath = getTrajectoryLedgerPath(projectRoot)
    mkdirSync(resolve(ledgerPath, ".."), { recursive: true })
    writeFileSync(ledgerPath, `${JSON.stringify(ledger, null, 2)}\n`, "utf-8")
    return ledgerPath
  }
  ```
- **Problem:** Direct `writeFileSync` without temp file + rename. A crash during write corrupts the ledger file. The agent-work-contract store (`store.ts:46-52`) correctly uses the atomic pattern (`writeFileSync(tmpFile) → renameSync(tmpFile, filePath)`), creating an **inconsistency** between the two modules.
- **Impact:** The trajectory ledger is the only file in `.hivemind/state/` without atomic write safety. Corrupt trajectory data after a crash.
- **Fix:** Adopt the same atomic pattern as `store.ts:46-52`.

#### DE-06: Missing Zod schema for trajectory record types
- **File:** `src/schema-kernel/trajectory.schema.ts` (entire file, 49 lines)
- **Comparison:** `src/schema-kernel/agent-work-contract.schema.ts` (148 lines) has full Zod schemas for all types (`AgentWorkContractSchema`, `AgentWorkContractStoreSchema`, etc.)
- **Problem:** The trajectory schema only validates tool input (`TrajectoryToolInputSchema`). There is NO Zod schema for `TrajectoryRecord`, `TrajectoryLedger`, `TrajectoryEvent`, `TrajectoryCheckpoint`, or `TrajectoryStatus`. The runtime validation in `ledger.ts:80-85` is an incomplete hand-rolled check:
  ```typescript
  function isTrajectoryLedger(value: unknown): value is TrajectoryLedger {
    return typeof value === "object" && value !== null
      && (value as { version?: unknown }).version === TRAJECTORY_LEDGER_VERSION
      && typeof (value as { trajectories?: unknown }).trajectories === "object"
      && (value as { trajectories?: unknown }).trajectories !== null
  }
  ```
  This validation does NOT check `checkpoints`, `events`, `evidenceRefs`, or any nested structure. Corrupt ledger data with malformed checkpoints or events would pass validation.
- **Impact:** Corrupt trajectory ledger data is silently loaded and re-persisted. The quarantine path is only triggered by JSON parse errors, not structural validation.
- **Fix:** Create `TrajectoryLedgerSchema`, `TrajectoryRecordSchema`, `TrajectoryEventSchema`, `TrajectoryCheckpointSchema` in `trajectory.schema.ts` and use them in `readTrajectoryLedger()`.

#### DE-07: `trajectoryId` attachment creates inconsistency window
- **File:** `src/features/agent-work-contracts/operations.ts:56-64`
- **Code:**
  ```typescript
  const persisted = upsertAgentWorkContract(input.projectRoot, contract)
  if (input.trajectoryId && trajectoryEvidenceRef) {
    attachTrajectoryEvidence({ ... })
  }
  ```
- **Problem:** The contract is persisted FIRST, then trajectory evidence is attached. If `attachTrajectoryEvidence` throws (disk error, invalid trajectory), the contract is already on disk with `trajectoryId` set but no corresponding trajectory entry. A subsequent read sees a dangling reference.
- **Impact:** Orphaned contract records with broken trajectory references. `hivemind-agent-work-export` would succeed but export a contract with an unresolvable `trajectoryId`.
- **Fix:** Either: (a) attach trajectory evidence before persisting the contract, or (b) wrap both in a try-catch that reverts the contract write on failure.

### MEDIUM

#### DE-08: `store-operations.ts` mix of read/write violates CQRS boundaries
- **File:** `src/task-management/trajectory/store-operations.ts:20-23`
- **Code:**
  ```typescript
  export function inspectTrajectoryLedger(input: { projectRoot: string; trajectoryId?: string }): { ... } {
    const ledger = readTrajectoryLedger(input.projectRoot)
    return { ledger, trajectory: input.trajectoryId ? ledger.trajectories[input.trajectoryId] : undefined }
  }
  ```
- **Problem:** Read and write operations coexist in the same module. Per ARCHITECTURE.md, CQRS requires separation. The `inspect` (read) and `attach`/`checkpoint`/`event`/`close` (write) operations are in the same file. The `hivemind-trajectory` tool also serves both read and write actions through a single tool, which violates the architectural principle that tools are the CQRS write-side boundary.
- **Fix:** Split `store-operations.ts` into read operations (`inspect`, `traverse`) and write operations (`attach`, `checkpoint`, `event`, `close`).

#### DE-09: `deriveSurface()` duplication not fully resolved
- **File:** `src/task-management/continuity/delegation-persistence.ts:22` (only remaining copy)
- **Context:** Phase 25 scope mentions deduplicating `deriveSurface()`. Previous Phase 23-06 assessment found it duplicated across 3 modules. It now exists in only 1 module (`delegation-persistence.ts`), but verification confirms it is NOT duplicated in trajectory or agent-work-contracts code.
- **Verdict:** RESOLVED. This finding can be removed from Phase 25 scope. Removed as a blocker.

#### DE-10: Cross-feature pressure dependency in agent-work-contracts
- **File:** `src/features/agent-work-contracts/operations.ts:3,27-31`
- **Code:**
  ```typescript
  import { detectRuntimePressure } from "../runtime-pressure/index.js"
  // ...
  const pressureDecision = detectRuntimePressure({ score: input.pressureScore, tier: input.pressureTier, toolName: "hivemind-agent-work-create" })
  ```
- **Problem:** `src/features/agent-work-contracts/` imports from `src/features/runtime-pressure/`. Per ARCHITECTURE.md, features should be standalone modules. This creates a hard dependency between agent-work-contracts (a write-side feature) and runtime-pressure (classified as "read-side analysis only").
- **Impact:** If the pressure module is refactored, agent-work-contracts breaks. The pressure dependency makes agent-work-contracts non-standalone.
- **Fix:** Inject the pressure decision as a parameter rather than importing directly. The tool layer (`hivemind-agent-work.ts`) should resolve pressure and pass results to the feature.

#### DE-11: `updatedAt` not enforced in `writeTrajectoryLedger`
- **File:** `src/task-management/trajectory/ledger.ts:67-72`
- **Code:**
  ```typescript
  export function writeTrajectoryLedger(projectRoot: string, ledger: TrajectoryLedger): string {
    const ledgerPath = getTrajectoryLedgerPath(projectRoot)
    mkdirSync(resolve(ledgerPath, ".."), { recursive: true })
    writeFileSync(ledgerPath, `${JSON.stringify(ledger, null, 2)}\n`, "utf-8")
    return ledgerPath
  }
  ```
- **Problem:** `writeTrajectoryLedger` does NOT stamp `updatedAt` itself. All 5 callers in `store-operations.ts` set `ledger.updatedAt = now` before calling write, but nothing enforces this. Compare with `store.ts:50` where `persistAgentWorkContracts` stamps `updatedAt: Date.now()` inline. If a future caller forgets to set `updatedAt`, the ledger timestamp becomes stale.
- **Fix:** Have `writeTrajectoryLedger` stamp `updatedAt` itself, or add validation that rejects ledgers with stale timestamps.

## 2. Feature Integration

### 2.1 Trajectory ↔ Session-Tracker Lineage Redundancy

| Aspect | Session-Tracker | Trajectory |
|--------|----------------|------------|
| Storage | `.hivemind/session-tracker/` (directory tree) | `.hivemind/state/trajectory-ledger.json` (flat file) |
| Lineage tracking | Hierarchy manifest with parent-child edges | `TrajectoryRecord` with `parentTrajectoryId` |
| Status | `status` field per session | `TrajectoryStatus` (`active`/`closed`) |
| Events | Full message/tool-call history in `.json` | `TrajectoryEvent` with `eventType` + `summary` |
| Test coverage | 418 tests in session-tracker alone | **0 tests** |
| Consumers | `hivemind-session-view` (3-root consolidation) | `hivemind-trajectory` tool (standalone) |

**Finding (INT-01):** Both systems track session lineage with parent-child relationships. The `hivemind-session-view` tool already consolidates trajectory data alongside session-tracker and delegation data, suggesting trajectory is a secondary, less-tested data source for lineage that session-tracker already provides with 418 tests.

**Recommendation:** Phase 25 should either:
- (a) **Deprecate** `TrajectoryRecord` lineage fields (`rootSessionId`, `sessionId`, `parentTrajectoryId`) and make trajectory purely an evidence/checkpoint/event store that references session-tracker IDs, or
- (b) **Document** the explicit boundary: trajectory owns *resumability evidence* (events + checkpoints), while session-tracker owns *session lineage* (hierarchy + status).

### 2.2 Trajectory ↔ Agent-Work-Contract Circular Evidence References

**Finding (INT-02):** The evidence reference pattern is circular:
1. Agent-work-contracts stores `trajectoryEvidenceRef: "agent-work-contract:{id}"` on the contract (`operations.ts:42`)
2. Agent-work-contracts calls `attachTrajectoryEvidence` with `evidenceRef: "agent-work-contract:{id}"` (`operations.ts:58-63`)
3. The trajectory stores the evidence ref on the trajectory record

This means: the contract references the trajectory, and the trajectory references the contract. Neither side has enough independent context to resolve the other without reading the other's store. This adds operational complexity — deleting a contract requires removing the trajectory evidence ref, and vice versa.

**Impact:** Cleanup is non-trivial. Orphaned references persist silently.

### 2.3 Trajectory ↔ Agent-Work-Contract ↔ Pressure Three-Way Coupling

**Finding (INT-03):** Three systems are coupled:
- `agent-work-contracts/operations.ts:27-31` calls `detectRuntimePressure()` from runtime-pressure
- `agent-work-contracts/operations.ts:58-63` calls `attachTrajectoryEvidence()` from trajectory
- `hivemind-pressure` tool has `attach_event` action that writes to trajectory

This creates a dependency triangle: agent-work → pressure + agent-work → trajectory + pressure → trajectory. Any change to one module's interface affects at least two others.

### 2.4 Tool Registration Discrepancy

**Finding (INT-04):** The `hivemind-session-view` tool (created in Phase 16) reads trajectory data alongside session-tracker and delegation data. However, the trajectory tool itself (`hivemind-trajectory`) is listed as a separate tool. There is no single "session intelligence" tool — consumers must compose data from 3 separate tool calls.

## 4. Overlap Analysis

### 4.1 Trajectory ↔ Session-Tracker

Both system tracks session lineage. The key difference:
- **Session-tracker** is session-lifetime-bound: created with session, updated on each tool call, message, status change. Has 418 tests. File-based (directory tree).
- **Trajectory** is designed for resumability evidence: events, checkpoints, optional lineage. Has 0 tests. Single-file ledger.

**Verdict:** Legitimate but redundant. The trajectory's value proposition is "resumability evidence across sessions" — events and checkpoints that survive compaction. The lineage fields (`rootSessionId`, `sessionId`, `parentTrajectoryId`) duplicate session-tracker data. Phase 25 should remove these lineage fields and make trajectory purely an event/checkpoint evidence store.

### 4.2 Trajectory ↔ Continuity Store

Both write to `.hivemind/state/`. Continuity owns session state; trajectory owns trajectory records. The `sessionId` field on trajectory records references continuity session IDs. No overlap here — this is a clean reference boundary.

### 4.3 Agent-Work-Contracts ↔ Delegation Records

Both track agent work:
| Aspect | Agent-Work-Contracts | Delegation Records |
|--------|---------------------|-------------------|
| Store file | `agent-work-contracts.json` | `delegations.json` |
| Status lifecycle | created→running→blocked→completed→cancelled | dispatched→running→completed→error→timeout |
| Consumers | `hivemind-agent-work-create/export` | `delegation-status`, `session-tracker` |

**Verdict:** Legitimate separation. Contracts are work *authorization* (what should be done, with what boundaries). Delegations are work *execution* (what was dispatched, with what result). The `trajectoryId` reference bridges them correctly.

### 4.4 Implemented vs Registered Tool Discrepancy

**Finding (OVL-01):** `hivemind-session-view` (`src/tools/hivemind/hivemind-session-view.ts`) reads trajectory data from `trajectory-ledger.json`. If trajectory store file format changes, `hivemind-session-view` breaks silently (it catches errors but returns partial data).
