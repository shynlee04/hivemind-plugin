## 1. Executive Summary

The delegation subsystem is the largest and most complex component of the Hivemind harness, spanning 21 source files in `src/coordination/delegation/` (3,866 LOC), 2 files in `src/coordination/completion/` (592 LOC), and 1 file in `src/coordination/concurrency/` (300 LOC). The architecture follows a layered DI pattern with a facade (`manager.ts`), runtime wiring (`manager-runtime.ts`), orchestration hub (`coordinator.ts`), and state store (`state-machine.ts`).

**Key findings:**
- The dual-signal completion protocol is correctly implemented but has no integration test validating the full dispatch→completion→cleanup lifecycle.
- `manager-runtime.ts` at 491 LOC is close to the 500 LOC module ceiling and contains the heaviest coupling (8+ direct imports from sibling modules).
- The concurrency queue (`queue.ts`) is well-isolated and thoroughly tested, but `SpawnReservation` has zero dedicated test coverage.
- Error handling is inconsistent: some modules throw `[Harness]`-prefixed errors, others silently return undefined or no-op.
- The state machine's terminal transition logic allows any terminal status to transition to itself (idempotent re-entry), which is intentional but undocumented in tests.

## 4. Internal State Analysis

### 4.1 `DelegationStateMachine` State

The state machine maintains an in-memory `Map<string, Delegation>` keyed by delegation ID.

**State fields per delegation record:**

| Field | Type | Purpose |
|-------|------|---------|
| `id` | `string` | Unique delegation identifier |
| `status` | `DelegationStatus` | Current lifecycle status |
| `parentSessionId` | `string` | Parent that initiated the delegation |
| `childSessionId` | `string \| undefined` | Child session executing the task |
| `agentName` | `string` | Target agent for delegation |
| `taskDescription` | `string` | Human-readable task description |
| `createdAt` | `number` | Unix timestamp (ms) |
| `lastMessageCount` | `number` | Latest observed message count for stability |
| `stablePollCount` | `number` | Consecutive polls with same message count |
| `nestingDepth` | `number` | Delegation nesting depth (for cycle detection) |
| `graceCleanupAt` | `number \| undefined` | Timestamp when grace cleanup fires |
| `safetyCeilingAt` | `number \| undefined` | Hard timeout ceiling |
| `queueKey` | `string \| undefined` | Concurrency lane key |
| `terminalKind` | `DelegationTerminalKind \| undefined` | How the delegation terminated |
| `error` | `string \| undefined` | Error message if status is 'error' |

**Status transition table:**

```
dispatched → running       (child session started)
dispatched → error         (preflight or spawn failure)
running    → completed     (successful completion)
running    → error         (runtime error)
running    → timeout       (safety ceiling exceeded)
completed  → completed     (idempotent re-entry)
error      → error         (idempotent re-entry)
timeout    → timeout       (idempotent re-entry)
```

**Constants:**
- `DEFAULT_SAFETY_CEILING_MS = 30 * 60 * 1000` (30 minutes) — hard upper bound
- Grace period: configurable, typically 5 seconds post-terminal for cleanup

### 4.2 `CompletionDetector` State

Five internal maps maintained:

| Map | Key | Value | Purpose |
|-----|-----|-------|---------|
| `watchers` | `sessionID` | `{ resolve, timeoutId }` | Pending single-signal watchers |
| `cachedResults` | `sessionID` | `CompletionResult` | Results for late watchers (idle NOT cached) |
| `dualSignalWatchers` | `delegationId` | `DualSignalWatcher` | Dual-signal state tracking |
| `messageCounts` | `sessionID` | `number` | Last known message count per session |
| `stabilityTimers` | `sessionID` | `timeoutId` | Active stability detection timers |

**Stability detection:**
- `stabilityTimeoutMs` default: 30,000ms (30 seconds)
- A session is considered stable when its message count remains unchanged for `stabilityTimeoutMs`
- On stability: fires `idle` signal to watcher or caches result

### 4.3 `DelegationConcurrencyQueue` State

Single internal map: `lanes: Map<string, Lane>`

**Lane structure:**

```typescript
type Lane = {
  active: number        // Currently executing delegations
  limit: number         // Maximum concurrent executions
  pending: PendingAcquire[]  // Waiting for slot release
  queued: {
    high: QueuedTask[]    // Priority queue (dequeued first)
    normal: QueuedTask[]  // Normal queue
  }
}
```

**Lane lifecycle:**
- Created on first `acquire()` or `enqueue()`
- Automatically cleaned up when `active === 0 && pending.length === 0 && queued.empty`
- Default limit: `DEFAULT_CONCURRENCY_LIMIT = 3`

## 5. Error Handling Patterns

### 5.1 Current Patterns

| Pattern | Used By | Example |
|---------|---------|---------|
| `[Harness]`-prefixed throws | `queue.ts`, `manager-runtime.ts` | `throw new Error("[Harness] Concurrency acquire timed out...")` |
| Silent undefined return | `state-machine.ts` getters, `queue.ts` peek/dequeue | `get(id): Delegation \| undefined` |
| Graceful no-op | `detector.ts` feed/FeedMessageCount | Early return on invalid inputs |
| Error status transition | `coordinator.ts`, `state-machine.ts` | Transition to `error` status with message |
| Timeout signal | `detector.ts`, `state-machine.ts` | Safety ceiling fires `timeout` signal |

### 5.2 Inconsistencies Found

**I1: Mixed error strategies in coordinator**
- `coordinator.ts` sometimes throws (preflight failures), sometimes transitions to error status
- Location: `src/coordination/delegation/coordinator.ts:dispatch()` vs `handleSessionError()`
- Impact: Callers must handle both thrown exceptions and error-status results
- Fix approach: Adopt error-status transitions for all delegation-level failures; reserve throws for programming errors

**I2: Silent no-op in CompletionDetector.feed()**
- `feed()` silently ignores unknown event types and undefined sessionIDs
- Location: `src/coordination/completion/detector.ts:39-47`
- Impact: Debugging difficulty when events are silently dropped
- Fix approach: Add optional debug logging for dropped events (behind a flag)

**I3: SpawnReservation errors are not wrapped**
- `reserveSubagentSpawn()` propagates raw errors from `taskState.reserveDescendant()`
- Location: `src/coordination/concurrency/queue.ts:268-275`
- Impact: Error message may lack `[Harness]` prefix, breaking error classification
- Fix approach: Wrap in `[Harness]`-prefixed error with context

**I4: Missing error context in state-machine transitions**
- `transition()` throws generic error on invalid status change
- Location: `src/coordination/delegation/state-machine.ts`
- Impact: Hard to debug when transition fails in production
- Fix approach: Include `from` status, `to` status, and delegation ID in error message

## 9. Cross-Cutting Concerns

### 9.1 Timer Management

The delegation system uses multiple timer mechanisms:

| Timer Type | Location | Default Duration | Cleanup |
|------------|----------|------------------|---------|
| Safety ceiling | `state-machine.ts` | 30 min | `clearTimeout` on terminal transition |
| Grace cleanup | `state-machine.ts` | 5 sec | `clearTimeout` on cancel |
| Stability detection | `completion/detector.ts` | 30 sec | `clearTimeout` on event or cancel |
| Watch timeout | `completion/detector.ts` | Configurable | `clearTimeout` on resolve or cancel |
| Concurrency acquire timeout | `concurrency/queue.ts` | Configurable | `clearTimeout` on resolve or reject |
| Polling interval | `monitor.ts` | Configurable | `clearInterval` on stop |
| Escalation timer | `escalation-timer.ts` | Configurable | `clearTimeout` on cancel |
| Periodic notification | `periodic-notifier.ts` | Configurable | `clearInterval` on stop |

**Concern:** No centralized timer registry. If the harness shuts down uncleanly, timers may leak.
**Recommendation:** Consider a timer registry in the state machine for graceful shutdown.

### 9.2 Memory Management

All state stores use in-memory Maps:
- `DelegationStateMachine.lanes` — delegation records
- `CompletionDetector.watchers/cachedResults/dualSignalWatchers/messageCounts/stabilityTimers` — 5 maps
- `DelegationConcurrencyQueue.lanes` — lane data

**Cleanup mechanisms:**
- `pruneExpired()` on state machine removes old terminal records
- Grace cleanup timer removes terminal delegations after delay
- `cleanupLane()` in concurrency queue auto-removes empty lanes
- `unwatch()` / `cancel()` in detector clears specific watchers

**Concern:** Long-running sessions may accumulate stale entries in `cachedResults` and `messageCounts` if completion signals are missed.
**Recommendation:** Add periodic sweep of stale entries in CompletionDetector.

### 9.3 Thread Safety

All modules are single-threaded (Node.js event loop). No `worker_threads` or `Atomics` usage.
- Maps are mutated synchronously within event loop ticks
- Async operations (watch, acquire) use Promise-based coordination
- No shared `ArrayBuffer` or `SharedArrayBuffer`

**Concern:** `DelegationConcurrencyQueue.makeRelease()` creates closures that capture lane references. If the same lane is cleaned up and recreated, stale closures may reference the old lane object.
**Recommendation:** Verify this is safe (analysis suggests it is, since cleanup only removes empty lanes and closures reference the same object).

### 9.4 Logging

The delegation system has **no structured logging**. Error conditions are either:
- Thrown as errors (caught by the runtime)
- Stored in delegation records (`error` field)
- Silently handled (no-op returns)

**Recommendation:** Add opt-in structured logging at key transition points:
- Status transitions (from → to, delegation ID)
- Completion signals (dual-signal arrival order)
- Concurrency slot acquisition/release
- Grace cleanup execution

## 10. Risk Assessment

### 10.1 High Risk

| Risk | Module | Description | Mitigation |
|------|--------|-------------|------------|
| R1: Dual-signal race | `detector.ts` | `signalCompletionEvent` and `signalTerminalStatus` arriving simultaneously | The `fired` flag prevents double-fire, but ordering is untested |
| R2: Timer leak on crash | All | No centralized timer cleanup on harness shutdown | Add shutdown hook in state machine |
| R3: State machine corruption | `state-machine.ts` | Direct Map mutation without defensive copy | Already mitigated by deep-clone-on-read in continuity store |

### 10.2 Medium Risk

| Risk | Module | Description | Mitigation |
|------|--------|-------------|------------|
| R4: Module size ceiling | `manager-runtime.ts` | 491 LOC approaching 500 LOC limit | Extract spawner wiring to separate module |
| R5: Cross-layer coupling | `state-machine.ts` | Imports from `task-management/` violate layer boundaries | Accept as pragmatic; document the exception |
| R6: Missing error context | `state-machine.ts` | Generic transition error messages | Add from/to/delegationId to error messages |

### 10.3 Low Risk

| Risk | Module | Description | Mitigation |
|------|--------|-------------|------------|
| R7: Priority starvation | `queue.ts` | High-priority tasks may starve normal queue | Acceptable for current usage; monitor |
| R8: Grace cleanup timing | `state-machine.ts` | Grace cleanup may fire before parent processes result | Accept; grace delay is configurable |
