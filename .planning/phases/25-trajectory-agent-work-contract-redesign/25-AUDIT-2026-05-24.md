# Phase 25 Audit: Trajectory + Agent-Work-Contract Redesign

**Audit Date:** 2026-05-24  
**Scope:** Full source-code audit of trajectory and agent-work-contract systems  
**Auditor:** gsd-codebase-mapper (TASK-A, round 3 verification)

---

## Executive Summary

This audit examines 9 source files across `src/task-management/trajectory/`, `src/features/agent-work-contracts/`, `src/schema-kernel/`, `src/tools/hivemind/`, and `src/shared/types.ts`, plus existing test files and planning documents.

**Overall assessment:** Both systems are structurally sound with clear CQRS boundaries and correct `.hivemind/` state-root compliance. However, **8 design errors**, **2 feature boundary violations**, and **3 mechanism-compliance gaps** were identified. Test coverage exists (12 tests across 4 files) but is thin — the ROADMAP claim of "zero trajectory tests" is outdated (3 ledger tests + 2 tool tests exist). The trajectory ledger lacks atomic write safety, there is a feature-layer boundary violation where agent-work-contracts directly imports trajectory mutation functions, and the agent-work-contracts store silently loses data on corruption.

**Phase 25 readiness: NOT YET READY.** 7 blocking issues (3 CRITICAL, 4 HIGH) must be resolved before Phase 25 can begin execution.

## 1. Design Errors

### 1.1 [CRITICAL] Trajectory ledger lacks atomic write — risk of partial file corruption

- **Files:** `src/task-management/trajectory/ledger.ts:67-72`
- **Issue:** `writeTrajectoryLedger()` calls `writeFileSync` directly on the target path. If the process crashes mid-write, the ledger file is left in a partial/corrupt state.
- **Evidence:** Compare with `src/features/agent-work-contracts/store.ts:46-52` which correctly uses `writeFileSync(tempFile)` + `renameSync(tempFile, filePath)` for atomic writes.
- **Fix approach:** Replace `writeFileSync(ledgerPath, ...)` with a write-to-temp + rename pattern identical to `store.ts:49-51`.

### 1.2 [CRITICAL] Agent-work-contracts store silently resets on corruption — data loss

- **Files:** `src/features/agent-work-contracts/store.ts:27-38`
- **Issue:** `readAgentWorkContracts()` uses `AgentWorkContractStoreSchema.safeParse()`. When parsing fails (corrupt JSON, schema mismatch), it calls `quarantineCorruptStore(filePath)` and **silently returns an empty store**. No error is thrown, no warning logged to the caller.
- **Evidence:**
  - Trajectory's `readTrajectoryLedger()` (`ledger.ts:41-58`) throws an error after quarantine — correct behavior.
  - Agent-work's `readAgentWorkContracts()` returns `createEmptyStore()` silently.
- **Impact:** If `hivemind-agent-work-create` is called while the store is corrupt, it overwrites the quarantined file with a fresh empty store, permanently losing all existing contracts.
- **Fix approach:** Throw after quarantine (like trajectory does), or log a CRITICAL warning and return a tombstone marker so callers can detect the data loss.

### 1.3 [HIGH] No rollback when trajectory attach fails after contract persist

- **Files:** `src/features/agent-work-contracts/operations.ts:57-64`
- **Issue:** `createAgentWorkContract()` calls `upsertAgentWorkContract()` (which writes to disk) BEFORE calling `attachTrajectoryEvidence()`. If the trajectory attach throws (disk full, ledger corrupt), the contract has already been durably written with a `trajectoryEvidenceRef` that points to a trajectory that doesn't exist. There is no rollback mechanism.
- **Fix approach:** Either (a) write contract + attach trajectory in a transaction-like sequence with rollback of the contract write on failure, or (b) defer the trajectory attachment to the tool layer so the feature module does not perform cross-sector mutations.

### 1.4 [HIGH] Trajectory ledger type guard is shallow — records pass through unvalidated

- **Files:** `src/task-management/trajectory/ledger.ts:80-85`
- **Issue:** `isTrajectoryLedger()` only checks that `version` matches `TRAJECTORY_LEDGER_VERSION` and `trajectories` is a non-null object. It does NOT validate:
  - The shape of individual `TrajectoryRecord` entries
  - That `evidenceRefs`, `checkpoints`, `events` exist and are arrays
  - That `status` is "active" or "closed"
- **Risk:** Corrupt or manually-edited ledger files with malformed records pass validation silently, causing runtime errors in `store-operations.ts` when accessing missing fields.
- **Fix approach:** Use Zod schema validation (similar to `AgentWorkContractStoreSchema.safeParse()` in `store.ts:31`) rather than a hand-rolled type guard.

### 1.5 [HIGH] In-place mutation of ledger records — temporal coupling

- **Files:** `src/task-management/trajectory/store-operations.ts:136-163`
- **Issue:** `upsertTrajectory()` mutates `ledger.trajectories[input.trajectoryId]` directly. When the trajectory already exists (`existing`), it overwrites `rootSessionId`, `sessionId`, and `parentTrajectoryId` with non-null-ish-coalescing (`input.X ?? existing.X`). The mutated object is the same reference stored in `ledger.trajectories`. Callers receive a reference to the in-memory live object, not a clone.
- **Risk:** Callers that hold a reference after the function returns still point to the live object, which can be mutated later by subsequent operations.
- **Fix approach:** Deep-clone before mutating (similar to `store.ts:122-146`).

### 1.6 [MEDIUM] `compareTrajectoryRecords` only handles direct parent-child edges

- **Files:** `src/task-management/trajectory/store-operations.ts:177-181`
- **Issue:** The sort comparator returns `1` if `a.parentTrajectoryId === b.id` and `-1` if `b.parentTrajectoryId === a.id`. For ancestry of depth 3+ (grandparent → parent → child), the sort between grandparent and child returns `a.createdAt - b.createdAt`, which may produce incorrect ordering if timestamps are identical or close.
- **Fix approach:** Use a topological sort that respects the full ancestry chain, or document the limitation.

### 1.7 [MEDIUM] Schema-tool arg type drift — inline `tool.schema` vs Zod schemas

- **Files:**
  - `src/tools/hivemind/hivemind-trajectory.ts:33-44` (inline args)
  - `src/schema-kernel/trajectory.schema.ts:11-23` (Zod schema)
  - `src/tools/hivemind/hivemind-agent-work.ts:33-54` (inline args)
  - `src/schema-kernel/agent-work-contract.schema.ts:87-109` (Zod schema)
- **Issue:** The inline `tool.schema` args and the Zod schemas are not validated against each other. Specific drifts:
  1. `trajectory.schema.ts:22`: `evidenceRefs: z.array(z.string().min(1))` — each string must be at least 1 char. Tool inline `hivemind-trajectory.ts:44`: `s.array(s.string())` — no min-length check.
  2. `agent-work-contract.schema.ts:108`: `pressureApproved: z.boolean().default(false)` — has a default value. Tool inline `hivemind-agent-work.ts:54`: `s.boolean().optional()` — no default, just optional.
  3. The Zod schema for `AgentWorkCreateToolInputSchema` defines `requiredProof` as `z.array(z.string().min(1)).default([])` but the tool inline at line 42 has `s.array(s.string()).optional()` — different optionality semantics.
- **Risk:** OpenCode uses the inline `tool.schema` args for the UI prompt and initial validation; the Zod schema is only applied in `execute()`. If the inline args allow a value that the Zod schema rejects, the user sees an opaque validation error instead of a clear schema violation.
- **Fix approach:** Generate or derive inline args from Zod schemas, or add a cross-validation test that ensures they agree.

### 1.8 [LOW] `quarantineCorruptStore` can race on duplicate timestamps

- **Files:** `src/features/agent-work-contracts/store.ts:98-100`
- **Issue:** The quarantine path is `${filePath}.corrupt-${Date.now()}-${process.pid}-${randomUUID()}`. The randomUUID provides uniqueness, but if two callers attempt to quarantine the same file (within the same process), the second `renameSync` will fail because the source file no longer exists (the first already moved it). This is not handled.
- **Fix approach:** Check if the file still exists before renaming, or wrap in try/catch.

## 2. Feature Integration / Boundary Violations

### 2.1 [CRITICAL] Feature module directly mutates trajectory state — CQRS boundary violation

- **Files:** `src/features/agent-work-contracts/operations.ts:3`
- **Import:** `import { attachTrajectoryEvidence } from "../../task-management/trajectory/index.js"`
- **Issue:** `src/features/` is defined as "Standalone runtime feature modules" per `src/features/AGENTS.md:9-12`. The architecture states that features "own computation" while tools "own CQRS mutation authority" (`src/features/AGENTS.md:29`). By importing `attachTrajectoryEvidence` directly, the agent-work-contracts feature is performing a CQRS mutation (writing to the trajectory ledger) without going through a tool.
- **Evidence:** `operations.ts:57-64` shows the feature calling `attachTrajectoryEvidence()` directly. This is the only place in `src/features/` that directly imports from `src/task-management/` for a mutation.
- **Fix approach:** Either (a) move the trajectory attach call to the tool layer (`hivemind-agent-work.ts`), or (b) if this is an intentional design decision, document it in `src/features/AGENTS.md` and accept the boundary relaxation.

### 2.2 [HIGH] Dual-truth risk between trajectory and session-tracker hierarchy manifest

- **Files:**
  - `src/task-management/trajectory/types.ts:53-76` — `TrajectoryRecord` has `parentTrajectoryId`, `rootSessionId`, `sessionId`
  - Session-tracker's hierarchy manifest (`hierarchy-manifest.json`) maintains parent-child delegation trees
- **Issue:** Both systems independently track session lineage:
  - Trajectory: `TrajectoryRecord.parentTrajectoryId` for evidence-chaining
  - Session-tracker: `hierarchy-manifest.json` for delegation hierarchy
- **Risk:** A session can be registered in the hierarchy manifest but not in the trajectory ledger, or vice versa. There is no cross-validation between the two stores.
- **Impact:** When recovering from a crash, the two stores may disagree about which sessions existed and their parent-child relationships.
- **Fix approach:** Add an integration test that creates a session/delegation, then verifies both stores agree on the hierarchy. Consider making trajectory the authoritative store for lineage and having session-tracker reference it.

## 3. Mechanism Compliance

### 3.1 [CRITICAL] Atomic write inconsistency between stores

| Store | Atomic write | File |
|-------|------------|------|
| Trajectory ledger | ❌ No — `writeFileSync` direct | `ledger.ts:67-72` |
| Agent work contracts | ✅ Yes — temp+rename pattern | `store.ts:46-52` |

- **Fix:** Unify on temp+rename pattern for both stores. The trajectory ledger is more critical because corrupt ledgers throw errors (losing all evidence), while agent-work-contracts silently resets (losing all contracts).

### 3.2 [HIGH] Error handling inconsistency on corrupt data

| Module | Behavior on corrupt data | File |
|--------|------------------------|------|
| `readTrajectoryLedger()` | ✅ Throw `[Harness]` error after quarantine | `ledger.ts:53-57` |
| `readAgentWorkContracts()` | ❌ Silent reset to empty store | `store.ts:31-38` |

- **Fix:** Align `readAgentWorkContracts()` with the trajectory pattern — throw after quarantine so callers must explicitly handle corruption rather than losing data silently.

### 3.3 [MEDIUM] `AgentWorkStatusSchema` defines 5 statuses, implementation uses only 1

- **Files:**
  - `src/schema-kernel/agent-work-contract.schema.ts:17` — `z.enum(["created", "running", "blocked", "completed", "cancelled"])`
  - `src/features/agent-work-contracts/operations.ts:47` — `status: "created"`
- **Issue:** The schema defines 5 lifecycle statuses (`created`, `running`, `blocked`, `completed`, `cancelled`) but the only place a status is ever set is `operations.ts:47` where it's hardcoded to `"created"`. There are no operations that transition contracts to `running`, `blocked`, `completed`, or `cancelled`.
- **Risk:** If a future tool or consumer reads a contract with `status: "created"` and expects it to be mutable, there is no operation to transition it. The status enum is aspirational — 4/5 values are dead code.
- **Fix approach:** Either implement lifecycle transitions (updateStatus, complete, cancel operations) or reduce the schema to only `z.enum(["created"])` until transitions are needed.

### 3.4 [LOW] `boundText` control char stripping may alter evidence refs

- **Files:** `src/task-management/trajectory/store-operations.ts:173-174`
- **Issue:** `boundText()` replaces control characters `[\u0000-\u001f\u007f]` with spaces and trims. Applied to evidence refs via `mergeStrings()` at line 170. Evidence references containing path components (e.g., `.hivemind/journal/2026-04-30.jsonl#L1`) are unlikely to contain control chars, but the transformation is lossy.
- **Fix:** Evidence refs should not be passed through `boundText()` at all — only use it for human-readable summaries.

## 4. Overlap Analysis

### 4.1 Trajectory ↔ Session-Tracker Hierarchy Manifest

| Aspect | Trajectory Ledger | Hierarchy Manifest |
|--------|------------------|-------------------|
| Purpose | Evidence/resumability lineage | Session delegation tree |
| Parent-child | `TrajectoryRecord.parentTrajectoryId` | `hierarchy-manifest.json` tree |
| Data root | `.hivemind/state/trajectory-ledger.json` | `.hivemind/state/session-tracker/hierarchy-manifest.json` |
| Records | Trajectory records | Session entries |
| Mutation | Through `store-operations.ts` | Through session-tracker capture handlers |

**Verdict:** Acceptable overlap by design. Trajectory is for cross-session resumability evidence; hierarchy manifest is for runtime session tracking. Both can coexist if integration tests verify they stay synchronized for the session IDs they share.

**Recommendation:** Add a cross-store consistency test that creates a session, records a trajectory, and verifies that both stores agree on the hierarchy structure.

### 4.2 Trajectory ↔ Delegation Persistence

- Trajectory correctly uses `EvidenceRef` strings to reference delegation records without owning them. ✅
- No overlap: delegation records live in `.hivemind/state/delegations.json`, trajectory in `.hivemind/state/trajectory-ledger.json`.

### 4.3 Agent Work Contracts ↔ Delegation Records

- Contracts create a "pre-delegation" agreement; delegation records capture actual dispatched work. ✅
- Contracts reference trajectories via `trajectoryId` but do not duplicate delegation data. ✅
- No overlap.

### 4.4 Agent Work Contracts ↔ Trajectory

- `operations.ts:42-51` sets `trajectoryEvidenceRef: agent-work-contract:${id}` on the contract and writes it to the trajectory ledger. This creates a bidirectional reference (contract → trajectory, trajectory → contract evidence ref). ✅
- The boundary violation on the write side (1.1) is the only concern here.

## 6. Recommendations

### 6.1 Pre-Phase-25 Fixes (blocking)

1. **[CRITICAL] Add atomic write to trajectory ledger** — `ledger.ts:67-72`: Replace direct `writeFileSync` with temp+rename pattern matching `store.ts:46-52`.

2. **[CRITICAL] Fix silent data loss in agent-work-contracts store** — `store.ts:27-38`: Throw after quarantine (like trajectory does) instead of silently returning empty store.

3. **[CRITICAL] Resolve feature boundary violation** — `operations.ts:3`: Either (a) move trajectory attach to tool layer, or (b) document intentional boundary relaxation in AGENTS.md.

4. **[HIGH] Add rollback or reorder for trajectory attach failure** — `operations.ts:57-64`: Ensure that if `attachTrajectoryEvidence` fails after `upsertAgentWorkContract` succeeds, either the contract write is rolled back or the sequence is reordered so the trajectory is attached first.

5. **[HIGH] Deepen trajectory ledger type guard** — `ledger.ts:80-85`: Use Zod schema for validation instead of shallow hand-rolled type guard.

6. **[HIGH] Deep-clone trajectory records on upsert** — `store-operations.ts:136-163`: Add deep-clone to prevent temporal coupling, matching `store.ts:122-146`.

7. **[HIGH] Align tool inline args with Zod schemas** — `hivemind-trajectory.ts:33-44`, `hivemind-agent-work.ts:33-54`: Add cross-validation tests or derive inline args from Zod schemas.

### 6.2 During Phase 25

8. Expand trajectory test coverage: traversal by sessionId, close on missing trajectory, concurrent writes, evidence ref deduplication, deep ancestry sorting.

9. Expand agent-work-contract test coverage: update existing contract, get non-existent contract, corrupt store behavior.

10. Implement agent-work-contract lifecycle transitions (running, blocked, completed, cancelled) or reduce schema to match actual usage.

11. Add cross-store consistency test between trajectory ledger and session-tracker hierarchy manifest.

12. Remove `boundText()` from evidence refs in `mergeStrings()` (`store-operations.ts:170`) — only apply to human-readable summaries.

13. Add `[Harness]` error prefix check across all trajectory operations for consistency.

### 6.3 Deferred

14. **Trajectory version migration support** — if `TRAJECTORY_LEDGER_VERSION` ever changes, there is no migration path. Document that version bumps require backward-compat migration.

15. **Topological sort for deep ancestry** — optional enhancement if deep delegation chains (>2 levels) become common.
