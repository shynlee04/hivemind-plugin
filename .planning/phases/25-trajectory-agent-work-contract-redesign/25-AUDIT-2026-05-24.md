## 1. Design Errors

### 1.1 DE-01: Agent Work Contract Has No Status Transition Machine

**Severity:** HIGH
**Files:** `src/schema-kernel/agent-work-contract.schema.ts:17`, `src/features/agent-work-contracts/operations.ts:43`

The `AgentWorkStatusSchema` defines five statuses (`created`, `running`, `blocked`, `completed`, `cancelled`) but there is **no state machine enforcement** anywhere in the codebase. The `createAgentWorkContract()` function hardcodes `status: "created"` at line 43 of `operations.ts` and there is no `updateAgentWorkContract()` operation to transition between statuses. The store supports `upsertAgentWorkContract()` at `store.ts:61` which blindly overwrites any contract — meaning any caller could set any status without validation.

**Impact:** A caller could skip `running` and go directly to `completed`, or set `cancelled` then back to `running`, violating the lifecycle contract that downstream consumers (compaction, handoff) depend on.

**Fix approach:** Add a `transitionAgentWorkContractStatus()` operation with a legal transition map: `created → running → {blocked, completed}`, `running → cancelled`, `blocked → running`. Enforce via Zod schema or runtime guard.

### 1.2 DE-02: Trajectory Ledger Has No Concurrent Write Protection

**Severity:** MEDIUM
**Files:** `src/task-management/trajectory/ledger.ts:28-43`, `src/task-management/trajectory/store-operations.ts:31-41`

All trajectory mutations (`attachTrajectoryEvidence`, `checkpointTrajectory`, `eventTrajectory`, `closeTrajectory`) follow the pattern: `readLedger → mutate in memory → writeLedger`. If two tool calls execute concurrently (e.g., in parallel agent dispatch), the second write will overwrite the first. The atomic write via `rename()` at `ledger.ts:43` prevents corrupt files but does not prevent lost updates.

**Impact:** Evidence refs, checkpoints, or events from concurrent operations can be silently dropped.

**Fix approach:** Either (a) serialize all trajectory writes through a queue, or (b) use file-locking (`proper-lockfile` or `flock`), or (c) accept the risk and document it as single-writer-only. Option (a) is most consistent with the existing concurrency queue in `src/coordination/concurrency/queue.ts`.

### 1.3 DE-03: Agent Work Contract Store Has Same Concurrency Issue

**Severity:** MEDIUM
**Files:** `src/features/agent-work-contracts/store.ts:27-38,46-52`

`readAgentWorkContracts()` reads the full store, then `upsertAgentWorkContract()` writes the full store back. Same read-modify-write race condition as the trajectory ledger. `persistAgentWorkContracts()` uses atomic rename at `store.ts:49-51` but this prevents corruption, not lost writes.

**Impact:** Two agents creating contracts simultaneously could lose one contract.

**Fix approach:** Same as DE-02 — serialize writes or add file locking.

### 1.4 DE-04: Trajectory `upsertTrajectory` Requires `rootSessionId` for Creation but Not for Updates

**Severity:** LOW
**Files:** `src/task-management/trajectory/store-operations.ts:136-163`

At line 145-147, `upsertTrajectory` throws if `rootSessionId` is missing for new trajectories. However, the `attach` action in the tool (`hivemind-trajectory.ts:73-81`) passes `rootSessionId` as optional. If a caller calls `attach` with only `trajectoryId` and no `rootSessionId`, and the trajectory doesn't exist yet, they get a runtime error. The schema validation at `trajectory.schema.ts:37-48` does not enforce `rootSessionId` for `attach` — it only checks `trajectoryId` is present for mutation actions.

**Impact:** Confusing runtime error when using `attach` to create a new trajectory without `rootSessionId`.

**Fix approach:** Either require `rootSessionId` in the schema for attach (when trajectory might not exist), or change the error message to be more descriptive about what's missing.

### 1.5 DE-05: `boundText` Truncation Is Silent

**Severity:** LOW
**Files:** `src/features/agent-work-contracts/operations.ts:110-113`, `src/task-management/trajectory/store-operations.ts:173-174`

Both `boundText` implementations silently truncate without any marker or warning. The trajectory version at `store-operations.ts:174` also strips control characters. If a caller passes a 3,000-character summary, it becomes 1,200 characters (contracts) or 2,000 characters (trajectory) with no indication that data was lost.

**Impact:** Silent data loss in compaction payloads and trajectory summaries. Debugging why injected context is incomplete becomes harder.

**Fix approach:** Add a truncation marker like `…[truncated]` at the boundary, or return a warning alongside the bounded value.

## 2. Feature Integration

### 2.1 FI-01: Agent Work Contract → Trajectory (One-Way Link)

**Status:** IMPLEMENTED
**Files:** `src/features/agent-work-contracts/operations.ts:57-64`

When a contract is created with a `trajectoryId`, the contract auto-attaches evidence to the trajectory via `attachTrajectoryEvidence()` at `operations.ts:58-64`. The evidence ref format is `agent-work-contract:${id}`.

**Gap:** The reverse link is weak. The trajectory only sees an evidence ref string — there is no typed contract reference on the trajectory side. A tool querying a trajectory cannot directly discover related contracts without string parsing.

**Recommendation:** Consider adding a `contracts` field to `TrajectoryRecord` or a dedicated lookup index.

### 2.2 FI-02: Agent Work Contract → Runtime Pressure

**Status:** IMPLEMENTED
**Files:** `src/features/agent-work-contracts/operations.ts:27-38`

The `createAgentWorkContract()` calls `detectRuntimePressure()` from `src/features/runtime-pressure/control-plane.ts` with the tool name `hivemind-agent-work-create`. If pressure is `block` or `require_approval` (without explicit approval), the write is prevented.

**Gap:** Only the **create** action is pressure-gated. There is no `updateAgentWorkContract()` or `transitionStatus()` operation, so there's no way to pressure-gate status transitions or compaction updates.

**Recommendation:** When adding status transitions (DE-01 fix), apply the same pressure gating pattern.

### 2.3 FI-03: Trajectory → Session Continuity / Delegation Records

**Status:** NOT INTEGRATED (BY DESIGN)
**Files:** `src/tools/hivemind/hivemind-trajectory.ts:32`

The tool description explicitly states: "without mutating continuity, delegation, journal, or document evidence authorities." The trajectory system operates as an independent evidence ledger. Integration with session continuity and delegation records is read-only through the `hivemind-session-view` tool which aggregates across roots.

**Observation:** This is correct CQRS separation. The trajectory is an append-only event timeline (Q3 decision) that complements but does not replace `continuity.ts`.

### 2.4 FI-04: Agent Work Contract → Session View

**Status:** NOT INTEGRATED
**Files:** `src/tools/hivemind/hivemind-session-view.ts`, `src/features/agent-work-contracts/`

The session-view tool (`hivemind-session-view`) aggregates data from session-tracker, delegations, and trajectory — but does **not** include agent work contracts. If an agent has an active work contract, the session view won't surface it.

**Recommendation:** Extend `hivemind-session-view` to include work contracts filtered by `owner.sessionId` when the session matches.

### 2.5 FI-05: Agent Work Contract → Delegation Manager

**Status:** NOT INTEGRATED (BY DESIGN)
**Files:** `src/coordination/delegation/manager.ts`, `src/features/agent-work-contracts/operations.ts`

Work contracts deliberately avoid mutating delegation records. The contract's `scope.allowedSurfaces` and `scope.nonGoals` define boundaries that the delegation manager does not currently enforce.

**Gap:** There is no runtime enforcement of `allowedSurfaces`. An agent with a work contract can still modify files outside its allowed surfaces because the delegation manager does not check contract boundaries.

**Recommendation:** This is a Phase 25 redesign consideration — should the delegation manager gate file mutations against active work contracts?

## 3. Mechanisms Review

### 3.1 MR-01: Trajectory Ledger Persistence

**Pattern:** JSON file at `.hivemind/state/trajectory-ledger.json`
**Write mechanism:** Atomic write via tmp file + `rename()` at `ledger.ts:30-43`
**Read mechanism:** Direct `readFileSync` + `JSON.parse` + no schema validation on read (unlike contracts)
**Deep clone:** No deep-clone on read — returns the parsed object directly

**Findings:**
- Unlike agent work contracts (`store.ts:31`), the trajectory ledger does NOT validate against a Zod schema on read. A corrupt or manually-edited ledger file will be loaded without validation.
- No quarantine mechanism for corrupt trajectory files (contracts have `quarantineCorruptStore()` at `store.ts:98-100`).
- No deep-clone on read means callers get references to the same object. Mutation of returned data could corrupt in-memory state.

**File:** `src/task-management/trajectory/ledger.ts:28-43`

### 3.2 MR-02: Agent Work Contract Store Persistence

**Pattern:** JSON file at `.hivemind/state/agent-work-contracts.json`
**Write mechanism:** Atomic write via tmp file + `rename()` at `store.ts:49-51`
**Read mechanism:** Direct `readFileSync` + `JSON.parse` + Zod schema validation + deep clone
**Deep clone:** Full deep clone via `cloneContract()` at `store.ts:122-145`

**Findings:**
- Schema validation on read is present and correct.
- Quarantine for corrupt stores is implemented.
- Deep clone prevents external mutation of persisted state.
- This is the **better** persistence implementation compared to the trajectory ledger.

**File:** `src/features/agent-work-contracts/store.ts:27-38`

### 3.3 MR-03: Compaction Bounding

**Pattern:** Pre-write bounding of text fields
**Limits:** briefing=1200, summary=1200, reinjectionPayload=2400, anchors=20 items max, sourceRefs=20 items max
**Implementation:** `operations.ts:93-101` — slices and trims

**Findings:**
- Bounding is applied correctly at create time.
- No bounding is applied if contracts are updated directly through `upsertAgentWorkContract()` (bypassing `createAgentWorkContract()`).
- The `boundText` function at `operations.ts:110-113` uses `value.slice(0, limit - 1).trimEnd()` — the `-1` is unnecessary since slice already respects the limit, but harmless.

### 3.4 MR-04: Evidence Reference Normalization

**Pattern:** Trajectory evidence refs are normalized (control chars stripped, whitespace collapsed, trimmed, deduplicated)
**Implementation:** `store-operations.ts:165-175`

**Findings:**
- `normalizeEvidenceRefs` correctly merges single + array refs.
- `mergeStrings` deduplicates via `Set`.
- `boundText` in trajectory strips control characters and collapses whitespace — more aggressive than the contract version.
- Trajectory evidence refs are capped at 2,000 chars; contract compaction fields have varying limits.

### 3.5 MR-05: Path Security

**Pattern:** Both stores use `assertPathWithinRoot()` from `src/shared/security/path-scope.js`
**Files:** `store.ts:18` (contracts), `ledger.ts:16` (trajectory — uses `resolve` directly)

**Finding:** The trajectory ledger at `ledger.ts:16-17` does NOT use `assertPathWithinRoot()`. It constructs the path via `resolve(projectRoot, ".hivemind", "state")` and appends the filename directly. While the project root is caller-provided and trusted by convention, this is a weaker security posture than the contract store.

**Severity:** LOW (project root is trusted by convention per CQRS model)

## 4. Existing Feature Overlap

### 4.1 OL-01: Trajectory ↔ Session View

**Overlap area:** Both systems provide session-centric read views.
- `hivemind-trajectory` traverse action returns trajectories filtered by `rootSessionId` or `sessionId`.
- `hivemind-session-view` aggregates session-tracker + delegations + trajectory data.

**Key difference:** Trajectory is write-side (append-only event timeline). Session-view is read-side (aggregation query). They follow CQRS correctly — trajectory owns the writes, session-view owns the reads.

**File references:**
- Trajectory tool: `src/tools/hivemind/hivemind-trajectory.ts`
- Session view tool: `src/tools/hivemind/hivemind-session-view.ts`

**Assessment:** No harmful overlap. Correct CQRS separation.

### 4.2 OL-02: Agent Work Contract ↔ Delegation Records

**Overlap area:** Both define work boundaries for agents.
- Delegation records in `src/task-management/continuity/delegation-persistence.ts` track dispatched tasks.
- Work contracts in `src/features/agent-work-contracts/` define scope, evidence requirements, and compaction.

**Key difference:** Delegation records track **dispatch** (who was sent to do what). Work contracts define **boundaries** (what's in scope, what evidence is required, what to preserve across compaction).

**Gap:** There is no foreign key or typed link between a delegation record and a work contract. An agent might be dispatched (delegation) with a work contract, but the two records are not linked.

**File references:**
- Delegation persistence: `src/task-management/continuity/delegation-persistence.ts`
- Work contracts: `src/features/agent-work-contracts/store.ts`

**Assessment:** Orthogonal concerns, but linking them would enable richer queries (e.g., "show me all delegations with active work contracts").

### 4.3 OL-03: Trajectory Evidence Refs ↔ Delegation Evidence

**Overlap area:** Both store evidence references.
- Trajectory stores `evidenceRefs: string[]` — arbitrary strings like `"journal:1"`, `"agent-work-contract:awc_xxx"`.
- Delegation records store structured status data.

**Gap:** Trajectory evidence refs are untyped strings with no schema. There is no validation that `"journal:1"` actually points to a valid journal entry, or that `"agent-work-contract:awc_xxx"` points to an existing contract.

**File references:**
- Trajectory types: `src/task-management/trajectory/types.ts:7` (`EvidenceRef` is just `string`)
- Contract schema: `src/schema-kernel/agent-work-contract.schema.ts:69-70`

**Assessment:** Weakness in the trajectory design. Evidence refs should have at least a prefix convention enforced by schema.

## 6. Context Injection Observations

### 6.1 CI-01: Trajectory as Context Injection Vector

The trajectory ledger is designed to be an append-only evidence timeline that agents can query to recover context after compaction or session breaks. The `compaction` fields in work contracts (briefing, summary, anchors, reinjectionPayload, sourceRefs) are explicitly designed for context reinjection.

**Observation:** The trajectory system does not currently consume contract compaction data. If an agent queries a trajectory, it gets evidence refs but not the contract's reinjection payload. To actually recover context, the agent would need to:
1. Query trajectory for evidence refs
2. Parse `agent-work-contract:*` prefixed refs
3. Query the contract store for the full reinjection payload

This three-step lookup is not documented anywhere and no tool automates it.

### 6.2 CI-02: Compaction Bounding vs. Actual Usefulness

The compaction limits (briefing=1200, summary=1200, reinjectionPayload=2400) are generous for structured data but tight for freeform context. A typical agent session produces far more context than 2,400 characters. The `anchors` field (max 20 items, 160 chars each) provides structured hooks but is limited.

**Observation:** The compaction system appears designed for a specific use case (bounded context recovery) but may be too constrained for real-world agent sessions. The Phase 25 redesign should validate whether these limits work in practice or need adjustment.
