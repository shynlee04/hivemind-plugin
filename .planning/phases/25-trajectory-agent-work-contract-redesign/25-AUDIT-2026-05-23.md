# Phase 25 Audit: Trajectory + Agent-Work-Contract Redesign

**Date:** 2026-05-23
**Auditor:** hm-l3-assessor (subagent)
**Scope:** `src/task-management/trajectory/`, `src/features/agent-work-contracts/`, consuming tools, schemas, and tests

## 1. Executive Summary

The trajectory ledger and agent-work-contract systems are **functional but structurally inconsistent**, with significant divergence in validation approaches, state management patterns, and cross-module coupling. The two modules were built in different phases with different design standards and now need reconciliation before Phase 25 redesign.

**Critical findings:**
- Validation inconsistency: trajectory uses manual JSON parse+validate; work-contracts uses Zod schema
- Read-modify-write concurrency hazard in both modules (no locking, full-ledger read/write per operation)
- Cross-module dependency: `agent-work-contracts/operations.ts` → `trajectory/store-operations.ts` is a one-way import creating tight coupling at the feature→task-management boundary
- Missing Zod schema for the trajectory ledger file format (trajectory has tool-input schema only)
- No event sourcing or append-only semantics despite trajectory conceptually being an event timeline
- Test coverage is minimal: 2 test cases for trajectory (62 LOC), 2 test cases for work-contracts (73 LOC)

## 2. Design Errors

### 2.1 Validation Approach Divergence

**Trajectory** (`src/task-management/trajectory/ledger.ts:18-35`):
- Reads file with `readFileSync` → `JSON.parse` → manual field checks in `validateLedgerStructure()`
- No Zod schema for the `TrajectoryLedger` file format
- `validateLedgerStructure()` is a manual assertion chain that does NOT produce typed error details

**Agent-work-contracts** (`src/features/agent-work-contracts/store.ts:25-36`):
- Reads file → `AgentWorkContractStoreSchema.safeParse()` (Zod)
- Produces structured `ZodError` with field-level detail
- Schema defined in `src/schema-kernel/agent-work-contract.schema.ts:78-82`

**Impact:** Trajectory ledger corruption produces vague errors; work-contract corruption produces actionable Zod diagnostics. Phase 25 should unify on Zod validation for both.

### 2.2 No Trajectory Ledger File Schema

The trajectory tool input has a Zod schema (`src/schema-kernel/trajectory.schema.ts:11-23`), but the **persisted ledger file** (`TrajectoryLedger` type in `src/task-management/trajectory/types.ts:51-59`) has no corresponding Zod schema. This means:
- File-format changes are not validated at the schema boundary
- Migration between versions requires manual code changes rather than schema evolution

**File:** `src/task-management/trajectory/types.ts:51-59`
```typescript
export interface TrajectoryLedger {
  version: 1
  updatedAt: number
  trajectories: Record<string, TrajectoryRecord>
}
```

### 2.3 Read-Modify-Write Without Locking

Both modules follow the same unsafe pattern:

**Trajectory** (`src/task-management/trajectory/store-operations.ts`):
- Every operation (`attachTrajectoryEvidence`, `recordCheckpoint`, `recordEvent`, `closeTrajectory`, `upsertTrajectory`) does: `readLedger()` → mutate in-memory → `writeLedger()`
- No file lock, no mutex, no queue

**Work-contracts** (`src/features/agent-work-contracts/store.ts:38-55`):
- `readStore()` → mutate → `writeStore()` in every operation
- `createAgentWorkContract` in `operations.ts:29-90` reads store, mutates, writes — all synchronous but unprotected

**Impact:** Concurrent tool invocations could corrupt the JSON files. Currently low risk because OpenCode runs tools sequentially per session, but cross-session scenarios (auto-loop, parallel delegations) could trigger this.

### 2.4 Trajectory Is Not Event-Sourced

The trajectory concept is described as "append-only event timeline" (per Q3 in locked decisions), but the implementation overwrites the entire JSON file on every mutation (`src/task-management/trajectory/store-operations.ts:15-22`). Each operation reads the full ledger, appends to in-memory arrays, and writes the entire ledger back.

This contradicts the Q3 decision: "Session Journal as Complement + Time-Machine — append-only event timeline, independent of continuity.ts"

### 2.5 Unsafe Type Assertions in Tool Layer

**File:** `src/tools/hivemind/hivemind-trajectory.ts:75,85,90,95,108`
```typescript
trajectoryId: args.trajectoryId as string,  // line 75
summary: args.summary as string,            // line 90
eventType: args.eventType as string,        // line 102
```

The `parseTrajectoryToolInput` function in `src/schema-kernel/trajectory.schema.ts:37-48` validates that `trajectoryId` is present for certain actions, but it throws a generic `Error` (not a Zod error), and the tool layer still uses `as string` assertions rather than relying on the validated type.

## 3. Feature Integration Analysis

### 3.1 Cross-Module Dependency Graph

```
src/tools/hivemind/hivemind-trajectory.ts
  → src/task-management/trajectory/index.ts (all operations)
  → src/schema-kernel/trajectory.schema.ts (validation)

src/tools/hivemind/hivemind-agent-work.ts
  → src/features/agent-work-contracts/index.ts (create/export)
  → src/schema-kernel/agent-work-contract.schema.ts (validation)

src/features/agent-work-contracts/operations.ts
  → src/task-management/trajectory/index.ts (attachTrajectoryEvidence)
  → src/features/runtime-pressure/index.ts (detectRuntimePressure)

src/features/agent-work-contracts/store.ts
  → src/schema-kernel/agent-work-contract.schema.ts (AgentWorkContractStoreSchema)
  → src/shared/security/path-scope.js (assertPathWithinRoot)

src/task-management/trajectory/ledger.ts
  → src/shared/security/path-scope.js (assertPathWithinRoot)
  → NO schema-kernel dependency (manual validation)
```

### 3.2 Boundary Violation: Feature → Task-Management

**File:** `src/features/agent-work-contracts/operations.ts:3`
```typescript
import { attachTrajectoryEvidence } from "../../task-management/trajectory/index.js"
```

Per architecture (`ARCHITECTURE.md:136-183`), `src/features/` is a standalone runtime feature sector. Importing directly from `src/task-management/` creates a tight coupling where a feature module reaches into the task-management layer. This violates the intended independence of feature modules.

**Recommendation:** Phase 25 should either:
1. Move trajectory attachment to a shared coordination layer, or
2. Accept the coupling but make it explicit via an injected dependency (factory pattern per OMO adaptation)

### 3.3 Pressure Integration

**File:** `src/features/agent-work-contracts/operations.ts:5`
```typescript
import { detectRuntimePressure } from "../runtime-pressure/index.js"
```

This is feature-to-feature coupling, which is architecturally acceptable but should be documented. The pressure gate is correctly implemented:
- `operations.ts:29-90`: Reads `pressureScore`/`pressureTier` from input, calls `detectRuntimePressure()`, blocks creation if outcome is "block" and no explicit `pressureApproved` override
- Pressure tier 8-9 (blocking band) → contract creation blocked unless `pressureApproved: true`

**Edge case found:** `operations.ts:43-47` — if `pressureScore` is undefined AND `pressureTier` is undefined, `detectRuntimePressure({})` defaults to tier 0 (steady), which always allows. This means contract creation with no pressure parameters always succeeds, which may not be the intended behavior when contracts should require explicit pressure acknowledgment.

## 4. Mechanism Compliance

### 4.1 CQRS Compliance

**Tools (write-side):** ✅ Correct
- `hivemind-trajectory.ts` and `hivemind-agent-work.ts` are proper tool entrypoints
- Both use schema validation before delegating to feature/task-management modules

**Store operations (mutation):** ⚠️ Partial
- Both modules perform direct file I/O in operations rather than through a shared mutation queue
- No event emission after mutation (no hook notification)

### 4.2 State Root Compliance

**Trajectory:** ✅ Writes to `.hivemind/state/trajectory-ledger.json`
**Work-contracts:** ✅ Writes to `.hivemind/state/agent-work-contracts.json`

Both correctly use `.hivemind/` as the canonical state root per Q6.

### 4.3 Path Safety

Both modules use `assertPathWithinRoot` from `src/shared/security/path-scope.js`:
- `src/task-management/trajectory/ledger.ts:8` — import
- `src/features/agent-work-contracts/store.ts:6` — import

Both construct paths as `join(projectRoot, ".hivemind", "state", fileName)` and validate with `assertPathWithinRoot`.

### 4.4 Error Handling

**Trajectory:**
- `ledger.ts:30-35`: Quarantines corrupt JSON by renaming to `.corrupt.{timestamp}.bak`
- Returns `{ version: 1, updatedAt: Date.now(), trajectories: {} }` as empty default on read failure

**Work-contracts:**
- `store.ts:31-35`: Similar quarantine pattern
- Returns `{ version: 1, updatedAt: Date.now(), contracts: {} }` as empty default

**Issue:** Both silently return empty stores on corruption. This means data loss is hidden — the agent gets an empty ledger/store with no error indicating that previous data existed. The quarantine file exists but nothing in the runtime surfaces this to the agent.

## 5. Overlap Analysis

### 5.1 Conceptual Overlap: Evidence References

Both systems maintain evidence references:

**Trajectory:** `TrajectoryRecord.evidenceRefs: string[]` (`types.ts:34`)
**Work-contracts:** `AgentWorkContract.trajectoryEvidenceRef?: string` (`schema:70`)

And `AgentWorkCompactionSchema.sourceRefs: string[]` (`schema:56`)

The work-contract's `trajectoryEvidenceRef` is set during creation (`operations.ts:64`) as the return value of `attachTrajectoryEvidence`. This creates a bidirectional evidence chain: contract → trajectory → evidence refs. However:
- The trajectory doesn't reference the contract back
- There's no integrity check that `trajectoryEvidenceRef` still exists in the trajectory

### 5.2 Overlap: Summary/Briefing Fields

Work-contracts have bounded compaction fields:
- `briefing` (max 1200 chars)
- `summary` (max 1200 chars)
- `reinjectionPayload` (max 2400 chars)
- `anchors` (max 20 items)

Trajectory events also have a `summary` field (`types.ts:44`). Both serve similar purposes but with different size constraints and lifecycle semantics.

### 5.3 Overlap: Status Management

**Trajectory statuses:** `TrajectoryStatus = "open" | "closed"` (`types.ts:1`)
**Work-contract statuses:** `AgentWorkStatus = "created" | "running" | "blocked" | "completed" | "cancelled"` (`schema:17`)

Work-contracts have a richer status lifecycle. Trajectory only tracks open/closed. There's no mechanism to sync trajectory closure with work-contract completion.

## 6. Phase 25 Readiness

### 6.1 What Can Be Redesigned Safely

1. **Add Zod schema for trajectory ledger file** — no breaking change, just wraps existing validation
2. **Unify quarantine error handling** — both modules already quarantine; standardize the pattern
3. **Add trajectory→contract back-reference** — extend `TrajectoryRecord` with optional `contractId`
4. **Inject trajectory dependency into work-contracts** — decouple via factory pattern

### 6.2 What Requires Caution

1. **Concurrent file access** — introducing locking requires careful design to avoid deadlocks in auto-loop scenarios
2. **Append-only semantics for trajectory** — changing from full-file-overwrite to true append-only requires file format migration
3. **Pressure gate default behavior** — changing the "no pressure params = allow" default could break existing contract creation flows

### 6.3 What Is Blocked

1. **True event sourcing** — blocked by the absence of an event journal module (Q3 references this but it's not built)
2. **Contract lifecycle state machine** — blocked by the absence of a lifecycle event system in work-contracts (status is set but never transitions)
3. **Cross-session contract tracking** — blocked by session-tracker flaws (documented in Phase 21 research)
