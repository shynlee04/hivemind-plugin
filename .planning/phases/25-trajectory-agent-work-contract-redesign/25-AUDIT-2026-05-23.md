# Phase 25 Audit: Trajectory + Agent-Work-Contract Systems

**Audit Date:** 2026-05-23  
**Files audited:** 14 source files + 2 test files + 3 governance files  
**Auditor:** gsd-codebase-mapper (subagent, TASK-A)

## Executive Summary

Phase 25 trajectory + agent-work-contract systems are **implemented and operational but have significant design gaps, incomplete test coverage, and minor CQRS/mechanism violations**. Both systems persist to `.hivemind/state/` correctly and are registered in `src/plugin.ts`. However:

- **11 design errors** identified (3 HIGH, 6 MEDIUM, 2 LOW)
- **0 unit tests** for internal modules — only 4 tool-level integration tests exist
- **1 JSDoc/implementation mismatch** — `boundText` claims "visible truncation marker" but never adds one
- **1 atomic-write inconsistency** — trajectory uses `writeFileSync` directly; agent-work-contracts uses tmp+rename
- **Cross-system validation missing** — no consistency checks between trajectory records, session-tracker, and delegation persistence

**Phase 25 readiness assessment:** PARTIAL — the systems function correctly at the tool level but lack unit-test coverage, cross-system validation, and CQRS-compliant atomic writes. The ROADMAP claim of "zero trajectory tests" is **inaccurate**: 2 tool-level tests exist in `tests/tools/hivemind-trajectory.test.ts`. However, the internal module unit tests (for `store-operations.ts`, `ledger.ts`, `operations.ts`, `store.ts`) are indeed **zero**.

## 1. Design Errors

### 1.1 [HIGH] `boundText` JSDoc/Implementation Mismatch — Missing Truncation Marker

**File:** `src/features/agent-work-contracts/operations.ts:104-113`  
**Issue:** The JSDoc at line 104 states *"Bound a text field with a visible truncation marker"* but the implementation does NOT append any truncation indicator (e.g., `…`, `[truncated]`, `...`). A truncated string is indistinguishable from a complete string of the exact same length.

```typescript
// Line 104: "visible truncation marker" promised
// Line 110-113: No marker added
function boundText(value: string, limit: number): string {
  if (value.length <= limit) return value
  return value.slice(0, Math.max(0, limit - 1)).trimEnd()  // No "..." appended
}
```

**Impact:** Consumers cannot detect data loss. A contract with a truncated `briefing` or `reinjectionPayload` returns with `status: "created"` and no warning, silently losing data.

**Fix:** Append `…` or `[truncated]` to truncated strings. Add at least 1-3 characters to the limit to accommodate the marker.

### 1.2 [HIGH] `closeTrajectory` Allows Double-Close Without Validation

**File:** `src/task-management/trajectory/store-operations.ts:99-112`  
**Issue:** `closeTrajectory` blindly sets `status = "closed"` regardless of current status. A double-close silently succeeds, overwriting the existing `closeSummary`.

```typescript
// Line 99-112: No check for already-closed
export function closeTrajectory(input: {...}): {...} {
  const trajectory = ledger.trajectories[input.trajectoryId]
  if (!trajectory) throw new Error(`[Harness] Trajectory "${input.trajectoryId}" not found`)
  trajectory.status = "closed"
  trajectory.closeSummary = input.summary ? boundText(input.summary) : trajectory.closeSummary
  // ...
}
```

**Impact:** A bug in an agent or tool could issue duplicate close calls. The second close silently discards the first `closeSummary`.

**Fix:** Add validation: if `trajectory.status === "closed"`, throw or return an idempotent success.

---

### 1.3 [HIGH] `isTrajectoryLedger` Type Guard Is Extremely Weak

**File:** `src/task-management/trajectory/ledger.ts:80-85`  
**Issue:** The type guard checks only `version` matches `TRAJECTORY_LEDGER_VERSION` and `trajectories` is a non-null object. It does NOT validate the shape or content of individual `TrajectoryRecord` entries.

```typescript
function isTrajectoryLedger(value: unknown): value is TrajectoryLedger {
  return typeof value === "object" && value !== null
    && (value as { version?: unknown }).version === TRAJECTORY_LEDGER_VERSION
    && typeof (value as { trajectories?: unknown }).trajectories === "object"
    && (value as { trajectories?: unknown }).trajectories !== null
}
```

**Impact:** A ledger file with correct `version` but corrupted or malformed trajectory records passes validation. Downstream operations (`traverse`, `close`, etc.) receive corrupted data with no error.

**Fix:** Add basic shape validation for each trajectory record in the ledger (at minimum validate `id`, `status`, `rootSessionId` exist as strings).

---

### 1.4 [MEDIUM] Trajectory Write Is NOT Atomic (Inconsistent With Agent-Work-Contracts)

**File:** `src/task-management/trajectory/ledger.ts:67-72`  
**Issue:** `writeTrajectoryLedger` uses direct `writeFileSync` without the tmp+rename atomic pattern. Compare with `src/features/agent-work-contracts/store.ts:47-52` which correctly uses tmp+rename.

```typescript
// trajectory/ledger.ts:70 — NOT atomic
writeFileSync(ledgerPath, `${JSON.stringify(ledger, null, 2)}\n`, "utf-8")

// agent-work-contracts/store.ts:49-51 — IS atomic
const tmpFile = `${filePath}.${process.pid}.${randomUUID()}.tmp`
writeFileSync(tmpFile, ...)
renameSync(tmpFile, filePath)
```

**Impact:** A crash or power loss mid-write can produce a truncated/corrupt `trajectory-ledger.json`. The quarantine path (`ledger.ts:74-78`) only fires on read/parse failure, not on write corruption.

**Fix:** Adopt the same tmp+rename pattern used in `store.ts`.

---

### 1.5 [MEDIUM] `mergeStrings` Applies `boundText` to Evidence References

**File:** `src/task-management/trajectory/store-operations.ts:169-171`  
**Issue:** `mergeStrings` calls `boundText` on each evidence reference before deduplication. Evidence references are artifact identifiers (e.g., `journal:abc-123`, `doc:phase-25-audit`) — they are NOT user text and should NOT be sanitized.

```typescript
function mergeStrings(existing: string[], incoming: string[]): string[] {
  return Array.from(new Set([...existing, ...incoming]
    .map((value) => boundText(value))  // strips control chars from ref strings!
    .filter(Boolean)))
}
```

**Impact:** Valid evidence references containing certain character sequences could be silently mangled. The `boundText` function strips all control characters (`\u0000-\u001f\u007f`), which is unnecessary for a reference string.

**Fix:** Use a simple trim+sanity check for evidence refs, or skip sanitization entirely for identifiers.

### 1.6 [MEDIUM] Silent Data Loss on Parse Failure (Both Systems)

**File:** `src/task-management/trajectory/ledger.ts:55-56`, `src/features/agent-work-contracts/store.ts:33-34`  
**Issue:** Both systems quarantine the corrupt file and return empty data. The quarantine path logs nothing to console or stderr. The caller receives an empty store with no indication of data loss.

```typescript
// ledger.ts:55-56 — silent data loss
quarantineCorruptLedger(ledgerPath)
throw new Error(`[Harness] Failed to parse trajectory ledger: ${message}`)
// Note: throws an error, so caller sees it — trajectory is better

// store.ts:33-34 — truly silent data loss
quarantineCorruptStore(filePath)
return createEmptyStore()  // No error thrown, no log
```

**Impact:** Agent-work-contracts `readAgentWorkContracts` returns an empty store on corrupt data. The caller proceeds as if no contracts have ever been created.

**Fix:** At minimum log a warning. Consider throwing like the trajectory equivalent does.

### 1.7 [MEDIUM] `boundCompaction` Silently Truncates Input

**File:** `src/features/agent-work-contracts/operations.ts:93-101`  
**Issue:** `boundCompaction` truncates `anchors` to 20 and `sourceRefs` to 20 items, and truncates text fields to their respective limits. No warning is returned to the caller.

```typescript
function boundCompaction(compaction: AgentWorkCompaction): AgentWorkCompaction {
  return {
    anchors: compaction.anchors.slice(0, ANCHOR_LIMIT).map(...),  // Silent truncation
    sourceRefs: compaction.sourceRefs.slice(0, ANCHOR_LIMIT).map(...),  // Silent truncation
    // ...
  }
}
```

**Impact:** A tool caller passes 50 anchors but silently receives 20. The response is `{ status: "created" }` with no indication of truncation. This is a data integrity concern — the caller doesn't know their data was trimmed.

**Fix:** Include a `warnings` field in the `AgentWorkCreateSuccess` type listing truncation details, or reject inputs exceeding limits with a clear error.

### 1.8 [MEDIUM] `upsertTrajectory` Allows Silent Field Overwrite on Existing Records

**File:** `src/task-management/trajectory/store-operations.ts:136-163`  
**Issue:** `upsertTrajectory` overwrites `rootSessionId`, `sessionId`, and `parentTrajectoryId` on existing records if the new input provides them. This means an attach/event/checkpoint call to an existing trajectory can silently change its identity.

```typescript
function upsertTrajectory(ledger, input, now): TrajectoryRecord {
  const existing = ledger.trajectories[input.trajectoryId]
  if (existing) {
    existing.rootSessionId = input.rootSessionId ?? existing.rootSessionId
    existing.sessionId = input.sessionId ?? existing.sessionId
    existing.parentTrajectoryId = input.parentTrajectoryId ?? existing.parentTrajectoryId
    return existing
  }
  // ... create new
}
```

**Impact:** If two callers use different `rootSessionId` values for the same `trajectoryId`, the last writer wins — corrupting lineage.

**Fix:** On existing trajectory, validate that identity fields match (throw or skip). Identity should only be set at creation time.

### 1.9 [MEDIUM] `createAgentWorkContract` Crosses Feature→Task-Management Boundary

**File:** `src/features/agent-work-contracts/operations.ts:3`  
**Issue:** The feature module imports `attachTrajectoryEvidence` from `../../task-management/trajectory/index.js`. While architecturally permitted, this couples a feature module directly to a task-management submodule. A future refactor of trajectory would break agent-work-contracts.

```typescript
import { attachTrajectoryEvidence } from "../../task-management/trajectory/index.js"
```

**Impact:** Tight coupling between feature and task-management layers. If trajectory changes its API or moves, agent-work-contracts must update.

**Fix:** Consider passing trajectory dependencies via dependency injection at composition time (in `src/plugin.ts`), or creating a thin interface layer.

---

### 1.10 [LOW] `trajectoryEvidenceRef` Format May Be Unresolvable

**File:** `src/features/agent-work-contracts/operations.ts:42`  
**Issue:** The evidence reference format `agent-work-contract:${id}` is a custom scheme. There is no resolver or index that can dereference this back to the actual contract.

```typescript
const trajectoryEvidenceRef = input.trajectoryId ? `agent-work-contract:${id}` : undefined
```

**Impact:** The reference is stored in trajectory evidence but cannot be resolved to the actual contract document. It's a human-readable label, not a machine-resolvable pointer.

**Fix:** The format itself is acceptable as an identifier. Document the resolution path in AGENTS.md.

---

### 1.11 [LOW] `AgentWorkContractStoreSchema` Uses `z.literal(1)` — Version Is Hard-Coded

**File:** `src/schema-kernel/agent-work-contract.schema.ts:79`  
**Issue:** The store schema uses `z.literal(1)` for version instead of referencing a constant. The types file has `STORE_VERSION = 1` in `store.ts:8`. If the version is incremented, the schema must be manually updated.

```typescript
// schema.ts:79
export const AgentWorkContractStoreSchema = z.object({
  version: z.literal(1),  // Hard-coded
  // ...
})
```

**Impact:** A version increment risks schema/source mismatch if only `store.ts` is updated.

**Fix:** Export the version from a shared location or add a comment cross-reference.

## 4. Overlap Analysis

### 4.1 Trajectory vs Session-Tracker

| Aspect | Trajectory (`task-management/trajectory/`) | Session-Tracker (`features/session-tracker/`) |
|--------|---------------------------------------------|-----------------------------------------------|
| Purpose | Execution lineage (parent/child trees, events, checkpoints) | Session lifecycle, messages, tool calls, hierarchy |
| State file | `.hivemind/state/trajectory-ledger.json` | `.hivemind/session-tracker/` (multi-file) |
| Hierarchy | Parent/child by `parentTrajectoryId` | HierarchyIndex with root tracking |
| Events | `TrajectoryEvent` appended to trajectory | Full message log with tool calls |
| Idempotent? | Events deduplicate by `eventId` | Uses `PendingDispatchRegistry` three-gate |

**Overlap:** Both track session hierarchies. A session tracked in session-tracker would also have a trajectory entry. There's no cross-reference validation.

### 4.2 Agent-Work-Contract vs Delegation Persistence

| Aspect | Agent-Work-Contract | Delegation Persistence (`task-management/continuity/delegation-persistence.ts`) |
|--------|--------------------|------------------------------------------------|
| Purpose | Durable work contracts for agent boundaries | Persist delegation records |
| State file | `.hivemind/state/agent-work-contracts.json` | `.hivemind/state/` (via ContinuityStoreFile) |
| Consumer | `hivemind-agent-work` tool | DelegationManager, recovery |
| Owner ref | `owner.agent`, `owner.sessionId` | `DelegationMeta.rootID`, `DelegationMeta.agent` |

**Overlap:** Both reference agents and sessions. Contracts exist independently of delegation records, which may lead to orphan contracts referencing deleted delegations.

### 4.3 Trajectory vs Journal

Trajectory stores `EvidenceRef` strings pointing to journal records but never validates or resolves them. The journal (`task-management/journal/`) exists as an append-only event timeline. Cross-referencing is by convention only — no integrity enforcement.

## 6. Recommendations

### Immediate (Phase 25 Scope)

1. **Fix `boundText` truncation marker** (`operations.ts:110-113`) — Add `…` to truncated strings. This is a data-integrity bug.

2. **Add `closeTrajectory` double-close guard** (`store-operations.ts:105`) — Throw on already-closed trajectories, or make idempotent with idempotency check.

3. **Adopt atomic write in trajectory ledger** (`ledger.ts:67-72`) — Use tmp+rename pattern matching `store.ts:49-51`.

4. **Fix `isTrajectoryLedger` type guard** (`ledger.ts:80-85`) — Validate individual record shapes.

5. **Add unit tests for internal modules** — Create `tests/lib/task-management/trajectory/` and `tests/features/agent-work-contracts/` directories with tests for `store-operations.ts`, `ledger.ts`, `operations.ts`, `store.ts`.

### Short-Term

6. **Replace `mergeStrings` `boundText` call with simple trim** (`store-operations.ts:169-171`) — Evidence refs should not be sanitized like user text.

7. **Add cross-system validation** — Trajectory `rootSessionId` should validate against session-tracker on creation. Agent-work-contract `trajectoryId` should verify the trajectory exists before linking.

8. **Implement agent-work-contract lifecycle transitions** — Currently only `created` is used. The other 4 statuses (`running`, `blocked`, `completed`, `cancelled`) in `AgentWorkStatusSchema` are dead state.

9. **Add truncation warnings to `createAgentWorkContract` response** — Return `warnings` array when compaction fields are truncated.

10. **Fix trajectory identity field overwrite** (`store-operations.ts:139-141`) — Only set identity fields on creation; validate on update.
