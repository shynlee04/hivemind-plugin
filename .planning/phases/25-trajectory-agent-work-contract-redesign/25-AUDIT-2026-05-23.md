# Phase 25 Audit: Trajectory + Agent-Work-Contract Redesign

**Date:** 2026-05-23
**Scope:** `src/task-management/trajectory/`, `src/features/agent-work-contracts/`, `src/schema-kernel/trajectory.schema.ts`, `src/schema-kernel/agent-work-contract.schema.ts`, `src/tools/hivemind/hivemind-trajectory.ts`, `src/tools/hivemind/hivemind-agent-work.ts`, `src/features/runtime-pressure/`, and associated tests

## Executive Summary

The trajectory ledger and agent-work-contract systems are **functional but structurally inconsistent**, built in different phases with different design standards. The two modules now need reconciliation before Phase 25 redesign.

**Critical findings:**
1. **Validation divergence** — trajectory uses manual `JSON.parse` + `isTrajectoryLedger` field checks; work-contracts uses full Zod schema validation via `AgentWorkContractStoreSchema.safeParse()`
2. **Read-modify-write concurrency hazard** — both modules read full file, mutate in-memory, and write entire file back with no locking
3. **Cross-boundary dependency** — `agent-work-contracts/operations.ts` imports `attachTrajectoryEvidence` from `task-management/trajectory/`, violating feature module independence per ARCHITECTURE.md
4. **No Zod schema for trajectory ledger file** — tool input has schema, but persisted `TrajectoryLedger` has no Zod schema
5. **Minimal test coverage** — 5 total test cases across both systems (2 tool + 3 unit for trajectory; 2 tool + 0 unit for work-contracts)

## 1. Design Errors

### 1.1 Validation Approach Divergence

**Trajectory** (`src/task-management/trajectory/ledger.ts:41-58`, `:80-85`):
- Uses `JSON.parse` + manual `isTrajectoryLedger()` field checks at line 80
- `isTrajectoryLedger` checks `version` (line 82) and `trajectories` object type (lines 83-84) — no deep validation
- No Zod schema for the `TrajectoryLedger` file format
- Corrupt JSON is quarantined (`quarantineCorruptLedger` at line 74) but produces only generic `"invalid schema"` error

**Agent-work-contracts** (`src/features/agent-work-contracts/store.ts:27-38`):
- Uses `AgentWorkContractStoreSchema.safeParse()` (line 31) — full Zod validation
- Produces structured `ZodError` with field-level detail
- Schema defined in `src/schema-kernel/agent-work-contract.schema.ts:78-82`

**Impact:** Trajectory ledger corruption produces unhelpful errors. Work-contracts produce actionable Zod diagnostics.

### 1.2 No Trajectory Ledger File Schema

The trajectory tool input has a Zod schema (`src/schema-kernel/trajectory.schema.ts:11-23`), but the persisted `TrajectoryLedger` type (`src/task-management/trajectory/types.ts:81-88`) has no corresponding Zod schema. This means:
- File-format changes are not validated at the schema boundary
- Version migration requires manual code changes rather than schema evolution
- `TRAJECTORY_LEDGER_VERSION` (types.ts:4) is a bare constant — no version migration logic exists

### 1.3 Read-Modify-Write Without Locking

Both modules follow the same unsafe pattern:

**Trajectory** (`src/task-management/trajectory/store-operations.ts`):
- Every operation does: `readTrajectoryLedger()` → mutate in-memory → `writeTrajectoryLedger()`
- Files: `attachTrajectoryEvidence` (line 31), `checkpointTrajectory` (line 49), `eventTrajectory` (line 73), `closeTrajectory` (line 99), `traverseTrajectory` (line 120)
- `writeTrajectoryLedger` (ledger.ts:67-72) writes full JSON, no tmp+rename atomic write pattern

**Work-contracts** (`src/features/agent-work-contracts/store.ts`):
- `upsertAgentWorkContract` (line 61) does: `readAgentWorkContracts()` → mutate → `persistAgentWorkContracts()`
- **However** work-contracts use tmp+rename atomic write pattern at `store.ts:49-51` — better than trajectory

**Impact:** Concurrent tool invocations could corrupt JSON files. Trajectory is worse since it lacks atomic write.

### 1.4 Trajectory Is Not Append-Only

The trajectory concept is described as tracking execution lineage with events/checkpoints, but:
- `eventTrajectory` (store-operations.ts:73) filters-and-replaces events array rather than appending-only
- `checkpointTrajectory` (store-operations.ts:49) same filter-and-replace pattern
- Full JSON file is overwritten per mutation (`writeTrajectoryLedger` at ledger.ts:67-72)
- No append-only log file (like session journal's `.jsonl` format)

This means large trajectory ledgers suffer growing write amplification.

### 1.5 Unsafe Type Assertions in Tool Layer

**File:** `src/tools/hivemind/hivemind-trajectory.ts:75,85,90,95,108`
```typescript
trajectoryId: args.trajectoryId as string,  // line 75
summary: args.summary as string,            // line 90
eventType: args.eventType as string,        // line 102
```

The `parseTrajectoryToolInput` function (trajectory.schema.ts:37-48) throws generic `Error` for action-specific validation, and the tool layer forces `as string` assertions rather than using validated discriminated unions.

### 1.6 Trajectory Status Enum Incorrect

**File:** `src/task-management/trajectory/types.ts:9`
```typescript
export type TrajectoryStatus = "active" | "closed"
```

This is a basic two-state lifecycle. Work-contracts have five states (`"created" | "running" | "blocked" | "completed" | "cancelled"` at `agent-work-contract.schema.ts:17`). There is no mapping or synchronization between the two status systems.

### 2.3 Pressure Integration Analysis

**File:** `src/features/agent-work-contracts/operations.ts:4,27-38`
```typescript
import { detectRuntimePressure } from "../runtime-pressure/index.js"
```

Feature-to-feature coupling is architecturally acceptable. The pressure gate:
- `operations.ts:27-31`: Classifies pressure from `pressureScore`/`pressureTier`
- `operations.ts:33-38`: Blocks if outcome is "block" or "require_approval" without explicit `pressureApproved`

**Edge case:** When both `pressureScore` and `pressureTier` are undefined, `detectRuntimePressure({})` defaults to tier 0 ("steady" band → "allow" outcome). This means contract creation with zero pressure parameters always succeeds, which may not be the intended behavior for contracts that should require explicit pressure acknowledgment.

### 3.6 Atomic Write Pattern Inconsistency

Trajectory's `writeTrajectoryLedger` (`ledger.ts:67-72`) uses direct `writeFileSync` without tmp+rename:
```typescript
mkdirSync(resolve(ledgerPath, ".."), { recursive: true })
writeFileSync(ledgerPath, `${JSON.stringify(ledger, null, 2)}\n`, "utf-8")
```

Work-contracts' `persistAgentWorkContracts` (`store.ts:46-52`) uses proper atomic write:
```typescript
const tmpFile = `${filePath}.${process.pid}.${randomUUID()}.tmp`
writeFileSync(tmpFile, `${JSON.stringify({ ...store, updatedAt: Date.now() }, null, 2)}\n`, "utf-8")
renameSync(tmpFile, filePath)
```

**Risk:** Trajectory ledger can be corrupted by a crash during write.

## 4. Overlap Analysis

### 4.1 Conceptual Overlap: Evidence References

Both systems maintain evidence references:
- **Trajectory:** `TrajectoryRecord.evidenceRefs: EvidenceRef[]` (types.ts:65)
- **Work-contracts:** `AgentWorkContract.trajectoryEvidenceRef?: string` (agent-work-contract.schema.ts:70)
- **Work-contracts (compaction):** `AgentWorkCompaction.sourceRefs: string[]` (schema:56)

The work-contract's `trajectoryEvidenceRef` is set during creation (operations.ts:42) and attached to trajectory via `attachTrajectoryEvidence` at line 57-63. However:
- The trajectory record has **no back-reference** to the contract — no `contractId` field on `TrajectoryRecord`
- No integrity check that `trajectoryEvidenceRef` still exists in the trajectory ledger

### 4.2 Overlap: Summary/Text Fields

Work-contracts have bounded compaction fields:
- `briefing` (max 1200 chars) — operations.ts:15
- `summary` (max 1200 chars) — operations.ts:16
- `reinjectionPayload` (max 2400 chars) — operations.ts:17
- `anchors` (max 20 items) — operations.ts:18

Trajectory events have their own `summary: string` field (types.ts:43) with 2000-char bound via `boundText` (store-operations.ts:173-174). Both serve similar purposes with different size constraints.

### 4.3 Overlap: Status Management

| Dimension | Trajectory | Work-Contracts |
|-----------|------------|----------------|
| Status type | `"active" \| "closed"` (types.ts:9) | `"created" \| "running" \| "blocked" \| "completed" \| "cancelled"` (agent-work-contract.schema.ts:17) |
| Status transitions | active → closed | created → (running/blocked) → (completed/cancelled) |
| Synchronization | None | None |

Work-contracts have a richer lifecycle. Trajectory only tracks active/closed. There is no mechanism to sync trajectory closure with contract completion.

### 4.4 Overlap with Session-Tracker

Trajectory's `TrajectoryRecord` (types.ts:53-76) has:
- `id`, `rootSessionId`, `sessionId`, `parentTrajectoryId`, `status`, `evidenceRefs`, `checkpoints`, `events`, timestamps, `closeSummary`

Session-tracker has its own hierarchy-manifest system with:
- Session hierarchy (parent/child), status tracking, turn counting
- Delegation records, fork handling, recovery paths

**Overlap finding:** Both trajectory and session-tracker maintain session lineage trees. Trajectory adds evidence references and checkpoints on top of the lineage, but the lineage information itself (`rootSessionId`, `parentTrajectoryId`) duplicates what session-tracker's `HierarchyIndex` already tracks.

### 4.5 Overlap with Delegation Persistence

Agent-work-contracts store contract metadata (owner, scope, evidence requirements, compaction payloads). Delegation persistence (`src/task-management/continuity/delegation-persistence.ts`) stores delegation records (dispatch ID, target agent, status, timestamps, results).

Both are persistence stores under `.hivemind/state/` but serve different purposes:
- Delegation persistence: operational tracking of in-flight/completed delegations
- Work-contracts: durable contracts for agent work boundaries and compaction

**No material overlap,** but there should be a documented conceptual boundary.

## 5. Phase 25 Readiness

### 5.1 What Can Be Redesigned Safely

1. **Add Zod schema for trajectory ledger file** — non-breaking, wraps existing `isTrajectoryLedger` with `safeParse`
2. **Fix trajectory write to use tmp+rename atomic pattern** — non-breaking, matches work-contracts pattern
3. **Unify quarantine error behavior** — both already quarantine; standardize with error surface instead of silent empty return
4. **Add trajectory→contract back-reference** — add optional `contractId` to `TrajectoryRecord`; non-breaking

### 5.2 What Requires Caution

1. **Concurrent file access** — introducing locking requires careful design to avoid deadlocks in auto-loop/parallel delegation
2. **Feature→task-management dependency decoupling** — requires factory injection pattern, which needs composition root changes in `plugin.ts`
3. **Pressure gate default behavior** — changing "no pressure params = allow" default could break existing tool callers that pass no pressure fields
4. **Trajectory → session-tracker deduplication** — lineage overlap needs architecture decision (Q1-Q6 locked decisions reference this)

### 5.3 What Is Blocked

1. **True event sourcing for trajectory** — blocked by the absence of an append-only journal format (would need `.jsonl` migration)
2. **Contract lifecycle state machine** — blocked by absence of lifecycle event system for work-contract status transitions (`created → running → completed/cancelled`)
3. **Cross-session contract tracking** — blocked by session-tracker flaws (documented in Phase 21 findings)
4. **Hook notification on mutation** — blocked by hook injection plane redesign (Phase 27)
