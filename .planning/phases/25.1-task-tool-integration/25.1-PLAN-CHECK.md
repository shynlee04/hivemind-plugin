## Risk Assessment

### Risk 1: Unused static type imports (BLOCKER)

**Severity:** BLOCKER — will fail `npm run typecheck`

Task 3's action section specifies adding static type imports:
```typescript
import type { TrajectoryMutationInput } from "../../task-management/trajectory/types.js"
import type { CreateAgentWorkContractInput } from "../../features/agent-work-contracts/types.js"
```

However, these types are NOT used anywhere in the file:
- The helper method signature uses primitives (`childSessionID: string`, etc.), not `TrajectoryMutationInput`
- The API calls inside the helper use dynamic `await import()` which doesn't reference these types
- `tsconfig.json` has `"noUnusedLocals": true` — unused imports cause typecheck errors

**Fix:** Remove the static type imports from Task 3's action. The helper method uses primitives, and the API calls use dynamic imports. No static type imports needed.

### Risk 2: `files_modified` frontmatter incomplete (WARNING)

**Severity:** WARNING — won't block execution but is inaccurate

The `files_modified` list in the plan frontmatter is missing:
- `tests/features/session-tracker/tool-delegation-integration.test.ts` (created by Task 5)
- `src/features/session-tracker/tool-delegation-utils.ts` (created by Task 1)

These files ARE mentioned in the task sections but not in the frontmatter manifest.

**Fix:** Add both files to `files_modified` in the frontmatter.

### Risk 3: `createAgentWorkContract()` return value ignored (INFO)

**Severity:** INFO — no functional impact

The function returns `AgentWorkCreateResult` (either `"created"` or `"pressure-blocked"`). The plan doesn't check this return value. Since the function is called inside a try/catch and the return type doesn't throw, this is acceptable. However, if the pressure gate blocks the contract (unlikely — verified as non-blocking when pressureScore/Tier are not provided), the plan won't detect it.

**Mitigation:** Already mitigated — research verified that `detectRuntimePressure({ score: undefined, tier: undefined })` returns `{ outcome: "allow" }`. The pressure gate will not block auto-created contracts.

### Risk 4: Dynamic imports add runtime overhead (INFO)

**Severity:** INFO — negligible impact

The plan uses `await import()` for trajectory and contract modules inside the helper method. Dynamic imports have ~1-5ms overhead on first call (module resolution + caching). Subsequent calls use the cached module. This is negligible compared to the file I/O operations these modules perform.

**Mitigation:** None needed — dynamic imports are the correct pattern here to avoid circular dependency concerns.
