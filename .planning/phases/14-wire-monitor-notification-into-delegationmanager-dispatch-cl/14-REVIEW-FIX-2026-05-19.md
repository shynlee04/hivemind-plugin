## Iteration 2: Live system_reminder Spam Remediation

**Fixed at:** 2026-05-19
**Source:** live runtime spam observed in `/Users/apple/hivemind-plugin-private/session-ses_1bfe.md`
**Commit:** `b5ffc7fe`
**Status:** `fixed: requires live human verification`

### Root Cause

Phase 14 wired monitor progress injection into `NotificationRouter.route({ type: "progress" })`, and the plugin delivery callback appended every routed notification through `appendTuiPrompt`. That made progress cadence/checkpoint signals become parent TUI `<system_reminder>` messages instead of remaining internal monitoring signals.

`DelegationCoordinator.markExecutionUnconfirmed()` also routed the 60s first-action deadline as a `progress` notification, creating another non-terminal path into the same parent-facing delivery surface.

### Applied Fix

**Files modified:** `src/plugin.ts`, `src/coordination/delegation/coordinator.ts`, `tests/integration/delegation-v2-integration.test.ts`, `tests/lib/coordination/delegation/coordinator.test.ts`

- Added a terminal-only gate in `setupDelegationModules()` so parent TUI append/toast delivery only runs for `success`, `failure`, and `timeout` notification types.
- Kept routed `progress` notifications acknowledged without appending to parent TUI, preventing pending-notification replay spam.
- Removed non-terminal progress routing from `markExecutionUnconfirmed()`; the record still moves to `executionState: "unconfirmed"`, but no parent-facing notification is emitted.
- Added a regression test proving a routed progress notification does not call `client.tui.appendPrompt`, while terminal completion still appends once.
- Updated coordinator unit coverage to assert unconfirmed execution state changes do not route parent-facing notifications.
