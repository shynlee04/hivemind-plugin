## Summary

The repository's core problem currently is discoverability, not lack of functionality. Critical harness and runtime behavior is spread across `src/cli/`, `src/control-plane/`, `src/shared/`, hooks, tools, and commands. This creates integration risk and makes the TUI extraction harder. The research confirms that adopting a feature-driven architecture (`src/features/*`) will centralize domain rules, allowing the OpenCode primitives (`tools/`, `hooks/`, `commands/`) to act as thin, easily testable adapters.

**Primary recommendation:** Relocate core logic into feature modules first, then rewrite adapters to strictly validate inputs via Zod and pass data to the new feature boundaries. Maintain `src/shared/contracts/` strictly for boundary typing.

## Architecture Patterns

### Recommended Project Structure
```text
src/
├── features/
│   ├── runtime-entry/       # Init, harness, doctor flows
│   ├── session-entry/       # Hook-based session initialization
│   ├── workflow/            # Workflow state management
│   ├── trajectory/          # Task progression and state tracking
│   ├── handoff/             # Agent handoff orchestration
│   ├── doc-intelligence/    # Document ingestion and markdown parsing
│   └── runtime-observability/ # Runtime status and state snapshots
├── shared/
│   └── contracts/           # EXACT shared types for backend and TUI
├── tools/                   # Thin adapters: Zod schema + feature delegation
├── hooks/                   # Thin adapters: Event interception + feature delegation
├── commands/                # Thin adapters: CLI routing + feature delegation
└── plugin/                  # Assembly and wiring only
```

### Pattern 1: Thin Tool Adapter (CQRS)
**What:** Tools only define Zod schemas, execute validation, and pass payloads to feature functions.
**When to use:** Refactoring any `src/tools/` implementation.
**Example:**
```typescript
import { tool } from '@opencode-ai/plugin';
import { executeWorkflowAction } from '../../features/workflow/action.js';

const s = tool.schema;

tool({
  description: 'Manage workflow tasks',
  args: {
    action: s.enum(['create', 'list']).describe('Action'),
  },
  async execute(args, context) {
    // 1. Validation/Adapter Logic Only
    const payload = { action: args.action, session: context.sessionID };
    // 2. Delegate to Feature
    const result = await executeWorkflowAction(payload);
    // 3. Return primitive response
    return JSON.stringify(result);
  }
});
```

### Anti-Patterns to Avoid
- **Leaking Business Logic into Tools/Hooks:** Defining state transitions or complex conditionals inside `execute()` or hook callbacks. Keep them strictly in `src/features/`.
- **Bypassing Shared Contracts:** Defining identical interfaces in the TUI and Backend separately. Use `src/shared/contracts/` for all boundary shapes.
- **God Features:** Putting unrelated logic into a single feature folder. Adhere to the designated folders (`runtime-entry`, `workflow`, etc.).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Tool Arguments Validation | Custom validation logic | `tool.schema` (Zod) | Already natively supported and expected by the OpenCode SDK. |
| Cross-Boundary Types | Duplicate `type` files in apps/opentui | `src/shared/contracts/*` | Guarantees type synchronization between the backend and Bun TUI. |

**Key insight:** The adapter layer must not invent state machines; it should only translate external SDK requests into internal feature calls and format the outputs.

## Common Pitfalls

### Pitfall 1: Broken Shared Contracts
**What goes wrong:** TUI and Backend go out of sync during the migration.
**Why it happens:** Moving files breaks relative imports or creates distinct interfaces.
**How to avoid:** Move the contracts first, verify `npx tsc --noEmit` and `npm run typecheck`, then migrate logic.

### Pitfall 2: Fat Adapters
**What goes wrong:** `src/tools/` or `src/hooks/` files retain business logic after refactoring.
**Why it happens:** Developers only move helper functions but leave core orchestrations in the `execute()` block.
**How to avoid:** Ensure the `execute()` block parses arguments, delegates to the feature, and returns the serialized result.
