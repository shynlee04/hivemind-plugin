**Researched:** 2026-03-18

## Summary

The repository's core problem currently is discoverability, not lack of functionality. Critical harness and runtime behavior is spread across `src/cli/`, `src/control-plane/`, `src/shared/`, hooks, tools, and commands. This creates integration risk and makes the TUI extraction harder.

**Primary recommendation:** Migrate to a hybrid folder structure where `src/features/*` owns the business logic and runtime state, while `tools/`, `hooks/`, `commands/`, and `plugin/` act exclusively as thin OpenCode adapter surfaces.

## User Constraints (from Audit Directives)

### Locked Decisions
- Must reorganize codebase to use `src/features/*` for ownership.
- `tools/`, `hooks/`, `commands/`, and `plugin/` must be thin adapter surfaces only.
- Consolidate runtime-entry, session-entry, workflow, trajectory, and handoff into `features/`.

## Architecture Patterns

### Recommended Project Structure
```
src/
  features/
    runtime-entry/
    session-entry/
    workflow/
    trajectory/
    handoff/
    doc-intelligence/
    runtime-observability/
  tools/
  hooks/
  commands/
  plugin/
  shared/contracts/
apps/
  opentui/
```

### Pattern 1: Feature Isolation
**What:** Each domain (workflow, handoff, runtime-entry) gets its own folder in `src/features/`.
**When to use:** When consolidating business logic that is currently scattered across CLI, shared, and tools.
**Why:** Improves discoverability and makes unit testing easier without needing the full OpenCode SDK context.

### Pattern 2: Thin Adapters
**What:** Tools and Hooks only validate input/output and immediately delegate to a Feature module.
**When to use:** For all OpenCode integration points (`tools/`, `hooks/`, `commands/`).

## Common Pitfalls

### Pitfall 1: Leaking Business Logic into Adapters
**What goes wrong:** A tool definition includes the actual implementation logic instead of delegating.
**Why it happens:** Convenience during initial development.
**How to avoid:** Enforce a rule that tools only handle Zod parsing and SDK `context` mapping, then call `FeatureX.execute()`.
