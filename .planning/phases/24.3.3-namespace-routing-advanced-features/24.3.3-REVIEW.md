## Summary

Reviewed 11 source files and 7 test files from Phase 24.3.3 (Namespace Routing + Advanced Features). The module extraction successfully reduced `execute-slash-command.ts` from 527 LOC to 402 LOC (-23%). The extracted modules are well-structured with clear separation of concerns. However, three BLOCKER issues were identified: a fire-and-forget dispatch pattern that returns success before execution, an unimplemented two-stage routing pipeline (SPEC compliance gap), and pervasive `any` types violating the project's strict TypeScript rules. Additionally, the semantic agent selector module exists as dead code in the execution flow — it was created but never wired into the tool.

## Critical Issues

### CR-01: Fire-and-forget dispatch returns `{ success: true }` before prompt executes (BLOCKER)

**File:** `src/tools/session/dispatch-command.ts:90-95`

**Issue:** The `dispatchCommand` function schedules `client.session.prompt(request)` in a `setTimeout(fn, 50)` and immediately returns `{ success: true }`. The caller receives a success signal before the actual dispatch has started. If the prompt call fails — due to an invalid session, network error, or SDK rejection — the error is silently swallowed by a `.catch()` that only writes to `console.error`. The caller has no mechanism to detect or react to the failure.

This creates a correctness gap: every call to `dispatchCommand` (used by synthetic parent prompt, subtask, and TUI-bypass paths in `execute-slash-command.ts`) reports success even when the underlying prompt never fires or fails.

**Fix:** Restructure to either:
  1. Await the prompt call directly (remove `setTimeout`), or
  2. Return a structured pending/deferred status (e.g., `{ success: true, deferred: true, dispatchId: "..." }`) that the caller can check, or
  3. Pass a callback/event emitter so errors are surfaced to the tool caller.

```typescript
// Option 1: Remove deferral for non-subtask paths
export async function dispatchCommand(context: {
  client: any
  sessionID: string
  agent?: string
  promptText: string
  subtask?: boolean
  description?: string
  commandSource?: string
  directory?: string
}): Promise<{ success: boolean; output?: string; error?: boolean }> {
  // ... validation ...

  const request = buildPromptRequest(context)
  try {
    await client.session.prompt(request)
    return { success: true }
  } catch (err) {
    const message = err instanceof Error ? err.message : String(err)
    console.error(`[Harness] Slash command prompt dispatch failed: ${message}`)
    return { success: false, error: true, output: message }
  }
}
```

### CR-02: Two-stage agent selection pipeline not wired into execution flow (BLOCKER)

**Files:**
- `src/tools/session/execute-slash-command.ts:118-180` — agent resolution not called
- `src/tools/session/semantic-agent-selector.ts` — entire module unused by tool

**Issue:** SPEC R5 and R7 require a two-stage routing pipeline: "Command resolved first, then agent selected independently" and "Use same logic for agent selection as command resolution." The `selectAgent` function was created in `semantic-agent-selector.ts` (115 LOC) as a full keyword-matching agent resolver, but it is **never called** from `execute-slash-command.ts`. The tool only validates agent format (`validateAgentFormat`) and existence (`validateAgentExists`) via `dispatch-command.ts` — it never performs semantic agent selection.

This means:
- The `selectAgent` module is dead code in the execution flow
- SPEC acceptance criteria "Two-stage sequential routing implemented: resolveCommand() → resolveAgent()" is not met
- The tool falls back to frontmatter-specified agents only, with no intent-based agent discovery

**Fix:** Wire `selectAgent()` into the execute flow, or formally defer agent selection and remove the dead module:

```typescript
// Option: Integrate after command resolution
const resolveResult = await resolveCommand({ ... })
if (!resolveResult.success) { /* ... */ }

// Stage 2: Resolve agent from intent
if (!validated.agent && !commandBundle?.agent) {
  const agentNames = await getAppAgents(client)
  const agentResult = await selectAgent(validated.command, agentNames)
  if (agentResult.agent) {
    validated.agent = agentResult.agent
  }
}
```

### CR-03: Pervasive `any` types on all new/modified modules (BLOCKER)

**Files:**
- `src/tools/session/dispatch-command.ts:16` — `client: any`
- `src/tools/session/dispatch-command.ts:22` — `a: any` in map callback
- `src/tools/session/validate-command.ts:8` — `args: any`
- `src/tools/session/semantic-agent-selector.ts:22` — `agents: any[]`
- `src/tools/session/execute-slash-command.ts:80` — `ExecuteSlashCommandSchema.shape as any`
- `src/tools/session/execute-slash-command.ts:81,103,113,135,162,214,289,393` — `as any` casts on return types

**Issue:** The project mandates "No `any` types on new code" (strict mode with `noImplicitAny`). Every new or refactored module in this phase introduces `any` types. The `client` parameter in `dispatchCommand` is typed `any`, the `agents` parameter in `selectAgent` is `any[]`, and `execute-slash-command.ts` uses pervasive `as any` casts on every return statement. This defeats TypeScript's type-checking benefits and makes future refactoring hazardous.

While some `as any` casts at API boundaries with the untyped OpenCode SDK client are unavoidable, the internal function signatures should use proper types.

**Fix:** Define and use proper interface types:

```typescript
// In dispatch-command.ts
import type { OpenCodeClient } from "../../shared/session-api.js"

export async function dispatchCommand(context: {
  client: OpenCodeClient
  sessionID: string
  // ...
}): Promise<DispatchResult> { /* ... */ }

// In semantic-agent-selector.ts
export async function selectAgent(
  intent: string,
  agents: Array<{ name: string; id?: string; description?: string }> = []
): Promise<AgentMatchResult> { /* ... */ }

// In validate-command.ts
import type { ExecuteSlashCommandInput } from "../../schema-kernel/commands.schema.js"

export function validateCommandContract(args: Partial<ExecuteSlashCommandInput>): ValidationResult { /* ... */ }
```

### WR-02: Duplicated keyword-scoring logic across two modules

**Files:**
- `src/tools/session/resolve-command.ts:141-157` — Stage 4 keyword scoring
- `src/tools/session/semantic-agent-selector.ts:97-114` — `calculateSimilarityScore`

**Issue:** The keyword-scoring algorithm (weight 1.0 for name matches, 0.5 for description matches, +1.0 for exact match) is independently implemented in two files with near-identical logic. This violates DRY and means any scoring change requires updating both copies. SPEC R7 explicitly states "Use same logic for agent selection as command resolution" but the implementation duplicates rather than shares.

**Fix:** Extract a shared scoring utility:

```typescript
// src/shared/keyword-scoring.ts
export function calculateKeywordScore(
  keywords: string[],
  name: string,
  description?: string
): number {
  let score = 0
  const nameLower = name.toLowerCase()
  keywords.forEach((keyword) => {
    if (nameLower.includes(keyword)) {
      score += 1.0
      if (nameLower === keyword) score += 1.0
    }
  })
  if (description) {
    const descLower = description.toLowerCase()
    keywords.forEach((keyword) => {
      if (descLower.includes(keyword)) score += 0.5
    })
  }
  return score
}
```

### WR-03: Cache miss counter inconsistency on failed discovery

**File:** `src/tools/session/resolve-command.ts:87-96`

**Issue:** If `discoverCommandBundles()` throws on line 87, the `misses` counter is pre-incremented in the local `cached` variable (line 94), but the cache entry is never saved (line 96 is skipped due to the thrown exception). The increment is lost. On the next call within TTL, the stale cache is used (a hit). The miss count reported by `getCacheStats()` will be lower than the actual number of failed discoveries.

**Fix:** Move miss counter increment after successful discovery:

```typescript
const discovery = await discoverCommandBundles({ projectRoot })
const prevHits = cached?.hits ?? 0
const prevMisses = cached?.misses ?? 0
cached = {
  commands: discovery.commands,
  timestamp: now,
  hits: prevHits,
  misses: prevMisses + 1, // Only incremented on success
}
discoveryCache.set(projectRoot, cached)
```

(Alternatively, wrap line 87 in a try/catch and handle null discovery explicitly.)

### WR-10: `extractEntities` uses a shared mutable stop-words Set on every call (minor)

**File:** `src/tools/session/resolve-command.ts:52`

**Issue:** The `stopWords` Set is created as a new instance on every call to `extractEntities`. While this is functionally correct, it allocates 13 string objects + a Set on every invocation. For a frequently-called function in hot paths, this creates unnecessary GC pressure.

**Fix:** Hoist to module level:

```typescript
const STOP_WORDS = new Set([
  "command", "the", "a", "an", "for", "to", "in", "on", "of", "with", "by", "at", "from"
])

export function extractEntities(text: string): string[] {
  return text
    .toLowerCase()
    .split(/\s+/)
    .map((word) => word.replace(/[^a-z0-9-]/g, ""))
    .filter((word) => word.length > 0 && !STOP_WORDS.has(word))
}
```
