# Phase 24.3.3: Execute-Slash-Command Advanced Features — Specification

**Created:** 2026-05-27

## Goal

Implement module extraction, semantic agent selection, and contract validation for execute-slash-command tool in a **single phase** (P24.3.3). Merged P24.3.3.1 (foundational) + P24.3.3.2 (advanced) for faster delivery.

## Background

The `execute-slash-command.ts` tool currently contains 527 LOC with inline command resolution and dispatch logic. Phase 24.3.2 deferred namespace routing, module extraction, and contract validation to this phase. Current state shows no semantic agent selection mechanism — agent is selected only if explicitly provided in command frontmatter or via `agent` field override. Users report hallucination when agent incorrectly assumes command execution is complete without following command instructions. Commands are loaded from `.opencode/commands/*.md` and `.opencode/commands/*.json` files.

## Requirements

1. **Phase scope**: Single phase P24.3.3 (merged P24.3.3.1 + P24.3.3.2)
    - Current: Phase split deferred due to namespace deferral
    - Target: Single phase with all features except namespace routing logic
    - Acceptance: Phase directory exists, ROADMAP.md updated to show merged phase

2. **Module extraction**: Extract command resolution and dispatch logic from execute-slash-command.ts
   - Current: execute-slash-command.ts (527 LOC) contains inline resolution/dispatch logic
   - Target: Three modules — `resolve-command.ts` (namespace routing), `dispatch-command.ts` (dispatch logic), `index.ts` (barrel re-exports)
   - Acceptance: `execute-slash-command.ts` under 500 LOC, imports from new modules, type-check passes

3. **Namespace field**: Add optional namespace field to command frontmatter (DEFERRED)
    - Current: No namespace field in command frontmatter
    - Target: `namespace?: string` field in YAML frontmatter (optional, placeholder only)
    - Acceptance: Schema accepts any string value, NO validation logic, NO routing logic, NO enforcement

4. **Contract validation**: Validate required command fields with dynamic schema
    - Current: No contract validation for command frontmatter
    - Target: `validateCommandContract()` returns `ValidationResult` with field-level errors, validates ONLY structural fields that MUST exist (name, description, triggers)
    - Acceptance: Returns errors for missing required fields, passes for valid commands, NO validation of SDK-driven fields (agent, subtask, model, tools)

5. **Two-stage routing**: Implement command resolution → agent selection pipeline
   - Current: Agent selected only if explicitly provided or from frontmatter; no fallback logic
   - Target: Stage 1 resolve command (semantic matching), Stage 2 resolve agent (semantic matching using same intent classification)
   - Acceptance: Command resolved first, then agent selected independently, semantic matching integrated into `resolveCommand()`

6. **Semantic agent selection**: Integrate semantic matching into existing resolveCommand()
   - Current: No semantic parsing, exact name matching only
   - Target: Extend `resolveCommand()` to parse natural language intent, match against command names/descriptions
   - Acceptance: User intent "gsd command" finds commands with "gsd" in name/description, no new module created

7. **Agent selection mechanism**: Use same logic for agent selection as command resolution
   - Current: Agent selection unclear, no research on OpenCode SDK agent discovery
   - Target: List available agents, parse user intent, select most relevant agent based on description/name
   - Acceptance: Agent selection works when user specifies intent, fallback to default agent if not specified

8. **Backward compatibility**: Zero breaking changes for existing commands
   - Current: Some commands may rely on current execution flow
   - Target: All existing commands work without changes, namespace field optional, legacy resolution fallback
   - Acceptance: Existing commands pass validation, type-check clean, no regression in tests

## Boundaries

**In scope:**
- Module extraction: `resolve-command.ts`, `dispatch-command.ts`, `index.ts`
- Namespace routing: optional field in YAML frontmatter, no hardcoded values
- Contract validation: dynamic schema validation, field-level error reporting
- Two-stage routing: command resolution → agent selection pipeline
- Semantic matching: integrate into `resolveCommand()`, no new module
- Phase 24.3.3.1: foundational features only (modules + namespace + validation)
- Phase 24.3.3.2: advanced features (semantic selection + shell/workflow)

**Out of scope:**
- Hardcoded namespace values (gsd/hf/test/core) — deferred to user-defined
- Shell command integration (`!bash` prefix) — Phase 24.3.3.2
- Workflow file parsing — Phase 24.3.3.2
- Natural language command auto-append — Phase 24.3.3.2
- List available commands to user — Phase 24.3.3.2
- Agent switching via command frontmatter `agent:` field — Phase 24.3.3.2
- Subtask mode (`subtask: true`) — Phase 24.3.3.2
- TUI bypass for child sessions — Phase 24.3.3.2

## Constraints

- `execute-slash-command.ts` target: under 500 LOC (flexible, no specific cap)
- Namespace field: optional, no hardcoded values, no enum constraints, `namespace?: string`
- Semantic matching: no new module, extend existing `resolveCommand()`
- Backward compatibility: zero breaking changes, legacy fallback for commands without namespace
- OpenCode SDK v1.15.5: validate against actual package.json versions
- Command files: support both `.md` and `.json` formats
