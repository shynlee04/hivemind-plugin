# Phase 21.1: Execute-Slash-Command SDK Redesign — Research

**Researched:** 2026-05-22
**Domain:** OpenCode plugin SDK + tool redesign
**Confidence:** HIGH

## Summary

`execute-slash-command` tool currently uses raw TUI prompt pipeline (`clearPrompt → appendPrompt → submitPrompt`) to dispatch slash commands. The OpenCode SDK v1.15.5 (installed) provides `client.session.command()` which is a dedicated command execution API that supports `agent`, `model`, `arguments`, and `command` parameters natively, with server-side `$1/$2/$ARGUMENTS` parsing.

The SDK v1 (main export used by `PluginInput`) and SDK v2 (`@opencode-ai/sdk/v2`) both expose `session.command()` — but with different parameter shapes. The **plugin** uses the v1 client (`createOpencodeClient` from `@opencode-ai/sdk`). The v1 `session.command()` has a simpler body that covers the 7 requirements from the SPEC. The v2 adds `variant`, `parts` (file-only), but the plugin client is v1.

**Key insight:** `session.command()` already handles agent switching (agent parameter), model override (model parameter), argument substitution (server-side from template), and command existence checking (server returns 404). The tool rewrite is primarily about wrapping the SDK call, adding preflight, reading frontmatter, and standardizing the result envelope — not about reimplementing command parsing.

**Primary recommendation:** Rewrite tool to call `client.session.command()` as primary path, with TUI fallback for backward compatibility; add preflight via `client.command.list()`; enrich from `loadPrimitives()` or command-engine for frontmatter awareness; return standard `ToolResponse` envelope.

## Architectural Responsibility Map

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| Command execution dispatch | Tool (`src/tools/session/execute-slash-command.ts`) | SDK server | Tool orchestrates SDK call + fallback logic |
| Command existence preflight | SDK `client.command.list()` | Command-engine `discoverCommandBundles()` | SDK provides native command list; command-engine provides local discovery |
| Frontmatter enrichment | Loader (`primitive-loader`) | SDK `Command.template` | SDK returns `Command` type with `agent`, `model`, `subtask`, `template` |
| Agent switching (TUI) | SDK `session.command({ agent })` | — | SDK handles agent switch server-side + fires `SessionNextAgentSwitched` event |
| Argument/template parsing | SDK server | — | SDK handles `$1/$2/$ARGUMENTS` server-side from `Command.template` |
| Subtask delegation | SDK `session.prompt({ parts: [SubtaskPartInput] })` | — | `session.command()` doesn't support subtask parts; use `session.prompt()` with SubtaskPartInput |
| Result rendering | Tool (`src/shared/tool-response.ts`) | — | Tool maps SDK response to standard `ToolResponse` envelope |

### Pattern 1: SDK-First Command Execution with TUI Fallback
**What:** Primary dispatch via `client.session.command()`. If that fails (session not available), fall back to TUI append path.
**When to use:** Always — backward compatibility guarantee from SPEC.
**Source:** [VERIFIED: node_modules/@opencode-ai/sdk/dist/gen/types.gen.d.ts:2373-2411]

### Pattern 2: Frontmatter-Aware Preflight via Enhanced CommandBundle
**What:** Use existing `discoverCommandBundles()` from command-engine to get full frontmatter (including `model`, `subtask`), then cross-reference with `client.command.list()` from SDK.
**When to use:** Always — SPEC requires frontmatter awareness.
**Source:** [VERIFIED: src/routing/command-engine/index.ts:27-41] — note current `CommandBundle` type at `types.ts:4-17` drops `model` and `subtask` from frontmatter.

### Pattern 3: Subtask Routing via SDK session.prompt()
**What:** When command frontmatter has `subtask: true` or `agent.mode === "subagent" && cmd.subtask !== false`, dispatch via `session.prompt()` with `SubtaskPartInput` instead of `session.command()`.
**When to use:** Only when frontmatter analysis resolves to subtask path.
**Source:** [VERIFIED: node_modules/@opencode-ai/sdk/dist/gen/types.gen.d.ts:2257] — `session.prompt()` accepts `SubtaskPartInput`.

### Pattern 4: Agent Switch + Restore (Two Mechanisms)
**What:** Mechanism (a) passes `agent` to `session.command()` — SDK switches agent server-side, fires `SessionNextAgentSwitched` event. Mechanism (b) records current agent from `ctx.agent`, dispatches under target agent, then sends follow-up command to restore.
**When to use:** Mechanism (a) for simple one-shot; mechanism (b) for `restore: true` flag.
**Source:** [VERIFIED: node_modules/@opencode-ai/sdk/dist/v2/gen/types.gen.d.ts:1567-1578] — `SyncEventSessionNextAgentSwitched` event.

### Anti-Patterns to Avoid
- **Reimplementing argument parsing in the tool layer:** SDK server handles `$1/$2/$ARGUMENTS` from `Command.template` — don't duplicate this logic.
- **Relying solely on TUI path:** The whole point of this phase is to move away from TUI append as the primary mechanism.
- **Ignoring the v1/v2 SDK gap:** The tool receives `PluginInput["client"]` which is v1 SDK. The v1 `session.command()` has different parameter shape than v2.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Command argument parsing (`$1/$2/$ARGUMENTS`) | Custom regex parser | SDK server (from `Command.template`) | SDK server already handles template placeholder substitution; tool just passes raw `arguments` string |
| Command discovery/listing | Custom file scanner | `client.command.list()` + `discoverCommandBundles()` | SDK provides native command list API; command-engine provides local frontmatter enrichment |
| Agent switching in TUI | Custom `@agent` text prepend | SDK `session.command({ agent })` | SDK handles agent switch server-side with proper event emission |
| Subtask delegate creation | Raw session fork | `session.prompt({ parts: [SubtaskPartInput] })` | SDK native `SubtaskPartInput` creates properly tracked child session |

## Common Pitfalls

### Pitfall 1: SDK v1 vs v2 Command API Mismatch
**What goes wrong:** The v1 SDK `session.command()` data type (`SessionCommandData` at `types.gen.d.ts:2373`) has `body: { arguments, command, agent?, model?, messageID? }` — no `parts` or `variant` fields. The v2 SDK has richer parameters. If the tool code references v2 fields, it won't type-check.
**Why it happens:** `PluginInput.client` is `ReturnType<typeof createOpencodeClient>` from `@opencode-ai/sdk` (v1), not from `@opencode-ai/sdk/v2`.
**How to avoid:** Always use v1 SDK types. The v1 `session.command()` supports all SPEC requirements (agent, model, arguments, command).
**Warning signs:** TypeScript errors about missing `parts` or `variant` fields.

### Pitfall 2: session.command() Queue Blocking
**What goes wrong:** Like the current JSDoc in `execute-slash-command.ts:14-20` warns, `session.command()` internally may queue if the session is busy. The old TUI path bypassed this blocking.
**Why it happens:** `SessionRunState.ensureRunning()` detects the session is busy and queues the command instead of executing it immediately.
**How to avoid:** Keep TUI fallback path. The SPEC says "SDK-first + TUI fallback" (hybrid approach). After SDK command call, if it queues (vs immediate execution), the tool still returns immediately — the command fires on the next turn.
**Warning signs:** Tool returns success but command doesn't appear to execute in the TUI.

### Pitfall 3: CommandBundle Type Missing Frontmatter Fields
**What goes wrong:** The existing `CommandBundle` type at `src/routing/command-engine/types.ts:4-17` has `name`, `source`, `filePath`, `description`, `agent`, `body`. It does NOT have `model` or `subtask`. If the tool relies on `discoverCommandBundles()` for frontmatter awareness, `model` and `subtask` won't be available.
**Why it happens:** The command-engine was designed before full frontmatter awareness was needed.
**How to avoid:** Either (a) use `client.command.list()` which returns the SDK's `Command` type with `subtask` field, or (b) extend `CommandBundle` to include `model` and `subtask` from the primitive loader's `CommandFile.frontmatter`.
**Warning signs:** Typing errors when accessing `command.model` or `command.subtask` on `CommandBundle`.

## Open Questions

1. **Does the v1 SDK `session.command()` queue when the session is busy?**
   - What we know: The current JSDoc says `session.prompt()` queues via `SessionRunState.ensureRunning()`. `session.command()` follows the same POST pattern.
   - What's unclear: Whether `session.command()` internally uses the same queue path.
   - Recommendation: Keep TUI fallback as the SPEC mandates. Test during implementation.

2. **Does the `CommandBundle` type need extension?**
   - What we know: `CommandBundle` at `src/routing/command-engine/types.ts:4-17` drops `model` and `subtask`.
   - What's unclear: Whether to extend it or use SDK `Command` type for frontmatter reading.
   - Recommendation: Use SDK `client.command.list()` for preflight (returns full `Command` type including `subtask`), and extend `CommandBundle` if local primitive loader data is needed.
