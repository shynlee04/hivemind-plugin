## Changes Made

### Task 1: Added getByParent() Accessor (REQ-23.2-03)

Added public `getByParent()` method to `PendingDispatchRegistry` class (line 303):

```typescript
getByParent(parentSessionID: string): PendingDispatchEntry | undefined {
  this.cleanupStale()
  const callIDs = this.byParent.get(parentSessionID)
  if (!callIDs || callIDs.size === 0) return undefined

  for (const callID of callIDs) {
    const entry = this.dispatches.get(`call:${callID}`)
    if (entry && Date.now() - entry.timestamp <= PendingDispatchRegistry.STALE_THRESHOLD_MS) {
      return entry
    }
  }
  return undefined
}
```

**Key design decisions:**
- Uses existing `byParent` Map (private, line 88) which maps `parentSessionID → Set<callID>`
- Iterates callIDs to find entries — doesn't resolve child IDs
- Reuses `STALE_THRESHOLD_MS` constant for consistency with `get()`
- Returns first non-stale match

### Task 2: Parent-Based Fallback + ManifestWriter.addChild() (REQ-23.2-03, REQ-23.2-04)

#### Change 1: Parent-based fallback for pending dispatch lookup (BUG#3)

Modified `writeImmediateChildFile()` at line 453-456:

```typescript
// REQ-23.2-03: Direct child ID lookup + parent-based fallback for race conditions
const pendingEntry =
  this.pendingRegistry?.get(sessionID) ??
  this.pendingRegistry?.getByParent(parentID)
```

The `??` (nullish coalescing) means: try child ID first, fall back to parent lookup only when `get()` returns undefined. This handles the race where `session.created` fires before `updateWithChildID()` has mapped the callID to the child session ID.

#### Change 2: manifestWriter.addChild() call (BUG#4 second call site)

Added `manifestWriter.addChild()` call at line 514-521 in `writeImmediateChildFile()`:

```typescript
// REQ-23.2-04: Populate hierarchy-manifest.json (was missing at this call site)
if (typeof this.manifestWriter.addChild === "function") {
```
