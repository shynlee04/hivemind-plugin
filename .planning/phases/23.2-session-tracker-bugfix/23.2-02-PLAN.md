    Per REQ-23.2-01. Replace the current `extractTextContent()` method (lines 316-322) with a more robust implementation that:

    1. **Extends the OutputPart interface** to include optional `content` field:
       ```typescript
       interface OutputPart {
       ```

    2. **Rewrites extractTextContent()** to try multiple fields per part:
       ```typescript
       private extractTextContent(parts: OutputPart[] | null | undefined): string {
         if (!Array.isArray(parts)) return ""

         const extracted: string[] = []
         for (const p of parts) {
           // Try multiple text-carrying fields in priority order
           const text =
             (typeof p.text === "string" && p.text.trim()) ? p.text :
             (typeof p.content === "string" && p.content.trim()) ? p.content :
             undefined

           if (text) {
             extracted.push(text)
           }
         }

         if (extracted.length === 0 && parts.length > 0) {
           // REQ-23.2-01: Log diagnostic warning when no text extracted from non-empty parts
           void this.client.app?.log?.({
             body: {
               service: "session-tracker",
               level: "debug",
               message: `[Harness] Session tracker: extractTextContent found no text in ${parts.length} part(s). Types: ${parts.map(p => p.type).join(", ")}`,
             },
           })
         }

         return extracted.join("\n")
       }
       ```

    Key design decisions:
    - Do NOT filter by `p.type === "text"` — any part type can carry text content. This fixes the core bug where delegate-task responses use a different part type.
    - Use `p.text` first, then `p.content` — matches OpenCode's observed field names.
    - Use `trim()` check to skip empty/whitespace-only strings.
    - Use `void` prefix on the log call to avoid floating promise warnings.
    - Log at `debug` level (not `warn`) since missing text is expected for some part types (e.g., tool_use, image). The log helps diagnose without noise.
  </action>
