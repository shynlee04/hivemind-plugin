# Phase 23.2: Session-Tracker Bugfix — Unified task/delegate-task

**Gathered:** 2026-05-24  
**Status:** Research Complete  
**Domain:** Session-Tracker runtime feature (src/features/session-tracker/)

---

## Summary

Phase 23.2 addresses 5 bugs in the session-tracker feature where the code was written exclusively for the `task` tool and never adapted for `delegate-task`. All 5 bugs share a common root cause: the session-tracker capture logic assumes `task` tool argument shapes and OpenCode SDK event patterns, breaking when `delegate-task` is used.

**Primary recommendation:** Fix all 5 bugs with surgical code changes that handle both delegation tools without restructuring the pending-dispatch-registry. The fixes are low-risk (no architecture changes), testable with existing unit test patterns, and verifiable via live UAT with both `task` and `delegate-task`.

## Technical Approach

### Bug 1: Assistant text extraction (HIGH priority)

**Current behavior:** `extractTextContent()` in `message-capture.ts:316-322` filters parts exclusively for `p.type === "text"` with `p.text` field. When OpenCode uses a different part schema (e.g., `p.content` instead of `p.text`), the function returns an empty string which is falsy at `content || undefined` (line 214 in session-writer.ts) and skipped entirely.

**Root cause:** Overly strict part type filtering that doesn't account for OpenCode SDK version variations or custom tool part schemas.

**Fix approach:** Extend `extractTextContent()` to handle multiple part type patterns:
```typescript
private extractTextContent(parts: OutputPart[] | null | undefined): string {
  if (!Array.isArray(parts)) return ""
  return parts
    .filter((p) => {
      // Accept both standard text parts and content-based parts
      const isTextPart = p.type === "text"
      const hasText = typeof p.text === "string" || typeof p.content === "string"
      return isTextPart && hasText
    })
    .map((p) => asString(p.text) ?? asString(p.content) ?? "")
    .join("\n")
}
```

**Verification strategy:** Add unit tests covering:
- `text` part with `.text` field
- `text` part with `.content` field  
- Unknown part type (graceful skip with warning log)

### Bug 2: Compaction writes raw JSON (MEDIUM priority)

**Current behavior:** `renderCompactionContext()` in `event-capture.ts:612-626` searches for summary text in the `session.compacted` event payload using `findCompactionText()`. The `session.compacted` event from OpenCode SDK contains only metadata (`sessionID`, `timestamp`), not actual compact summary text. `findCompactionText()` searches for preferred keys (`summary`, `compactSummary`, `compactionSummary`, etc.), finds nothing, and falls back to `stringifyEvent(event)` which dumps raw JSON.

**Root cause:** Misunderstanding of OpenCode SDK `session.compacted` event schema — it's a metadata-only event, not a content-rich event.

**Fix approach:** Read session message history via `client.session.messages()` to extract the compact summary from the last assistant message:
```typescript
private async handleSessionCompacted(
  sessionID: string,
  event: Record<string, unknown> | undefined
): Promise<void> {
  // Try to find summary in event first (for backward compatibility)
  const summary = this.findCompactionText(event)
  
  // If not found, read from message history
  if (!summary) {
    try {
      const messages = await getSessionMessages(this.client, sessionID)
      if (messages && messages.length > 0) {
        // Get last assistant message
        const lastMsg = messages.find(m => 
          this.messageRole(m) === "assistant" 
        ) ?? messages[messages.length - 1]
        summary = this.extractTextFromSdkMessage(lastMsg)
      }
    } catch (err) {
      // Fallback to note
      summary = "[Compaction occurred but summary unavailable]"
    }
  }
  
  // Write summary (not raw JSON)
  if (summary) {
    await this.sessionWriter.appendJourneyEntry(sessionID, {
      type: "session_compacted",
      summary,
      metadata: { capturedFrom: "session.compacted" }
    })
  }
}
```

**Verification strategy:** Test with simulated `session.compacted` event (metadata-only) → verify message history read → verify summary extraction → verify .md write contains plain text, not JSON.

### Bug 3: Tool attribution race condition (MEDIUM priority)

**Current behavior:** `writeImmediateChildFile()` in `event-capture.ts:453,472` looks up pending dispatch by child session ID using `dispatches.get(childID)` and `dispatches.get('call:{childID}')`. However, pending dispatch is registered with key `call:{callID}` (a different value) at PreToolUse time. If async polling hasn't called `updateWithChildID()` yet, lookup returns `undefined` → tool defaults to `"task"`.

**Root cause:** Race condition between session.created event firing and pending registry updateWithChildID().

**Fix approach:** Add fallback lookup by parent session ID using the existing `byParent` reverse index:
```typescript
private async writeImmediateChildFile(
  event: SessionCreatedEvent
): Promise<void> {
  const { childSessionID, parentSessionID } = event.metadata
  
  // Try direct child ID lookup first
  const pending = this.pendingDispatchRegistry.get(childSessionID)
  
  // If not found, fallback to search by parent
  if (!pending) {
    const byParent = this.pendingDispatchRegistry.byParent.get(parentSessionID)
    if (byParent && byParent.length > 0) {
      // Take first match (should be unique in practice)
      pending = byParent[0]
    }
  }
  
  // Use pending.tool if found, fallback to "task"
  const tool = pending?.tool ?? "task"
  
  // ... rest of implementation using tool variable
}
```

**Verification strategy:** Mock session.created event arriving before updateWithChildID() → verify fallback by parent lookup → verify tool attribute shows correct tool name.

### Bug 4: Hierarchy manifest not populated (HIGH priority)

**Current behavior:** `manifestWriter.addChild()` in `hierarchy-manifest.ts:62-94` is **never called** anywhere in the codebase. Only `manifestWriter.updateChildStatus()` is called (4 locations), which silently no-ops when the child isn't found. Meanwhile, `sessionIndexWriter.addChild()` IS called, so `session-continuity.json` gets the child but `hierarchy-manifest.json` does not.

**Root cause:** Missing call site — `addChild()` was implemented but never wired into the delegation recording path.

**Fix approach:** Add `manifestWriter.addChild()` calls at two locations:
1. In `tool-delegation.ts:303` — immediately after `sessionIndexWriter.addChild()` in `recordChildTaskDelegation()`
2. In `event-capture.ts` — in `writeImmediateChildFile()` when a new child is detected via `session.created` event

```typescript
// In tool-delegation.ts:303
await this.sessionIndexWriter.addChild(childSessionID, {
  parentSessionID,
  tool: "delegate-task",
  actor: subagentType,
  depth: childDepth,
  status: "active"
})
await this.manifestWriter.addChild(childSessionID, {
  parentSessionID,
  tool: "delegate-task",
  actor: subagentType,
  depth: childDepth,
  status: "active"
})
```

**Verification strategy:** Dispatch child via `delegate-task` → verify `hierarchy-manifest.json` contains child entry with correct metadata → verify `session-continuity.json` also has entry (existing behavior preserved).

### Bug 5: Actor field extraction (LOW priority)

**Current behavior:** `recordChildTaskDelegation()` in `tool-delegation.ts:237` extracts `subagentType` from `args.subagent_type` (the `task` tool's argument key). The `delegate-task` tool uses `args.agent`. When dispatched via `delegate-task`, `args.subagent_type` is `undefined` → defaults to `"unknown"` → propagated to `actor` field at line 337.

**Root cause:** Hardcoded argument key that doesn't support both delegation tools' different argument schemas.

**Fix approach:** Check BOTH argument keys with fallback order:
```typescript
const subagentType = 
  args.subagent_type ??  // task tool
  args.agent ??          // delegate-task tool
  "unknown"              // fallback
```

**Verification strategy:** Dispatch via `delegate-task` with `args.agent = "hm-debugger"` and no `args.subagent_type` → verify actor field shows `"hm-debugger"`, not `"unknown"`.

## Constraints & Pitfalls

### Constraint 1: No architecture changes

Phase 23.2 is explicitly scoped to bugfixes only. The pending-dispatch-registry keying scheme (`call:{callID}`) is NOT being changed — we're just adding a fallback lookup path. This keeps the risk profile low and avoids regression testing of architectural changes.

### Constraint 2: OpenCode SDK event shapes

We must adapt to existing OpenCode SDK hook event shapes, not change them. The `session.compacted` event is metadata-only by design in OpenCode SDK v1.14.44. Our workaround (reading message history) is an Hivemind-side adaptation, not an SDK change.

### Constraint 3: Test isolation

All fixes must pass existing unit tests in `tests/features/session-tracker/`. The session-tracker has 256+ tests that must not regress. New unit tests should be added for each bug fix, following the existing test patterns.

### Pitfall: Race condition timing

BUG #3's race condition fix relies on the `byParent` reverse index existing. Verify this index is populated before `writeImmediateChildFile()` is ever called. The index should be built during pending dispatch registration at PreToolUse time.

### Pitfall: Message history availability

BUG #2's fix assumes `client.session.messages()` is available for all sessions. Test this assumption: what happens for child sessions where message history might be incomplete or unavailable? Add fallback handling.

### Pitfall: Part type variations

BUG #1's fix handles `p.text` and `p.content` fields. However, OpenCode SDK might introduce new part types in future versions. Consider making the filter more defensive:
```typescript
const isTextPart = p.type === "text"
const hasText = typeof p.text === "string" || typeof p.content === "string" || typeof p.value === "string"
```

## Recommendations

### 1. Fix order (priority-based)

Execute fixes in this order to maximize impact and minimize risk:

1. **BUG #4 (HIGH)** — Hierarchy manifest not populated
   - No code logic changes, just missing call sites
   - Immediate visible impact
   - Low regression risk

2. **BUG #1 (HIGH)** — Assistant text extraction
   - Affects session readability

3. **BUG #3 (MEDIUM)** — Tool attribution race condition
   - Requires understanding of pending registry internals
   - Test with race condition timing
   - Medium regression risk

4. **BUG #2 (MEDIUM)** — Compaction summary
   - Requires SDK message history access
   - May have edge cases (child sessions, unavailable history)
   - Test thoroughly before merge

5. **BUG #5 (LOW)** — Actor field extraction
   - Simple fix, minimal impact
   - Can be done last

### 2. Test strategy

For each bug fix:
1. Add unit test covering the specific scenario
2. Run existing test suite to verify no regressions
3. Run typecheck (`npm run typecheck`)
4. If feasible, run live UAT with both `task` and `delegate-task`

### 3. Integration with Phase 23.1 findings

Phase 23.1 investigated why `session.created` events aren't firing for custom tool dispatches. The findings suggest this may be an architectural limitation of OpenCode SDK (can't observe SDK sessions created via `client.session.create()`). 

**Action:** Document this in Phase 23.2's limitations. The bugfixes assume `session.created` events DO fire (which they do for OpenCode's native delegation tools). If a custom tool creates sessions via `client.session.create()`, those sessions won't be captured by the session-tracker regardless of these bugfixes.

### 4. Documentation updates

After fixes are implemented:
