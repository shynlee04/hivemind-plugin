## Executive Summary

Four interconnected bugs were identified in the session-tracker capture system through live UAT testing. All four root causes are now confirmed with file:line evidence. The bugs share a common theme: **child session metadata is incomplete because the capture system lacks a reliable mechanism to propagate delegation context (agent name, model, subagent type) from the dispatch site to the event hooks.**

## Bug A: Root `.md` Frontmatter `children: []` — Never Populated

### Symptom
Root session `.md` file shows `children: []` even though child sessions exist and are tracked in `hierarchy-manifest.json`.

### Root Cause
**No code path ever updates the root `.md` frontmatter `children` array after it is initialized to `[]`.**

### Evidence

1. **Initialization** — All session creation paths set `children: []`:
   - `event-capture.ts:487` — `writeImmediateChildFile` creates child .json with `children: []`
   - `tool-delegation.ts:269` — `recordChildTaskDelegation` sets `children: []`
   - `tool-capture.ts:291` — sets `children: []`

2. **`updateFrontmatter` calls** — Only update specific fields, never `children`:
   - `event-capture.ts:306` — `updateFrontmatter(sessionID, { status: "completed" })` — **status only**
   - `event-capture.ts:358` — `updateFrontmatter(sessionID, { status: "error" })` — **status only**
   - `event-capture.ts:410` — `updateFrontmatter(sessionID, { status })` — **status only**
   - `message-capture.ts:224` — `updateFrontmatter(sessionID, { lastMessage: content })` — **lastMessage only**

3. **Grep confirmation** — Searching for `children` updates in frontmatter across the entire session-tracker directory returns ZERO results for any update/append pattern.

### Impact
Root `.md` file is unreliable for determining child session count. Consumers must read `hierarchy-manifest.json` instead.

### Fix Direction
`writeImmediateChildFile` (event-capture.ts:442) or `child-writer.ts:mergeMetadata` should call `updateFrontmatter` on the root `.md` to push a `ChildRef` into the `children` array. Alternatively, remove `children` from the `.md` frontmatter entirely and use `hierarchy-manifest.json` as the sole source of truth.

## Bug B: Compaction Shows "summary unavailable"

### Symptom
When a child session is compacted, the journey entry contains `"**Compaction occurred — summary unavailable.**\n"` instead of the actual compaction summary.

### Root Cause
**Two-part issue:**

1. **`handleSessionCompacted` child route returns early** (event-capture.ts:604-611) — When the compacted session is a child, it writes a journey entry and `return`s at line 611 WITHOUT calling `appendCompactionBlock`. This is architecturally correct (children don't have `.md` files), but the journey entry gets the fallback text.

2. **`resolveCompactionFromMessages` fails for child sessions** (event-capture.ts:764-793) — `getSessionMessages(client, sessionID)` likely returns empty/null for child sessions because the SDK's message API operates on the parent session context, not the child's. When messages are empty, the function falls through to the default at line 792: `"**Compaction occurred — summary unavailable.**\n"`.

### Evidence

1. `handleSessionCompacted` flow:
   ```
   :590 — eventSummary = findCompactionText(event)    // Likely null for metadata-only event
   :596 — compactContext = resolveCompactionFromMessages(sessionID)  // Returns fallback
   :603 — childRoute = resolveChildLifecycleRoute(sessionID)  // Found for child
   :605-610 — appendJourneyEntry with compactContext (fallback text)
   :611 — return  // EARLY RETURN — appendCompactionBlock never called
   ```

2. `resolveCompactionFromMessages` (line 792) — Default fallback is hardcoded `"**Compaction occurred — summary unavailable.**\n"`.

3. `getSessionMessages` catch block (line 789-791) — Silently swallows errors, falls through to default.

### Impact
Compaction history for child sessions is lost. Journey entries contain unhelpful placeholder text.

### Fix Direction
Either (a) ensure `getSessionMessages` works for child sessions by passing parent context, or (b) extract the compaction summary from the event payload before falling back to message history, or (c) read the last assistant turn from the child `.json` file as the compaction summary.

## Bug C: Manifest `turnCount: 0` for All Children

### Symptom
`hierarchy-manifest.json` shows `turnCount: 0` for every child, even when child `.json` files contain 4-12 turns.

### Root Cause
**`addChild()` hardcodes `turnCount: 0` and no code path ever updates it.**

### Evidence

1. **`hierarchy-manifest.ts` — `addChild()` method**:
   - Sets `turnCount: 0` unconditionally when registering a new child.
   - **No `updateTurnCount()` method exists** anywhere in the class.

2. **`hierarchy-manifest.ts` — `generateFromContinuity()` method** (line 204):
   - Also hardcodes `turnCount: 0` when generating manifest from continuity data.

3. **Grep confirmation** — Searching for `turnCount` update/set patterns across all session-tracker files shows only initialization at `0`. No increment or update logic exists.

### Live Test Data
```json
{
  "sessionID": "ses_1a5cee94affeiQgh6J82I7PVIo",
  "turnCount": 0,  // WRONG — child .json has 12 turns
  "status": "completed"
}
```

### Impact
Manifest `turnCount` is unreliable. Consumers must read child `.json` files and count `turns.length`.

### Fix Direction
Add `updateTurnCount(sessionID, count)` to `HierarchyManifest`. Call it from `ChildWriter.appendChildTurn` after each turn is added, or add a periodic reconciliation pass that reads all child `.json` files and updates manifest turn counts.

## Bug D: Actor `"unknown"`, Model `""` in Child Turns

### Symptom
Child `.json` turns show `actor: "unknown"` and `model: ""` instead of the actual agent name and model.

### Root Cause
**Race condition in pending-dispatch-registry + hook payload lacks delegation metadata.**

### Evidence

1. **Race condition in `pending-dispatch-registry.ts`**:
   - `add()` stores entry on `PreToolUse` (before tool execution)
   - `remove()` fires on `PostToolUse` (after tool execution)
   - Child session creation event fires AFTER `PostToolUse`
   - So `find()` returns `null` when `writeImmediateChildFile` looks up the entry
   - Default at event-capture.ts:457: `const subagentType = explicitSubagentType ?? pendingEntry?.subagentType ?? "unknown"`

2. **`child-recorder.ts` line 105**:
   ```typescript
   actor: input.agent || "unknown",
   ```
   The `input.agent` comes from the `chat.message` hook payload, which does NOT carry delegation metadata (agent name from the `task`/`delegate-task` tool call).

3. **`child-recorder.ts` line 120**:
   ```typescript
   model: typeof input.model === "string" ? input.model : input.model?.modelID,
   ```
   If `input.model` is undefined (hook doesn't carry model), `model` becomes `undefined` → serialized as `""`.

4. **Live test data confirms**:
   - Task child: 4 turns, all `actor: "unknown"`, `model: "unknown"`
   - Delegate-task child: 12 turns, all `actor: "unknown"`, `model: ""`

### Impact
Child session attribution is completely lost. Cannot determine which agent or model produced which turns.

### Fix Direction
Three options (pick one or combine):
1. **Registry fix**: Don't `remove()` on PostToolUse — keep entries for the child session's full lifetime. Add TTL-based cleanup instead.
2. **Hook enrichment**: Pass delegation metadata (agent name, model) through the OpenCode hook payload so `ChildRecorder` can access it.
3. **Write-through on dispatch**: When `delegate-task`/`task` tool fires, immediately write a "delegation context" file that `ChildRecorder` reads on first message.

```
Bug D (actor=unknown)
  └── pending-dispatch-registry race + hook payload lacks delegation metadata
       ├── affects writeImmediateChildFile (event-capture.ts:457)
       └── affects ChildRecorder (child-recorder.ts:105, :120)

Bug A (root .md children=[])
  └── writeImmediateChildFile creates child .json but never updates root .md
       └── updateFrontmatter only called for status/lastMessage fields

Bug C (turnCount=0)
  └── addChild() hardcodes turnCount=0, no update method exists
       └── No code path reconciles manifest with actual child .json turn count

Bug B (compaction summary unavailable)
  └── resolveCompactionFromMessages fails for child sessions
       └── getSessionMessages likely returns null for child sessionID
       └── handleSessionCompacted child route returns early with fallback text
```

### Common Theme
**Child sessions are second-class citizens in the capture pipeline.** The system was designed around root-session `.md` files and only partially adapted for child `.json` files. Key gaps:
- Root `.md` frontmatter never reflects child state
- Manifest turn counts are never reconciled
- Hook payloads don't carry delegation context
- Message history API doesn't work for child session IDs
