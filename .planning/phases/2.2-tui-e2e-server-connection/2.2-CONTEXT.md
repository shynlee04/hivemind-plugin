<domain>
## Phase Boundary

Extract the OpenTUI dashboard into an isolated Bun app boundary at `apps/opentui`. Wire it to the OpenCode server using shared runtime-status contracts and prove innate OpenCode capabilities function correctly between client and server.
</domain>

<decisions>
## Implementation Decisions

### TUI Extraction
- Ensure `apps/opentui` is entirely decoupled from Node-specific constraints and relies on Bun execution to support OpenTUI's `.scm` and native features.

### Client-Server Wiring
- Wire the TUI dashboard to fetch and render the exact `RuntimeStatus` interface established in `src/shared/contracts/runtime-status.ts`.
- Ensure `package.json` scripts (`build`, `test`) correctly orchestrate the TUI using `bun`.

### E2E Validation
- Implement a test/proof that demonstrates a live connection between the TUI client and the OpenCode server. This must verify that "innate OpenCode work functions without problem between 2 sides".
</decisions>
