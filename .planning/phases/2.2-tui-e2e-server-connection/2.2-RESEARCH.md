## Summary

The TUI is currently an orphan spike. It builds, but it is not wired into package exports, scripts, or CLI entrypoints; `src/cli.ts` has no TUI route and `package.json` exposes no TUI entry. Furthermore, `apps/opentui` fails under Node because OpenTUI is Bun-exclusive. 

**Primary recommendation:** Keep the core package on Node, move the dashboard to a real Bun/OpenTUI app boundary at `apps/opentui`, and share typed contracts with the backend.

## User Constraints (from Audit Directives)

### Locked Decisions
- Dashboard must be extracted fully into the `apps/opentui` Bun app boundary.
- TUI must be wired to the OpenCode server using shared runtime-status contracts.
- Must prove innate OpenCode work functions without communication errors between the client and server.

## Architecture Patterns

### Pattern 1: Bun-Exclusive Client App
**What:** `apps/opentui` is run exclusively via Bun.
**Why:** OpenTUI uses `.scm` and other Bun-specific features that fail in Node.

### Pattern 2: Shared Contracts
**What:** `src/shared/contracts/` is used by both the Node backend and the Bun frontend.
**Why:** Ensures the client and server agree on the shape of `RuntimeStatus` and events without duplicating types.

## Common Pitfalls

### Pitfall 1: Running OpenTUI with Node
**What goes wrong:** `ERR_UNKNOWN_FILE_EXTENSION` for `.scm` files.
**How to avoid:** Ensure all TUI run scripts and tests use `bun` instead of `node` or `tsx`.

### Pitfall 2: Silent Communication Failures
**What goes wrong:** The TUI renders, but data from the backend is missing or stale.
**How to avoid:** Build a concrete End-to-End test that validates a round-trip message (e.g., status request from TUI -> Backend -> TUI renders correct state).
