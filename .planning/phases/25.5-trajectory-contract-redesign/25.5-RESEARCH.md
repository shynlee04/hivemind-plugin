# Phase 25.5: Trajectory + Contract Redesign — Research

**Researched:** 2026-05-30
**Domain:** Trajectory per-PHASE redesign + Contract runtime enforcement via hooks
**Confidence:** HIGH

## Summary

Phase 25.5 redesigns two existing modules — trajectory (per-delegation → per-PHASE) and agent-work-contracts (passive records → active governance with runtime enforcement). The current codebase has 10 files across both modules (4 trajectory + 6 contract), 54 tests that need complete rewriting, and 2 tool registrations that need updating. The hook infrastructure for contract enforcement (`tool.execute.before`) and system prompt injection (`experimental.chat.system.transform`) already exists and is well-patterned. The primary technical challenge is wiring contract enforcement into the existing `tool-before-guard.ts` chain without breaking the session-tracker's proactive child discovery, and designing the 5-state trajectory lifecycle machine with both automatic (event-based) and manual (tool-call) transitions.

**Primary recommendation:** Implement in 3 layers — (1) types + schemas first (foundation), (2) operations + lifecycle (business logic), (3) hooks + tools + system prompt injection (integration). Each layer is independently testable.

### Pattern 1: Phase-Level Trajectory Model

**What:** Trajectory records represent phase lifecycles, not individual delegations. One trajectory per phase with ID format `traj-phase-{N}`.

**When to use:** When orchestrator starts a new phase, it calls `createPhaseTrajectory(phaseNumber, rootSessionId)`.

**Key type changes:**
```typescript
// Source: current types.ts analysis

// AFTER: 5 states
export type TrajectoryStatus = "planning" | "executing" | "verifying" | "completed" | "closed"

// NEW: Phase-level trajectory record extends existing record
export type PhaseTrajectoryRecord = TrajectoryRecord & {
  phaseNumber: number          // Phase number from ROADMAP
  phaseName?: string           // Optional human-readable phase name

// Jump link format
```

**ID format convention:**
- Phase trajectory: `traj-phase-25.5` (phase number from ROADMAP)
- Delegation trajectory: `traj-{childSessionID}` (linked via `parentTrajectoryId`)

### Pattern 2: Lifecycle State Machine (5 States)

**What:** Trajectory transitions through 5 states with both automatic (event-based) and manual (tool-call) transitions.

**State machine:**

```
planning ──→ executing ──→ verifying ──→ completed ──→ closed

    └──(cancel)───┴──(cancel)───┘
```

### Pattern 4: Contract Enforcement Hook

**What:** `tool.execute.before` hook checks active contract's `allowedSurfaces` before tool execution. Blocks with hard error if file not in scope.

**Integration point:** `src/hooks/transforms/tool-before-guard.ts` — add as third step in chain after toolGuard and sessionTracker.

**Key insight from SDK types:**
```typescript
// Source: @opencode-ai/plugin SDK index.d.ts line 234-240
"tool.execute.before"?: (input: {
    tool: string;      // Tool name (e.g., "write", "edit")
    sessionID: string;  // Session executing the tool
    callID: string;     // Unique call identifier
}, output: {
    args: any;          // Tool arguments (contains file paths)
}) => Promise<void>;
```

**Enforcement algorithm:**
1. Look up active contract for the session's agent (via `sessionID` → contract store)
2. If no active contract → allow (D-26: no contract = no restriction)
3. If contract exists, extract file paths from `output.args` (tool-specific)
4. Check each file path against `contract.scope.allowedSurfaces`
5. If any path outside scope → throw `[Harness] contract violation: agent {name} not allowed to modify {file}. Allowed: {surfaces}`

**File path extraction by tool name:**
| Tool | Path field | Notes |
|------|-----------|-------|
| `write` | `args.filePath` | Absolute path |
| `edit` | `args.filePath` | Absolute path |
| `bash` | N/A | Cannot enforce (arbitrary commands) |
| `glob` | N/A | Read-only, skip |
| `read` | N/A | Read-only, skip |

**Agent name resolution:** The hook receives `sessionID` but not agent name. Resolution path:
1. Check in-memory delegation state (`src/shared/state.ts` → `getDelegationMeta(sessionID)`)
2. If not found, check delegation-persistence records
3. If agent name found, look up active contract by agent name

**Performance consideration:** Contract lookup on every tool call. Cache active contracts in memory (agent's discretion area per CONTEXT.md).

## Common Pitfalls

### Pitfall 1: Agent Name Resolution in Hook
**What goes wrong:** The `tool.execute.before` hook receives `sessionID` but not agent name. Contract lookup requires agent name.
**Why it happens:** OpenCode SDK hook signature only provides `{ tool, sessionID, callID }`.
**How to avoid:** Build a sessionID → agent name resolution path:
1. First check in-memory delegation state (`getDelegationMeta(sessionID)?.agent`)
2. Fall back to delegation-persistence records
3. If no agent found, skip enforcement (no contract = no restriction per D-26)
**Warning signs:** Hook silently skipping enforcement for sessions it can't resolve.

### Pitfall 2: File Path Extraction Across Tool Types
**What goes wrong:** Different tools have different arg shapes for file paths. `write` uses `filePath`, `edit` uses `filePath`, `bash` has no extractable path.
**Why it happens:** No standardized file path field across tool schemas.
**How to avoid:** Create a `extractFilePaths(toolName: string, args: Record<string, unknown>): string[]` helper that handles known file-modifying tools. Return empty array for unknown/read-only tools.
**Warning signs:** Enforcement triggering on read-only tools or missing enforcement on write tools.

### Pitfall 3: Trajectory Ledger Corruption on Concurrent Writes
**What goes wrong:** Two sessions writing to the same trajectory ledger simultaneously could corrupt the file.
**Why it happens:** File-based persistence with read-modify-write cycle.
**How to avoid:** The existing ledger pattern already handles this via atomic `writeFileSync`. Node.js single-threaded event loop prevents true concurrent writes within a single process. Cross-process writes are rare (different OpenCode instances on same project).
**Warning signs:** Corrupt ledger quarantined files appearing in `.hivemind/state/`.

### Pitfall 4: Lifecycle State Transition Race Conditions
**What goes wrong:** Event-based automatic transitions could conflict with manual orchestrator transitions.
**Why it happens:** Both paths can trigger the same transition.
**How to avoid:** Idempotent transitions — if already in target state, no-op instead of error. Validate against `ALLOWED_TRANSITIONS` before applying.
**Warning signs:** `[Harness] invalid trajectory transition` errors in logs during normal operation.

### Pitfall 5: Pressure Removal Breaking Existing Callers
**What goes wrong:** Removing pressure fields from `CreateAgentWorkContractInput` breaks existing tool callers that pass `pressureScore`, `pressureTier`, `pressureApproved`.
**Why it happens:** Tool schema still accepts these fields (backward compat per D-43/D-44).
**How to avoid:** Keep fields in Zod schema as `.optional()`, strip them in the operation layer before processing. The tool accepts them but the operation ignores them.
**Warning signs:** Tool calls failing with validation errors after schema changes.

### Pitfall 6: Test Rewrite Scope Creep
**What goes wrong:** Rewriting 54 tests turns into adding new tests for edge cases not in the original scope.
**Why it happens:** New design reveals untested scenarios.
**How to avoid:** Rewrite existing tests 1:1 for new API first. Flag new test scenarios as follow-up items. The SPEC says "all 54 rewritten, 0 skipped" — not "add new tests."
**Warning signs:** Test count growing beyond 54 during rewrite phase.

### Contract Enforcement in tool-before-guard
```typescript
// Source: current tool-before-guard.ts pattern, extended
export function createContractEnforcementHook(deps: {
  contractStore: { getActiveContractByAgent: (agent: string) => AgentWorkContract | undefined }
  resolveAgentName: (sessionID: string) => string | undefined
}): (input: unknown, output: unknown) => Promise<void> {
  return async (input, output) => {
    const inputRecord = input as Record<string, unknown>
    const toolName = inputRecord.tool as string
    const sessionID = inputRecord.sessionID as string

    // Only enforce on file-modifying tools
    const filePaths = extractFilePaths(toolName, output)
    if (filePaths.length === 0) return

    // Resolve agent name from session
    const agentName = deps.resolveAgentName(sessionID)
    if (!agentName) return  // No agent = no contract = no restriction (D-26)

    // Look up active contract
    const contract = deps.contractStore.getActiveContractByAgent(agentName)
    if (!contract) return  // No contract = no restriction (D-26)

    // Check each file path against allowed surfaces
    const allowed = contract.scope.allowedSurfaces
    for (const filePath of filePaths) {
      if (!isPathAllowed(filePath, allowed)) {
        throw new Error(
          `[Harness] contract violation: agent ${agentName} not allowed to modify ${filePath}. Allowed: ${allowed.join(", ")}`
        )
      }
    }
  }
}

function extractFilePaths(toolName: string, output: unknown): string[] {
  const args = (output as Record<string, unknown>)?.args as Record<string, unknown> | undefined
  if (!args) return []
  
  switch (toolName) {
    case "write":
    case "edit":
      return args.filePath ? [String(args.filePath)] : []
    default:
      return []  // Read-only or unknown tools
  }
}

function isPathAllowed(filePath: string, allowedSurfaces: string[]): boolean {
  return allowedSurfaces.some(surface => 
    filePath.startsWith(surface) || filePath.includes(surface)
  )
}
```

### Progressive Disclosure Implementation
```typescript
// Source: current traverseTrajectory, adapted
export type TrajectoryDepth = "summary" | "detailed" | "full"

export type TrajectorySummary = {
  id: string
  status: TrajectoryStatus
  eventCount: number
  checkpointCount: number
  durationMs: number
  phaseName?: string
}

export function traverseTrajectory(input: {
  projectRoot: string
  rootSessionId?: string
  sessionId?: string
  trajectoryId?: string
  depth?: TrajectoryDepth  // NEW: default "full" for backward compat
}): TrajectoryTraversal | { summaries: TrajectorySummary[] } {
  const ledger = readTrajectoryLedger(input.projectRoot)
  const all = Object.values(ledger.trajectories)
  const selected = filterTrajectories(all, input)
  const depth = input.depth ?? "full"

  if (depth === "summary") {
    return { summaries: selected.map(projectSummary) }
  }
  
  if (depth === "detailed") {
    return {
      trajectories: selected.map(projectDetailed),
      edges: computeEdges(selected),
    }
  }
  
  // depth === "full" — return everything (current behavior)
  return { trajectories: selected, edges: computeEdges(selected) }
}

function projectSummary(record: TrajectoryRecord): TrajectorySummary {
  return {
    id: record.id,
    status: record.status,
    eventCount: record.events.length,
    checkpointCount: record.checkpoints.length,
    durationMs: record.updatedAt - record.createdAt,
  }
}

function projectDetailed(record: TrajectoryRecord): TrajectoryRecord {
  return {
    ...record,
    evidenceRefs: [],  // Strip full evidence refs in detailed mode
    checkpoints: record.checkpoints.map(cp => ({
      ...cp,
      evidenceRefs: [],  // Strip checkpoint evidence refs
    })),
  }
}
```

## Open Questions

1. **Agent name resolution robustness**
   - What we know: `getDelegationMeta(sessionID)` in `src/shared/state.ts` provides delegation metadata including agent name for active delegations
   - What's unclear: Whether all sessions that create contracts will have delegation records (e.g., root sessions)
   - Recommendation: Fall back gracefully — no agent name = no enforcement (per D-26)

2. **Trajectory event types for automatic transitions**
   - What we know: Events have `eventType` string field (D-32 to D-34 mention "first delegation event", "execution complete event", "verification pass event")
   - What's unclear: Exact event type strings that trigger automatic transitions
   - Recommendation: Define constants in `types.ts` — `EVENT_DELEGATION_STARTED`, `EVENT_EXECUTION_COMPLETE`, `EVENT_VERIFICATION_PASS`

3. **Contract enforcement scope for non-file tools**
   - What we know: `allowedSurfaces` checks file paths for `write`/`edit` tools
   - What's unclear: Should enforcement apply to non-file tools (e.g., `bash` commands that could modify files)?
   - Recommendation: Start with file-modifying tools only (`write`, `edit`). Document that `bash` is not enforced. Future phases can add command pattern matching.
