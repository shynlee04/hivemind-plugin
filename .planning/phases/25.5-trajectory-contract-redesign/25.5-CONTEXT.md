**Gathered:** 2026-05-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Redesign trajectory as a per-PHASE orchestrator mindmap (not per-delegation) and redesign agent-work-contracts as active governance tools with runtime enforcement via `tool.execute.before` hook (not passive records). Remove all pressure integration from both modules.

**Critical constraint:** "The harness can only OBSERVE and INJECT, not CONTROL." Trajectory and contracts work through hooks and system prompt injection, not by forcing agents to call tools.

**Pressure system removed:** All pressure integration is removed from trajectory/contracts. The pressure module (runtime-pressure) exists independently but is NOT integrated with trajectory or contracts.

<decisions>
## Implementation Decisions

### Trajectory Scope (U1: Both Layered)
- **D-01:** Trajectory is PER-PHASE (not per-delegation). One trajectory per phase.
- **D-02:** Phase-level trajectory aggregates delegation-level records.
- **D-03:** Delegation-level records linked to phase trajectory via jump links (not data duplication).
- **D-04:** Phase trajectory ID format: `traj-phase-{N}`.
- **D-05:** Delegation trajectory ID format: `traj-{childSessionID}` (linked to phase trajectory via `parentTrajectoryId`).

### Contract Enforcement (U2: Enforce + Document)
- **D-06:** Contracts enforce tool restrictions at runtime via `tool.execute.before` hook.
- **D-07:** `scope.allowedSurfaces` is checked before tool execution ‚Äî agents violating scope are blocked.
- **D-08:** Contracts also document scope for humans (export as Markdown).

### Pressure Removal (U3: Remove Everything)
- **D-09:** Remove pressure check from `createAgentWorkContract()` in `operations.ts`.
- **D-10:** Remove pressure integration from contract lifecycle transitions.
- **D-11:** Remove `pressureScore`, `pressureTier`, `pressureApproved` from contract creation input.
- **D-12:** Remove `detectRuntimePressure()` call from contract operations.
- **D-13:** Keep `runtime-pressure` module as independent module (not integrated with contracts).

### CQRS Pragmatism (U4: Acceptable)
- **D-14:** Hooks writing to trajectory/contracts is acceptable. CQRS boundary is a guideline, not hard rule.

### Trajectory Creation (U5: Orchestrator-Driven)
- **D-15:** Trajectory is created when orchestrator explicitly calls `createPhaseTrajectory(phaseNumber, rootSessionId)` via the trajectory tool.
- **D-16:** NOT automatic via hooks ‚Äî the orchestrator decides when a phase starts.
- **D-17:** After creation, events trigger agents to use trajectory tools for updates, queries, reads, modifications.
- **D-18:** The trajectory tool is specifically designed for CRUD operations on the phase mindmap.
- **D-19:** Agents are instructed (via system prompt injection) to call trajectory tools when phase-relevant events occur.

### Trajectory vs Delegation Persistence (U6: Link via Jump Links)
- **D-20:** Trajectory links TO delegation-persistence via jump links.
- **D-21:** No data duplication between trajectory and delegation-persistence.
- **D-22:** Jump links use `delegation:{childSessionID}` format (standardized).

### Hook Wiring ‚Äî tool.execute.before Enforcement (GA-1)
- **D-23:** Hard block (throw error), not best-effort (log warning).
- **D-24:** Error format: `[Harness] contract violation: agent {name} not allowed to modify {file}. Allowed: {allowedSurfaces.join(", ")}`.
- **D-25:** The hook checks the agent's active contract before each tool execution.
- **D-26:** If no active contract exists, tool execution proceeds (no contract = no restriction).

### Progressive Disclosure ‚Äî Depth Parameter (GA-2)
- **D-27:** All 3 levels required now (summary, detailed, full).
- **D-28:** `summary` returns: status + event count + phase name + duration.
- **D-29:** `detailed` returns: event types + summaries + delegation chain.
- **D-30:** `full` returns: all evidence refs + checkpoints + jump links + full event data.

### Lifecycle State Transitions (GA-3)
- **D-31:** Both automatic (event-based) and manual (orchestrator tool call) transitions supported.
- **D-32:** `planning‚Üíexecuting`: triggered by first delegation event OR orchestrator explicit call.
- **D-33:** `executing‚Üíverifying`: triggered by execution complete event OR orchestrator explicit call.
- **D-34:** `verifying‚Üícompleted`: triggered by verification pass event OR orchestrator explicit call.
- **D-35:** `completed‚Üíclosed`: triggered by orchestrator explicit call ONLY (requires explicit closure).

### Phase Spawn Detection (GA-4)
- **D-36:** The orchestrator explicitly calls `createPhaseTrajectory(phaseNumber, rootSessionId)` via the trajectory tool.
- **D-37:** NOT automatic via hooks ‚Äî the orchestrator decides when a phase starts.
- **D-38:** The trajectory tool is available to the orchestrator via system prompt injection.
- **D-39:** Phase number comes from the orchestrator's knowledge of the ROADMAP.

### System Prompt Injection (GA-5)
- **D-40:** Use `experimental.chat.system.transform` hook to inject trajectory tool instructions.
- **D-41:** Instructions tell agents: "When managing a phase, call createPhaseTrajectory to start tracking. Use addTrajectoryEvent to record milestones. Use traverseTrajectory to review progress."
- **D-42:** Instructions tell agents: "When receiving a contract, check allowedSurfaces before modifying files. Stay within scope."

### Backward Compatibility
- **D-43:** Old schema fields (`pressureScore`, `pressureTier`, `pressureApproved`) silently ignored, not rejected.
- **D-44:** Existing callers using old fields continue to work without error.

### Agent's Discretion
- Exact error message wording within the `[Harness] contract violation` format
- System prompt injection text refinement (wording, placement)
- Performance optimization for hook checks (caching active contracts)

</decisions>

<specifics>
## Specific Ideas

- Trajectory is the orchestrator's mindmap ‚Äî high-level view of phase lifecycle
- Progressive disclosure: summary ‚Üí detailed ‚Üí full depth levels (all 3 required now)
- Contracts enforce boundaries: `allowedSurfaces` checked at runtime via hard block
- Pressure is completely removed from trajectory/contracts
- Jump links connect trajectory events to delegation records without duplication
- Session-tracker is the primary data source for delegation-level details
- Orchestrator explicitly creates phase trajectories (not automatic via hooks)
- System prompt injection via `experimental.chat.system.transform` instructs agents on tool usage

</specifics>
