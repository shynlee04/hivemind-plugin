# Phase 25.5: Trajectory + Contract Redesign — Specification

**Created:** 2026-05-30
**Ambiguity score:** 0.07 (gate: ≤ 0.20)
**Requirements:** 12 locked

## Goal

Redesign trajectory as a per-PHASE orchestrator mindmap (not per-delegation) and redesign agent-work-contracts as active governance tools with runtime enforcement via `tool.execute.before` hook (not passive records). Remove all pressure integration from both modules.

## Background

Phase 25 delivered trajectory and contract modules, but the design is flawed per deep-architecture-research-2026-05-29.md:

- **Trajectory is per-session/per-delegation** — should be per-PHASE (one trajectory per phase lifecycle)
- **Contracts are passive records** — should enforce `scope.allowedSurfaces` at runtime via `tool.execute.before` hook
- **Pressure integration is baked into contracts** — user explicitly requested removal ("complete product of hallucination, get them out of the system")
- **Only 2 lifecycle states** (active/closed) — needs 5 states (planning→executing→verifying→completed→closed)
- **No progressive disclosure** — orchestrator needs summary/detailed/full depth levels
- **No jump links** to delegation-persistence — data duplication risk

Current codebase:
- `src/task-management/trajectory/` — 4 files (types, store-operations, ledger, index)
- `src/features/agent-work-contracts/` — 6 files (types, operations, lifecycle, bounds, store, index)
- `src/tools/hivemind/hivemind-trajectory.ts` — CRUD tool registered in plugin.ts
- `src/tools/hivemind/hivemind-agent-work.ts` — create/export tools registered in plugin.ts
- 34 trajectory tests + 20 contract tests (all for old design)

## Requirements

1. **Phase trajectory creation**: Trajectory is created when a phase spawns (per-PHASE, not per-delegation).
   - Current: Trajectory is per-session/per-delegation — no phase-level concept exists
   - Target: One trajectory per phase, created by hook on phase spawn, ID format `traj-phase-{N}`
   - Acceptance: Creating a phase spawns a trajectory with ID `traj-phase-{N}` in the ledger

2. **Delegation trajectory linking**: Delegation records are linked to phase trajectory via jump links.
   - Current: No linking mechanism between delegation records and trajectory
   - Target: Delegation trajectories have `parentTrajectoryId` pointing to phase trajectory
   - Acceptance: Delegation trajectory record has `parentTrajectoryId` set to `traj-phase-{N}`

3. **Contract enforcement**: `tool.execute.before` hook checks `scope.allowedSurfaces` and blocks violations.
   - Current: Contracts are passive records — no runtime enforcement
   - Target: `tool.execute.before` hook checks active contract's `allowedSurfaces` before tool execution; blocks with error if file not in scope
   - Acceptance: Attempting to modify file outside `allowedSurfaces` returns `[Harness] contract violation` error

4. **Contract evidence gating**: `completeContract` requires proof when `requiredProof` is non-empty.
   - Current: `completeContract` accepts optional proof, no enforcement
   - Target: `completeContract` throws if `requiredProof` is non-empty and no proof provided
   - Acceptance: Calling `completeContract` with empty proof when `requiredProof` is non-empty throws error

5. **Trajectory CRUD tool**: Orchestrator can create/update/query/read/modify phase trajectory.
   - Current: `hivemind-trajectory` tool exists with inspect/traverse/attach/checkpoint/event/close
   - Target: Same operations work for phase-level trajectory (not per-delegation)
   - Acceptance: Tool operations work with `traj-phase-{N}` IDs

6. **Progressive disclosure**: `traverseTrajectory` supports depth parameter (summary/detailed/full).
   - Current: `traverseTrajectory` returns full data only
   - Target: `traverseTrajectory` accepts `depth` parameter, returns appropriate detail level
   - Acceptance: `summary` returns phase status + event count, `detailed` adds event types + summaries, `full` returns all evidence refs

7. **Trajectory lifecycle states**: 5 states — planning→executing→verifying→completed→closed.
   - Current: `TrajectoryStatus` is `"active" | "closed"` (2 states)
   - Target: 5 states: `planning`, `executing`, `verifying`, `completed`, `closed`
   - Acceptance: Trajectory transitions through states via events (automatic) or manual tool calls

8. **Pressure removal**: All pressure integration removed from trajectory and contract operations.
   - Current: `createAgentWorkContract` calls `detectRuntimePressure`, has `pressureScore`/`pressureTier`/`pressureApproved` fields
   - Target: No pressure calls in trajectory or contract operations; pressure module remains independent
   - Acceptance: `createAgentWorkContract` has no pressure logic; `detectRuntimePressure` import removed from operations.ts

9. **Cross-session persistence**: Trajectory readable/writable from any session with same project root.
   - Current: Trajectory stored in `.hivemind/state/trajectory-ledger.json` (already cross-session)
   - Target: Same persistence model verified for phase-level trajectories
   - Acceptance: Different sessions can read/write same phase trajectory via `projectRoot`

10. **Jump link format**: Standardized string references between trajectory and delegation-persistence.
    - Current: No standardized format
    - Target: `delegation:{childSessionID}` format for jump links in trajectory events
    - Acceptance: Jump links use consistent `delegation:{childSessionID}` format

11. **Backward compatibility**: Old schema fields silently ignored.
    - Current: `createAgentWorkContract` accepts `pressureScore`/`pressureTier`/`pressureApproved`
    - Target: Fields silently ignored, no error on existing callers
    - Acceptance: Calling `createAgentWorkContract` with pressure fields succeeds without error

12. **Test rewrite**: All existing trajectory/contract tests rewritten for new design.
    - Current: 34 trajectory tests + 20 contract tests for old per-delegation design
    - Target: All tests rewritten to match new per-PHASE design with lifecycle states
    - Acceptance: `npm test` passes with all rewritten tests, 0 skipped

## Boundaries

**In scope:**
- Trajectory types redesign (5 lifecycle states, phase-level ID format)
- Trajectory store operations update (phase-level CRUD)
- Trajectory progressive disclosure (summary/detailed/full depth parameter)
- Contract operations redesign (remove pressure, add enforcement)
- Contract enforcement hook (`tool.execute.before` surface check)
- Contract evidence gating (requiredProof enforcement)
- Jump link format standardization
- Tool registration updates in plugin.ts
- All 54 existing tests rewritten (34 trajectory + 20 contract)
- Typecheck clean

**Out of scope:**
- Phase spawn hook implementation — that is Phase 25.1 (Task Tool Integration)
- Delegation-persistence changes — existing delegation records are unchanged
- Pressure module changes — `runtime-pressure` remains independent
- Agent profile changes — `.opencode/agents/` unchanged
- Session-tracker changes — existing session-tracker is unchanged
- Documentation updates — that is Phase 26.1 (Artifact Naming)

## Constraints

- **CQRS pragmatism**: Hooks writing to trajectory/contracts is acceptable (D-14)
- **Observe/inject, not control**: Trajectory and contracts work through hooks and system prompt injection, not by forcing agents to call tools
- **No data duplication**: Trajectory links TO delegation-persistence via jump links, does not copy data
- **Backward compatibility**: Old schema fields silently ignored, not rejected
