# Phase 24.3.2: Revamp execute-slash-command — Research

**Researched:** 2026-05-26
**Domain:** OpenCode plugin tooling, command dispatch, namespace routing, CQRS
**Confidence:** HIGH

## Summary

Phase 24.3.2 revamps the `execute-slash-command` tool by fixing 6 critical flaws, adding namespace routing with a hybrid facade (public tool → internal resolve + dispatch), and adopting ToolResponse error handling. The phase encompasses 12 SPEC requirements and 6 Governance Agreement (GA) decisions from CONTEXT.md.

**The core architectural shift is from a monolith with thrown errors to a hybrid facade:** `execute-slash-command.ts` stays as the public tool surface, `resolve-command.ts` and `dispatch-command.ts` are extracted as internal modules, and namespace routing is grafted onto the existing command-engine read-side. No new packages are needed — the full stack already exists in the project.

**Primary recommendation:** Refactor in three ordered waves — (1) ToolResponse + error classification + execution tracking (fixes 5 of 6 flaws), (2) hybrid facade extraction + namespace routing (fixes the remaining flaw + new capability), (3) move session tools + narrow import + comprehensive tests.

### Pattern 1: ToolResponse Error Flow (new for this tool)
**What:** All errors return via `error()` ToolResponse envelope instead of throwing, with classified error types.
**When to use:** Every error path in execute-slash-command and its extracted modules.
**Example:**
```typescript
// Source: Existing pattern in command-engine.ts, replicated for this tool
import { error } from '../../shared/tool-response.js';

// Instead of: throw new HarnessError('Command not found');
// Return:
return renderToolResult(
  error('not_found', {
    command: commandName,
    namespace: namespace || undefined,
    searched: availableCommands,
  })
);
```

### Pattern 2: Hybrid Facade Extraction
**What:** Public tool is a thin orchestrator; real work lives in focused internal modules.
**When to use:** When a tool has grown to cover multiple responsibilities.
**Pattern:**
```typescript
// execute-slash-command.ts — thinned to orchestration
export function createExecuteSlashCommandTool(deps: ToolDeps) {
  return tool(
    { ... },
    async (input, ctx) => {
      const resolved = await resolveCommand(input, deps);       // extracted
      const dispatched = await dispatchCommand(resolved, deps); // extracted
      return renderToolResult(dispatched);
    }
  );
}
```

### Pattern 3: Namespace Resolution + Fallback
**What:** exact match → namespace-scoped exact → fuzzy within namespace → global fuzzy → error.
**When to use:** Lookup flow for namespace-aware commands.
**Pattern:**
```typescript
// resolve-command.ts
function resolveCommand(args: { name, namespace? }, deps): ResolvedCommand {
  const engine = deps.commandEngine;
  if (namespace) {
    // Try exact within namespace first
    const nsCmd = engine.resolveCommandNamespace(name, namespace);
    if (nsCmd) return nsCmd;
  }
  // Fallback: existing resolution (exact → fuzzy → substring)
  const fallback = engine.resolveCommand(name, fallbackStrategy);
  return fallback;
}
```

### Anti-Patterns to Avoid
- **Throwing errors instead of ToolResponse:** All 6 error types must return via `error()`, not thrown exceptions.
- **Namespace as hard block:** If a command has no namespace field, it must still resolve via existing fallback (zero breakage — GA-6).
- **Synchronous dispatch for agent override paths:** The 50ms setTimeout defer must be preserved for synthetic/subtask paths (SPEC REQ-12).
- **Duplicating command-engine logic:** Namespace resolution belongs on command-engine read-side, not in the tool module.
- **Hardcoding agent list:** `AGENT_DEFAULTS` and `AGENT_TOOLS` live in `src/plugin.ts`, not hardcoded in the tool.

## Common Pitfalls

### Pitfall 1: Breaking 50ms Deferred Dispatch
**What goes wrong:** Removing the `setTimeout(..., 50)` defer makes the tool block until session.prompt() completes, breaking the TUI synchronous return contract.
**Why it happens:** The async/await refactoring temptation — "just await the prompt directly."
**How to avoid:** SPEC REQ-12 mandates: only synthetic/subtask paths use setTimeout; TUI path returns immediately (tool returns after injection). Keep the two paths separate.
**Warning signs:** Test suite shows timeout failures for agent-override test cases.

### Pitfall 2: Namespace as Hard Block
**What goes wrong:** Adding namespace resolution but not falling back to legacy resolution, breaking all commands that lack `namespace:` frontmatter.
**Why it happens:** "New system, new rules" mindset rather than GA-6's explicit fallback.
**How to avoid:** The resolve flow must always be `resolve(namespace?)` → if namespace and miss → `resolve()` with legacy fallback. Never return `not_found` without trying the fallback first.
**Warning signs:** Tests for legacy (non-namespaced) commands start failing.

### Pitfall 3: Missing delegation_id for Subtask Paths
**What goes wrong:** Delegation is correctly dispatched but the ToolResponse doesn't include `delegation_id` for parent sessions to track.
**Why it happens:** The delegation_id is available inside DelegationManager but not threaded back to the tool response.
**How to avoid:** After DelegationManager.dispatch() resolves, extract the delegation ID and include it in the ToolResponse metadata.
**Warning signs:** Session tracker shows delegations without parent linkage.

### Pitfall 4: Narrow Import Migration Breaks
**What goes wrong:** Changing from `@opencode-ai/plugin` to `@opencode-ai/plugin/tool` import causes type issues.
**Why it happens:** Different export surfaces between the two entry points.
**How to avoid:** Verify the narrow import exports exactly what the tool needs. Read the existing 17 tools that already use narrow imports for the pattern.
**Warning signs:** TypeScript compiler errors on `tool()` or `ToolConfig` types.

### Pitfall 5: Overlapping Wave Dependencies
**What goes wrong:** Wave 2 (namespace routing) depends on Wave 1 (ToolResponse), and Wave 3 (comprehensive tests) depends on both — but tests are written in parallel and fail against incomplete code.
**Why it happens:** Writing comprehensive tests at the end instead of incrementally per wave.
**How to avoid:** Each wave adds/updates tests for its own changes. Wave 1 adds ToolResponse + error tests. Wave 2 adds namespace routing tests. Wave 3 adds edge-case and integration tests.

### resolveCommandNamespace — Read-Side Query
```typescript
// Source: Pattern from existing resolveCommand in command-engine
// NEW method — CQRS compliant (read-only)

class CommandEngine {
  // Existing
  resolveCommand(name: string, strategy: ResolutionStrategy): Command | null;

  // NEW — namespace-scoped resolution
  resolveCommandNamespace(name: string, namespace: string): Command | null {
    const commands = this.loadPrimitives(); // cached, read-only
    return commands.find(
      (cmd) => cmd.name === name && cmd.frontmatter?.namespace === namespace
    ) ?? null;
  }
}
```

### Dispatch Flow — Subtask vs User-Session
```typescript
// Source: Existing pattern from execute-slash-command.ts
// Preserve the two-path dispatch architecture

async function dispatchCommand(
  resolved: ResolvedCommand,
  deps: ToolDeps
): Promise<DispatchResult> {
  const { sessionApi, delegationManager } = deps;

  if (resolved.subtask && resolved.agent) {
    // Synthetic/subtask path — 50ms defer
    const delegationId = await delegationManager.dispatch({
      agent: resolved.agent,
      prompt: resolved.expandedBody,
    });
    return { path: 'subtask', delegationId };
  }

  // TUI path — synchronous (tool returns immediately after injection)
  sessionApi.appendToSession(resolved.expandedBody, {
    agent: resolved.agent,
  });
  return { path: 'user-session', dispatchId: crypto.randomUUID() };
}
```

## Open Questions

1. **Exact namespace resolution strategy — parallel vs sequential**
   - What we know: CONTEXT GA-6 requires legacy fallback; `resolveCommandNamespace()` is read-only.
   - What's unclear: When a user types a bare command name without namespace prefix, should the resolver try all namespaces in parallel? Or try current namespace, then exact, then fuzzy?
   - Recommendation: Start with sequential (check namespace-scoped exact → global exact → global fuzzy → substring). This minimizes behavior change. This is at the agent's discretion per CONTEXT.

2. **Test infrastructure — do we need new fixtures?**
   - What we know: Existing `execute-slash-command.test.ts` has 9 tests and 435 LOC with fixtures.
   - What's unclear: Whether the existing fixtures support namespace-annotated commands or need extension.
   - Recommendation: Inspect test fixtures during Wave 1. Add namespace-aware fixture helpers in Wave 2 before writing namespace routing tests.

3. **Session tools move — what is the dedicated surface?**
   - What we know: SPEC REQ-7 mandates moving session tools out of execute-slash-command.
   - What's unclear: Where they move to — new `src/tools/session/` directory? Or integrated into `src/task-management/`?
   - Recommendation: Defer to planner discretion based on existing tool categorization patterns.
