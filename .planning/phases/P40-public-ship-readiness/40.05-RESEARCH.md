# Phase P40.05: Fix State Persistence & Module-Level Singletons — Research

**Researched:** 2026-05-31
**Domain:** State persistence, module-level caching, process lifecycle management
**Confidence:** HIGH

## Summary

This phase addresses three interrelated issues in the Hivemind harness's state management layer. Through source code analysis of the continuity store, config subscriber, plugin composition root, and process signal handlers, the research reveals a nuanced picture: the `atomic_commit:false` data-loss risk is **partially mitigated** by existing `flushAllStores()` on process exit handlers (L485-498 of continuity/index.ts), but the mitigation is fragile — it depends on Node.js process signal delivery, does NOT work during test runs (guarded by `VITEST` env check), and has zero test coverage. The `CANONICAL_STATE_DIR` issue described in the original plan is **already fixed** — the code uses `getCanonicalStateDir()` as a function (not a module-level constant) since at least the current codebase revision. The config subscriber singleton issue is **already fixed** — the cache is a `Map<string, HivemindConfigs>` keyed by project root. However, `persistStore()` calls `getCachedConfig()` with **no project root argument** (L329), which returns defaults when cache is cold — this is the real bug.

**Primary recommendation:** Fix `persistStore()` to accept and pass through `projectRoot` so `getCachedConfig()` can look up the correct cache entry. Add a dedicated flush test. Remove the `VITEST` guard from process handlers and instead control registration via explicit initialization. Do NOT remove the `atomic_commit:false` code path — it has valid use cases (batch mode during high-frequency delegation updates).

### Pattern 1: Deferred Path Resolution
**What:** `getCanonicalStateDir()` is already a function (not a module-level constant), resolving paths lazily per call.
**When to use:** Already implemented — L23-26 of continuity/index.ts.
**Example:**
```typescript
// VERIFIED: src/task-management/continuity/index.ts L23-26
export function getCanonicalStateDir(projectRoot?: string): string {
  const root = projectRoot || process.cwd()
  return resolve(root, ".hivemind", "state")
}
```

### Pattern 2: Atomic Write via Temp+Rename
**What:** `writeStoreToDisk()` writes to a temp file first, then renames atomically.
**When to use:** Already implemented — prevents corrupt reads on crash mid-write.
**Example:**
```typescript
// VERIFIED: src/task-management/continuity/index.ts L312-322
function writeStoreToDisk(filePath: string, store: ContinuityStoreFile): void {
  mkdirSync(dirname(filePath), { recursive: true })
  const tmpFile = `${filePath}.${process.pid}.${randomUUID()}.tmp`
  const redactedStore = redactBoundaryFields(store, { ... })
  writeFileSync(tmpFile, `${JSON.stringify(redactedStore, null, 2)}\n`, "utf8")
  renameSync(tmpFile, filePath)
}
```

### Pattern 3: Process Exit Flush Guard
**What:** `flushAllStores()` registered on `exit`, `SIGINT`, `SIGTERM`.
**When to use:** Already implemented — but guarded by `!process.env.VITEST` and has no tests.
**Example:**
```typescript
// VERIFIED: src/task-management/continuity/index.ts L474-499
export function flushAllStores(): void {
  const caches = getAllStoreCaches()
  for (const [filePath, store] of caches.entries()) {
    try { writeStoreToDisk(filePath, store) }
    catch (err) { console.error(`[Harness] Failed to flush: ${err}`) }
  }
}

if (typeof process !== "undefined" && !process.env.VITEST) {
  process.on("exit", () => { flushAllStores() })
  process.on("SIGINT", () => { flushAllStores(); process.exit(130) })
  process.on("SIGTERM", () => { flushAllStores(); process.exit(143) })
}
```

### Anti-Patterns to Avoid
- **Removing `atomic_commit:false` entirely:** The batch mode has valid performance use cases during rapid delegation state updates. The right fix is ensuring a flush path exists, not removing the toggle.
- **Adding shutdown hooks to the OpenCode plugin SDK:** The SDK has no `dispose`/`shutdown` hook (verified from `@opencode-ai/plugin` index.d.ts — `Hooks` interface has `event`, `config`, `tool`, `chat.message`, `tool.execute.before/after`, `system.transform`, `shell.env` only). Process signal handlers are the correct approach.
- **Module-level `const` for paths:** Already fixed. The original plan description mentions `CANONICAL_STATE_DIR` as a module-level constant, but it's actually a function `getCanonicalStateDir()` that takes `projectRoot` parameter.

## Common Pitfalls

### Pitfall 1: `getCachedConfig()` called without projectRoot returns defaults
**What goes wrong:** `persistStore()` at L329 calls `const config = getCachedConfig()` with no argument. If the cache is cold (first call, or after invalidation), `getCachedConfig()` returns `getDefaultConfigs()` which has `atomic_commit: true` — so this bug is masked in the default case. But if the user's config has `atomic_commit: false`, and the cache was invalidated between store load and persist, the default (true) would be returned, causing an unexpected write.
**Why it happens:** `getCachedConfig()` has a fallback path: if no projectRoot matches and no lastProjectRoot exists, it returns `getDefaultConfigs()`.
**How to avoid:** Pass `projectRoot` through from all callers of `persistStore()`.
**Warning signs:** Tests that mock `getCachedConfig` to return `atomic_commit: false` — those pass, but production code path is different because production calls `getCachedConfig()` without args.

### Pitfall 2: Process exit handlers skipped in test environment
**What goes wrong:** The `VITEST` guard at L485 means `flushAllStores()` is never registered during tests. This means tests cannot verify shutdown behavior.
**Why it happens:** The guard was added to prevent test interference with process signals.
**How to avoid:** Extract signal registration into an explicit `registerShutdownHandlers()` function that tests can call, rather than relying on module-level side effects.

### Pitfall 3: `flushAllStores()` has no test coverage
**What goes wrong:** The most critical safety mechanism — flushing state on exit — has zero tests. If it breaks, nobody notices until data loss occurs in production.
**Why it happens:** Tests use `vi.resetModules()` and `VITEST=true`, so the process handlers never fire.
**How to avoid:** Add explicit unit tests for `flushAllStores()` that mock `writeStoreToDisk` and verify it's called for all cached stores.

### Pitfall 4: Config subscriber cache can grow unbounded
**What goes wrong:** The `configCache` Map in subscriber.ts is never automatically pruned. In long-running processes that serve multiple projects, memory grows.
**Why it happens:** No LRU or max-size limit on the Map.
**How to avoid:** This is LOW severity — OpenCode typically serves one project per process. Not in scope for this phase but worth noting.

## Code Examples

### The Real Bug: `persistStore()` without projectRoot
```typescript
// VERIFIED: src/task-management/continuity/index.ts L324-338
function persistStore(projectRoot?: string): void {
  // BUG: getCachedConfig() called without projectRoot
  const config = getCachedConfig()  // ← returns defaults if cache cold
  const store = ensureStoreLoaded(projectRoot)
  store.updatedAt = Date.now()
  if (!config.atomic_commit) {
    return  // Skip disk write — state remains in memory for later flush
  }
  const filePath = getContinuityFile(projectRoot)
  writeStoreToDisk(filePath, store)
}
```

### The Fix: Pass projectRoot through
```typescript
function persistStore(projectRoot?: string): void {
  const config = projectRoot
    ? getCachedConfig(projectRoot)
    : getCachedConfig()
  const store = ensureStoreLoaded(projectRoot)
  store.updatedAt = Date.now()
  if (!config.atomic_commit) {
    return
  }
  const filePath = getContinuityFile(projectRoot)
  writeStoreToDisk(filePath, store)
}
```

### flushAllStores() — Already Exists but Untested
```typescript
// VERIFIED: src/task-management/continuity/index.ts L474-483
export function flushAllStores(): void {
  const caches = getAllStoreCaches()
  for (const [filePath, store] of caches.entries()) {
    try {
      writeStoreToDisk(filePath, store)
    } catch (err) {
      console.error(`[Harness] Failed to flush store at ${filePath} during process exit: ${err}`)
    }
  }
}
```

## Open Questions

1. **Is `atomic_commit:false` actually used in production?**
   - What we know: Default is `true`. Tests verify both paths work.
   - What's unclear: Whether any user config sets it to `false`.
   - Recommendation: Keep the toggle — it's a valid optimization for high-frequency updates.

2. **Should the VITEST guard be removed in favor of explicit registration?**
   - What we know: The guard prevents test coverage of the flush path.
   - What's unclear: Whether removing it would cause test flakiness.
   - Recommendation: Extract to explicit `registerShutdownHandlers()` and call it from plugin.ts init, not module scope.
