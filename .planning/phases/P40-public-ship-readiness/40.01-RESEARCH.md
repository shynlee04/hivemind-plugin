# Phase 40.01: Fix Fire-and-Forget Dispatch Paths - Research

**Researched:** 2026-05-30
**Domain:** Hivemind dispatch pipeline — execute-slash-command and dispatch-command modules
**Confidence:** HIGH

## Summary

Investigation of 3 fire-and-forget dispatch paths in the Hivemind harness reveals that all 3 are **intentional** deferred dispatch patterns using `setTimeout(fn, 50)`, originally designed to avoid reentrant deadlock when a tool returns while the same session is processing a prompt. However, the "fire-and-forget" behavior creates real reliability problems: dispatch failures are swallowed (only `console.error`), callers receive `{ success: true }` before the SDK call completes, and there is no mechanism to propagate errors back to the tool result.

The root cause is identical in all 3 paths: `setTimeout` wrapping prevents the `async` callback's rejection from being caught by the caller. The fix pattern is the same for all 3 — replace `setTimeout` with a proper deferred execution mechanism that still prevents reentrant deadlock but allows error propagation.

**Primary recommendation:** Replace `setTimeout` fire-and-forget with a deferred-promise pattern that resolves AFTER the SDK dispatch completes but schedules execution on the next macrotask. This preserves the deadlock-avoidance intent while enabling error propagation.

## Path 1: dispatch-command.ts:64-114 — setTimeout Fire-and-Forget

### Root Cause Analysis

**File:** `src/tools/session/dispatch-command.ts`
**Lines:** 64-114
**Classification:** Intentional deferred dispatch with swallowed errors

```typescript
// Line 64: setTimeout wraps the ENTIRE dispatch logic
setTimeout(async () => {
  try {
    if (subtask) {
      // Line 78: SDK call — session.prompt() with subtask parts
      await client.session.prompt({ ... })
    } else {
      // Line 91: SDK call — session.prompt() with text parts + agent override
      await client.session.prompt({ ... })
      // Line 100: RESTORE agent — SECOND prompt call, also fire-and-forget
      if (restoreAgent && restoreAgent !== agent) {
        await client.session.prompt({ ... })
      }
    }
  } catch (caughtError: unknown) {
    // Line 112: ONLY console.error — error NEVER reaches the caller
    console.error(`[Harness] Slash command dispatch failed: ${message}`)
  }
}, 50)

// Line 116: Returns IMMEDIATELY with success — before dispatch fires
return { success: true }
```

**Why it's fire-and-forget:** The `setTimeout(async () => { ... }, 50)` wraps the entire async dispatch in a macrotask callback. The returned Promise from the async callback is never awaited. Line 116 returns `{ success: true }` before `client.session.prompt()` is even called.

**Comment explains intent (lines 60-63):**
> "Deferred dispatch: schedule on next macrotask (setTimeout) so the tool returns before session.prompt() fires on the same session — prevents reentrant deadlock."

**The reentrant deadlock concern is real:** When `execute-slash-command` calls `dispatchCommand` from inside its own `execute()` handler, and the dispatch fires `session.prompt()` on the SAME session that's currently executing the tool, OpenCode's SDK can deadlock because the session is busy processing the tool call.

### Impact Assessment

1. **Silent failure:** If `client.session.prompt()` throws (network error, invalid session, SDK error), the only signal is a `console.error` line. The tool result already returned `{ success: true }`.
2. **Lost agent restore:** The restore prompt at line 100 is a SECOND fire-and-forget call inside the first. If the initial dispatch succeeds but the restore fails, the session permanently stays on the overridden agent.
3. **State inconsistency:** The caller (execute-slash-command) reports "dispatched: true" in metadata, but the dispatch may have failed. Downstream systems (session-tracker, delegation monitor) believe the command executed.
4. **No retry mechanism:** Once the setTimeout callback fails, there is no mechanism to retry or report the failure.

### Solution Approaches

**Approach A: Deferred Promise with macrotask scheduling**
- Replace `setTimeout` with a pattern that schedules on next macrotask but returns a Promise
- Use `Promise.resolve().then(() => new Promise(resolve => setTimeout(resolve, 50)))` or a helper
- The function returns the Promise, caller can await it
- **Complexity:** LOW — modify dispatchCommand signature and 3 callers
- **Risk:** LOW — preserves deadlock avoidance, adds error propagation
- **Breakage:** Callers that currently fire-and-forget dispatchCommand (line 313 in execute-slash-command) must handle the new async behavior

**Approach B: Event emitter / callback pattern**
- dispatchCommand emits a 'dispatch-complete' or 'dispatch-failed' event
- Subscribers (execute-slash-command, session-tracker) react to events
- Decouples dispatch from tool result
- **Complexity:** MEDIUM — requires event bus wiring
- **Risk:** MEDIUM — introduces new event dependency chain, harder to test
- **Breakage:** All callers need event subscription

**Recommended: Approach A** — minimal change, preserves intent, adds error propagation.

## Path 2: execute-slash-command.ts:500-515 — session.command() Fire-and-Forget

### Root Cause Analysis

**File:** `src/tools/session/execute-slash-command.ts`
**Lines:** 500-515
**Classification:** Intentional deferred dispatch with swallowed errors (copy of Path 1 pattern)

```typescript
// Line 499: Triggered when resolvedAgent is set (from selectAgent or command frontmatter)
// and we're NOT in a child session, NOT in subtask mode, NOT in synthetic-prompt mode
if (resolvedAgent) {
  // Line 500: setTimeout wraps client.session.command()
  setTimeout(async () => {
    try {
      await client.session.command({
        path: { id: ctx.sessionID },
        body: {
          command: commandBundle.name,
          arguments: validated.arguments ?? "",
          agent: resolvedAgent,
        },
        ...(ctx.directory ? { query: { directory: ctx.directory } } : {}),
      })
    } catch (caughtError: unknown) {
      // Line 513: ONLY console.error
      console.error(`[Harness] session.command dispatch failed: ${message}`)
    }
  }, 50)

  // Line 517: Returns IMMEDIATELY
  return {
    output: `✓ Command ${cmdDisplay} dispatched via SDK command API.`,
    metadata: { ..., dispatched: true },
    error: false,
  }
}
```

**Why it's fire-and-forget:** Identical pattern to Path 1. The `setTimeout(async () => { ... }, 50)` wraps `client.session.command()`. Error is caught but only logged.

**When this path activates:** When `resolvedAgent` is set (from `selectAgent` semantic matching or command frontmatter `agent:`), the command is not a subtask, the command is not a synthetic prompt, and we're not in a child session. This is the "Path 3 — TUI pipeline with agent routing" case in the codebase.

### Impact Assessment

1. **Silent failure:** `client.session.command()` failure is only logged to console
2. **False positive metadata:** `dispatched: true` in metadata is set at line 534 BEFORE the dispatch fires
3. **No agent restore needed here** (unlike Path 1) — `session.command()` handles agent routing internally
4. **Less severe than Path 1** — `session.command()` is a higher-level API that may have its own retry logic in the SDK

### Solution Approaches

**Approach A: Awaiting session.command() directly (remove setTimeout)**
- Since `session.command()` is a different API than `session.prompt()`, it may not trigger reentrant deadlock
- Remove setTimeout, just `await client.session.command()` inline
- Return real success/failure in the tool result
- **Complexity:** LOW — remove 15 lines, add error handling inline
- **Risk:** MEDIUM — must verify that `session.command()` on the current session during tool execution doesn't deadlock
- **Breakage:** Tool return is delayed by the SDK call duration, but this is correct behavior

**Approach B: Deferred promise pattern (same as Path 1 Approach A)**
- Keep setTimeout for deadlock safety, but propagate errors through a deferred promise
- **Complexity:** LOW — same pattern as Path 1 fix
- **Risk:** LOW — preserves deadlock avoidance
- **Breakage:** None if callers already handle async

**Recommended: Approach B** — safer, consistent with Path 1 fix pattern. If deadlock testing confirms `session.command()` is safe to call synchronously, Approach A can be applied later as a simplification.

## Path 3: execute-slash-command.ts:453 — Child Session Dispatch (NOT fire-and-forget)

### Root Cause Analysis

**File:** `src/tools/session/execute-slash-command.ts`
**Lines:** 451-493
**Classification:** FALSE POSITIVE — this path is NOT fire-and-forget

```typescript
// Line 451: Child session path
if (isChildSession) {
  // Line 453: AWAITED dispatch — NOT fire-and-forget
  const childDispatchResult = await dispatchCommand({
    client,
    sessionID: ctx.sessionID,
    promptText,
    subtask: false,
    directory: ctx.directory,
  })

  // Line 461: Error check on result
  if (!childDispatchResult.success) {
    return {
      output: childDispatchResult.output ?? `Failed to dispatch ${cmdDisplay}`,
      metadata: { error: true, errorType: "dispatch_failed", command: args.command },
      error: true,
    }
  }
  // Line 473: Returns success with real dispatch result
  return { output: ..., metadata: { ..., dispatched: true }, error: false }
}
```

**Why this is NOT fire-and-forget at this level:** Line 453 uses `await dispatchCommand(...)`. The call IS awaited at the execute-slash-command level. HOWEVER, `dispatchCommand` internally uses `setTimeout` (Path 1), so the `await` resolves immediately when dispatchCommand returns `{ success: true }` — BEFORE the actual SDK dispatch fires.

**The fire-and-forget is inherited from Path 1** — `dispatchCommand` always returns `{ success: true }` because its internal setTimeout hasn't fired yet. So while line 453 appears to await the result, it's awaiting a pre-determined `{ success: true }`.

### Impact Assessment

1. **Inherited problem:** The `await` at line 453 gives a FALSE sense of error handling — it will ALWAYS see `{ success: true }`
2. **Error check is dead code:** The `if (!childDispatchResult.success)` branch at line 461 can NEVER be reached because dispatchCommand always returns `{ success: true }`
3. **Same failure modes as Path 1:** If the inner setTimeout callback fails, this path has no way to know

### Solution Approaches

**Approach A: Fix Path 1 → this path auto-fixes**
- If dispatchCommand is fixed to properly await the SDK dispatch (Path 1 Approach A), then line 453's `await` will correctly resolve with real success/failure
- The error check at line 461 will become live code
- **Complexity:** ZERO — no changes needed here if Path 1 is fixed
- **Risk:** NONE
- **Breakage:** None — existing error check becomes functional

**Approach B: Direct SDK call bypassing dispatchCommand**
- Remove dispatchCommand indirection, call `client.session.prompt()` directly
- Full control over error handling
- **Complexity:** MEDIUM — duplicates dispatchCommand logic
- **Risk:** MEDIUM — must handle agent validation, format checking
- **Breakage:** Bypasses agent validation in dispatchCommand

**Recommended: Approach A** — fixing Path 1 automatically fixes this path. The existing error handling at line 461 becomes real and functional.

## SDK API Surface: WaiterModel Dispatch and Dual-Signal Completion

### How dispatch works in the harness

The DelegationManager facade (`manager.ts`) supports two dispatch paths:

| Path | Entry Point | Mechanism | Returns |
|------|------------|-----------|---------|
| SDK delegation | `dispatch(params: DelegateParams)` | Creates child session → `sendPromptAsync()` → monitors via dual-signal | `DelegationResult` with childSessionId |
| Command delegation | `dispatchCommand(params)` | Creates PTY or headless process → monitors via completion handler | `DelegationResult` with status |

### V2 Coordinator dispatch flow (coordinator.ts:151-244)

```
coordinator.dispatch(params)
  → dispatcher.preflightCheck(params)      // Validate agent, acquire slot
  → createRecord(delegationId, params)      // Register delegation
  → lifecycle.transition(id, "dispatched")  // State machine
  → childSessionStarter.start(...)          // Create child session
  → attachChildSession(id, childSessionId)  // Link delegation → session
  → lifecycle.transition(id, "running")     // State machine
  → monitor.start(id, parentId)             // Start polling
  → detector.watchDualSignal(id, childSessionId, callback)  // Dual-signal
  → return DelegationResult                 // With childSessionId
```

### Dual-signal completion protocol

From `coordinator.ts:239`:
```typescript
this.deps.detector.watchDualSignal(delegationId, record.childSessionId, (result) => {
  this.handleCompletion(delegationId, result)
})
```

Two independent signals converge to mark completion:
1. **Session lifecycle hooks** — `handleSessionIdle`, `handleSessionError`, `handleSessionDeleted` fire from OpenCode event hooks
2. **SDK message polling** — SdkDelegationHandler polls child session messages for terminal conditions

When either signal fires, `detector.signalCompletionEvent()` or `detector.signalTerminalStatus()` is called, which triggers the `watchDualSignal` callback.

### Key difference: dispatch-command vs. coordinator

The fire-and-forget paths in `dispatch-command.ts` and `execute-slash-command.ts` do NOT participate in the dual-signal completion protocol. They call `client.session.prompt()` or `client.session.command()` directly, bypassing the coordinator entirely. This means:

- No delegation record is created
- No dual-signal monitoring is set up
- No completion callback is registered
- The dispatch is invisible to the delegation manager

This is by design — these paths are for "quick command execution" (not long-running delegation), but the fire-and-forget timing means even quick command failures are invisible.

### session-api.ts SDK surfaces used

| Method | Purpose | Returns |
|--------|---------|---------|
| `client.session.prompt()` | Synchronous prompt — blocks until response | Response data or empty string |
| `client.session.promptAsync()` | Async prompt — returns 204 immediately | void |
| `client.session.command()` | Execute slash command with agent override | Response data |
| `client.tui.appendPrompt()` | Append text to TUI input | Response data |
| `client.tui.submitPrompt()` | Submit TUI input | Response data |

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Deferred execution with error propagation | Custom setTimeout + callback system | `Promise` with macrotask scheduling | Promise composition handles errors naturally |
| Reentrant deadlock prevention | Per-session locks in dispatchCommand | macrotask scheduling (setImmediate / setTimeout(0)) | OpenCode SDK serializes per-session prompts |
| Dispatch result tracking | Custom state tracking in dispatchCommand | Existing DelegationManager dual-signal | Coordinator already tracks dispatch lifecycle |

## Common Pitfalls

### Pitfall 1: Removing setTimeout Causes Reentrant Deadlock
**What goes wrong:** If `dispatchCommand` calls `session.prompt()` synchronously on the same session that's executing the tool, OpenCode's internal session lock may deadlock.
**Why it happens:** OpenCode serializes prompt processing per session. A tool executing inside session A cannot send a new prompt to session A until the current prompt completes.
**How to avoid:** Keep macrotask scheduling but capture the Promise for error propagation. Test with: `session.prompt()` called from tool `execute()` on the same session.
**Warning signs:** Tool hangs indefinitely, no response returned to agent.

### Pitfall 2: False Positive dispatchResult.success
**What goes wrong:** Callers check `dispatchResult.success` thinking it reflects actual dispatch outcome, but it always returns `true`.
**Why it happens:** dispatchCommand returns before setTimeout fires.
**How to avoid:** Fix dispatchCommand to return a real Promise. All 3 callers already have error handling for `!success` — it's just never triggered today.

### Pitfall 3: Agent Restore Failure
**What goes wrong:** The restore prompt at dispatch-command.ts:100 fires inside the same setTimeout callback. If the initial dispatch succeeds but restore fails, the session stays on the wrong agent permanently.
**Why it happens:** Both calls are in the same try/catch — but restore failure doesn't undo the initial dispatch success.
**How to avoid:** Separate restore into its own error-handling path. If restore fails, at minimum log a prominent warning and potentially re-throw.

## Code Examples

### Current fire-and-forget pattern (dispatch-command.ts:64-116)
```typescript
// FIRE-AND-FORGET — returns before dispatch completes
setTimeout(async () => {
  try {
    await client.session.prompt({ path: { id: sessionID }, body })
    if (restoreAgent && restoreAgent !== agent) {
      await client.session.prompt({ path: { id: sessionID }, body: { parts, agent: restoreAgent } })
    }
  } catch (caughtError) {
    console.error(`[Harness] Slash command dispatch failed: ${message}`)
  }
}, 50)
return { success: true }
```

### Proposed fix: Deferred Promise pattern
```typescript
// DEFERRED BUT TRACKED — returns real success/failure after macrotask
return new Promise<{ success: boolean; output?: string; error?: boolean }>((resolve) => {
  setTimeout(async () => {
    try {
      await client.session.prompt({ path: { id: sessionID }, body })
      if (restoreAgent && restoreAgent !== agent) {
        await client.session.prompt({ path: { id: sessionID }, body: { parts, agent: restoreAgent } })
      }
      resolve({ success: true })
    } catch (caughtError) {
      const message = caughtError instanceof Error ? caughtError.message : String(caughtError)
      console.error(`[Harness] Slash command dispatch failed: ${message}`)
      resolve({ success: false, output: message, error: true })
    }
  }, 50)
})
```

### Coordinator dispatch (coordinator.ts:151-244) — properly awaited pattern
```typescript
async dispatch(params: DispatchParams): Promise<DelegationResult> {
  const preflight = await this.deps.dispatcher.preflightCheck(params)
  // ... creates delegation record, starts monitoring ...
  this.deps.detector.watchDualSignal(delegationId, record.childSessionId, (result) => {
    this.handleCompletion(delegationId, result)
  })
  return { childSessionId, delegationId, status: "dispatched" }
}
```

## Open Questions

1. **Does session.command() trigger reentrant deadlock?**
   - What we know: session.prompt() does (documented in comments)
   - What's unclear: session.command() is a different API — it may not have the same locking behavior
   - Recommendation: Test `session.command()` called from tool execute() on the same session before choosing Path 2 approach

2. **What is the acceptable latency for tool return?**
   - What we know: setTimeout adds 50ms delay before dispatch fires
   - What's unclear: If we await the full SDK dispatch, tool return could take 200ms-2s
   - Recommendation: The deferred-promise pattern preserves the 50ms delay, so this is not a concern for the recommended approach
