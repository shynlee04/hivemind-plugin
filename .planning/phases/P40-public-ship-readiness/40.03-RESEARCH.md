# P40.03 Research: Wire Auto-Loop / Ralph-Loop to Hook Chain

**Researched:** 2026-05-31
**Domain:** Loop orchestration primitives → hook chain wiring
**Confidence:** HIGH (all code read, tests verified passing)

## Summary

Phase 39 designed two pure orchestration primitives — `runAutoLoop` (self-referential dispatch loop) and `runRalphLoop` (validate-fix-redispatch cycle) — in `src/coordination/spawner/`. Both are imported into `plugin.ts` (lines 74-75) and passed into the `deps` object (line 473) as `runAutoLoop`, `runRalphLoop`, and `escalationMessage`. The `HookDependencies` type (`hooks/types.ts`) declares them as optional fields (lines 33-35). **However, no hook consumer actually calls them.** The existing `session-hooks.ts` (lines 157-215) contains a hand-rolled inline auto-loop that only handles `delegationPacket`-type sessions and uses `lifecycleManager.requestAutoLoopRetry()` directly. The spawner primitives are wired for injection but never invoked — they are dead-letter dependencies.

**Primary recommendation:** Replace the inline auto-loop in `session-hooks.ts` with `runAutoLoop`, and add `runRalphLoop` for validate-fix escalation on delegation failure. This is primarily a refactoring task in one file (`session-hooks.ts`) with minor adjustments to how `deps.runAutoLoop` and `deps.runRalphLoop` are used.

## Gap Analysis

### Gap 1: Inline Auto-Loop Does Not Use Spawner Infrastructure

**Current (session-hooks.ts:157-215):**
- Triggers on `session.idle` for sessions with `delegationPacket` in continuity
- Tracks state via `AutoLoopState` map (iterations, lastMessageCount, retryPending, exhausted)
- Checks for `completionSignal` (`<promise>DONE</promise>`) in assistant text
- Calls `lifecycleManager.requestAutoLoopRetry()` which sends a prompt via SDK
- Has its own `maxIterations` (default 5), backoff, and exhaustion tracking

**What `runAutoLoop` provides that inline doesn't:**
- Generic typed result `T` (inline loop has no result type)
- Pluggable `dispatcher` and `verifier` callbacks (inline hardcodes SDK prompt + signal check)
- Structured `AutoLoopResult<T>` with `completed|exhausted|failed` status
- Clean error wrapping with `[Harness]` prefix
- Infrastructure-level failures re-thrown, task-level failures returned as `failed`

**The wiring gap:** Replace the inline state-tracking + retry logic with a `runAutoLoop` call where:
- `dispatcher` = `lifecycleManager.requestAutoLoopRetry` (or equivalent SDK prompt)
- `verifier` = checks for completion signal in assistant text
- `maxIterations` = from `autoLoopConfig`

### Gap 2: No Ralph-Loop Invocation at All

There is no validate-fix-redispatch cycle in the current runtime. When a delegation fails, the session just ends. `runRalphLoop` should be invoked when:
- A delegation result needs validation (e.g., "did the fix actually work?")
- An auto-loop dispatch produces output that needs correction

**The wiring gap:** Add a `runRalphLoop` invocation in the auto-loop verifier — when the verifier finds partial success, it can run a ralph-loop to validate and fix before continuing.

### Gap 3: No Delegation Completion → Auto-Loop Chain

The current inline auto-loop only handles `delegationPacket` sessions. There is no auto-loop behavior for:
- Parent sessions waiting for child delegation completion
- Multi-step orchestration (task A completes → trigger task B)

**This is out of scope for P40.03** (noted in existing PLAN.md: "New loop strategies") but should be documented as a future extension point.

## Dead-Loop Risk Assessment

### Risk 1: Inline Auto-Loop — Infinite Iteration
**Severity:** LOW (mitigated)
**Details:** The inline loop has a `maxIterations` guard (default 5) at line 174: `if (state.iterations >= autoLoopConfig.maxIterations)`. It also checks `state.retryPending` to prevent concurrent retries (line 159) and `state.exhausted` to stop permanently (line 159).
**Assessment:** Well-guarded. No infinite loop risk.

### Risk 2: Inline Auto-Loop — Stale Message Count
**Severity:** MEDIUM
**Details:** The loop tracks `lastMessageCount` (line 170) and skips if `messages.length <= state.lastMessageCount`. If the session compacts between idle events, the message count could reset, potentially allowing extra iterations.
**Mitigation:** The `maxIterations` cap still prevents unbounded growth.

### Risk 3: `runAutoLoop` — No Infinite Loop Risk
**Severity:** NONE
**Details:** `runAutoLoop` validates `maxIterations > 0` (line 99), increments `attempt` monotonically (line 110), and the while loop condition is `attempt < options.maxIterations` (line 109). The loop is bounded by construction.

### Risk 4: `runRalphLoop` — No Infinite Loop Risk
**Severity:** NONE
**Details:** Same pattern — validates `maxCorrectionCycles > 0` (line 105), bounded while loop at line 134. Error cases return immediately (status: "error").

### Risk 5: Session Idle Event Storm
**Severity:** MEDIUM
**Details:** If OpenCode fires multiple `session.idle` events in rapid succession, the inline loop guards with `retryPending` (line 159). After wiring `runAutoLoop`, ensure the dispatcher call is still guarded against concurrent invocations.
**Mitigation:** Keep the `retryPending` guard or use a session-scoped mutex.

### Risk 6: Terminal Session Events After Loop Start
**Severity:** LOW
**Details:** `TERMINAL_SESSION_EVENTS` (session.deleted, session.error) clean up auto-loop state (lines 141-144). After wiring `runAutoLoop`, need to ensure that a loop in-flight when the session is deleted doesn't hang.
**Mitigation:** The dispatcher (`requestAutoLoopRetry`) sends via SDK — if the session is deleted, the SDK call will reject, and `runAutoLoop` wraps it as a `[Harness]` error.

## Recommended Approach

### Step 1: Refactor session-hooks.ts Inline Auto-Loop to Use `runAutoLoop`

**File:** `src/hooks/lifecycle/session-hooks.ts`
**Change:** Replace lines 157-215 (the inline delegation-packet auto-loop) with a `deps.runAutoLoop` call.

**Wiring:**
```typescript
// Pseudocode for the replacement
if (continuity.metadata.delegationPacket) {
  const result = await deps.runAutoLoop({
    initialPrompt: buildAutoLoopPrompt({...}),
    maxIterations: autoLoopConfig.maxIterations,
    dispatcher: async (prompt, attempt) => {
      await waitForRetry(autoLoopConfig.backoffMs, deps.sleep)
      await lifecycleManager.requestAutoLoopRetry({ sessionID, promptText: prompt })
      return { attempt, sessionID }
    },
    verifier: async (result, attempt) => {
      const messages = await getSessionMessages(client, sessionID)
      const assistantText = extractAssistantText(messages)
      if (assistantText.includes(autoLoopConfig.completionSignal)) {
        return { outcome: "completed", result }
      }
      if (attempt >= autoLoopConfig.maxIterations) {
        return { outcome: "failed", result, error: "exhausted" }
      }
      return {
        outcome: "needs_continuation",
        result,
        nextPrompt: buildAutoLoopPrompt({...})
      }
    },
  })
  // Handle result.status: completed | exhausted | failed
}
```

**Key considerations:**
- The inline loop's `AutoLoopState` map and `retryPending` guard are replaced by `runAutoLoop`'s internal iteration tracking
- `session.idle` fires once per iteration (the dispatcher sends a prompt → model responds → session.idle fires again → auto-loop checks completion signal)
- The loop is **reactive** (event-driven per idle), not proactive — `runAutoLoop` drives iterations internally but the trigger is still the `session.idle` event

### Step 2: Add Ralph-Loop for Error Escalation

**File:** `src/hooks/lifecycle/session-hooks.ts`
**Change:** In the auto-loop's verifier, when a delegation fails, invoke `deps.runRalphLoop` for validate-fix escalation.

**Wiring:**
```typescript
// In the verifier, when the dispatch result indicates failure:
if (deps.runRalphLoop) {
  const ralphResult = await deps.runRalphLoop({
    initialResult: dispatchResult,
    maxCorrectionCycles: 3,
    validator: async (result) => /* check if result passes */,
    fixer: async (fixPrompt) => /* dispatch fix session */,
  })
  if (ralphResult.status === "exhausted" && deps.escalationMessage) {
    stateManager.addWarning(sessionID, deps.escalationMessage(ralphResult))
  }
}
```

### Step 3: Remove or Document `AutoLoopState` Map

**Decision point:** The `autoLoopStates` map (line 129) and `getAutoLoopState` helper (lines 60-77) track per-session iteration state. If `runAutoLoop` manages its own state internally, this map becomes unnecessary for the delegation-packet path. However:
- It's still used by `experimental.session.compacting` (lines 237-258) to inject context
- It may be needed for the `retryPending` guard against concurrent idle events

**Recommendation:** Keep the `AutoLoopState` map for session-level tracking (retryPending, exhausted) but remove the iteration counting (now handled by `runAutoLoop` internally). Alternatively, use the `AutoLoopResult` returned by `runAutoLoop` to populate the compaction context.

## Architectural Decision: Reactive vs Proactive Loop

The current inline auto-loop is **reactive**: it fires on each `session.idle` event, checks completion, and sends a retry prompt. `runAutoLoop` is **proactive**: it drives iterations internally in a while loop.

**Problem:** If we call `runAutoLoop` directly in the `session.idle` handler, the loop would drive all iterations synchronously within a single idle event. But the dispatcher (`requestAutoLoopRetry`) sends a prompt to the session — we need to wait for the model to respond and fire another `session.idle` before checking completion.

**Resolution options:**

1. **Keep reactive pattern, use `runAutoLoop` per-iteration:** Call `runAutoLoop` with `maxIterations: 1` on each idle event. The verifier checks completion; if `needs_continuation`, return and let the next idle event trigger the next iteration. State tracking moves to the `AutoLoopState` map or continuity metadata.

2. **Make dispatcher await model response:** The `requestAutoLoopRetry` currently fires-and-forgets (returns void). To use `runAutoLoop` with `maxIterations > 1`, the dispatcher would need to poll for the model's response (like `DelegationManager` does for child sessions). This is a bigger change.

3. **Hybrid:** Use `runAutoLoop` as the state machine but break out of the while loop after each dispatch, relying on the next `session.idle` event to re-enter. This defeats the purpose of `runAutoLoop`'s internal loop.

**Recommendation:** Option 1 is the safest and most aligned with the current architecture. It uses `runAutoLoop` as a typed, structured single-iteration primitive while preserving the reactive event-driven pattern. The `AutoLoopState` map still tracks cross-event state, but the dispatch/verify logic is delegated to the spawner primitive's callbacks.

## Open Questions

1. **Is the reactive or proactive pattern intended?** The `runAutoLoop` design suggests proactive (internal while loop), but the current runtime is reactive (session.idle events). This determines whether `maxIterations: 1` or `maxIterations: N` is correct.
   - **Recommendation:** Use reactive with `maxIterations: 1` per idle event, preserving current behavior while gaining the typed result structure.

2. **Should `runRalphLoop` wrap individual auto-loop iterations or the entire delegation result?** I.e., is ralph-loop "inner" (validate each dispatch result) or "outer" (validate the final delegation outcome)?
   - **Recommendation:** Inner — use ralph-loop as the verifier's escalation path within auto-loop.

3. **Should the inline `AutoLoopState` map be removed or kept?** It serves both the auto-loop and the compaction hook.
   - **Recommendation:** Keep for compaction context, but simplify (remove `iterations` tracking, use `AutoLoopResult` instead).
