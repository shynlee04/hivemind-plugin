# Phase P40.07: End-to-End Integration Verification — Research

**Researched:** 2026-05-31
**Domain:** E2E integration testing, npm package verification, public repo sync validation
**Confidence:** HIGH

## Summary

P40.07 is the **terminal ship gate** for the Hivemind harness. All 6 prior phases (P40.01–P40.06) must pass before this phase begins. Its job is to verify that the complete user journey — install → plugin load → tool availability → bootstrap → delegation — works end-to-end from the perspective of an external consumer installing the npm package.

**Current state:** 2,990 tests pass across 247 test files. A single `user-install.test.ts` (78 lines) simulates the postinstall flow but covers only asset extraction — not plugin registration, tool execution, hook firing, or bootstrap primitive loading. The existing `40.07-PLAN.md` was flagged as `draft-violated` (created outside GSD workflow).

**Primary recommendation:** Write 4 new integration tests (user-install upgrade, plugin registration, tool smoke, bootstrap flow) plus validate sync-oss dry-run — each test targeting a specific user-facing surface that has zero coverage today.

### Pattern 1: Simulated Consumer Project

**What:** Create a temp directory, populate with minimal package.json, run sync-assets.js as subprocess with `INIT_CWD` pointing to temp dir.

**When to use:** Any test that verifies the user-facing install experience.

**Example:**


### Anti-Patterns to Avoid

- **Testing internal implementation details:** E2E tests verify external behavior, not internal state. Don't test that `primitive-loader.ts` calls specific functions — test that loaded primitives exist in the registry.
- **Coupling to source file paths:** Tests should work from `dist/`, not `src/`. The consumer never sees `src/`.
- **Assuming GSD primitives exist:** GSD agents/skills/commands live in `.opencode/get-shit-done/` and are NOT shipped. Assets are the sole source of truth.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Temp directory management | Custom temp dir logic | `os.tmpdir()` + `mkdtempSync()` | Race conditions, cleanup |
| Package content validation | Custom file walker | `npm pack --dry-run` output parsing | `files` field in package.json controls this |
| Primitive file counting | Hard-coded counts | Dynamic counts from `assets/` directory | Counts change between versions |
| Version comparison | Custom semver | String comparison (already used in sync-assets.js) | Project uses simple version strings |

**Key insight:** The existing `user-install.test.ts` already established the pattern for simulated consumer projects. New tests should extend this pattern, not invent new ones.

## Common Pitfalls

### Pitfall 1: Testing from `src/` instead of `dist/`

**What goes wrong:** Tests import from `src/` directly, passing locally but failing for consumers who only have `dist/`.
**Why it happens:** Developer convenience — `src/` is always available in the dev environment.
**How to avoid:** All E2E tests must import from `dist/` or test via subprocess execution (like sync-assets.js).
**Warning signs:** Test imports contain `../../src/` paths.

### Pitfall 2: GSD primitive leak

**What goes wrong:** GSD agents (gsd-*) appear in the npm package or in `.opencode/` after install.
**Why it happens:** GSD primitives live in `.opencode/get-shit-done/` (developer tooling) and are NOT in `assets/`. But if sync-assets.js or the `files` field is misconfigured, they could leak.
**How to avoid:** Verify `package.json files` field explicitly lists only `assets/` subdirs (not `.opencode/`). Verify sync-assets.js only copies from `assets/`.
**Warning signs:** `ls` of npm pack output shows `.opencode/get-shit-done/` or `gsd-*` files.

### Pitfall 3: Hard-coded primitive counts

**What goes wrong:** Tests assert "there are exactly 43 agents" but the count changes in the next version.
**Why it happens:** Tests written to match current state without considering evolution.
**How to avoid:** Read counts dynamically from `assets/` directory, or assert minimum thresholds (e.g., "at least 30 agents").
**Warning signs:** Tests contain literal numbers like `expect(agents.length).toBe(43)`.

### Pitfall 4: Version drift false negatives

**What goes wrong:** sync-assets.js skips extraction when version matches, so tests pass but don't actually verify extraction works.
**Why it happens:** Version drift check in sync-assets.js short-circuits the entire flow.
**How to avoid:** E2E tests must run in a fresh temp directory (no pre-existing version.json).
**Warning signs:** Tests reuse temp directories or don't clean up between runs.

### Pitfall 5: CI-only failures

**What goes wrong:** Tests pass locally (macOS) but fail on CI (Ubuntu) due to path differences.
**Why it happens:** `INIT_CWD`, `process.cwd()`, and temp dir paths differ across platforms.
**How to avoid:** Use `path.join()` everywhere, never hard-code `/tmp/` or platform-specific paths.
**Warning signs:** Tests use `path.resolve()` without considering platform differences.

## Code Examples

### Simulated User Install (existing pattern from user-install.test.ts)

```typescript
// Source: tests/integration/user-install.test.ts [VERIFIED: codebase]
import { execSync } from "node:child_process";
import { existsSync, readFileSync } from "node:fs";
import { mkdtempSync, rmSync } from "node:fs";
import { join } from "node:path";
import { tmpdir } from "node:os";

const PROJECT_ROOT = join(import.meta.dirname, "../..");

describe("User Install E2E", () => {
  let consumerDir: string;

  beforeEach(() => {
    consumerDir = mkdtempSync(join(tmpdir(), "hivemind-e2e-"));
  });

  afterEach(() => {
    rmSync(consumerDir, { recursive: true, force: true });
  });

  it("copies all primitive directories to .opencode/", () => {
    execSync(`node "${join(PROJECT_ROOT, "scripts/sync-assets.js")}" --mode=install`, {
      env: { ...process.env, INIT_CWD: consumerDir },
      cwd: PROJECT_ROOT,
      stdio: "pipe",
    });

    for (const dir of ["agents", "skills", "commands", "workflows", "references", "templates"]) {
      expect(existsSync(join(consumerDir, ".opencode", dir))).toBe(true);
    }
  });
});
```

## Open Questions

1. **Tool smoke test dependency injection**
   - What we know: Tool factory functions require coordinator instances, project roots, and SDK clients as parameters.
   - What's unclear: Whether minimal mocks suffice or if real (in-memory) instances are needed.
   - Recommendation: Start with minimal mocks for smoke tests; upgrade to real instances only if mocks mask real failures.

2. **Plugin registration test isolation**
   - What we know: `dist/plugin.js` wires real delegation system on import.
   - What's unclear: Whether importing plugin.js in test context triggers side effects (file I/O, port binding).
   - Recommendation: Test plugin export shape without executing registration logic; verify registration via mock SDK.

3. **sync-oss validation test scope**
   - What we know: sync-oss.yml is a GitHub Actions workflow with `workflow_dispatch`.
   - What's unclear: Whether we should test the workflow logic locally or rely on CI-only testing.
   - Recommendation: Extract the banned-directory and required-path logic into a testable function that can be unit-tested, then also verify via CI.
