# Phase 40.02: Unify Delegation State Machines — Research

**Researched:** 2026-05-30
**Domain:** Delegation state management — dual v1/v2 state machine consolidation
**Confidence:** HIGH

## Summary

The Hivemind delegation subsystem currently operates **two separate in-memory state stores** that track overlapping delegation lifecycle state. The v1 store lives inside `DelegationStateMachine` (owned by `RuntimeDelegationManager` in `manager-runtime.ts`) and the v2 store lives inside `DelegationLifecycle` (wrapping the same `DelegationStateMachine` class, but potentially instantiated independently). The public facade `DelegationManager` (in `manager.ts`) bridges both, concatenating and deduplicating records from both stores in `listDelegations()`.

The coordinator path (v2) is the intended future: it has dual-signal completion, execution evidence tracking, periodic notifications, slot management, and retry handling. The runtime path (v1) has legacy SDK dispatch via `spawnDelegatedSession`, command delegation via `CommandDelegationHandler`, concurrency queuing, and behavioral guardrails. Both paths share the same `DelegationStatus` enum and `VALID_DELEGATION_TRANSITIONS` map, but they manage their own in-memory Maps, persistence calls, and timer lifecycles independently.

**Primary recommendation:** Consolidate to a single `DelegationStateMachine` instance owned by the coordinator path. The v1 `RuntimeDelegationManager` should be stripped to a thin dispatch adapter that delegates state tracking to the shared machine. The facade `DelegationManager` should drop its v1 fallback path once all dispatch lanes flow through the coordinator.

| Capability | Primary Tier | Secondary Tier | Rationale |

| State storage & transitions | `state-machine.ts` | — | Single source of truth for delegation records |

## Current Architecture: Dual State Machines

### State Machine A: `DelegationStateMachine` (v1, owned by `RuntimeDelegationManager`)

**File:** `src/coordination/delegation/state-machine.ts` (443 LOC)
**Owner:** `RuntimeDelegationManager` in `manager-runtime.ts` (507 LOC)

**Store:**
- `delegations: Map<string, Delegation>` — all v1 delegations by ID
- `delegationsBySession: Map<string, string>` — child session ID → delegation ID reverse lookup
- `safetyTimers: Map<string, NodeJS.Timeout>` — 30-min safety ceiling per delegation
- `gracePeriodTimers: Map<string, NodeJS.Timeout>` — post-terminal cleanup timers

**Transition map (verified from source):**
```
dispatched → [running, completed, error, timeout]
running    → [completed, error, timeout]
completed  → []  (terminal)
error      → []  (terminal)
timeout    → []  (terminal)
```

**Key methods:**
- `registerDelegation(delegation, scheduleSafetyCeiling)` — insert into store, arm safety timer
- `transition(id, nextStatus)` — guarded non-terminal transition
- `transitionToTerminal(id, newState, error?, terminalDetail?)` — unified terminal path: set status, persist, clear timers, schedule grace cleanup, notify parent
- `persistAll()` — write all delegations to disk via `delegation-persistence.ts`
- `pruneCompletedDelegations(maxAgeMs?)` — remove old terminal records

**Dispatch flow (v1):**
1. `RuntimeDelegationManager.dispatch(params)` validates agent, resolves nesting depth
2. Acquires semaphore slot from `DelegationConcurrencyQueue`
3. Calls `spawnDelegatedSession()` → gets `childSessionId`
4. Creates `Delegation` record with `status: "dispatched"`, `executionMode: "sdk"`
5. Calls `state.registerDelegation(delegation, true)` — arms safety ceiling
6. Calls `state.transition(id, "running")` after SDK spawn
7. Returns `buildDelegationResult(delegation)`

### State Machine B: `DelegationLifecycle` → `DelegationStateMachine` (v2, owned by `DelegationCoordinator`)

**Files:**
- `src/coordination/delegation/lifecycle.ts` (93 LOC) — thin adapter
- `src/coordination/delegation/state-machine.ts` (same class, different instance)
- `src/coordination/delegation/coordinator.ts` (556 LOC) — orchestrator

**Store:**
- `active: Map<string, ActiveDelegation>` — coordinator's own tracking (record + slotHandle)
- `delegationByChildSession: Map<string, string>` — coordinator's session → ID mapping
- PLUS whatever `DelegationLifecycle.stateMachine` owns (could be same or different `DelegationStateMachine` instance)

**Interface bridge:**
```typescript
// lifecycle.ts defines:
interface DelegationLifecycleStateMachine {
  get(delegationId: string): Delegation | undefined
  getAll?(): Delegation[]
  registerDelegation?(delegation: Delegation): void
  transition(delegationId: string, toStatus: DelegationStatus): boolean
  transitionToTerminal?(delegationId: string, toStatus: DelegationStatus, error?: string): void
}
```

The `DelegationLifecycle` class wraps ANY implementation of this interface. Currently `DelegationStateMachine` implements it. The coordinator receives a `lifecycle` dependency injection.

**Dispatch flow (v2):**
1. `DelegationCoordinator.dispatch(params)` runs `dispatcher.preflightCheck(params)`
2. Creates `Delegation` record, stores in `this.active`
3. Calls `lifecycle.register(record)` → delegates to `stateMachine.registerDelegation()`
4. Calls `lifecycle.transition(id, "dispatched")`
5. Starts `childSessionStarter.start()` → gets real `childSessionId`
6. Calls `lifecycle.transition(id, "running")`
7. Wires dual-signal completion via `detector.watchDualSignal()`
8. Starts periodic notifier and monitor
9. Returns `DelegationResult`

**Terminal flow (v2):**
- `handleCompletion(id, result)` → transitions lifecycle, routes notification, cleans up active/slot
- `handleTimeout(id)` → `lifecycle.markTimeout()`, cleanup
- `failDispatch(id, error)` → `lifecycle.transition(id, "error")`, cleanup

### The Facade: `DelegationManager` (manager.ts, 428 LOC)

This is the **critical bridging layer**. It:

1. **Conditionally instantiates** `RuntimeDelegationManager` (v1) only when an `OpenCodeClient` is provided
2. **Optionally receives** a `coordinator` (v2) via dependency injection
3. **Optionally receives** a `lifecycle` (v2) via dependency injection
4. **Dual-dispatches**: `dispatch()` checks `this.options.coordinator` first, falls back to `this.runtime`
5. **Merges lists**: `listDelegations()` concatenates v2 + v1, deduplicates by ID (v2 wins)
6. **Forwards events**: `handleSessionIdle/Error/Deleted` fans out to both v1 and v2

**plugin.ts wiring (verified):**
```typescript
// Both are instantiated independently
const coordinator = new DelegationCoordinator({ ... })
const manager = new DelegationManager(client, { coordinator, lifecycle, ... })
```

## Overlap Analysis

| Concern | v1 (RuntimeDelegationManager) | v2 (DelegationCoordinator) | Overlap? |
|---------|------------------------------|---------------------------|----------|
| In-memory store | `DelegationStateMachine.delegations` | `DelegationLifecycle.stateMachine.*` | **YES** — two Map instances |
| Session → ID lookup | `delegationsBySession` | `delegationByChildSession` | **YES** — two reverse-lookup Maps |
| Status transitions | `state.transition()`, `state.transitionToTerminal()` | `lifecycle.transition()`, `lifecycle.markTimeout()` | **YES** — same transition map |
| Persistence | `state.persistAll()` | `lifecycle.transition()` → `stateMachine.persistAll()` | **YES** — same `persistDelegations()` call |
| Safety timers | `safetyTimers` in state-machine | Not used (coordinator uses `handleTimeout`) | Partial — coordinator delegates timeout to lifecycle |
| Grace period cleanup | `gracePeriodTimers` in state-machine | Same code path via lifecycle | **YES** — shared `DelegationStateMachine` code |
| Dispatch entry | `dispatch(params: DelegateParams)` | `dispatch(params: DispatchParams)` | **YES** — two dispatch signatures |
| Concurrency queue | `DelegationConcurrencyQueue` | `slot-manager.ts` (SlotHandle) | **YES** — two concurrency mechanisms |
| Monitoring | `DelegationMonitor.start()` | `deps.monitor.start()` + `watchDualSignal()` | v2 is strictly richer |
| Notifications | `NotificationRouter.register()` | `notificationRouter.register()` + `periodicNotifier` | v2 is strictly richer |
| Evidence tracking | Not present | `recordExecutionSignal()`, evidence levels | v2-only |
| Command delegation | `CommandDelegationHandler` | Not present | v1-only |
| PTY management | `PtyManager` integration | Not present | v1-only |

## What Lives Only in v1 (must migrate or preserve)

1. **`DelegationConcurrencyQueue`** — semaphore-based concurrency with queue-key support
2. **`CommandDelegationHandler`** — PTY and headless command delegation
3. **`SdkDelegationHandler`** — legacy SDK dispatch with stability polling
4. **`spawnDelegatedSession()`** — actual child session creation via SDK
5. **`applyBehavioralGuardrail()`** — concurrency cap based on behavioral profile
6. **`recoverPending()`** — recovery of delegations from persistence after restart
7. **PTY session management** — `getDelegationForPtySession()`, `markCommandCancellationForPtySession()`

## What Lives Only in v2 (must be preserved)

1. **Dual-signal completion** — `detector.watchDualSignal()` with message + tool observation
2. **Execution evidence tracking** — `recordExecutionSignal()`, evidence levels, execution states
3. **Periodic notifications** — `PeriodicNotifier` for long-running delegations
4. **Slot management** — `SlotHandle` for concurrency via `SlotManager`
5. **Preflight checks** — `DelegationDispatcher.preflightCheck()` with depth, agent validation
6. **Retry handling** — `DelegationRetryHandler.persistWithRetry()`
7. **Parent model inheritance** — extracts provider/model from parent session messages

## Recommended Consolidation Path

### Phase 1: Single State Machine Instance
- Ensure `plugin.ts` creates ONE `DelegationStateMachine` instance
- Both v1 runtime and v2 coordinator receive the same instance (or the same `DelegationLifecycle`)
- Eliminates the dual-store merge in `listDelegations()`

### Phase 2: Coordinator Absorbs Runtime Dispatch
- Move `spawnDelegatedSession()`, concurrency queue, and agent validation INTO the coordinator
- The coordinator's `childSessionStarter` dependency already abstracts session creation
- `RuntimeDelegationManager` becomes a thin shim that calls coordinator methods

### Phase 3: Command Delegation Lane
- Command delegation (`dispatchCommand()`) needs either:
  - A parallel coordinator lane (separate dispatch method), OR
  - Migration to use the same coordinator dispatch with `executionMode: "pty"` / `executionMode: "headless"`
- The `Delegation.executionMode` field already supports this distinction

### Phase 4: Facade Simplification
- Once all dispatch lanes flow through coordinator, `DelegationManager` becomes a pure facade
- Remove `RuntimeDelegationManager` entirely
- Remove v1 fallback paths from all facade methods

## Risks

| Risk | Severity | Mitigation |
|------|----------|------------|
| Dual-store state divergence (v1 and v2 out of sync) | **HIGH** | Phase 1 first: share one state-machine instance |
| Command delegation breakage during migration | **MEDIUM** | Keep command dispatch behind a separate method until fully tested |
| Timer lifecycle confusion (two sets of safety/grace timers) | **MEDIUM** | Single state-machine owns all timers |
| Persistence race conditions (two writers to same file) | **HIGH** | Single `persistAll()` caller |
| Test coverage gaps (many tests may only exercise v1 path) | **MEDIUM** | Audit test coverage before each consolidation step |
