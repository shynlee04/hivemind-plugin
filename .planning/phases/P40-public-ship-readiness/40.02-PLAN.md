# Phase P40.02: Consolidate dual v1/v2 delegation state machines

## Problem

Two delegation state stores exist simultaneously:

1. **v2 lifecycle** — An inline `DelegationLifecycleStateMachine` object in `plugin.ts` `setupDelegationModules()` backed by a local `records` Map (plugin.ts L273–L308). Used by `DelegationCoordinator` for delegate-task v2 dispatch.
2. **v1 runtime** — A `DelegationStateMachine` instance created inside `RuntimeDelegationManager` constructor (manager-runtime.ts L95–L101). Used by SDK and command dispatch paths.

The facade (`manager.ts`) merges both stores in `listDelegations()` (L139–L151) and `canSessionAccessDelegation()` (L328–L330), creating dual-writer conflicts and stale-read risk.

## Goal

Share ONE `DelegationStateMachine` instance between both paths, eliminate the dual-store merge, and convert `RuntimeDelegationManager` to a stateless dispatch-only adapter.

## Task 1: Wire single shared DelegationStateMachine in plugin.ts

**file_targets:** `src/plugin.ts`

### Current State

`setupDelegationModules()` creates:
- A local `records` Map (L273) used by an inline `DelegationLifecycleStateMachine` POJO (L291–L308)
- A `DelegationCoordinator` wired to that lifecycle (L353)
- A `DelegationManager` facade which, because `enableRuntimeAdapter=true`, constructs a `RuntimeDelegationManager` internally (manager.ts L56) — which creates its OWN `new DelegationStateMachine(...)` (manager-runtime.ts L95–L101)

Result: two independent in-memory delegation stores.

### Action

1. **Import `DelegationStateMachine`** from `./coordination/delegation/state-machine.js` in plugin.ts.

2. **Create ONE shared instance** inside `setupDelegationModules()`, before the lifecycle POJO:
   - Instantiate `new DelegationStateMachine({ client: options.client })`.
   - The state machine's constructor takes `DelegationStateMachineOptions` which needs `client` and optional `clearExternalTimers`. Since `RuntimeDelegationManager` currently provides `clearExternalTimers` that clears SDK + command handler timers, and the shared instance will outlive both, wire `clearExternalTimers` as a no-op initially (the runtime manager's handler wiring will be relocated in Task 5).

3. **Replace the inline lifecycle POJO** (L291–L308) with a `DelegationLifecycleStateMachine` adapter that delegates to the shared `DelegationStateMachine`:
   - `get(id)` → `sharedStateMachine.get(id)`
   - `getAll()` → `sharedStateMachine.getAll()`
   - `registerDelegation(delegation, scheduleSafetyCeiling?)` → `sharedStateMachine.registerDelegation(delegation, scheduleSafetyCeiling ?? false)`
   - `transition(id, status)` → `sharedStateMachine.transition(id, status)`
   - `transitionToTerminal(id, status, error?)` → `sharedStateMachine.transitionToTerminal(id, status, error)`

4. **Remove the `records` Map** — no longer needed since the shared state machine owns storage.

5. **Pass the shared state machine** to `DelegationManager` via a new option (e.g., `sharedStateMachine`) so the facade can forward it to `RuntimeDelegationManager` instead of the runtime manager creating its own.

6. **Preserve ALL existing tool registrations** — the `registerDelegationTools()` call must receive the same `delegationManager` instance with the same surface.


## Task 3: Remove direct persistDelegations from coordinator, route through lifecycle

**file_targets:** `src/coordination/delegation/coordinator.ts`

### Current State

`DelegationCoordinator.cleanup()` (L454–L470) calls:
```typescript
void this.deps.retryHandler.persistWithRetry(
  this.deps.lifecycle.list?.() ?? [...this.active.values()].map((entry) => entry.record)
)
```

This bypasses the lifecycle's `registerDelegation` and calls the retry handler's persistence directly. The coordinator also maintains its own `active` Map (L145) as a secondary delegation store for in-flight records.

The lifecycle adapter's `registerDelegation` now supports `scheduleSafetyCeiling` (Task 4), but `coordinator.dispatch()` at L156 calls `this.deps.lifecycle.register?.(record)` without passing `scheduleSafetyCeiling`, so the safety ceiling timer is never armed for v2 delegations.

### Action

1. **In `dispatch()` (L156)**, pass `true` for `scheduleSafetyCeiling` when calling register:
   ```typescript
   // Change from:
   this.deps.lifecycle.register?.(record)
   // To:
   this.deps.lifecycle.register?.(record, true)
   ```
   This ensures the shared `DelegationStateMachine` arms the 30-minute safety ceiling for v2 delegations, matching the v1 behavior (manager-runtime.ts L221).

2. **In `cleanup()` (L469)**, replace `retryHandler.persistWithRetry()` with a call through the lifecycle/state machine's own `persistAll()` mechanism. The shared `DelegationStateMachine` already calls `persistDelegations()` inside `persistAll()` and `transitionToTerminal()`. Since `transitionToTerminal` already persists (state-machine.ts L289), and `transition` persists (L248), the explicit `persistWithRetry` call is redundant when using the shared state machine.
   - Change to: delegate persistence to the state machine by calling `this.deps.lifecycle.register?.(updatedRecord)` for any record mutated in cleanup, or simply rely on the transition calls already persisting.
   - If `retryHandler.persistWithRetry` provides retry semantics needed for durability, keep the call but route the data through `this.deps.lifecycle.list?.()` only (remove the fallback to `this.active`).

3. **Verify** that `cleanup()` no longer has a code path that writes to persistence without going through the lifecycle adapter.


### Done When

- `grep -c "persistDelegations" src/coordination/delegation/coordinator.ts` returns 0

## Task 2: Remove dual-store merge from facade (manager.ts)

**file_targets:** `src/coordination/delegation/manager.ts`

### Current State

The facade merges two stores:
- `listDelegations()` (L139–L151): Concatenates `v2List` (from `options.lifecycle.list()`) and `v1List` (from `this.runtime.getAllDelegations()`), then deduplicates by ID.
- `canSessionAccessDelegation()` (L328–L330): Concatenates both lifecycle and runtime delegation lists to walk the parent-child chain.

After Task 1, both paths share the same `DelegationStateMachine`. The v2 lifecycle adapter reads from it, and the runtime manager's `getAllDelegations()` also reads from it. They are the SAME store — the merge is now redundant and masks the consolidation.

### Action

1. **Simplify `listDelegations()`**: Remove the v1/v2 concatenation and dedup logic. Read from a single source:
   ```typescript
   listDelegations(sessionId?: string): Delegation[] {
     const delegations = this.options.lifecycle?.list() ?? this.runtime?.getAllDelegations() ?? []
     return sessionId ? delegations.filter((d) => d.parentSessionId === sessionId) : delegations
   }
   ```

2. **Simplify `canSessionAccessDelegation()`**: Remove the dual-store concatenation at L328–L330. Use a single source:
   ```typescript
   const all = this.options.lifecycle?.list() ?? this.runtime?.getAllDelegations() ?? []
   ```
   Remove the `byId` Map construction since there's only one source.

3. **Preserve the public API** — `getStatus()`, `getAllDelegations()`, `listDelegations()`, `canSessionAccessDelegation()`, `getVisibleDelegationsForSession()` must all return the same results as before (same delegation records visible to callers).

## Task 5: Convert RuntimeDelegationManager to dispatch-only adapter

**file_targets:** `src/coordination/delegation/manager-runtime.ts`

### Current State

`DelegationManager` (manager-runtime.ts) is a 507-line class that:
- Creates its own `DelegationStateMachine` (L95–L101) — state storage
- Owns concurrency via `DelegationConcurrencyQueue` (L80) — dispatch concern
- Owns `CommandDelegationHandler` and `SdkDelegationHandler` — dispatch concern
- Has `getStatus()`, `getAllDelegations()`, `getVisibleDelegationsForSession()` — state read concern
- Calls `this.state.persistAll()` directly (L107, L113, L222) — state write concern

After Tasks 1–2, the shared `DelegationStateMachine` owns all state. This class should become a thin adapter that only does dispatch + concurrency.

### Action

1. **Accept the shared state machine via constructor options**:
   - Add `sharedStateMachine?: DelegationStateMachine` to `DelegationManagerOptions`
   - If provided, use it instead of creating a new one: `this.state = options.sharedStateMachine ?? new DelegationStateMachine(...)`
   - When the shared instance is provided, the constructor no longer owns state lifecycle

2. **Remove direct `persistAll()` calls** from this module:
   - L107 (`persistAllDelegations: () => dm.state.persistAll()`) in `CommandDelegationHandler` callbacks
   - L113 (`persistAllDelegations: () => dm.state.persistAll()`) in `SdkDelegationHandler` callbacks
   - L222 (`this.state.persistAll()`) after `registerDelegation` in `dispatch()`
   - These are now handled by the shared state machine's own lifecycle (it persists on `registerDelegation`, `transition`, and `transitionToTerminal`).

3. **Remove `persistDelegations` import** from `manager-runtime.ts` (L4):
   ```typescript
   import { readPersistedDelegations } from "../../task-management/continuity/delegation-persistence.js"
   ```
   Keep `readPersistedDelegations` for `recoverPending()` but remove `persistDelegations` if imported.

4. **Verify background delegation path**: The `dispatch()` method handles SDK delegations (L172–L252) which is used for background agent dispatch. After removing direct persist calls, verify the delegation round-trip still works:
   - `registerDelegation(delegation, true)` persists via the state machine
   - `transition(delegation.id, "running")` persists via the state machine
   - `transitionToTerminal(...)` persists via the state machine

5. **Keep `recoverPending()`** — it still needs `readPersistedDelegations()` to hydrate from disk, but should forward hydrations to the shared state machine via `hydrateFromPersistence()`.
