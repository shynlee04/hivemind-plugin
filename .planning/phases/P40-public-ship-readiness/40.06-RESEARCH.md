### F3: bun-pty Import Pattern — Graceful Degradation Works

The bun-pty integration uses a **three-layer graceful degradation** pattern:

1. **`pty-runtime.ts`** (entry): Dynamic `await import("./pty-manager.js")` wrapped in try/catch → returns `null` on failure [VERIFIED: pty-runtime.ts lines 13-20]
2. **`pty-manager.ts`** (implementation): Direct `import { spawn } from "bun-pty"` + `isSupported()` checks `"Bun" in globalThis` [VERIFIED: pty-manager.ts lines 1, 47]
3. **`bun-pty.d.ts`** (type shim): Ambient module declaration allows `tsc` to pass without bun-pty installed [VERIFIED: bun-pty.d.ts lines 22-55]

**Plugin integration:** `plugin.ts` line 371 calls `const ptyManager = await createPtyManagerIfSupported()` and passes the result (nullable) through the dependency chain. Tools that use PTY receive `null` and gracefully degrade to headless mode.

**Risk assessment:** LOW. The pattern is correct. If `bun-pty` fails to install (e.g., on Node.js without optional deps), the dynamic import fails, `pty-runtime.ts` returns `null`, and all PTY features are disabled.

### F5: PACKAGE BLOAT — CRITICAL SHIPPING BLOCKER

**npm pack --dry-run produces:**
- Package size: **23.4 MB** packed
- Unpacked size: **102.6 MB**
- Total files: **2504**
- Name: `hivemind-opencode-0.1.0.tgz`

[VERIFIED: npm pack --dry-run output]

**Root cause:** The `files` array includes `assets`, which captures the ENTIRE `assets/` directory — including a hidden `assets/.hivemind/` directory:

| Directory | Size | Files | Contents |
|-----------|------|-------|----------|
| `assets/.hivemind/skills/` | **89 MB** | 410 .md | Internal governance skills (gate-l3-*, hf-l2-*, hm-l2-*) |
| `assets/.hivemind/agents/` | 616 KB | ~12 .md | Internal meta-builder agents |
| `dist/` | 5.4 MB | ~980 | Compiled output + source maps |
| `assets/` (visible) | 7.2 MB | ~60 | Intended primitives (agents, skills, commands, workflows, references, templates) |

**The `assets/.hivemind/` directory is 90 MB of internal-only governance and meta-builder primitives that should NOT be shipped to end users.** These are development-time tooling (quality gates, meta-builder authoring skills, brainstorm skills) that have no use in a consumer's project.

**Resolution options:**
1. **Add `.npmignore` with `assets/.hivemind/`** — simplest, most targeted
2. **Make `files` array more specific** — replace `"assets"` with explicit subdirs: `"assets/agents"`, `"assets/skills"`, `"assets/commands"`, `"assets/workflows"`, `"assets/references"`, `"assets/templates"` (but this excludes `assets/agent-instructions/`)
3. **Move `assets/.hivemind/` out of `assets/`** — e.g., to `.hivefiver-meta-builder/` (which is already gitignored from sync-oss.yml) — but this changes the development workflow

**Recommendation:** Option 1 (`.npmignore`) is the least disruptive. Option 2 is also clean but requires maintaining the explicit list. Both should be done — `.npmignore` as belt, explicit files as suspenders.

## Common Pitfalls

### Pitfall 1: Hidden Directories in files Array
**What goes wrong:** `"files": ["assets"]` captures `assets/.hivemind/` (a hidden directory) because npm's `files` field is not path-aware — it includes everything under the specified directory, including dot-directories.
**Why it happens:** `assets/.hivemind/` was likely created for local development convenience without considering it would be bundled.
**How to avoid:** Use `.npmignore` to exclude `assets/.hivemind/`, or use explicit directory entries in `files` instead of broad globs.
**Warning signs:** `npm pack --dry-run` shows a package > 10 MB for a library that should be < 5 MB.

### Pitfall 2: publishConfig Access Default
**What goes wrong:** Without `"access": "public"` in `publishConfig`, scoped packages default to restricted (private). Even unscoped packages can hit this if npm configuration changes.
**Why it happens:** npm's default for new packages depends on account settings and registry state.
**How to avoid:** Always explicitly set `"publishConfig": { "access": "public" }` for open-source packages.

### Pitfall 3: postinstall Running During npm pack
**What goes wrong:** `prepack` runs `npm run build` which triggers `sync-assets.js` in build mode, modifying `.opencode/` during the pack process. This is correct behavior but can be surprising.
**Why it happens:** npm lifecycle scripts are designed to prepare the package before packing.
**How to avoid:** Ensure build mode is idempotent and doesn't create side effects in the consumer's project.

### Pitfall 4: Missing CLI Entry in Published Package
**What goes wrong:** `bin/hivemind.cjs` references `../dist/` paths. If `dist/` is not built before pack (or if the files array doesn't include `bin/`), the CLI breaks.
**Why it happens:** The `prepack` script handles this, but only if `npm pack` or `npm publish` triggers it.
**How to avoid:** Verify with `npm pack --dry-run` that both `bin/` and `dist/` appear, and test the CLI after extraction.

### publish.yml Template
```yaml
# Source: npm publish best practices
name: Publish to npm

on:
  push:
    tags:
      - 'v*'
  workflow_dispatch:
    inputs:
      dry_run:
        description: 'Dry run (skip actual publish)'
        type: boolean
        default: true

jobs:
  publish:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm
          registry-url: 'https://registry.npmjs.org'
      - run: npm ci
      - run: npm run typecheck
      - run: npm test
      - run: npm run build
      - name: Verify package contents
        run: npm pack --dry-run
      - name: Publish (dry run)
        if: ${{ github.event.inputs.dry_run != 'false' }}
        run: npm publish --dry-run
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
      - name: Publish
        if: ${{ github.event.inputs.dry_run == 'false' || github.event_name == 'push' }}
        run: npm publish
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
```

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `.npmignore` only | `files` array + `.npmignore` | npm 5+ | `files` is whitelist, `.npmignore` is blacklist from that whitelist |

| `dependencies` for optional | `optionalDependencies` | npm 3+ | Graceful degradation when dep fails to install |
