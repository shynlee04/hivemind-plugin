# P40.05: Fix State Persistence & Module-Level Singletons

## Problem

### Issue A: `atomic_commit: false` causes silent data loss
When `atomic_commit` is false (default if configs.json missing), ALL state changes stay in memory. There is NO flush mechanism — on process restart, everything is lost.

### Issue B: Module-level singleton state dirs
`CANONICAL_STATE_DIR` and `LEGACY_STATE_DIR` are resolved at import time using `process.cwd()`. If the plugin loads from a different cwd than the project root, state goes to wrong location.

### Issue C: Singleton config cache
`subscriber.ts` caches one config at a time. Switching project roots mid-process serves stale config.

## Scope

### Task 1: Add shutdown flush for atomic_commit:false
- **File:** `src/task-management/continuity/index.ts`
- **Approach:** Register a process shutdown handler (or hook into OpenCode lifecycle) that flushes pending state to disk
- **Alternative:** Remove the `atomic_commit:false` code path entirely — always persist
- **Decision gate:** Research which approach is safer

### Task 2: Defer state dir resolution to first use (lazy init)
- **File:** `src/task-management/continuity/index.ts:22-23`
- **Change:** Replace `const CANONICAL_STATE_DIR = resolve(...)` with a getter function that resolves on first access
- **Consider:** Accept `projectRoot` parameter instead of relying on `process.cwd()`

### Task 3: Support multi-project config cache
- **File:** `src/config/subscriber.ts:22-25`
- **Change:** Use `Map<string, HivemindConfigs>` keyed by project root instead of single cached value
- **Ensure:** Config lookups always use the correct project root

### Task 4: Respect OPENCODE_HARNESS_STATE_DIR override
- **File:** Check if environment variable override is actually used
- **Ensure:** `OPENCODE_HARNESS_STATE_DIR` takes precedence over `process.cwd()` resolution
