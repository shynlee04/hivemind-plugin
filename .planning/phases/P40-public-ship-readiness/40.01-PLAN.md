# P40.01: Fix 3 Fire-and-Forget Dispatch Paths

**Goal:** Replace all 3 `setTimeout` fire-and-forget dispatch paths with deferred-promise pattern that preserves reentrant deadlock prevention while enabling error propagation.

**Why:** All 3 paths return `{ success: true }` before the SDK dispatch fires, making failures invisible to callers, session-tracker, and users. The fix pattern is identical for all 3: wrap `setTimeout` body in `new Promise()` that resolves/rejects after SDK call completes.

## Context

- **Research:** @.planning/phases/P40-public-ship-readiness/40.01-RESEARCH.md — full root cause analysis of all 3 paths, SDK surface, dependency map
- **Source (Path 1):** `src/tools/session/dispatch-command.ts:64-116` — `setTimeout(async () => { ... }, 50)` wraps `client.session.prompt()`, always returns `{ success: true }` at line 116
- **Source (Path 2):** `src/tools/session/execute-slash-command.ts:500-515` — same pattern wraps `client.session.command()`, returns at line 517
- **Source (Path 3):** `src/tools/session/execute-slash-command.ts:313` — calls `dispatchCommand()` without `await`, so even though Path 1 returns a result, it's discarded here. Path 3 at line 453 (child session) does `await` but inherits Path 1's premature return

**Key invariant:** The 50ms `setTimeout` delay is INTENTIONAL to prevent reentrant deadlock. It MUST be preserved. The fix wraps the setTimeout in a Promise, not removes it.

## Risk Section

### CRITICAL RISK: Reentrant Deadlock Regression

**What:** The entire reason `setTimeout` exists is to prevent OpenCode SDK deadlock when `session.prompt()` is called on the same session currently executing a tool. If the deferred-promise pattern somehow allows the SDK call to fire before the tool returns, the session's internal lock deadlocks.

**Why the fix is safe:** The deferred-promise pattern preserves `setTimeout(fn, 50)`. The Promise wraps the setTimeout — it does NOT remove it. The tool still returns (resolving the tool call) before the setTimeout callback fires. The only change is that the Promise captures the result instead of discarding it.

**Regression test:** After implementation, verify that the 50ms delay still exists by checking that `vi.advanceTimersByTime(50)` is required to resolve the Promise. If the Promise resolves immediately (without timer advance), the deadlock protection has been removed.

### MEDIUM RISK: Tool Return Latency Increase

**What:** Callers that previously got immediate `{ success: true }` will now wait for the SDK dispatch to complete (~50ms + SDK round-trip).

**Mitigation:** This is correct behavior. The tool return SHOULD reflect actual dispatch status. The 50ms minimum from setTimeout is preserved. SDK round-trip for `session.prompt()` is typically 100-500ms — acceptable for a tool that reports dispatch status.

### LOW RISK: Agent Restore Timing

**What:** The restore prompt at line 100 fires inside the same Promise as the initial dispatch. If the initial dispatch takes a long time, the restore is delayed proportionally.

**Mitigation:** This is the current behavior (both are in the same setTimeout callback). No regression. The fix only changes error propagation, not timing.
