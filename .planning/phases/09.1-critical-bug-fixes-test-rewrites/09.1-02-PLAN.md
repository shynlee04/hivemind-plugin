    **Step 1: Create `tests/lib/helpers/in-memory-client.ts`**

    Create the directory `tests/lib/helpers/` first (`mkdir -p tests/lib/helpers`).

    Implement `createInMemoryClient()` that returns an object matching the `OpenCodeClient` surface with in-memory Maps. The adapter must provide:

    - `session.get({ path: { id } })` → looks up session in sessions Map, returns `{ data: session }` or throws
    - `session.create({ body })` → creates session with auto-generated ID, stores in Map, returns `{ data: { id, ...body } }`
    - `session.messages({ path: { id } })` → returns messages from messages Map for that session
    - `session.status()` → builds status map from sessions Map
    - `session.prompt(...)` → no-op, returns `{ data: {} }`
    - `session.promptAsync(...)` → no-op, returns `{ status: 204 }`
    - `session.abort(...)` → no-op, returns `{ data: {} }`
    - `tui.showToast(...)` → no-op vi.fn()

    Test helper methods for programmatic state manipulation:
    - `_sessions: Map<string, ...>` — direct access to sessions store
    - `_messages: Map<string, unknown[]>` — direct access to messages store
    - `_setStatus(sessionID: string, type: string)` — mutate session status
    - `_addMessage(sessionID: string, msg: unknown)` — add message to session
    - `_setGetSessionError(error: Error)` — inject SDK failure for error-path testing

    All SDK methods should be `vi.fn(async ...)` so tests can assert call counts.

    The file should be under 120 LOC. Use the example from RESEARCH.md Pattern 1 as the starting point but adapt it to match the actual `OpenCodeClient` surface from `src/lib/session-api.ts`.

    **Step 2: Rewrite `tests/lib/lifecycle-background-observer.test.ts`**

    DELETE the entire existing content (576 LOC of mock-heavy tests). Replace with a clean test file under 300 LOC that:

    1. Imports `observeBackgroundCompletion` from `src/lib/lifecycle-background-observer.ts`
    2. Imports `createInMemoryClient` from `./helpers/in-memory-client.ts`
    3. Uses `vi.useFakeTimers()` for time control
    4. Does NOT use `vi.mock()` for ANY internal module — only the in-memory client replaces SDK transport

    Test cases to include (each with WHY/WITH/WHAT/HOW rationale comment per D-23):

    1. **"completes when session goes idle after producing assistant output"** — Core happy path. Session starts busy, transitions to idle after assistant messages exist. Verify patchLifecycle is called with status="completed".

    2. **"does NOT complete when session is idle but has no assistant messages"** (Bug 3 regression D-17) — Session goes idle with only user prompt. Observer must continue polling, NOT complete.

    3. **"reports error when session is deleted during observation"** — Session disappears from getSession. Observer must call patchLifecycle with status="error" and error containing "deleted".

    4. **"reports error when session enters retry state"** — Session status becomes "retry". Observer must call patchLifecycle with status="error" and error containing "retry".

    5. **"times out when poll deadline expires"** — Session stays busy indefinitely. Observer must call patchLifecycle with status="error" and error containing "timed out".

    6. **"does not set seenBusy for unknown status"** (Bug 2 regression D-16) — Session returns { type: "unknown" }. Observer must NOT set seenBusy. Later transitioning to idle without ever being "busy" should still complete (if startup window elapsed and evidence exists).

    7. **"releases concurrency queue on completion"** — Verify releaseQueue is called exactly once regardless of completion path (completed, error, timeout).

    Helper setup for each test:
    - `beforeEach`: create fresh in-memory client, set up mock continuity/patchLifecycle/releaseQueue functions
    - Create a `setupObserver` helper that wires all args with defaults, returning controls for advancing time and inspecting calls

    Use `vi.advanceTimersByTime(3000)` to advance through poll intervals (DEFAULT_POLL_INTERVAL_MS = 3000).
    Use a custom `sleepFn` that resolves after `vi.advanceTimersByTime()` to control the polling loop.
