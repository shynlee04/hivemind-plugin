# Phase 24.1 — Agent Hierarchy Restructure — Research

**Researched:** 2026-05-26
**Domain:** OpenCode Agent Schema — Frontmatter Standardization, Hierarchy Optimization, 45→30 Agent Consolidation
**Confidence:** HIGH

## Summary

Phase 24.1 restructures the Hivemind hm-* agent hierarchy from 45 agents (1 L0 + 1 L1 + 43 L2/L3) to 30 agents (1 L0 + 0 L1 + 29 L2) per the decisions in CONTEXT.md. The L1 coordinator layer is removed entirely (merged into programmatic features). All 43 L2/L3 agents are consolidated into 29 domain-focused specialists with non-overlapping domains.

**Key finding:** The current frontmatter format includes **16 fields** (`name`, `temperature`, `steps`, `color`, `permission`, `skills`, `instruction`, `reasoningEffort`, `depth`, `lineage`, `domain`, `delegation_routing`, `intent_classification`, `mode`, `description`, `tools`) — far exceeding the correct OpenCode schema which specifies only `description`, `mode`, `hidden`, `temperature`, `steps`, `permission`, `model`, `color`, `disable`, `top_p`. Per CONTEXT.md D-24-02, shipped files should be **minimal**: only `description`, `mode: all`, `hidden: true`. Runtime overrides (temperature, steps, permission) are set by orchestrator, not stored in files.

**Primary recommendation:** Execute a two-pass restructuring: **Pass 1** (Phase 24.1) — consolidate 45→30 agents, correct frontmatter to minimal format, delete L1, rename agents to drop `-lX-` tier suffix. **Pass 2** (Phase 24.2) — write full agent profile bodies with proper execution flows, success criteria, and artifact contracts.

**Note:** GSD repo change (original abandoned, new maintainer at open-gsd/get-shit-done-redux) does NOT affect this phase — it deals exclusively with OpenCode's native agent schema, which is verified via Context7 against anomalyco/opencode.

## OpenCode Schema Validation Findings

**Source:** [CITED: anomalyco/opencode docs — agent definition](https://github.com/anomalyco/opencode/blob/dev/packages/web/src/content/docs/agents.mdx)
**Verified:** 2026-05-26 via Context7 query on `/anomalyco/opencode`

### Valid Frontmatter Fields

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `description` | string | ✅ YES | One-liner role description |
| `mode` | `all` / `primary` / `subagent` | ❌ (default: `all`) | `all` = Tab cycle + @ invocation; `primary` = Tab only; `subagent` = @ only |
| `hidden` | boolean | ❌ | Hides from @ autocomplete (only applies with `mode: subagent`) |
| `temperature` | 0.0-1.0 | ❌ | Runtime override — best practice: set in orchestrator config, not file |
| `steps` | integer | ❌ | Max agentic iterations — runtime override |
| `permission` | object | ❌ | Fine-grained tool control — runtime override by orchestrator |
| `model` | string | ❌ | Model override — runtime override |
| `color` | string (hex/theme) | ❌ | UI concern — optional |
| `disable` | boolean | ❌ | Disable agent entirely |
| `top_p` | 0.0-1.0 | ❌ | Response diversity |
| `tools` | array | ❌ | **DEPRECATED** per OpenCode docs — use `permission` instead |

### Key Findings

1. **`mode: all` + `hidden: true` is the correct combination** for L2 specialists that should be callable by orchestrator but hidden from user @ menu. OpenCode docs confirm: `hidden: true` hides from @ autocomplete but agents remain invocable via task tool if permissions allow.

2. **The `tools` field is DEPRECATED.** Current hm-* agents do NOT use `tools` (good), but this confirms the field is correct to omit.

3. **`name` field is redundant** — filename (without `.md`) is the agent name. Many current agents include `name:` in frontmatter. This should be removed.

4. **`temperature`, `steps`, `color`, `permission`** are all valid OpenCode fields BUT CONTEXT.md D-24-02 specifies they should not be in shipped files. Runtime overrides are set by orchestrator based on task type. This is architecturally correct — the orchestrator should decide temperature (high for creative tasks like planning, low for deterministic tasks like execution), not the agent file.

5. **`depth`, `lineage`, `domain`, `skills`, `instruction`** are custom Hivemind fields. They have no OpenCode schema meaning. They should be removed from shipped files. Skills loading and instruction references are orchestrator concern.

## Common Pitfalls

### Pitfall 1: Forgetting to Update Cross-References
**What goes wrong:** hm-orchestrator.md references agents by their old names (e.g., `hm-l2-critic`, `hm-l1-coordinator`) 
**Why it happens:** Orchestrator file is large (806 LOC) with many hardcoded agent name references
**How to avoid:** After all file operations, grep for every old agent name in the codebase and update references
**Warning signs:** Orchestrator attempts to dispatch to nonexistent agent names

### Pitfall 2: Accidental Deletion of Agent Body Content
**What goes wrong:** Deleting an agent file permanently removes its domain knowledge
**Why it happens:** "Remove" = `rm` is fast and irreversible
**How to avoid:** Before deleting, extract domain knowledge to `.planning/references/` or a skill file. Archive deleted files to `.planning/archive/agents/` instead of hard-deleting

### Pitfall 3: Phase 24.1 / 24.2 Boundary Blur
**What goes wrong:** Phase 24.1 starts writing full agent profile bodies, scope-creeping into Phase 24.2 territory
**Why it happens:** The temptation to "just write the body while we're here" + many agents will look empty
**How to avoid:** Strict pass separation. Phase 24.1 = file operations + minimal frontmatter ONLY. Phase 24.2 = full profile writing. Use TODO comments as handoff

### Pitfall 4: L1 Coordinator Removal Breaks Orchestration Flow
**What goes wrong:** The L1 coordinator's wave management, checkpoint gates, and result consolidation functionality is lost
**Why it happens:** The agent file is deleted but the programmatic replacement doesn't exist yet (it's spread across session-tracker, delegation-status, coordination tools)
**How to avoid:** Before deleting L1, confirm that ALL its coordination functionality is available through programmatic features (session-tracker for tracking, delegation-status for monitoring, native `task` tool for dispatch). The orchestration flow becomes: L0 → L2 (fast-path) or L0 → coordination features → L2 (coordinated path)
