<objective>
Fix F-07 (recovery blindness) by making `childToRootMain` mapping survive restart via continuity tree rebuild, and add F-13 stack overflow protection to `ensureAncestorRoute()`.

**Purpose:** After a restart, `buildFromDisk()` repopulates `childToParent` and `parentToChildren` from `session-continuity.json` files, but `childToRootMain` mapping is NOT rebuilt — it's only populated during live registration via `registerChild()`. This means all previously registered children have no rootMain after restart, causing `resolveWriteParent()` to fall back to the immediate parent instead of the root main directory. Combined with `MAX_DEPTH=20`, the recursion in `ensureAncestorRoute()` is protected against corrupt SDK data or deep chains.

**Design decision for DAG hierarchies:** `rebuildChildToRootMain()` uses the existing `resolveRootMain()` which walks `childToParent` chain until finding a session NOT in `childToParent` (the root). For DAG structures (child under multiple parents), the FIRST registered parent determines the root. This matches existing runtime behavior.

**Output:** `hierarchy-index.ts` with `rebuildChildToRootMain()` called at end of `buildFromDisk()`; `session-tracker/index.ts` with `MAX_DEPTH=20` guard.
</objective>

## Source Context

### hierarchy-index.ts L94-137 — buildFromDisk()

The existing second pass (L127-136) iterates `childToParent` entries and resolves rootMain for those NOT already in the map:

```typescript
for (const [childID] of this.childToParent) {
  if (!this.childToRootMain.has(childID)) {
    const root = this.resolveRootMain(childID)
    if (root) this.childToRootMain.set(childID, root)
  }
}
```

BUT this code path only fires for children whose parent chain was fully reconstructed during the first pass. Children whose continuity files were read AFTER their parent's rootMain was resolved (or children registered via reverse-order) are already in `childToRootMain` and skipped. The gap: children whose parent was NOT in `childToRootMain` at first pass time are also skipped if the second pass condition `!this.childToRootMain.has(childID)` is true but `resolveRootMain()` returns undefined because the chain is incomplete.

After `buildFromDisk()`, ALL children in `childToParent` should have a rootMain.

### hierarchy-index.ts L259-268 — resolveRootMain()

```typescript
private resolveRootMain(childID: string): string | undefined {
  let current = childID
  const visited = new Set<string>()
  while (this.childToParent.has(current) && !visited.has(current)) {
    visited.add(current)
    current = this.childToParent.get(current)!
  }
  return this.childToParent.has(current) ? undefined : current
}
```

This walks up the chain until it finds a session that is NOT in `childToParent`. Returns `undefined` if the walk loops back to a visited node (cycle detected).

### session-tracker/index.ts L375-388 — ensureAncestorRoute()

```typescript
private async ensureAncestorRoute(sessionID: string, seen: Set<string>): Promise<void> {
  if (seen.has(sessionID)) return
  seen.add(sessionID)
  const session = await this.getSessionSafely(sessionID)
  const parentID = session && typeof session === "object" && "parentID" in session
    ? (session as { parentID?: string }).parentID
    : undefined
  if (!parentID) return
  await this.ensureAncestorRoute(parentID, seen)
  if (!this.hierarchyIndex.isChild(sessionID)) {
    this.hierarchyIndex.registerChild(parentID, sessionID)
  }
}
```

NO `MAX_DEPTH` guard. No `depth` parameter. Recursively calls itself, going deeper for each ancestor.

    ## Step 1: Add rebuildChildToRootMain() method

    Add this method after `getDepth()` (after L300), before `get size()` (L305):

    ```typescript
    /**
     * Rebuilds the childToRootMain mapping for all registered children.
     *
     * Called at the end of buildFromDisk() to ensure that every child in the
     * hierarchy index has a root main session ID. Uses resolveRootMain() to
     * walk each child's parent chain.
     *
     * For DAG hierarchies (child under multiple parents), uses first-found-wins
     * strategy — the first parent chain that resolves determines the rootMain.
     * This matches existing runtime behavior where registerChild() propagates
     * rootMain from the parent at registration time.
     *
     * Per G-2: No new file format. Rebuilds from the in-memory childToParent
     * map which was populated from session-continuity.json files.
     */
    rebuildChildToRootMain(): void {
      for (const [childID] of this.childToParent) {
        if (!this.childToRootMain.has(childID)) {
          const root = this.resolveRootMain(childID)
          if (root) {
            this.childToRootMain.set(childID, root)
          }
        }
      }
    }
    ```

    ## Step 2: Call rebuildChildToRootMain() at the end of buildFromDisk()

    At the end of `buildFromDisk()` — after the second pass at L131-136 (which resolves rootMain for children already in the map during pass 1) — add a third step:

    ```typescript
      // Second pass: resolve root main for each registered child (D-03, D-08).
      // After all child→parent relationships are loaded, walk each chain
      // to determine the root main session ID. This handles reverse-order
      // registration (e.g., L1→L2 registered before root→L1).
      for (const [childID] of this.childToParent) {
        if (!this.childToRootMain.has(childID)) {
          const root = this.resolveRootMain(childID)
          if (root) this.childToRootMain.set(childID, root)
        }
      }

      // Third pass (G-2 / REQ-21-05): Ensure EVERY registered child has
      // a rootMain. The second pass only resolves children NOT already in
      // childToRootMain. This pass catches any children whose parent chain
      // was fully resolved after their iteration in the second pass.
      this.rebuildChildToRootMain()
    ```

    The key insight: the second pass at L131-136 only handles children where `!this.childToRootMain.has(childID)` is true at that moment. If a child's parent chain resolves during the iteration (unlikely but possible with complex topologies), the child is added to `childToRootMain` via `set`. The new third pass catches any remaining unassigned children.

    ## Replace ensureAncestorRoute() with MAX_DEPTH guard

    Replace the existing method (L375-388):

    ```typescript
    /**
     * Recursively registers a parent session's own parent chain from SDK data.
     *
     * Includes MAX_DEPTH guard (F-13 / REQ-21-07) to prevent stack overflow
     * on corrupt SDK data or deep ancestor chains.
     *
     * @param sessionID - Session whose ancestors should be registered.
     * @param seen - Cycle guard for defensive SDK data handling.
     * @param depth - Current recursion depth (internal, pass 0 on first call).
     */
    private async ensureAncestorRoute(
      sessionID: string,
      seen: Set<string>,
      depth: number = 0,
    ): Promise<void> {
      const MAX_DEPTH = 20

      // F-13: Stack overflow guard — return gracefully with warning
      if (depth > MAX_DEPTH) {
        // hierarchy-index.ts has no client reference, but this code IS in
        // session-tracker/index.ts which has access to this.client
        void this.client.app?.log?.({
          body: {
            service: "session-tracker",
            level: "warn",
            message: `[Harness] Session tracker: ensureAncestorRoute exceeded MAX_DEPTH=${MAX_DEPTH} at "${sessionID}" — truncating to prevent stack overflow`,
          },
        })
        return
      }

      if (seen.has(sessionID)) return
      seen.add(sessionID)

      const session = await this.getSessionSafely(sessionID)
      const parentID =
        session && typeof session === "object" && "parentID" in session
          ? (session as { parentID?: string }).parentID
          : undefined
      if (!parentID) return

      await this.ensureAncestorRoute(parentID, seen, depth + 1)
      if (!this.hierarchyIndex.isChild(sessionID)) {
        this.hierarchyIndex.registerChild(parentID, sessionID)
      }
    }
    ```
