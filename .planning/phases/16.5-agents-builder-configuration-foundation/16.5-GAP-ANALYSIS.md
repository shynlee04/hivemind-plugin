# Phase 16.5 Gap Analysis: PRD vs Implementation

## Executive Summary

Phase 16.5 delivered a solid foundational schema kernel, cross-primitive validator, config compiler, and workflow skill, but falls significantly short of the original PRD's vision. The implementation covers approximately 45-50% of the PRD's scope, with critical gaps in natural-language auto-activation depth, advanced framework stacking, builtin object loading, runtime validation, eval harness metrics, and post-restart verification. The system works for basic create/update operations but cannot reliably detect indirect intent, validate across framework boundaries, or guarantee ecosystem coherence.

## Critical Gaps (Top 10)

### 1. No Builtin/Existing Object Discovery — BLOCKER
**PRD Expected:** "Load and inspect ALL relevant objects — builtin + custom agents, commands, tools, skills, permissions, config, MCP tools, workflows, prompts, rules, hooks, runtime settings, orchestration metadata."
**Implemented:** The system can only CREATE new primitives. It never reads existing ones. The `configure-primitive` tool writes files but cannot read them. The workflow skill mentions "scanning" in Turn 0/1 but has no actual file-system scanning implementation. The cross-primitive validator accepts pre-loaded `PrimitiveMap` but provides no mechanism to populate that map from disk.
**Impact:** Users cannot modify existing agents, audit current configurations, or detect conflicts against already-installed primitives. The system is write-only.
**Evidence:** `src/tools/configure-primitive.ts:43-112` — only write path exists. `src/lib/cross-primitive-validator.ts:37-44` — `PrimitiveMap` is input-only, no loader. `.opencode/skills/hivefiver-agent-config/SKILL.md:25-36` — describes scanning but delegates to non-existent implementation.

### 2. Zero Framework-Aware Stacking Logic — BLOCKER
**PRD Expected:** "Advanced Stacking Compatibility — Hivemind, GSD, BMAD, Speckit, other frameworks. Must parse their metadata, reason about boundaries, avoid conflicts."
**Implemented:** No framework metadata parsing exists. No boundary detection. The skill is literally named `hivefiver-agent-config` and references `hivefiver-orchestrator`, making it framework-specific rather than framework-compatible.
**Impact:** Cannot safely coexist with GSD's `.planning/` directory, BMAD's `bmad.yaml`, Speckit's configs, or any other framework. Conflicts will go undetected. The branding actively prevents adoption by non-Hivemind users.
**Evidence:** `.opencode/skills/hivefiver-agent-config/SKILL.md:1-3` — name and description contain "hivefiver". `16.5-CRITICAL-AUDIT.md:56-62` — D6 section documents this as BLOCKER. `src/lib/cross-primitive-validator.ts` — no framework metadata fields in `ValidationCategory`.

### 3. Schema Rigid `.strict()` Rejects Future Fields — BLOCKER
**PRD Expected:** Configurations should work after OpenCode updates and remain compatible with future extensions.
**Implemented:** All 7 schemas use `.strict()` or fixed enums (`AgentModeEnum`, `PermissionKeySchema`, `ConfigPrecedenceLevelSchema`, `THEME_COLOR_NAMES`). If OpenCode adds a new frontmatter field, permission key, or config level, the harness will reject valid configs.
**Impact:** This is a compatibility time-bomb. A single OpenCode update could break all user configurations across all installed projects. The CRITICAL-AUDIT identified this as 6 BLOCKERs (D7).
**Evidence:** `src/schema-kernel/agent-frontmatter.schema.ts:124` — `.strict()`. `src/schema-kernel/permission.schema.ts:19-34` — hardcoded 14-key enum. `16.5-CRITICAL-AUDIT.md:64-74` — D7 audit findings.

### 4. Missing "Work Well Together" Runtime Validation — CRITICAL
**PRD Expected:** "Harness-Build Validation Layer — loading order, resolution order, hierarchy, inheritance, pipeline position, chain connectivity."
**Implemented:** Only static structural validation exists (agent-command binding, permission deadlock, role overlap, missing skill, MCP gap, rule file gap). No runtime or topological validation.
**Impact:** Users can create configurations that pass validation but fail at runtime due to loading order issues, circular dependencies, or pipeline misalignment. The "valid" boolean is misleading.
**Evidence:** `src/lib/cross-primitive-validator.ts:50-62` — 6 detectors, all static. Categories `missing-model` and `command-skill-ref` declared but unwired (`16.5-VERIFICATION.md:97-98`). No loading-order, inheritance, or pipeline validation functions exist.

### 5. No Eval Harness Success Metrics — CRITICAL
**PRD Expected:** "Eval Harness Success Metrics — correctness, stability, no crash, no overlap, no conflict, successful orchestration, ecosystem coherence."
**Implemented:** Unit tests verify schema parsing and compilation round-trips. No metrics for: correctness of generated configs, stability under load, runtime crash detection, overlap quantification, conflict resolution success, orchestration success rate, or ecosystem coherence score.
**Impact:** No way to measure if the system actually works in production. No regression detection for config quality. No benchmarking.
**Evidence:** `tests/` — 680 tests, all unit/integration. No eval/ directory. No benchmark files. `16.5-VERIFICATION.md:123-124` — metrics are test counts, not quality metrics.

### 6. Single-Primitive-Per-Call Batch Limitation — CRITICAL
**PRD Expected:** "Batch and Group Configuration — individual + grouped + batch across ALL primitive types."
**Implemented:** `batchCompile()` accepts an array but each element is a single primitive type. The `configure-primitive` tool only accepts one `primitive` enum value per call. No mixed-primitive batch (e.g., create agent+command+skill in one transaction).
**Impact:** Users must make separate calls for each primitive type, losing atomicity across types. Cannot configure a complete feature slice (agent + its command + its skill) in one operation.
**Evidence:** `src/tools/configure-primitive.ts:18` — `primitive: z.enum(["agent", "command", "skill"])` singular. `src/lib/config-compiler.ts:229-260` — `batchCompile` iterates but doesn't mix types in validation.

### 7. No Post-Restart Validation — CRITICAL
**PRD Expected:** "All generated configs work after OpenCode restart" + "remain compatible under advanced stacking" + "work after restart without conflict."
**Implemented:** No restart simulation. No validation that compiled `.md` files are actually discoverable by OpenCode. No check that agents appear in `@` autocomplete. No verification that commands are registered.
**Impact:** Configurations may compile successfully but fail to load at runtime. Users only discover this after restarting OpenCode.
**Evidence:** `16.5-VERIFICATION.md:100-105` — explicitly flagged as requiring human verification. No programmatic test exists. `src/tools/configure-primitive.ts:99-112` — writes file but doesn't verify OpenCode discovery.

### 8. Literal-Keyword Intent Detection Only — HIGH
**PRD Expected:** "Natural-Language Auto-Activation — Detect explicit/implicit/indirect intent, no slash commands needed."
**Implemented:** Keyword-based trigger lists with exact/literal matching ("configure" + "agent", "update" + "agent", etc.). No semantic understanding, no indirect intent detection (e.g., "my agents are too slow" → tune temperature), no context from conversation history.
**Impact:** Users must use specific keywords. Natural variations, indirect requests, or context-aware triggers won't work. The "auto" in auto-activation is overstated.
**Evidence:** `.opencode/agents/hivefiver-orchestrator.md:163-192` — trigger lists are regex/keyword based. No NLP, no embedding matching, no semantic classification.

### 9. Missing Extensibility for "Other Fields" — HIGH
**PRD Expected:** "ALL OpenCode primitives... and any other relevant primitives or metadata fields treated broadly."
**Implemented:** 7 schema files cover: agent, command, permission, skill, MCP server, tool definition, config precedence. Missing schemas for: rules, prompts, workflows, hooks, injection points, runtime settings, orchestration metadata. No plugin architecture for adding new schemas.
**Impact:** Cannot configure or validate rules, prompts, workflows, hooks, or runtime settings. As OpenCode adds new primitive types, the harness cannot support them without code changes.
**Evidence:** `src/schema-kernel/` — no `rules.schema.ts`, `prompt.schema.ts`, `workflow.schema.ts`, `hook.schema.ts`, `runtime-setting.schema.ts`. `src/schema-kernel/index.ts` — no exports for missing schemas.

### 10. Insufficient Build-Time Validation Tooling — HIGH
**PRD Expected:** "Test Scripts and Bin Tools for Hivefiver — Build-time validation tools, metadata load-order validators, runtime path validators."
**Implemented:** Single script `bin/validate-agent-config.sh` runs typecheck + vitest + schema export check. No metadata load-order validator. No runtime path validator. No framework-specific validation.
**Impact:** CI pipelines cannot validate configuration loading order, path correctness, or framework compatibility. The "build-time" promise is unfulfilled.
**Evidence:** `bin/validate-agent-config.sh` — 38 lines, generic checks only. No `bin/validate-load-order.sh`, `bin/validate-runtime-paths.sh`, `bin/validate-framework-compatibility.sh`.

**What to build:**
1. Remove `.strict()` from all schemas, replace with `.passthrough()` + unknown-key warnings

3. Create `src/lib/primitive-loader.ts` — filesystem scanner that populates `PrimitiveMap` from `.opencode/`, `~/.config/opencode/`, and `opencode.json`
4. Add `read`/`inspect` operations to `configure-primitive` tool (currently write-only)
