# Phase 16.5: Agents Builder Configuration Foundation

## Goal

Build the foundation for a configurable agents builder system wired to the hivefiver module, enabling:
1. Auto-detection of agent configuration intent through natural language
2. Programmatic compilation and validation of OpenCode agent configurations
3. Conflict-free stacking with other frameworks (GSD, BMAD, Speckit, etc.)
4. Support for both JSON and YAML config formats
5. Batch and group configuration of agents and other OpenCode primitives

## Requirements

### R-01: Zod Schema Foundation (CYCLE 1)
Separate Zod schemas for all OpenCode primitives in `src/schema-kernel/`:
- Agent frontmatter schema (14+ fields validated)
- Command frontmatter schema
- Permission ruleset schema (pattern-based + rules-based)
- Skill metadata schema
- MCP server configuration schema
- Tool definition schema
- Config precedence schema

### R-02: Cross-Primitive Validator (CYCLE 2)
`src/lib/cross-primitive-validator.ts` detecting conflicts across:
- Agent↔Command bindings
- Agent↔Skill dependencies
- Agent↔Agent role overlaps
- Permission conflicts (global vs agent-level)
- MCP server references
- Rule file references

### R-03: Config Compiler Tool (CYCLE 3)
`src/tools/configure-primitive.ts` + `src/lib/config-compiler.ts`:
- JSON/YAML → .md compilation
- Reverse compilation (.md → JSON)
- Dry-run mode
- Atomic batch operations

### R-04: Agent Config Workflow Skill (CYCLE 4)
`.opencode/skills/hivefiver-agent-config/SKILL.md`:
- Investigate → Collect → Proposal → Validate → Compile → Test → Save
- Turn-based, bite-size, guided configuration
- Supports individual and batch operations

### R-05: Enhanced Orchestrator + Auto-Detection (CYCLE 5)
- Intent classification for "configure agents" in hivefiver-orchestrator
- New `hf-configure` command with --from-file, --scope, --dry-run flags
- Auto-activation without requiring slash commands

### R-06: Test Harness (CYCLE 6)
- Unit tests for all schemas
- Integration tests for compiler round-trips
- End-to-end workflow tests
- CLI validation script for CI

## Connection to Phase 16.4

Phase 16.4 established the architecture baseline with:
- 14 locked requirements for the harness
- Module boundaries (src/lib, src/tools, schema-kernel)
- State ownership rules
- Lifecycle boundaries
- Code/tree organization standards

Phase 16.5 builds on top of that baseline without conflicting:
- New schemas extend schema-kernel/ (established in 16.4)
- New tools follow the plugin tool contract (established in 16.4)
- New lib modules follow the dependency rules (established in 16.4)
- Max module size: 500 LOC (16.4 standard)
