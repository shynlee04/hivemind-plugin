## Summary

Phase 24.2 analyzes the GSD packed repository (`open-gsd/get-shit-done-redux`, outputId `8f249957d3b9d873`) to synthesize what Hivemind must adopt from GSD's mature architecture and what it must build uniquely. GSD has evolved through ~3,700 changesets over 4+ years, resulting in a battle-tested meta-prompting framework with 33 agents, 44 commands, ~80 workflows, and 183 references/templates/hooks. Hivemind, by contrast, is building a **programmatic runtime composition engine** — not a meta-prompting layer — using TypeScript tools, hooks, and OpenCode plugin primitives.

**Key finding:** GSD excels at *what* to do (workflow orchestration, artifact conventions, agent specialization) but handles *how* to do it through procedural agent profiles and markdown workflows. Hivemind must programmatize these patterns — replacing procedural agent logic with tool-based delegation, session tracking, and lifecycle hooks. GSD's artifact naming conventions and command→workflow→agent routing layer are mature patterns worth adopting; its agent profile format (tools/color in frontmatter) is deprecated by the OpenCode schema and should NOT be copied.

**Primary recommendation:** Hivemind should adopt GSD's three-layer routing (Command→Workflow→Agent), artifact naming conventions, and agent specialization model, but build programmatic replacements for GSD's procedural looping/gating/hierarchy logic. Phase 24.2 specifically covers agent profile body quality enforcement — ensuring the 30 hm-* agents defined in Phase 24.1 have proper execution flows, success criteria, and artifact contracts that work with Hivemind's programmatic features rather than containing inline orchestration logic.

## 1. GSD Architecture Overview

### 1.1 Three-Layer Routing Architecture

GSD uses a clean three-layer routing architecture that Hivemind should adopt as a pattern:

```
User Command              Layer 1: COMMAND
  │                       commands/gsd/*.md (44 files)
  │                       YAML frontmatter + execution_context routing
  ▼
Workflow Orchestration    Layer 2: WORKFLOW
  │                       get-shit-done/workflows/*.md (~80 files)
  │                       Step-by-step orchestration with gate sub-files
  ▼
Specialist Agents         Layer 3: AGENT
  │                       agents/gsd-*.md (33 files)
  │                       Focused role definitions with tool access
  ▼
CLI Tools Layer           gsd-tools.cjs + bin/lib/*.cjs (33 modules)
  │                       CommandRoutingHub, state, phase, roadmap, config, etc.
  ▼
File System               .planning/ (PROJECT.md, ROADMAP.md, STATE.md,
                          config.json, phases/, research/, templates/)
```

**Key observations:**
- Commands are **thin** — they parse args, load context, then delegate to workflow files via `execution_context`
- Workflows are **procedural** — step-by-step markdown with SDK calls and agent spawn instructions
- Agents are **specialized** — each has a focused domain, limited tools, and produces specific artifacts
- CLI tools layer provides **programmatic utilities** — state management, phase operations, config, templates
- File system is the **canonical state store** — no database, no server

### 1.2 Command Frontmatter Format (GSD)

```yaml
---
name: gsd:plan-phase
description: Create detailed phase plan (PLAN.md) with verification loop
argument-hint: "[phase] [--auto] [--research] [--skip-research] [--research-phase <N>] [--view] [--gaps] [--skip-verify]"
allowed-tools:
  - Read
  - Write
  - Bash
  - Glob
  - Grep
  - Agent
  - AskUserQuestion
  - WebFetch
  - mcp__context7__*
requires: [discuss-phase, phase, review, update]
---
```

**Notable:** GSD uses `allowed-tools` (not `tools` — the deprecated field). It also uses `requires` for dependency tracking between commands. The `name:` field uses colon form (`gsd:plan-phase`) for Gemini compatibility. Commands route to workflows via `execution_context` frontmatter: `@~/.claude/get-shit-done/workflows/plan-phase.md`

### 1.3 Agent Frontmatter Format (GSD)

```yaml
---
name: gsd-executor
description: Executes GSD plans with atomic commits, deviation handling, checkpoint protocols, and state management. Spawned by execute-phase orchestrator or execute-plan command.
tools: Read, Write, Edit, Bash, Grep, Glob, mcp__context7__*
color: yellow
---
```

**Critical finding for Phase 24.2:** GSD agents use `tools:` and `color:` in frontmatter. Per the OpenCode schema (validated in Phase 24.1), **`tools:` is DEPRECATED** — OpenCode uses `permission:` instead. Hivemind's D-24-02 decision removes BOTH `tools:` and `color:` from shipped files. **GSD's format is NOT the target for Hivemind** — Hivemind's format is the minimal OpenCode format from D-24-02.

### 1.4 Agent Body Format (GSD)

GSD agents use extensive XML-style tag sections in their body:

```
<role>     — Defines what the agent is and does
<step>     — Execution steps with priority ordering
<process>  — Full process definition
<context>  — Context loading instructions
<rule>     — Behavioral rules (numbered)
```

Each agent body is 200-800 LOC, containing detailed procedural instructions for the agent to follow. This is **the opposite** of Hivemind's D-24-02 goal of minimal agent files. GSD puts orchestration logic inside agent profiles; Hivemind puts orchestration in programmatic features.

### 1.5 Two-Stage Routing (Namespace Skills)

GSD v1.40 introduced six namespace meta-skills to reduce the eager skill listing token cost:
- `gsd-workflow` — Phase pipeline: discuss/plan/execute/verify/phase/progress
- `gsd-project` — Project lifecycle: new/complete/audit
- `gsd-quality` — Quality: code-review/debug/audit/security
- `gsd-context` — Context: map/sync/inbox
- `gsd-manage` — Management: config/settings/workspace
- `gsd-ideate` — Ideation: explore/spike/sketch

The model sees 6 namespace routers (~120 tokens) instead of ~86 skills (~2,150 tokens). This is relevant to Hivemind's future command system (Phase 24.3+).

## 2. GSD's Strengths for Hivemind to Adopt

### 2.1 Command→Workflow→Agent Routing

**Adopt as pattern:** GSD's clean three-layer separation (thin command → procedural workflow → specialist agent) is mature and battle-tested. Hivemind's equivalent would be:
- **Hm Command** (`.opencode/commands/hm-*.md`) — Parses args, routes to workflow
- **Hm Workflow** (`.opencode/workflows/hm-*.md` or similar) — Step-by-step orchestration
- **Hm Agent** (`.opencode/agents/hm-*.md`) — Specialist with focused domain

**GSD's `allowed-tools` field** is a useful pattern for command files (to restrict which tools an agent can use during that command), but Hivemind should use OpenCode's `permission` instead of `allowed-tools` since `tools`/`allowed-tools` is deprecated.

### 2.2 Agent Specialization (One Domain Per Agent)

GSD's 33 agents each have a single, non-overlapping domain. This matches Hivemind's D-24-03 decision exactly. GSD's agent categories are instructive:
- **Researchers** (3): project, phase, ui
- **Planners/Roadmappers** (2): planner, roadmapper
- **Executors** (1): executor
- **Verifiers/Checkers** (4): plan-checker, verifier, integration-checker, ui-checker
- **Auditors** (3): nyquist-auditor, ui-auditor, security-auditor
- **Debuggers** (1): debugger
- **Doc Writers** (2): doc-writer, doc-verifier
- **Mappers** (1): codebase-mapper
- **Profilers** (1): user-profiler

Hivemind's 30-agent roster in CONTEXT.md aligns well with these categories, but adds cross-cutting agents not present in GSD (intent-loop, ecologist, shipper, specifier, intel-updater).

### 2.3 Artifact Naming Conventions

GSD has a mature `Canonical Artifact Registry` (`get-shit-done/templates/README.md`) that defines naming, lifecycle, location, and consumption for every artifact. Key conventions:

| Artifact | Pattern | Location |
|----------|---------|----------|
| CONTEXT.md | `{NN}-CONTEXT.md` | `.planning/phases/{NN}-name/` |
| RESEARCH.md | `{NN}-RESEARCH.md` | `.planning/phases/{NN}-name/` |
| PLAN.md | `{NN}-{MM}-PLAN.md` | `.planning/phases/{NN}-name/` |
| SUMMARY.md | `{NN}-{MM}-SUMMARY.md` | `.planning/phases/{NN}-name/` |
| REVIEW.md | `{NN}-REVIEW.md` | `.planning/phases/{NN}-name/` |
| VERIFICATION.md | `{NN}-VERIFICATION.md` | `.planning/phases/{NN}-name/` |
| SECURITY.md | `{NN}-SECURITY.md` | `.planning/phases/{NN}-name/` |
| UI-SPEC.md | `{NN}-UI-SPEC.md` | `.planning/phases/{NN}-name/` |

**Adopt as pattern:** Hivemind should maintain a similar `Canonical Artifact Registry` (in `.planning/references/` or equivalent) that documents every artifact type, its naming convention, lifecycle, valid locations, and which agents consume/produce it.

### 2.4 Workflow Pipeline Structure

GSD workflows follow a consistent pattern:
1. **Pre-requisites/Required reading** — Load references
2. **Initialize** — Load context via `gsd-sdk query init.*`
3. **Core steps** — Numbered procedural steps with gates
4. **Gate checks** — Sub-steps or sub-files for quality gates
5. **State updates** — Write results back to STATE.md / filesystem
6. **Error handling** — Recovery paths for common failures

Hivemind's workflows should follow a similar structure but with programmatic features replacing procedural agent instructions.

### 2.5 Gate Sub-File Pattern

GSD extracts complex gate logic into sub-files under `workflows/<workflow>/steps/`:
- `execute-phase/steps/codebase-drift-gate.md`
- `execute-phase/steps/per-plan-worktree-gate.md`
- `execute-phase/steps/post-merge-gate.md`

**Adopt as pattern:** Hivemind's gate logic (spec-compliance, evidence-truth, lifecycle-integration) could be structured as workflow step sub-files, keeping the main workflow file focused on orchestration flow.

### 2.6 Two-Stage Routing (Namespace Meta-Skills)

GSD's namespace routing pattern (6 meta-skills → ~86 concrete skills) is a proven token-budget optimization. Hivemind's future command system (Phase 24.3+) should adopt a similar pattern if the agent listing becomes large enough to warrant it.

### 2.7 Procedural Agent Profile Body (For Hivemind's Adaptation)

GSD agents contain detailed XML-tagged execution instructions. While Hivemind's D-24-02 removes orchestration logic from agent files, **remaining** useful body sections from GSD include:
- `<role>` — Concise domain definition
- Artifact output specs (what, where, format)
- Success criteria
- Anti-patterns to avoid

These are **not** orchestration logic — they describe *what the agent does and produces*, not *how the orchestrator sequences it*.

## 3. GSD's Gaps that Hivemind Must Fill

### 3.1 No Programmatic Session Tracking

**GSD:** Session state is entirely file-based (STATE.md). There is no programmatic session history, no delegation chain tracking, no event journal. If a user runs `/gsd-planner`, then `/gsd-executor`, the only connection is files written to `.planning/`.

**Hivemind fills this with:** `session-tracker` — programmatic session persistence with delegation hierarchy, event journals, and trajectory records. This is a core Hivemind differentiator.

### 3.2 No Delegation Status Monitoring

**GSD:** Agent subagent spawning via `Agent()` or `task` tool has no built-in status monitoring. The orchestrator spawns and assumes completion when the agent returns. There is no way to query "what are all running delegations and their status?"

**Hivemind fills this with:** `delegation-status` tool — queryable delegation state, completion detection, dual-signal protocol.

### 3.3 Procedural Looping/Gating in Workflow Files

**GSD:** Looping and gating logic is embedded in workflow markdown files as procedural steps (e.g., "iterate up to 3 times", "check gate X, if failed go back to step Y"). This is brittle — if the agent misreads the workflow, the loop breaks.

**Hivemind fills this with:** `completion-detection`, `iterative-loop`, `wave-execution` — programmatic looping and gating controlled by tools and hooks, not workflow prose.

### 3.4 No Trajectory or Progress Tracking

**GSD:** Progress is tracked via STATE.md text updates. There is no programmatic trajectory record showing which agent did what, in what order, with what results.

**Hivemind fills this with:** `trajectory` — append-only event chain with checkpoint snapshots for context resume across sessions.

### 3.5 No Circuit Breaker or Pressure Monitoring

**GSD:** If an agent enters a loop or produces nonsense, the workflow continues until it hits a procedural "max iterations" check. There is no runtime health monitoring.

**Hivemind fills this with:** `pressure` — runtime health classification, circuit-breaker thresholds, tool authority inspection.

### 3.6 No Agent Work Contracts

**GSD:** Agent scope is defined in the agent profile body and the spawning orchestrator's prompt. There is no formal contract specifying allowed surfaces, dependencies, evidence requirements.

**Hivemind fills this with:** `agent-work-contract` — durable scope boundaries, evidence requirements, verification commands per delegation.

### 3.7 Agent Profile Format is Deprecated

**GSD:** Uses `tools:` and `color:` in frontmatter. Per OpenCode schema, `tools:` is deprecated (replaced by `permission:`). GSD also uses `name:` in frontmatter (redundant with filename).

**Hivemind fills this with:** D-24-02 minimal format — `description`, `mode: all`, `hidden: true` only. No `tools`, `color`, or `name` in shipped files.

### 3.8 No Primitives Management

**GSD:** Commands, agents, and workflows are plain files in the repo. There is no primitives schema, no batch compilation, no conflict detection across primitives.

**Hivemind fills this with:** `configure-primitive` tool — schema-based YAML/JSON compilation, batch configuration, cross-primitive conflict detection, dry-run validation.

## 5. Recommendations for Hivemind Architecture

### 5.1 Agent Profile Body Format

Per D-24-02, agent files contain: role description, execution flow, deviation rules, artifact specs, success criteria. Based on GSD analysis, the recommended format is:

```markdown
---
description: >
  [One sentence: what this agent does. Second sentence: what artifact it produces.
  Third sentence: when/why the orchestrator calls it.]
mode: all
hidden: true
---

# hm-{name} — {Domain}

## Role

[2-3 sentence description of this agent's domain and responsibilities.
WHAT it does, not HOW the orchestrator sequences it.]

## Delegation Target

This agent is dispatched by: [which orchestrator/workflow]
Dispatches to: [sub-agents if any — empty if leaf]
Accepts from: [workflow/command that spawns this agent]

## Execution Flow

1. **Load context** — What init command to run, what files to read
2. **Process** — Core work steps (NOT orchestration — just what this agent does)
3. **Produce output** — What artifact(s) to write, where, naming convention
4. **Verify** — Self-check: did output meet spec?

## Deviation Rules

- [What constitutes a deviation and how to handle it]
- [When to stop and request human input]

## Artifact Specs

| Artifact | Location | Format | Contents |
|----------|----------|--------|----------|
| {name} | {path} | {markdown/json/etc} | {description} |

## Success Criteria

- [Falsifiable statement: the agent has succeeded when...]
- [Evidence required: what files exist, what output was produced]

## Anti-Patterns

- [What this agent should NOT do]
- [What to avoid]

<!-- Delegation boundary: This agent does NOT perform orchestration, looping, gating, or hierarchy management. Those functions belong to the orchestrator and programmatic features. -->
```

**Rationale:** This format:
- Is **minimal** (no orchestrator logic, no temperature/steps/color/tools)
- Provides **clear delegation targets** (who spawns this, who it spawns)
- Documents **artifact contracts** (what, where, format)
- Sets **falsifiable success criteria** (verifiable by gate agents)
- Includes **anti-patterns** (learned from real execution failures)
- Uses a **delegation boundary comment** to prevent scope creep

### 5.2 Programmatic Features vs Agent Profile Responsibilities

### 5.4 What Hivemind Should NOT Copy from GSD

1. **`tools:` in frontmatter** — Deprecated by OpenCode schema. Use `permission` at runtime.
2. **`color:` in frontmatter** — UI concern, not agent metadata.
3. **`name:` in frontmatter** — Redundant with filename.
4. **Procedural looping in agent bodies** — Belongs in programmatic features.
5. **Procedural gating in agent bodies** — Belongs in gate triad (lifecycle→spec→evidence).
6. **Spawning multiple researchers in parallel** (GSD spawns 4 parallel researchers) — Hivemind's single-agent-per-domain model dispatches one researcher per phase.
7. **gsd-sdk procedural CLI tools** — Hivemind uses TypeScript plugin tools, not bash CLI SDK.

### 5.5 Key Architecture Insight: Separating Agent Profile from Orchestration

The most critical insight from comparing GSD and Hivemind:

**GSD conflates agent definition with orchestration logic.** A GSD agent profile (e.g., `gsd-executor.md`) contains both "I am an executor" AND "here's how to execute plans step by step." The step-by-step instructions ARE the orchestration.

**Hivemind must separate these.** The agent profile says "I am an executor who produces SUMMARY.md from PLAN.md." The orchestration (which plan to execute, when, in what sequence, with what gates) lives in programmatic features or workflow files. The agent profile is a contract, not a program.

This separation is what D-24-02's "no looping/gating/hierarchy logic in agent profile body" enforces. The agent profile tells the agent WHAT it is and WHAT it produces. The orchestrator tells the agent WHEN to do it and HOW it fits with other agents' work.

## Common Pitfalls

### Pitfall 1: Copying GSD's Agent Format
**What goes wrong:** Writing GSD-style agent profiles with `tools:` and `color:` in frontmatter, violating D-24-02.
**Why it happens:** GSD is the most visible reference, and its format "looks correct."
**How to avoid:** Always validate against the OpenCode schema. Hivemind's format is `description`, `mode: all`, `hidden: true` only.
**Warning signs:** Agent file has `tools:`, `color:`, or `name:` in frontmatter.

### Pitfall 2: Putting Orchestration Logic in Agent Profiles
**What goes wrong:** Agent profiles contain "if X then spawn Y" or "loop until Z" instructions, duplicating programmatic features.
**Why it happens:** It's natural to write "how to do the job" in the agent's description.
**How to avoid:** The agent profile should answer "WHAT do you produce?" not "HOW does the system sequence you?" Add a `<!-- Delegation boundary -->` comment as a reminder.
**Warning signs:** Agent body references `task` tool, `delegate-task`, looping, checkpoints, or wave execution.

### Pitfall 3: Not Documenting Artifact Contracts
**What goes wrong:** Agents create artifacts with different naming or location than what downstream agents expect.
**Why it happens:** Artifact specs are implicit (assumed from agent name/role).
**How to avoid:** Every agent profile must have an explicit `Artifact Specs` table. Create a cross-reference registry.
**Warning signs:** Agent profile says "produces research output" without specifying filename, location, or format.

### Pitfall 4: Over-Minimal Agent Profiles
**What goes wrong:** Following D-24-02 so strictly that agent profiles become useless — just a description with no execution guidance.
**Why it happens:** "Minimal" is misinterpreted as "empty."
**How to avoid:** D-24-02 says "Agent body contains: role description, execution flow, deviation rules, artifact specs, success criteria." The frontmatter is minimal; the body has substance. GSD's body pattern (minus orchestration logic) is the right reference.
**Warning signs:** Agent profile has only frontmatter with no body content.

## Open Questions

1. **Should Hivemind agents use XML-style tags (like GSD) or plain markdown sections?**
   - What we know: GSD uses `<role>`, `<step>`, `<process>` XML tags that provide clear structure but add verbosity
   - What's unclear: Whether XML tags improve agent parsing vs. plain markdown headings
   - Recommendation: Use plain markdown `## Section` headings for minimalism. GSD's XML tags are a convention, not a requirement.

2. **How detailed should agent profile anti-patterns be?**
   - What we know: GSD doesn't use explicit anti-patterns sections; Hivemind's format includes them
   - What's unclear: Whether detailed anti-patterns improve agent behavior or just add token cost
   - Recommendation: Start with 3-5 anti-patterns per agent, adjust based on observed execution failures.

3. **Should the Canonical Artifact Registry be a reference file or programmatic schema?**
   - What we know: GSD uses a human-readable reference (templates/README.md)
   - What's unclear: Whether Hivemind needs programmatic artifact validation
   - Recommendation: Start as reference file, evolve to programmatic schema if validation becomes a pain point.
