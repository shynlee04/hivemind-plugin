### D-58.9-01: `pane-captured` event shape — extend existing `PaneCapturedEvent` with optional `content` field

**Decision:** Reuse the existing `PaneCapturedEvent` interface at `src/features/tmux/observers.ts:72-78` and add an OPTIONAL `content?: string` field. The existing metadata-only shape remains valid (backward compat for any subscriber that doesn't want full content). New subscribers that want full content read `event.content`. The P53 hook at `src/hooks/pane-monitor.ts:107-115` is updated to read `event.content` if present (falling back to the existing metadata-only path for older observers).

**Rationale:**
- The existing `PaneCapturedEvent` is consumed by `createTmuxEventObserver` at `observers.ts:201-203` via `observer.onPaneCaptured` registration.
- The P53 hook subscribes via `tmuxObserver.onPaneCaptured` at `src/plugin.ts:772-786` (per the sticky bug report). It currently receives metadata-only.
- Adding `content?: string` as optional preserves the metadata-only contract for non-journal subscribers.
- The 7-field `JournalEntry` schema in `pane-monitor.ts:107-115` does NOT include `content` (only `contentLength`); P58.9 does NOT change that schema. The full content is available to the hook for any future consumer that wants it.

**Wire path (per D-58.9-01):**
1. `SessionManager.startPolling` `tick()` after `this.latestCapture.set(tracked.paneId, capture)` (line 338): emit `pane-captured` event with shape `{ type: "pane-captured", sessionId: tracked.sessionId, paneId: tracked.paneId, contentLength: capture.byteLength, timestamp: capture.capturedAt, content: capture.content }`.
2. `SessionManager` gets a new `private observer: PaneObserver | null = null` field.
3. `SessionManager.setObserver(observer)` method wires the observer in.
4. In `src/plugin.ts`, find the place that constructs `SessionManager` and call `sessionManager.setObserver(tmuxObserver)` immediately after.
5. The observer is typed as `PaneObserver = { onPaneCaptured: (event: PaneCapturedEvent) => void }` (matches the existing interface at `observers.ts:102`).

**Implementation touchpoints:**
- `src/features/tmux/observers.ts:72-78` — add `content?: string` to `PaneCapturedEvent`.
- `src/features/tmux/session-manager.ts:328-356` — add `observer` field, `setObserver()` method, and `observer?.onPaneCaptured(...)` call in tick() AFTER `this.latestCapture.set(...)` (line 338).
- `src/plugin.ts` — find `new SessionManager(` and add `sessionManager.setObserver(tmuxObserver)` immediately after.
- `src/hooks/pane-monitor.ts` — extend the journal write to include `content` in the persisted journal entry as an 8th optional field, OR keep the 7-field shape and write `content` to a sibling file `<ts>-pane-content.txt`. **Decision (per D-58.9-01):** keep the 7-field shape unchanged; write `content` to a sibling `<ts>-pane-content.txt` file in the same directory. This preserves the P53 schemaVersion contract (no `schemaVersion: 2` bump needed) and avoids re-validating journal consumers.
- `tests/scripts/tmux/75-pane-captured-journal.bats` (new slot 75) — assert both `<ts>-pane.json` (7 fields) AND `<ts>-pane-content.txt` (full content) exist after the polling tick fires.

**Backward compat:** The `PaneCapturedEvent` extension is additive. Existing subscribers (if any) that read only metadata fields continue to work. The P53 hook is the ONLY known subscriber; it's updated to write the sibling content file.

**Risk:** Event flooding. Mitigation: emit ONLY on hash CHANGE (not on every tick). The existing `lastCaptureHash` detection at `session-manager.ts:339-344` is the natural gate: emit only inside the `if (prevHash !== hash)` block (line 341), so the journal only captures NEW content, not unchanged re-captures.

### D-58.9-02: vitest timeout root cause — `SessionManager.startPolling` timer pollution

**Decision:** The most likely root cause of the 5 full-suite-only timeouts is **timer pollution** from `SessionManager.startPolling` (introduced in P58.8). The polling loop uses `setTimeout` chains (not `setInterval`) at `session-manager.ts:353, 355` that are NOT cleaned up by vitest's default `afterEach` between test files. When the next test file's tests run, the still-pending `setTimeout` callbacks interfere with the new test's mock timer setup, causing timeouts.

**Fix path (primary):**
- Identify the 5 affected test files (`tests/eval/coherence.test.ts`, `tests/plugin/bootstrap-tools-registration.test.ts`, `tests/tools/delegate-task.test.ts`).
- Add `afterEach(() => { vi.useRealTimers() })` at the top of each test file's `describe` block. This is the standard vitest pattern for tests that interact with `setTimeout`/`setInterval` (mock or real).
- For tests that explicitly USE mock timers (`vi.useFakeTimers()`), add `afterEach(() => { vi.useFakeTimers({ shouldAdvanceTime: false }) })` cleanup so the next test's setup is clean.
- For tests that construct a `SessionManager` instance (via test seam or factory), call `sessionManager.stopPolling()` in `afterEach` to clear any pending polling timers.

**Fix path (secondary, if primary doesn't resolve):**
- Increase the vitest timeout for these specific tests via `it(name, () => {...}, 15000)` if the root cause is slow-but-correct (e.g., real tmux subprocess being slow in CI).

**Fix path (tertiary, last resort):**
- Skip with documented reason: `it.skip(name, () => {...})` with a `// SKIP-P58.9-REASON: <root cause + retry plan>` comment cross-linked to this REQ.

**Rationale:**
- The 5 affected tests all time out in full-suite but pass in isolation. This is the **classic test pollution signature**.
- `SessionManager.startPolling` was added in P58.8 (commit that introduced `setTimeout` chains).
- The pollution is most likely timer-based, not file-handle-based (no `fs` operations in the polling tick).
- `vi.useRealTimers()` is the standard vitest escape hatch for timer pollution.

**Verification:**
- Run `npx vitest run` (full suite) and confirm 5/5 fixes PASS.
- Run `npx vitest run -t "<name>"` (isolation) and confirm 5/5 still PASS.
- The pre-existing 2 fails (`tool-registration.test.ts:185`, `plugin-lifecycle.test.ts:175`) remain — they have the same pollution signature but are pre-P58.8 and out of scope for P58.9.

**Risk:** Some test failures may have non-timer root causes (e.g., module-level state pollution). Mitigation: investigate each failure individually; if a non-timer root cause is found, document the actual fix.

### D-58.9-03: BATS slot 75 design — real `opencode attach` TUI in pane

**Decision:** Create `tests/scripts/tmux/75-pane-real-runtime.bats` that uses a **real `opencode attach` TUI in a real tmux pane**. The scenario is the canonical BATS-vs-runtime closure: it exercises the same code path that the user exercises in their TUI, and verifies the polling tick updates pane content within 1 second.

**Design:**

```bash
# tests/scripts/tmux/75-pane-real-runtime.bats
setup() {
  # Skip if no tmux server is running (CI environments)
  [[ -z "${TMUX:-}" && -z "$(tmux list-sessions 2>/dev/null)" ]] && skip "no tmux server"
  
  # Skip if no real opencode binary is available
  command -v opencode >/dev/null 2>&1 || skip "no opencode binary"
  
  # Setup: spawn a real tmux session running opencode attach
  SESSION_NAME="p58-9-real-runtime-$$"
  tmux new-session -d -s "$SESSION_NAME" -x 200 -y 50 "opencode attach --server ${OPENCODE_TEST_SERVER:-http://localhost:8080} --session test-session-$$ --dir ${PWD}"
  
  # Wait for the TUI to be ready
  sleep 2
  
  # Discover the live paneId
  PANE_ID=$(tmux list-panes -t "$SESSION_NAME" -F '#{pane_id}' | head -1)
  
  # Start the SessionManager polling loop on this pane
  # (via tmux_node_eval or direct Node import — see helpers.bash for the bridge)
  start_session_manager_polling "$PANE_ID" "test-session-$$"
}

teardown() {
  tmux kill-session -t "$SESSION_NAME" 2>/dev/null || true
  stop_session_manager_polling
}

@test "75-pane-real-runtime: pane content updates within 1s of new event" {
  # Send a synthetic chat.message event via the SDK to the test session
  inject_chat_message_event "test-session-$$" "E2E-PROBE-P58-9-$(date +%s)"
  
  # Wait 1 second for the polling tick to capture
  sleep 1
  
  # Assert the probe text is in the captured pane content
  CAPTURED=$(capture_pane_via_session_manager "$PANE_ID")
  echo "$CAPTURED" | grep -c "E2E-PROBE-P58-9-" | grep -qE "^[1-9]" || {
    echo "FAIL: probe text not found in pane content"
    echo "Captured content: $CAPTURED"
    return 1
  }
}
```

**Skip patterns (per AC-58.9-03-06):**
- `[[ -z "${TMUX:-}" && -z "$(tmux list-sessions 2>/dev/null)" ]] && skip "no tmux server"` — covers environments without a tmux server.
- `command -v opencode >/dev/null 2>&1 || skip "no opencode binary"` — covers environments without the opencode CLI.
- Both skips produce exit 0 (BATS convention) so the test doesn't pollute the full-suite run.

**Bridge from BATS to Node:** Reuse the existing `tmux_node_eval` helper from `helpers.bash` (used by BATS 66) to invoke the SessionManager polling loop. If `helpers.bash` lacks a `start_session_manager_polling` helper, add it as a thin wrapper around `dist/features/tmux/session-manager.js`.

**Rationale:**
- The S1 PUSH complaint "live update tại panel tmux vẫn chưa hoạt động" can ONLY be reproduced with a real TUI in a real pane. `cat` mocks pass the BATS but miss the real bug.
- The 1s assertion is a hard upper bound: the polling tick fires at 5s cadence by default but the test injects the event and waits 1s for the NEXT tick. If the polling is healthy, content updates within 1s of the event injection. If the polling is broken (e.g., `startPolling` never started), the content does NOT update and the test fails.

**Risk:** Flaky in CI. Mitigation: skip patterns above; the test is only run when a real opencode + tmux server are available (typically only in dev/laptop environments, not in CI).

### D-58.9-04: Regression guard placement — 4 new vitest integration tests

**Decision:** Place the 5 regression guards in `tests/integration/` (existing directory pattern). One new test file per guard category:

| Guard | Test file | Pattern |
|-------|-----------|---------|
| 27-tool-key | `tests/integration/tool-key-invariant.test.ts` | Parse `src/plugin.ts` for `tool({` helper calls; count; assert `=== 27` |
| AC#11 manualOverride FIRST in tmux-copilot | `tests/integration/manualoverride-invariant.test.ts` | Regex parse the `case "forward-prompt":` block in `src/tools/tmux-copilot.ts`; assert FIRST statement is `getManualOverrideState(...)` |
| AC#10 manualOverride FIRST in plugin.ts | Same test file as above | Regex parse `appendTuiPrompt` function in `src/plugin.ts`; assert FIRST statement in the loop body is `getManualOverrideState(...)` |
| P20 no-new-deps | `tests/integration/no-new-deps.test.ts` | Read `package.json`; compare against `tests/fixtures/package-deps-baseline.json` snapshot; assert equal |
| P51 in-tree tmux single-source-of-truth | `tests/integration/in-tree-tmux.test.ts` | Grep `src/` for `import .* @opencode-tmux` and `import .* opencode-tmux-`; assert zero runtime imports (allow `ORIGIN:` attribution comments) |

**Snapshot file for no-new-deps:** `tests/fixtures/package-deps-baseline.json` is a JSON file with shape:
```json
{
  "dependencies": { "name": "version", ... },
  "devDependencies": { "name": "version", ... },
  "peerDependencies": { "name": "version", ... }
}
```
The test reads `package.json` (current) and the snapshot (baseline), compares `Object.keys(deps).sort()` and value-for-value, asserts equal. If a contributor adds a new dep, the test fails until the snapshot is intentionally updated (via atomic commit with `// UPDATE-SNAPSHOT-P58.9: <reason>` marker).

**Rationale:**
- `tests/integration/` is the existing pattern for cross-module guards (see `tests/integration/hook-registration.test.ts:86-103` for the P20/P49 invariants baseline).
- The 5 guards are independent files with no shared state; they can be committed in parallel.
- The no-new-deps snapshot is git-tracked so a contributor who legitimately wants to add a dep must explicitly update the snapshot (which becomes part of the PR review).

**Risk:** False positives. The 27-tool-key guard might be brittle if `src/plugin.ts` uses a different registration pattern (e.g., a loop). Mitigation: the regex should match the actual pattern observed in `src/plugin.ts:137,138,144,157-163,177-185,198-203,872,876` per the P58.8 verification report.

**Risk:** 27-tool-key count drift from 27 (e.g., P59 legitimately adds a new tool). Mitigation: the test should compare against the current count and a documented "expected count" comment at the top of the test file. Updating the expected count is a deliberate atomic commit.
