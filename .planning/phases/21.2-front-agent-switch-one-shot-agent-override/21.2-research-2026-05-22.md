# Phase 21.2 Research — One-Shot Front-Agent Mode Override Feasibility

**Created:** 2026-05-22
**Status:** Research complete, prototype required before implementation
**Research quality:** B — live OpenCode source plus local codebase scan and subagent research agreement

## Research Question

Can Hivemind run a command under a configured `mode: subagent` agent as a main/front-facing agent for exactly one execution, without mutating either the command primitive or the agent primitive?

## Key Source Finding

OpenCode command execution resolves the command agent and computes subtask behavior from the resolved agent mode and the command's `subtask` config:

```ts
const agent = agentName ? yield* agents.get(agentName) : yield* agents.defaultInfo()
const isSubtask = (agent.mode === "subagent" && cmd.subtask !== false) || cmd.subtask === true
const userAgent = isSubtask ? (input.agent ?? (yield* agents.defaultInfo()).name) : agent.name
```

Implication:

- Passing `agent: "gsd-executor"` alone does not prove main/front-facing execution, because `gsd-executor.mode === "subagent"` drives `isSubtask` unless the command has `subtask:false`.
- SDK `CommandInput` has no explicit `mode`, `agentMode`, or temporary agent config overlay field.
- Command-level `subtask:false` is the closest native lever, but the SDK request shape does not expose `subtask` as an input field; it is command configuration, not request-time agent configuration.

## Feasibility Verdict

**Verdict:** PARTIAL / NEEDS PROTOTYPE.

The platform supports one-shot subtask dispatch using `SubtaskPartInput`, and it supports command configuration where `subtask:false` prevents automatic subtask conversion for subagent-mode agents. The platform does not expose a confirmed one-shot agent-mode override field in the SDK request body.

## Candidate Solutions

### Option A — Command Overlay With `subtask:false`

Build a transient command object or command-dispatch payload where the command's effective config has `subtask:false` and `agent: gsd-executor`, without changing the file on disk.

**Pros:** Aligns with OpenCode's native `isSubtask` formula.
**Cons:** Requires access to a command overlay seam. No SDK request field currently proves this can be injected directly.

### Option B — Synthetic Main Prompt With Agent Override

Load the command file, expand `$ARGUMENTS`, and call `session.prompt({ agent: "gsd-executor", parts: [{ type: "text", text: expandedBody }] })` instead of `session.command()`.

**Pros:** Non-mutating and narrow. Does not need command registry overlay.
**Cons:** May still be rejected or semantically unsafe if OpenCode disallows subagent-mode agents as main prompt agents. It also bypasses native command parser events and command hooks.

### Option C — One-Shot Agent Config Overlay

Create an in-memory overlay where `gsd-executor.mode` is `all` or `primary` only for the duration of one command dispatch.

**Pros:** Closest to user intent.
**Cons:** No SDK/API seam found yet. Likely requires plugin/config lifecycle interception or upstream changes. Highest implementation risk.

### Option D — File Mutation / Alias Agent

Create a separate primary/all alias for `gsd-executor` or mutate the agent file temporarily.

**Pros:** Most platform-native if persisted.
**Cons:** Rejected by the user requirement because command and agent configuration must not be changed.

## Recommended Path

Option A is the best first prototype if a command overlay seam can be found, because it matches OpenCode's own `isSubtask` formula and keeps native command execution semantics. If Option A cannot be implemented through the SDK or plugin hooks, Option B should be prototyped as a fallback because it is non-mutating and can be tested with a single synthetic prompt payload. Option C should be deferred until A and B fail because it likely crosses deeper OpenCode config lifecycle boundaries and may require unsupported runtime mutation.

Option D should remain rejected. It would satisfy front-agent execution only by changing persistent configuration or adding alias primitives, which violates the one-shot requirement and creates governance drift between command intent and actual primitive state.

## Prototype Questions

1. Can `session.command()` be made to see an effective command config with `subtask:false` without writing the command file?
2. Does `session.prompt({ agent: "gsd-executor" })` accept an agent whose resolved mode is `subagent`?
3. If accepted, does it truly run in the parent session, or does OpenCode still route it as a task/subagent?
4. Can `command.execute.before` mutate parts only, or can it influence `cmd.subtask`/agent mode before `isSubtask` is computed?
5. Is there a plugin config hook that can inject a one-shot agent override safely and then clear it before the next command?

## Blocking Condition

Do not implement a permanent feature until a prototype proves at least one non-mutating route with L1 live behavior evidence.
