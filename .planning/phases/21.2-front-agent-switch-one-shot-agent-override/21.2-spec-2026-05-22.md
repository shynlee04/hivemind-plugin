# Phase 21.2 Spec — Front-Agent Switch One-Shot Agent Override

**Created:** 2026-05-22
**Status:** Research locked; prototype plan created, production implementation still gated by live UAT
**Ambiguity score:** 0.18

## Goal

Determine whether `execute-slash-command` can support a one-shot main/front-facing command execution override where a configured `mode: subagent` agent is temporarily treated as `all` or `primary` for exactly one command invocation, without mutating command or agent configuration files.

## Requirements

1. **Agent discovery scope:** The phase must discover agents from project and global OpenCode primitive roots, supporting both singular and plural directories: `agent/`, `agents/`, `command/`, and `commands/`.

2. **Non-mutating override:** The phase must not write to `.opencode/commands/**`, `.opencode/agents/**`, global config command/agent files, or generated primitive source files to simulate one-shot mode.

3. **Two-mode target case:** The phase must evaluate `.opencode/commands/gsd-progress.md` with `gsd-executor` in two modes: subtask dispatch and main/front-facing dispatch.

4. **SDK/source evidence:** The phase must cite OpenCode SDK/source evidence for command execution, `agent` handling, `subtask` handling, and agent mode enforcement.

5. **Prototype before implementation:** If no direct SDK field exists for temporary agent mode, the phase must produce a prototype plan before any production implementation.

6. **No false runtime claim:** Unit tests, docs, and source inspection cannot claim front-agent switch success. Live UAT must prove whether the command ran in the parent session or as a subtask.

## Acceptance Criteria

- [x] Research artifact identifies whether SDK `session.command()` exposes a one-shot `mode` or `subtask` request field.
- [x] Research artifact identifies whether command-level `subtask:false` is usable without mutating the command primitive.
- [x] Research artifact evaluates whether synthetic `session.prompt({ agent, parts })` can run a subagent-mode agent in the parent session.
- [x] Prototype plan includes a stop condition if OpenCode rejects subagent-mode agents as primary prompt agents.
- [x] Governance docs clearly state feasibility: SUPPORTED, PARTIAL, NOT SUPPORTED, or NEEDS PROTOTYPE.

## Explicit Non-Goals

- Do not implement a persistent `agent.mode` mutation.
- Do not create alias agents to work around the issue.
- Do not claim `@agent /command` switches front-facing agent; Phase 21.1 live UAT already disproved that for this harness path.
- Do not bypass command governance by silently running arbitrary expanded command bodies unless the prototype explicitly labels it as a synthetic fallback.

## Initial Verdict

Phase 21.2 starts with verdict **PARTIAL / NEEDS PROTOTYPE**. OpenCode has a native `subtask:false` command-level concept, but no confirmed SDK request-time agent mode override field has been found.

## Current Verdict

Phase 21.2 remains **PARTIAL / NEEDS PROTOTYPE** after live MCP/source/type validation. `SessionCommandData` does not expose request-time `subtask` or `mode`; the only non-mutating prototype path currently identified is a synthetic parent-session `session.prompt({ agent, parts })` fallback using the expanded command body.

## Recommended Prototype Order

1. Prototype command-level overlay or SDK command payload equivalent to `subtask:false`.
2. If unavailable, prototype synthetic parent-session prompt with `agent: gsd-executor` and expanded `gsd-progress` body.
3. If both fail, document the requirement as unsupported without upstream OpenCode changes.
