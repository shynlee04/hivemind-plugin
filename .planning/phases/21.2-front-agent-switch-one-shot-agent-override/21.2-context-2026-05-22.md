# Phase 21.2 Context — Front-Agent Switch One-Shot Agent Override

**Created:** 2026-05-22
**Status:** Research/spec-only, implementation not authorized
**Evidence level:** L5 planning plus live-source research references

## User Intent

The user wants to know whether `execute-slash-command` can temporarily run a command under an agent as if that agent had `mode: all` or `mode: primary`, without mutating either the command file or the agent file.

The motivating case is `.opencode/commands/gsd-progress.md` with `gsd-executor`:

1. Run as a subtask with `gsd-executor`.
2. Run in the main/front-facing context by temporarily treating `gsd-executor` as a primary/all agent for that one command execution.

## Current Proven Behavior From Phase 21.1

Phase 21.1 proved the following:

- One-shot `agent` + `subtask: true` can dispatch a command as a subtask without mutating command or agent configuration.
- Awaiting `session.prompt()` inside a tool execution hangs because the session is busy running the tool.
- Deferred subtask scheduling avoids the hang and returns immediately.
- TUI `@agent /command` text injection does not switch the front-facing agent; it is treated as prompt text/mention behavior.

## Known Target Primitive State

`gsd-progress.md` has no `agent` or `subtask` frontmatter. It is a complex command with progress, next-step routing, and freeform `--do` modes.

`gsd-executor.md` declares `mode: subagent`. That makes it valid for subtask execution, but not obviously valid as a main/front-facing agent.

## Phase Boundary

Phase 21.2 is a feasibility and prototype-planning phase. It must not implement runtime code until the research gate establishes whether the desired behavior is supported, partially supported, or impossible without upstream OpenCode changes.
