## Summary

**Root cause: VALIDATED.** The previous investigation was correct. The race window is deterministic and matches all symptoms across 4 failed UAT rounds.

The startup sequence in `plugin.ts` is:
1. `new SessionTracker()` — constructor sets `client`/`projectRoot` only; ALL deps remain `undefined` (TypeScript `!` assertion)
2. `setupDelegationModules({onChildSessionCreated: ...})` — captures `sessionTracker` ref in a callback
3. `void sessionTracker.initialize()` — fire-and-forget
4. Inside `initialize()`: sync `constructDependencies()` creates deps in local vars, then `await deps.hierarchyIndex.buildFromDisk()` (async I/O, 10-100ms+), then `Object.assign(this, deps)` — ONLY NOW are deps available

Step 2 fires the callback from `coordinator.ts:138` when `delegate-task` creates a child. If this happens before `Object.assign`, `this.eventCapture` is `undefined` and `handleSessionEvent` silently skips at line 140. The event is dropped forever. The child `.json` is never written. Hierarchy is permanently broken.

**Fix scope validation:** ~15-25 LOC, 3 files. The `constructDependencies()` function in `initialization.ts` is fully synchronous — extracting it is trivial. The `.then(() => cleanup())` chain on plugin.ts:327 is a separate bug (kills retry flush interval).

**Primary recommendation:** Extract `constructDependencies()` (or a wrapper) to run before `setupDelegationModules()` in plugin.ts. The 3-requirement spec is validated and complete.

## Phase 1: File-by-File Analysis

### 1.1 `src/features/session-tracker/index.ts` (585 lines)

**Key method — constructor (lines 98-101):**
```typescript
constructor(deps: { client: OpenCodeClient; projectRoot: string }) {
  this.client = deps.client
  this.projectRoot = deps.projectRoot
}
```
All 15+ dependencies declared with TypeScript definite assignment (`!`) operators (lines 68-94). They are `undefined` until `initialize()` runs and calls `Object.assign(this, deps)`.

**Key method — `handleSessionEvent` (lines 134-182):**
```typescript
async handleSessionEvent(event) {
  try {
    if (this.eventCapture) {                          // Line 140 — GUARD
      await this.eventCapture.handleSessionEvent(event)
    }
    // Fork handling — also guarded by this.projectIndexWriter / this.sessionIndexWriter
    ...
  }
}
```
Line 140: `if (this.eventCapture)` is a truthiness check. Before `Object.assign`, `this.eventCapture` is `undefined` — the entire body is silently skipped. **No log, no error, no diagnostic signal.**

**Key method — `initialize` (lines 456-533):**
```typescript
async initialize(): Promise<void> {
  try {
    await this.hierarchyIndex?.buildFromDisk?.()       // Line 458 — ALWAYS NO-OP (undefined)
    const deps = constructDependencies(                // Line 459 — SYNC constructor calls
      this.client, this.projectRoot, { ... }
    )
    await deps.hierarchyIndex.buildFromDisk()           // Line 468 — ASYNC I/O (10-100ms+)
    Object.assign(this, deps)                            // Line 470 — DEPS NOW AVAILABLE
    // ... more async work follows (retry queue, recovery, etc.)
  }
}
```

**Critical finding — line 458 is dead code:** `this.hierarchyIndex?.buildFromDisk?.()` — since `this.hierarchyIndex` is `undefined` at this point (before `Object.assign`), the optional chaining makes this a no-op every time. This was either a copy-paste error or attempted "try before construct" that can never work.

**Race window:** Between `void sessionTracker.initialize()` (plugin.ts:327) and `Object.assign(this, deps)` (index.ts:470). The sync `constructDependencies()` runs in ~1ms, but `await deps.hierarchyIndex.buildFromDisk()` is disk I/O reading potentially hundreds of `.json` files. Conservative estimate: 10-100ms. During this window, ALL deps are unavailable on `this`.

### 1.2 `src/features/session-tracker/initialization.ts` (202 lines)

**Key function — `constructDependencies` (lines 85-192):**
```typescript

**Fully synchronous.** No `await`, no I/O, no `Promise`. Pure object construction. This means extracting it (or a subset) to run before delegation wiring is technically trivial.

These are not needed for the critical deps (`eventCapture`, `pendingRegistry`, `childWriter`, `hierarchyIndex`) — those only need `client`, `projectRoot`, and each other.

**The race window in detail:**
```

t0    createPtyManagerIfSupported() — async, may take some time
t1    new SessionTracker() — deps all undefined
t2    setupDelegationModules() — captures callback with sessionTracker ref
t3    delegationManager.recoverPending() — more async work starts
t4    createHarnessLifecycleManager(), hydrateFromContinuity()

t5c     Object.assign(this, deps) — deps now available
t6    ... rest of plugin init continues

If delegate-task fires between t2 and t5c:
  → onChildSessionCreated calls sessionTracker.handleSessionEvent()
  → this.eventCapture is undefined
  → Guard at line 140 skips — event silently dropped
  → Child .json is never written
  → Hierarchy permanently broken
  → U4/U5 symptoms for this session's lifetime

## Phase 2: Root Cause Validation

### Claim: Race condition in `sessionTracker.initialize()` — CONFIRMED ✓

**The investigation claim is VALID and ACCURATE.** After reading ALL 15 specified source files (4,000+ LOC), the race window exists exactly as described.

| Investigation Claim | Status | Evidence |
|---|---|---|
| `plugin.ts:327` calls `void sessionTracker.initialize()` — fire-and-forget | **CONFIRMED** | plugin.ts:327 |
| Critical deps assigned inside async `initialize()` via `Object.assign` | **CONFIRMED** | index.ts:459-470 |
| `this.eventCapture` is `undefined` before `Object.assign` | **CONFIRMED** | index.ts:68 (`private eventCapture!`) |
| `handleSessionEvent` guard at index.ts:140 silently skips | **CONFIRMED** | index.ts:140-142 |
| Race matches U4/U5 symptoms | **CONFIRMED** | Child `.json` never written → hierarchy broken → `childCount:0` |

### Minor Correction Needed

The investigation claims the race is: `void initialize()` → `await this.hierarchyIndex?.buildFromDisk?.()` (async) → `constructDependencies()` (sync) → `Object.assign`. 

Actually the sequence is:
1. `void initialize()` starts
2. `await this.hierarchyIndex?.buildFromDisk?.()` — is **always a NO-OP** (hierarchyIndex is undefined), so this resolves immediately
3. `constructDependencies()` — SYNC, creates deps in local var (~1ms)
4. `await deps.hierarchyIndex.buildFromDisk()` — ASYNC I/O (10-100ms+)
5. `Object.assign(this, deps)` — deps available

**This makes the window SMALLER than the investigation claimed** (only the async buildFromDisk separates construction from assign, not the full initialize), but it still exists and is still long enough to be hit. The I/O scan of potentially hundreds of `.json` files in `buildFromDisk()` is the primary source of delay.

### Additional Race Window: PreToolUse

**New finding:** The same race affects `handleToolExecuteBefore()` at tool-delegation.ts:94:
```typescript
if (!this.pendingRegistry) return
```

If `this.pendingRegistry` is `undefined` (before `Object.assign`), the pending dispatch is NEVER registered. This means even AFTER the race is fixed for `handleSessionEvent`, if a PreToolUse event fires during the same window, Gate 0 would have no pending entries to match against. The `tool-before-guard.ts` fires synchronously during tool execution, which can happen at any time after plugin init.

**However**, for `delegate-task`, the `onChildSessionCreated` callback is the PRIMARY path, and the PreToolUse registration is the SECONDARY path. The callback is the one that directly feeds session IDs. The fix for `handleSessionEvent` is sufficient to fix U4/U5. The `handleToolExecuteBefore` race is a lesser concern that will also be fixed by making deps available earlier.

## Phase 3: New Findings

### Finding A: Dead code at index.ts:458

```typescript
await this.hierarchyIndex?.buildFromDisk?.()
```

`this.hierarchyIndex` is `undefined` at this point (before `Object.assign`). The optional chaining (`?.`) makes this a permanent no-op. This is either a bug (intended to be `deps.hierarchyIndex.buildFromDisk()` after line 470) or dead code that was never functional.

**Recommendation:** Remove line 458. It contributes nothing and is misleading.

### Finding B: Compounding silent skip in tool-delegation.ts

```typescript
// tool-delegation.ts:94
if (!this.pendingRegistry) return
```

Same root cause — `this.pendingRegistry` is `undefined` before `Object.assign`. This means PreToolUse registration of pending dispatch entries is ALSO silently dropped during the race window. This compounds the effect: even if `handleSessionEvent` somehow did reach `eventCapture`, Gate 0 would find no pending entries.

**Recommendation:** The primary fix (sync deps before delegation wiring) addresses this automatically since `pendingRegistry` is one of the critical deps.

### Finding C: `plugin.ts:286` — ptyManager is awaited before delegation wiring

```typescript
const ptyManager = await createPtyManagerIfSupported()  // line 274-275
```

This `await` on line 275 happens BEFORE `setupDelegationModules()` on line 281. This adds time between `new SessionTracker()` and the delegation wiring capture. But it doesn't affect the race — the session tracker's `initialize()` hasn't even started yet at this point.

### Finding D: No existing test for the race condition

All 40+ test files in `tests/features/session-tracker/` manually wire deps via `(tracker as any).eventCapture = ...`. No test:
1. Creates a `SessionTracker` without calling `initialize()`
2. Calls `handleSessionEvent` immediately after `new SessionTracker()`
3. Verifies the event is dropped

This is why the bug survived 4 UAT rounds — no test caught it.

## Phase 4: Fix Recommendation

### Approach A: Extract `constructCoreDependencies()` — RECOMMENDED

**Strategy:** Add a new public method `constructCoreDependencies()` on SessionTracker that constructs only the critical deps (`eventCapture`, `pendingRegistry`, `childWriter`, `hierarchyIndex`) synchronously. Call it in plugin.ts **BEFORE** `setupDelegationModules()`.

```typescript
// NEW in index.ts
constructCoreDependencies(): void {
  const deps = constructDependencies(this.client, this.projectRoot, {
    getSessionSafely: (id) => this.getSessionSafely(id),
    ensureChildRoute: (p, c) => this.ensureChildRoute(p, c),
    bootstrappedSessions: this.bootstrappedSessions,
  })
  // Only assign critical-path deps needed for handleSessionEvent
  this.eventCapture = deps.eventCapture
  this.pendingRegistry = deps.pendingRegistry
  this.childWriter = deps.childWriter
  this.hierarchyIndex = deps.hierarchyIndex
  this.sessionWriter = deps.sessionWriter
  this.sessionIndexWriter = deps.sessionIndexWriter
  this.projectIndexWriter = deps.projectIndexWriter
  this.retryQueue = deps.retryQueue
  this.manifestWriter = deps.manifestWriter
}
```

Modified `initialize()` then skips `constructDependencies` and only does the remaining async setup.

### Approach B: Move constructDependencies() call before delegation wiring

**Strategy:** Call `constructDependencies()` directly from plugin.ts before `setupDelegationModules()`, assign results via a setter. Simpler but breaks encapsulation.

## Common Pitfalls

### Pitfall 1: Not validating the full startup sequence before changing code
**What goes wrong:** Fix one guard but miss the compounding effect (e.g., fix `handleSessionEvent` guard but not `handleToolExecuteBefore` guard).
**How to avoid:** Read ALL session-tracker files, not just the one with the obvious guard.

### Pitfall 2: Breaking `initialize()` by extracting deps
**What goes wrong:** `constructDependencies()` is called twice (once sync, once in `initialize()`), creating duplicate objects.
**How to avoid:** Ensure `initialize()` detects that deps are already assigned and skips `constructDependencies()`.

### Pitfall 3: Not verifying `handleToolExecuteBefore` also benefits
**What goes wrong:** Fix primary path (`handleSessionEvent`) but `tool-delegation.ts:94` still has `if (!this.pendingRegistry) return`.
**How to avoid:** The primary fix (sync deps before delegation wiring) automatically fixes both — `this.pendingRegistry` is available when `handleToolExecuteBefore` fires.
