# Phase 23.1: Session-Tracker SDK Dispatch Investigation

## Summary

**Root cause:** Race condition — `sessionTracker.initialize()` is fire-and-forget (`void` at `plugin.ts:328`), but its critical-path dependencies (`eventCapture`, `pendingRegistry`, `childWriter`, `hierarchyIndex`) are only assigned via `Object.assign(this, deps)` inside the async initialize method. Any `onChildSessionCreated` callback that fires before `Object.assign` completes silently drops the `session.created` event, because `this.eventCapture` is still `undefined` and the `handleSessionEvent` guard at `index.ts:140` short-circuits.

**Confidence:** HIGH — the timing window is deterministic and matches the "Session status not found" symptom across all 4 failed UAT rounds.

### Event Expected

For the **native `task` tool**: OpenCode runtime creates the child session natively and fires `session.created` through the lifecycle event system. The `event` hook in core-hooks.ts receives this and routes to `consumeSessionTrackerFact` → `sessionTracker.handleSessionEvent()` → `eventCapture.handleSessionCreated()`.

For **`delegate-task`** (custom tool): The plugin calls `client.session.create()` directly via the SDK. **It is uncertain whether this SDK call triggers a `session.created` lifecycle event** that reaches the `event` hook. The `onChildSessionCreated` callback (coordinator.ts:138) is the backup path, calling `sessionTracker.handleSessionEvent()` directly to bypass the event hook.

## Phase 3: Root Cause

### Primary Root Cause (RACE CONDITION) — CONFIDENCE: HIGH

**The `sessionTracker.initialize()` method is fire-and-forget** (`void` at `plugin.ts:328`), and its critical-path dependencies are only assigned **after** asynchronous I/O completes inside `initialize()`.

#### The Race Window

```
Plugin init timeline:
  Line 280:  sessionTracker = new SessionTracker({...})
             → this.eventCapture = undefined  (TypeScript definite assignment assertion)
             → this.pendingRegistry = undefined
             → this.childWriter = undefined
             → this.hierarchyIndex = undefined

  Line 282:  setupDelegationModules({...onChildSessionCreated: (id, pid) => {
               sessionTracker.handleSessionEvent({...})  // captures sessionTracker ref
             }})

  Line 328:  void sessionTracker.initialize()  ← FIRE-AND-FORGET, no await
             ├── await this.hierarchyIndex?.buildFromDisk?.()
             │   └── NO-OP (hierarchyIndex is still undefined)
             ├── const deps = constructDependencies(...)
             │   └── SYNC — creates EventCapture, PendingDispatchRegistry, etc.
             ├── await deps.hierarchyIndex.buildFromDisk()  ← ASYNC I/O (could be slow)
             ├── Object.assign(this, deps)                  ← CRITICAL: deps now available
             │   └── this.eventCapture = EventCapture{...}
             │   └── this.pendingRegistry = PendingDispatchRegistry{...}
             │   └── this.childWriter = ChildWriter{...}
             │   └── this.hierarchyIndex = HierarchyIndex{...}
             ├── await this.retryQueue.flush()              ← more async I/O
             ├── await this.recovery.initialize()            ← more async I/O
             ├── await this.projectIndexWriter.initializeIndex()
             ├── await this.cleanupOrphanedTmpFiles()
             ├── await this.cleanupOrphanDirectories()
             └── await this.projectContinuityChecker.ensureCompleteness()
```

**Between `void sessionTracker.initialize()` (line 328) and `Object.assign(this, deps)` (inside initialize):**
- `this.eventCapture` is `undefined`
- `this.pendingRegistry` is `undefined`
- `this.childWriter` is `undefined`
- `this.hierarchyIndex` is `undefined`

**If `onChildSessionCreated` fires during this window:**

```typescript
// src/features/session-tracker/index.ts:134-142
async handleSessionEvent(event) {
  try {
    if (this.eventCapture) {              // → undefined! SKIPPED
      await this.eventCapture.handleSessionEvent(event)
    }
    if (
      event.eventType === "session.created" &&
      this.projectIndexWriter &&            // → undefined! SKIPPED
      this.sessionIndexWriter               // → undefined! SKIPPED
    ) {
      // Fork handling — also skipped
    }
  }
}
```

**The event is silently dropped.** No error is logged. No child `.json` is written. The session-tracker never learns about the child session.

#### Why This Matches U4/U5 Symptoms

- **U4 "Session status not found"**: The child session `.json` is never created. Status queries return nothing.
- **U5 "childCount:0, depth:0"**: No child records exist in the hierarchy index because `writeImmediateChildFile()` was never called.
- **Failed ALL 4 UAT rounds**: Every fresh plugin reload starts the race anew. If the first `delegate-task` dispatch happens in the first few seconds after plugin init (typical agent behavior), the race is almost guaranteed to trigger.
- **Intermittent success possible**: If `buildFromDisk()` completes quickly (few sessions) AND no delegation happens during the window, it works — explaining why tests might pass but UAT fails.

### Secondary Issues

1. **`sessionTracker.cleanup()` stops retry flush** (`plugin.ts:329`): The `.then(() => sessionTracker.cleanup())` chain stops the retry queue's periodic flush interval. Any child writes enqueued after cleanup never auto-flush.

2. **`client.session.create()` event uncertainty**: Even if the race is fixed, it's unclear whether SDK-created sessions fire `session.created` lifecycle events. The `onChildSessionCreated` callback is the only reliable path for `delegate-task`.

3. **No logging for dropped events**: `handleSessionEvent` silently returns when `eventCapture` is undefined. No warning, no error. This made debugging the race extremely difficult.

### Why Previous Fixes Failed

| Fix | File | Why It Didn't Work |
|-----|------|-------------------|
| Added `delegate-task` to tool filter | `tool-before-guard.ts:40` | Correctly populates PendingDispatchRegistry, but Gate 0 is never reached because `handleSessionCreated` isn't called when `eventCapture` is undefined |
| `onChildSessionCreated` callback | `coordinator.ts:138` + `plugin.ts:288-290` | Correctly bypasses event hook, but `sessionTracker.handleSessionEvent()` drops the event when deps are uninitialized |

## Phase 4: Fix Scope

### Primary Fix: Eliminate the Race Window

**Approach A (Minimum — RECOMMENDED):** Move critical-path dependency construction **before** `setupDelegationModules()` captures the `onChildSessionCreated` callback.

Move the sync portion of `constructDependencies()` out of `initialize()` and into the constructor or an init phase 1:

```typescript
// In plugin.ts:
const sessionTracker = new SessionTracker({ client, projectRoot })

// NEW: Construct critical deps synchronously before wiring delegation
sessionTracker.constructCoreDependencies()  // creates pendingRegistry, eventCapture, etc.

const delegationModules = setupDelegationModules({
  ...
  onChildSessionCreated: (id, pid) => {
    // Now this WILL work — eventCapture is available
    sessionTracker.handleSessionEvent(...)
  }
})

// Existing: async init for I/O-heavy setup
void sessionTracker.initialize()  // builds hierarchy, flushes retry queue, etc.
```

**Files to change:** 2-3
- `src/features/session-tracker/index.ts` — add `constructCoreDependencies()` public method
- `src/features/session-tracker/initialization.ts` — optionally extract sync portion
- `src/plugin.ts` — call `constructCoreDependencies()` before `setupDelegationModules()`

**LOC estimate:** 15-25 lines
**Complexity:** SIMPLE

### Secondary Fix: Add Defensive Logging

Add a warning log when `handleSessionEvent` is called but `eventCapture` is undefined:
- **File:** `src/features/session-tracker/index.ts:140` — log warning instead of silent skip
- **LOC:** 3 lines
- **Complexity:** TRIVIAL

### Tertiary Fix: Remove premature cleanup

Remove the `.then(() => sessionTracker.cleanup())` chain — let the retry queue live for the plugin's lifetime:
- **File:** `src/plugin.ts:328-339`
- **LOC:** 2 lines
- **Complexity:** TRIVIAL

## Recommendation

**Fix within this phase (23.1).** The fix is SIMPLE (15-25 LOC) and addresses the confirmed root cause directly. No major architectural changes needed.

### Implementation Order

1. **Primary fix:** Move `constructDependencies()` (or a sync subset) before the delegation module wiring in `plugin.ts` — ensures `eventCapture` and `pendingRegistry` are available before `onChildSessionCreated` can fire
2. **Defensive logging:** Add warning when `handleSessionEvent` skips due to uninitialized deps — prevents silent debugging in future regressions
3. **Cleanup fix:** Remove `.then(() => sessionTracker.cleanup())` — keeps retry queue alive

### Verification

After fix, verify via:
1. **Unit test:** Call `onChildSessionCreated` immediately after `new SessionTracker()` — should write child `.json`
2. **Typecheck + test suite:** `npm run typecheck && npx vitest run tests/features/session-tracker/`
3. **Live UAT:** Delegate a task via `delegate-task` in the first second after plugin load — `session-tracker` should show child session with correct hierarchy
