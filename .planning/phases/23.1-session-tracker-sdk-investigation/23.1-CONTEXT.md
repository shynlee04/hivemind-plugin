<domain>
## Phase Boundary

Fix race condition in `sessionTracker.initialize()` causing `onChildSessionCreated` callbacks from `delegate-task` dispatch to be silently dropped. This is the root cause of U4/U5 (session-tracker cannot find child sessions, hierarchy broken) that failed 4 rounds of live UAT. Fix scope: ~30 LOC, 3-4 files, SIMPLE complexity.

</domain>

<decisions>
## Implementation Decisions

### Fix Architecture
- **D-01:** Move sync `constructDependencies()` (or sync subset of critical deps: `eventCapture`, `pendingRegistry`, `childWriter`, `hierarchyIndex`) to execute BEFORE `setupDelegationModules()` in `plugin.ts`. This ensures these deps are available before any `onChildSessionCreated` callback can fire.
- **D-02:** Keep existing `initialize()` async method for remaining async setup (hierarchy index build, retry queue start). Only the critical-path deps need to be synchronous.
- **D-03:** Add defensive `[Harness]` warning log at `index.ts:140` when `handleSessionEvent` skips due to uninitialized deps — prevents silent debugging in future.
- **D-04:** Remove `.then(() => sessionTracker.cleanup())` chain from `plugin.ts:328` — retry queue must live for plugin lifetime.

### the agent's Discretion
- Exact implementation of sync subset extraction from `constructDependencies()`
- Whether to expose a new sync method or refactor `initialize()` to split sync/async parts
- Variable naming in plugin.ts for the refactored initialization

</decisions>

<specifics>
## Specific Ideas

- The race window is deterministic: `void sessionTracker.initialize()` at `plugin.ts:328` starts async deps assignment, but `setupDelegationModules()` at `plugin.ts:282` captures `sessionTracker` ref BEFORE deps are ready. Any `onChildSessionCreated` before `Object.assign(this, deps)` inside initialize() hits the guard.
- Fix must ensure the FIRST call to `onChildSessionCreated` after `new SessionTracker()` has its deps available — no exceptions.

</specifics>
