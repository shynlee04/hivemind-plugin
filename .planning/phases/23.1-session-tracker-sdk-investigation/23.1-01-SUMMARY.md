# Phase 23.1 Plan 01: Fix Session-Tracker Race Condition

**Sync constructCoreDependencies() before delegation wiring eliminates the race window where onChildSessionCreated callbacks silently dropped session events; adds defensive [Harness] logging for uninitialized deps; removes premature cleanup chain killing the retry queue**

- **Completed:** 2026-05-24T02:48:47Z
- **Tasks:** 3 (all committed; working tree restored)
- **Files modified:** 2 source files + 1 research doc

## Accomplishments

- **REQ-01:** Added `constructCoreDependencies()` public sync method on SessionTracker — called in plugin.ts BEFORE `setupDelegationModules()` so `onChildSessionCreated` callbacks immediately find `eventCapture`, `pendingRegistry`, etc. available. Race window eliminated.
- **REQ-02:** Added `[Harness]` warning log at `handleSessionEvent` guard (index.ts:140 else branch) — fires when `eventCapture` is unexpectedly falsy, providing diagnostic signal for future timing regressions.
- **REQ-03:** Removed `.then(() => sessionTracker.cleanup())` chain from plugin.ts — retry flush interval now lives for plugin lifetime instead of being killed on init completion.
- **Dead code removal:** `await this.hierarchyIndex?.buildFromDisk?.()` at index.ts:458 was a permanent no-op (hierarchyIndex undefined pre-Object.assign). Removed as part of `initialize()` cleanup.
- **Research doc updated:** Both Open Questions marked (RESOLVED) with rationale.

## Decisions Made

- **constructCoreDependencies() is fully synchronous** — calls the existing `constructDependencies()` factory and assigns all deps via `Object.assign(this, deps)`. Previously this was done inside async `initialize()` after an I/O await. Safe to call multiple times.
- **initialize() no longer calls constructDependencies()** — after `constructCoreDependencies()` has been called first, `initialize()` directly calls `await this.hierarchyIndex.buildFromDisk()` (no optional chaining needed — hierarchyIndex is already defined).
- **cleanup is not called after initialize()** — the retry flush interval started by `initialize()` runs for the plugin lifetime. Cleanup is handled by lifecycle shutdown hooks instead.
