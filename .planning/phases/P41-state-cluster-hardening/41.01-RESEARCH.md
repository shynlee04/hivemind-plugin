# Phase 41.01: State Store Redundancy Analysis — Research

**Researched:** 2026-05-31
**Domain:** State persistence architecture, deduplication, and canonical data model
**Confidence:** HIGH — all claims verified against actual source code and live on-disk files

## Summary

This investigation definitively answers whether `.hivemind/state/delegations.json` and `.hivemind/state/session-continuity.json` are redundant with session-tracker data under `.hivemind/session-tracker/`.

**Primary finding:** Neither file is fully redundant, but there IS partial data overlap that causes cross-contamination and unnecessary complexity. Both files serve distinct purposes from session-tracker's files, though with overlapping identity concepts.

| File | Redundant? | Verdict |
|------|-----------|---------|
| `delegations.json` | **PARTIALLY** | ~40% overlap with session-tracker child .json files, but 60% unique data (executionMode, surface, recoveryGuarantee, queueKey, nestingDepth, polling state). Cannot be replaced. |
| `state/session-continuity.json` | **NO** — but misnamed | ZERO overlap with session-tracker's per-session `session-continuity.json`. They are entirely different data with the same filename. Contains unique rich metadata (pendingNotifications, lifecycle, delegationPacket, compactionCheckpoint, governance) that session-tracker does not store. |
| Session overlap between state and project | **ZERO** | 0 out of 114+18 sessions overlap between `.hivemind/state/session-continuity.json` and `.hivemind/session-tracker/project-continuity.json`. The state file uses a different namespace (test/parent/v1 IDs) |

**Primary recommendation:** Keep all three systems, but act on TWO things: (1) rename `state/session-continuity.json` to something distinguishable from the per-session one, and (2) push `pendingNotifications` and `delegationPacket` into the session-tracker child .json files to eliminate the only real cross-system duplication.

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| Delegation operation state (dispatch → poll → terminal) | `delegations.json` | — | Delegation lifecycle is operation-oriented, not session-oriented. Session IDs are properties of a delegation, not its key. |

| Child session conversation data (turns, journey, agent metadata) | session-tracker `{childID}.json` files | — | Per-child file captures the actual work done. |

## Redundancy Verdicts

### 1. Is `delegations.json` redundant?

**Verdict: NOT fully redundant — overlap exists but 60% of fields are unique.**

Overlapping fields (found in both delegations.json AND session-tracker child .json):
- `parentSessionId` ↔ `parentSessionID` ✓
- `childSessionId` ↔ `sessionID` (child file's own id) ✓
- `agent` ↔ `delegatedBy.agentName` / `mainAgent.name` ✓
- `status` ↔ `status` ✓
- `createdAt` ↔ `created` (but format differs: epoch ms vs ISO string) ✓
- `error` ↔ `status="error"` (but delegations has full error string, session-tracker has no error field) ⚠️

Unique to `delegations.json` (not found in any session-tracker file):
- `executionMode` (sdk/pty/headless) — affects dispatch strategy
- `surface` (agent-delegation/command-process) — affects recovery
- `recoveryGuarantee` (resumable/best-effort/non-resumable-after-restart) — affects lifecycle
- `queueKey` — concurrency control
- `nestingDepth` — separate from session-tracker's `delegationDepth` (different computation)
- `stablePollCount`, `lastMessageCount`, `lastMessageCountChangeAt` — polling state machine
- `gracePeriodExpiresAt` — cleanup timing
- `explicitCancellation` — user intent tracking
- `terminationSignal` — OS signal tracking
- `workingDirectory` — filesystem context
- `ptySessionId` — PTY delegation tracking
- `fallbackReason` — fallback mode tracking
- `id` (delegation UUID) — keyed by delegation, not session

**The delegation ID is the critical difference:** `delegations.json` is keyed by delegation operation UUID (`id` field). Session-tracker is keyed by session ID. A single delegation maps one parent session to one child session, but the delegation operation itself (with its polling state, timers, and dispatch metadata) has no equivalent in session-tracker.

### 2. Is `session-continuity.json` (at `.hivemind/state/`) redundant?

**Verdict: NOT redundant — ZERO field overlap with session-tracker's `session-continuity.json`.**

Despite sharing the same filename, these are entirely different data structures:

| Aspect | `.hivemind/state/session-continuity.json` | `.hivemind/session-tracker/{id}/session-continuity.json` |
|--------|-------------------------------------------|-----------------------------------------------------------|
| Schema | `ContinuityStoreFile` (v1) | `SessionContinuityIndex` (v2.0) |
| Key | Session ID → rich metadata | Root session ID → hierarchy tree |
| Contains | pendingNotifications, lifecycle, governance, delegationPacket, compactionCheckpoint | hierarchy.children, turnCount, toolSummary |
| Fields named "sessionID" | Session IDs come from the delegation/notification context | Session IDs are the OpenCode SDK session IDs |
| Session overlap | **ZERO** overlap with project-continuity.json sessions (18 state sessions vs 114 project sessions, 0 shared IDs) | Shares IDs with project-continuity.json |

**The naming is the ONLY problem.** These should be called different things:
- `.hivemind/state/session-metadata.json` (or `session-state.json`)
- `.hivemind/session-tracker/{id}/session-hierarchy.json` (or `hierarchy-index.json`)

### 3. What's the clean architecture?

```
Canonical state model (if redesigned):

.hivemind/
├── state/
│   ├── delegations.json              # KEEP as-is — delegation operations
│   └── session-metadata.json          # RENAME from session-continuity.json
│                                      # (keeps lifecycle, notifications, governance,
│                                       compaction checkpoints, delegation packets)
└── session-tracker/
    ├── project-continuity.json        # KEEP as-is — master session index
    ├── {sessionID}/
    │   ├── hierarchy-index.json       # RENAME from session-continuity.json
    │   │                               # (per-session hierarchy tree)
    │   ├── hierarchy-manifest.json    # KEEP as-is — flat delegation tree snapshot
    │   └── {childID}.json             # KEEP as-is — child session records
    └── {sessionID}/...
```

Key design principles:
1. **`state/` = runtime operation state** (delegation lifecycle, governance rules, session lifecycle hooks). Written synchronously by control-plane code. Survives restarts but is not the primary data store.
2. **`session-tracker/` = session knowledge base** (hierarchy, turns, journey, agent info). Written asynchronously by hook observers. The canonical source for "what happened in this session."
3. **Identity separation**: `delegations.json` keys by delegation UUID → crossing into session-tracker. Session-tracker keys by session ID. The difference is fundamental — don't merge them.
4. **Eliminate the `status` field duplication**: Currently `status` exists in delegations.json (delegation status), state/session-continuity.json (session metadata status), and project-continuity.json (session index status). Each serves a different purpose but the naming overlap causes confusion. Use distinct field names or accept it as intentional read-model denormalization.

## Common Pitfalls

### Pitfall 1: Assuming identically named files store the same data
**What goes wrong:** Developers see "session-continuity.json" in two locations and assume redundancy.
**Why it happens:** The naming was not coordinated between the legacy continuity module (v1) and session-tracker (v2.0).
**How to avoid:** Rename one or both to make the distinction obvious from the filename.
**Warning signs:** Confusion in code reviews about "which session-continuity.json" is being used.

### Pitfall 2: Attempting to merge delegations.json into session-tracker
**What goes wrong:** Loss of delegation operation data (executionMode, polling state, timers) that session-tracker does not store.
**Why it happens:** Superficial overlap in parentSessionId/childSessionId fields suggests duplication.
**How to avoid:** Recognize that delegations are operations (keyed by UUID), not sessions (keyed by session ID).
**Warning signs:** The identity mismatch — delegation UUIDs don't appear anywhere in session-tracker.

### Pitfall 3: Pending notifications stored in two places
**What goes wrong:** `state/session-continuity.json` stores `pendingNotifications` as session metadata. Some notification/event data also flows through session-tracker. Duplicated notification delivery or missed notifications on recovery.
**Why it happens:** The notification system evolved independently from session-tracker.
**How to avoid:** Migrate `pendingNotifications` into child .json files or a dedicated notification state file.

### Pitfall 4: Zero session overlap creates recovery blind spots
**What goes wrong:** On restart, `recoverPending()` reads `delegations.json` and `readPersistedDelegations()` finds delegation records whose session IDs don't exist in any session-tracker file. The recovery path has incomplete context.
**Why it happens:** Test and notification-only sessions (`parent-1`, `ses_cleanup`, `ses_parent`) are created in the state store without corresponding session-tracker entries.
**How to avoid:** Either (1) ensure all sessions written to state store are also registered in project-continuity.json, or (2) accept the namespace separation as intentional.

## Open Questions

1. **Should pendingNotifications be migrated to session-tracker child .json files?**
   - What we know: Currently stored only in `state/session-continuity.json`. Not accessible from session-tracker tools.
   - What's unclear: Would this require schema changes to `ChildSessionRecord` or a new dedicated notification file?
   - Recommendation: Investigate as minor migration — add `pendingNotifications` field to `ChildSessionRecord` or create a separate `notifications.json` per session.

2. **Should governance state be moved out of session-continuity.json?**
   - What we know: Governance (rules + violations) is stored under `sessions.governance` in the v1 continuity file. It has no relationship to sessions.
   - What's unclear: Whether governance needs its own storage location, or if it should stay as a parallel key in the metadata store.
   - Recommendation: Keep it for now — it's a small payload and not causing problems. List as future optimization.

3. **What is the migration cost of renaming `session-continuity.json` files?**
   - What we know: The two identically-named files have different schemas and different consumers.
   - What's unclear: Whether existing files on disk should be renamed, or only future writes use new names.
   - Recommendation: Only change future writes + add backwards-compatible reads. Do not migrate existing on-disk files.
