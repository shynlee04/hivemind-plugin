# SPEC: Session-metadata.json (renamed from session-continuity.json)

**Spec version:** 1.0
**Date:** 2026-05-31
**Status:** Draft for redesign phase
**Requirement IDs:** P41-SM-01 through P41-SM-15

## 1. Purpose (one sentence)

`session-metadata.json` is the harness's durable, single-source-of-truth file that stores rich per-session metadata (status, delegation context, pending notifications, lifecycle state, governance records) for all sessions across the project lifetime, enabling crash recovery, notification replay, and delegation continuity without depending on in-memory state or the OpenCode SDK session API.

| **Process exit / SIGINT / SIGTERM** | `continuity/index.ts` → `flushAllStores()` | Flush cached stores to disk | **Yes** (forced write regardless of `atomic_commit`) |

### What DOES NOT trigger writes

- **Hooks SHALL NOT directly write `.hivemind/state/` files** — enforced by CQRS boundary (`classifyHookEffect()`). Hooks call continuity library functions which own persistence, but hooks themselves never write raw files.

### `SessionContinuityMetadata` (the rich metadata payload)

```typescript
interface SessionContinuityMetadata {
  // ---- Core status ----
  status: TaskStatus             // "pending" | "queued" | "running" | "completed" | "failed" | "error" | "cancelled" | "interrupt"
  description: string            // Human-readable session purpose
  title?: string                 // Optional session title

  // ---- Delegation context ----
  delegation: DelegationMeta | null  // rootID, depth, budgetUsed, agent, model, queueKey, runtimePolicyOverride
  category?: string              // Session category (e.g., "implementation", "review", "investigation")
  constraints: string[]          // Task constraints (e.g., ["no parallel delegation"])
  route?: string                 // Session router classification result

  // ---- Lifecycle ----
  lifecycle?: SessionLifecycleState  // phase, launchedAt, completedAt, runMode, queue, observation, error

  // ---- Pending notifications (recovery-critical) ----
  pendingNotifications: PendingNotification[]  // Undelivered delegation notifications
    // Each PendingNotification: { sessionID, description, agent, status, error,
    //   resultPreview, briefSummary, outputLink, duration, elapsedHuman,
    //   metadata: { delegationId, terminalState, recoveryGuarantee, summaryPreview },
    //   createdAt, delivered, retryCount, maxRetries, ... }

  // ---- Delegation packet (for auto-loop) ----
  delegationPacket?: DelegationPacket  // { id, createdAt, spec, plan, artifacts, commits, parentChain, status, updatedAt }

  // ---- Result capture (for delegation result retrieval) ----
  resultCapture?: CapturedResult      // resultText, artifactPaths, gitCommits, toolCallSummary, messageCount, capturedAt, partial

  // ---- Compaction checkpoint ----
  compactionCheckpoint?: CompactionCheckpointData  // agent, model, tools, delegationMeta, warnings, sessionStats, capturedAt

  // ---- Timestamps ----
  lastToolActivityAt?: number    // Unix ms of last tool use
  updatedAt: number              // Unix ms of last record update
}
```

## 5. Name Disambiguation

**The core problem that this rename solves: two completely different data structures both named `session-continuity.json`.**

| File | Path | Content | Owned By | Schema |
|------|------|---------|----------|--------|
| **NEW: `session-metadata.json`** (was `session-continuity.json`) | `.hivemind/state/session-metadata.json` | Rich per-session metadata: status, delegation, pending notifications, lifecycle, governance violations | `src/task-management/continuity/index.ts` | `ContinuityStoreFile` (versioned envelope with `sessions{}` map, governance section) |
| **Unchanged: `session-continuity.json`** (session-tracker per-session) | `.hivemind/session-tracker/{sessionID}/session-continuity.json` | Parent-child hierarchy tree for one main session | `src/features/session-tracker/` | `SessionContinuityIndex` (hierarchy tree, turn count, tool summary) |

**Why the rename is safe:**
- The session-tracker's `project-continuity.ts` explicitly skips `session-continuity.json` when scanning for child records (`child.name === "session-continuity.json"` — see `project-continuity.ts:110`).
- The two files have never referenced each other's data.
- The only shared naming collision is the string `"session-continuity.json"` — they parse to different shapes.

**Migration:**
- Rename `.hivemind/state/session-continuity.json` → `.hivemind/state/session-metadata.json`
- Update `resolveContinuityFilePath()` in `continuity/index.ts:39-52` to look for `session-metadata.json`
- Keep backward compatibility: try `session-metadata.json` first, fall back to `session-continuity.json` (legacy) if missing
- The environment variable `OPENCODE_HARNESS_CONTINUITY_FILE` (if set) should accept the new name or redirect
- Legacy path still at `.opencode/state/hivemind/session-continuity.json` — this was the pre-Q6 path and should continue to be checked for backward compatibility during a transition window

**Spec for backward-compat load order (priority):**
1. `OPENCODE_HARNESS_CONTINUITY_FILE` env var (exact path — caller updates at migration)
2. `.hivemind/state/session-metadata.json` (new canonical)
3. `.hivemind/state/session-continuity.json` (legacy canonical, phased out)
4. `.opencode/state/hivemind/session-continuity.json` (legacy pre-Q6, phased out)

## 6. Lifecycle States

### File-level states

| State | Description | When |
|-------|-------------|------|
| **Empty** | File does not exist, or exists as `{}` (which normalizes to empty store) | Clean install, first run |
| **Active** | File exists with valid JSON, sessions `{}` map may contain 0-N records | Normal operation |
| **Stale (atomic_commit: false)** | In-memory cache has data but disk file is outdated or missing | Config has `atomic_commit: false` |
| **Corrupt** | File exists but contains unparseable JSON | Crash during write, manual corruption |
| **Quarantined** | Corrupt file renamed to `session-metadata.json.corrupt-{timestamp}-{pid}-{uuid}` | Detected on read, auto-quarantined |

### Per-record lifecycle states

The `metadata.status` field follows the `TaskStatus` lifecycle (from `src/shared/types.ts:3`):

```
pending → queued → running → completed
                    ↓   ↓
                 failed  error
                    ↓
                cancelled
                    ↓
               interrupt (can return to previous state)
```

### Actor-triggered state transitions

```
session.created          → record created with status: "pending"
session.queued           → patch status: "queued"
delegation dispatched    → patch status: "running"
session.deleted          → patch status: "completed" (or delete record)
session.error            → patch status: "error"
delegation timeout       → patch status: "failed"
user cancellation        → patch status: "cancelled"
auto-loop interrupt      → patch status: "interrupt" (preserves previous state)
```

## 7. Cross-File Dependencies

### Direct data links

| Other File | Path | Relationship | Direction |
|-----------|------|-------------|-----------|
| **delegations.json** | `.hivemind/state/delegations.json` | Pending notifications in session-metadata reference `delegationId` values that exist in delegations.json. When `replayPendingDelegationNotifications()` replays, it reads session-metadata, looks up delegation IDs, and appends to TUI. | session-metadata → delegations.json (read) |
| **session-tracker hierarchy** | `.hivemind/session-tracker/{sessionID}/session-continuity.json` | NO direct data link — different data structure with name collision only. The session-tracker explicitly skips `session-continuity.json` during child scans. | None (name collision only) |
| **session-tracker project index** | `.hivemind/session-tracker/project-continuity.json` | NO direct data link — session-metadata stores rich per-session data; project-continuity stores session discovery index. However, `hydrateFromContinuity()` in the lifecycle manager reads session-metadata at plugin init. | session-metadata → lifecycle manager (read) |
| **trajectory-ledger.json** | `.hivemind/state/trajectory-ledger.json` | NO direct data link as of Phase 41. Trajectory events reference session IDs that may or may not appear in session-metadata's `sessions{}` map. Indirect: trajectory events link to delegation IDs which could cross-reference session-metadata's pending notification delegationIds. | None (indirect via delegationId) |
| **agent-work-contracts.json** | `.hivemind/state/agent-work-contracts.json` | NO direct data link. Agent work contracts reference session IDs independently. | None |

### Startup sequence dependency

```
plugin init
  ├── SessionTracker.constructCoreDependencies()
  │     └── (no read from session-metadata)
  ├── setupDelegationModules()
  │     └── delegationManager.recoverPending() — reads delegations.json, NOT session-metadata
  ├── lifecycleManager.hydrateFromContinuity()
  │     └── READS session-metadata.json — rehydrates lifecycle state
  └── replayPendingDelegationNotifications()
        └── READS session-metadata.json — drains pending notifications
```

### What breaks if session-metadata.json is missing

1. **Cold start with no file:** `loadStoreFromDisk()` returns `emptyStore()` with zero sessions. `hydrateFromContinuity()` finds nothing — no lifecycle state rehydration, no auto-loop recovery, no pending notification replay. This is graceful degradation.
2. **Corrupt file:** `loadStoreFromDisk()` quarantines the file and throws `[Harness]` error. The plugin init catches this — but any synchronous call to `listSessionContinuity()` or `getSessionContinuity()` will throw. (`ensureStoreLoaded()` on line 247-257 is called by the public API and will throw on corrupt data.)
3. **Missing delegations.json + pending notifications:** `replayPendingDelegationNotifications()` reads session-metadata for pending notifications. If session-metadata's `pendingNotifications[]` has records but delegations.json is absent, replay proceeds with whatever fields exist (sessionID, description, status, error) and logs to TUI.

## 8. Acceptance Criteria

### P41-SM-01: Rename to session-metadata.json
- **Behavior:** The continuity module reads and writes `.hivemind/state/session-metadata.json` instead of `.hivemind/state/session-continuity.json`
- **Backward compatibility:** On first run with the new name, if `session-metadata.json` does not exist but `session-continuity.json` does, the continuity module reads the old file, migrates in memory, and writes to the new file.
- **Environment variable:** `OPENCODE_HARNESS_CONTINUITY_FILE` still works as an exact path override. `OPENCODE_HARNESS_STATE_DIR` still works, writing to `{stateDir}/session-metadata.json`.
- **Verification:** `continuity.getContinuityStoragePath()` returns a path ending in `session-metadata.json`.

### P41-SM-02: Legacy path fallback
- **Behavior:** The module checks paths in order: env var → `.hivemind/state/session-metadata.json` → `.hivemind/state/session-continuity.json` → `.opencode/state/hivemind/session-continuity.json`
- **Verification:** Test that `loadStoreFromDisk()` finds data on the second (legacy canonical) fallback when the new file doesn't exist.

### P41-SM-03: Name collision resolved
- **Behavior:** No file named `session-continuity.json` exists under `.hivemind/state/`.
- **Verification:** `ls .hivemind/state/session-continuity.json` returns "no such file" after migration.

### P41-SM-04: Write path updated
- **Behavior:** `resolveContinuityFilePath()` returns path ending in `session-metadata.json`.
- **Verification:** After importing continuity module, `getContinuityStoragePath()` returns the new path.

### P41-SM-05: Quarantine uses new name
- **Behavior:** Corrupt files are renamed to `session-metadata.json.corrupt-{ts}-{pid}-{uuid}`.
- **Verification:** Write garbage to new path, call `listSessionContinuity()`, verify `.corrupt-` file with new prefix exists.

### P41-SM-06: atomic_commit toggle still works
- **Behavior:** When `atomic_commit: false`, writes do not create `session-metadata.json`. When `true`, writes create it.
- **Verification:** Existing tests for atomic_commit toggle pass with the new filename.

### P41-SM-07: flushAllStores writes new name
- **Behavior:** `flushAllStores()` writes all cached stores using the new path.

### P41-SM-08: All existing functionality preserved

## 9. Non-Goals (Out of Scope)

- **No schema changes:** The `ContinuityStoreFile`, `SessionContinuityRecord`, and `SessionContinuityMetadata` type shapes remain identical. This is purely a rename + path update.
- **No session-tracker changes:** The session-tracker's own `session-continuity.json` files (hierarchy trees) are NOT affected.
- **No API surface changes:** All exported functions (`recordSessionContinuity`, `patchSessionContinuity`, `listSessionContinuity`, etc.) keep their existing names and signatures.
- **No legacy migration tool:** The rename leverages existing fallback logic (`loadStoreFromDisk` tries multiple paths). No separate migration script.
- **No hook changes:** hooks that read continuity data need no code changes — they import module-level functions, not file paths.

## 10. Open Questions

1. **Should the exported function names also be renamed for consistency?**
   - Current: `recordSessionContinuity()`, `patchSessionContinuity()`, `listSessionContinuity()`, etc.
   - Would become: `recordSessionMetadata()`, `patchSessionMetadata()`, etc.
   - Risk: Higher migration cost (every caller in 6+ files + tests must update)
   - Recommendation: Defer to a follow-up phase. The SPEC is about the *file* rename, not the API.

2. **Should the type names `ContinuityStoreFile` / `SessionContinuityRecord` be renamed?**
   - Same consideration as above. Defer. Types are internal to the continuity module and re-exported from `shared/types.ts`.

3. **Is there a single migration file rename that flips the switch, or do we need a gradual rollout?**
   - Single atomic change recommended: update `getContinuityFile()` → try new path first, fall back to old, add backward-compat read. This is a one-commit change.
