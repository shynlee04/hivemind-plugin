    For each of the 5 coordinator skills, add a `## Self-Correction` section. Position it AFTER the main workflow/procedure content and BEFORE any reference/appendix sections (or at the end of the file if no references exist).

    Each Self-Correction section must contain 4 subsections as `###` headings with 3-5 sentences of practical guidance per subsection. Tailor the content to each skill's specific domain.

    **Template structure:**
    ```markdown
    ## Self-Correction

    ### When the Task Keeps Failing
    (3-5 sentences specific to this skill's domain — what to check, what to try, when to escalate)

    ### When Unsure About the Next Step
    (3-5 sentences — how to diagnose uncertainty, what information to gather, default safe action)

    ### When the User Contradicts Skill Guidance
    (3-5 sentences — how to handle user override, when to defer, how to document the deviation)

    ### When an Edge Case Is Encountered
    (3-5 sentences — how to recognize edge cases, when to improvise vs. when to stop, how to report)
    ```

    **Tailored content per skill:**

    **hm-coordinating-loop** — Multi-agent dispatch coordination:
    - Failing: Check if subagent files_to_read actually exist on disk. Verify the agent name matches an .opencode/agents/ file. Try a simpler task decomposition with fewer agents.
    - Unsure: Default to sequential dispatch instead of parallel. Log the decision point and proceed with the safest option.
    - Contradiction: If the user wants a different agent than what the skill recommends, use the user's choice but note it may affect parallel safety.
    - Edge case: If two agents need the same output file, serialize their execution instead of failing on file conflict.

    **hm-phase-loop** — Iterative phase loops:
    - Failing: Check if the phase entry gate criteria are actually met before looping. Verify STATE.md reflects real progress. If the same gate fails 3 times, stop and ask the user.
    - Unsure: Read the phase PLAN.md again for the verification criteria. If no PLAN.md exists, the phase may not be planned yet — suggest planning first.
    - Contradiction: If the user says the phase is done but verification fails, present the failing truths and ask which to accept or rework.
    - Edge case: If a phase has zero plans, it was likely skipped — suggest planning before execution.

    **hm-phase-execution** — Wave-based execution:
    - Failing: If a wave plan fails, do not continue to the next wave. Report the failure, check if it's a dependency issue, and retry only the failed plan. If the plan file references non-existent files, validate against disk first.
    - Unsure: Check the PLAN.md frontmatter for `wave` and `depends_on` fields. If dependencies aren't met, skip this plan and come back.
    - Contradiction: If the user wants to run plans out of wave order, warn about dependency risks but allow if the user confirms.
    - Edge case: If two same-wave plans touch the same file, they cannot run in parallel — serialize them and note the conflict.

    **hm-completion-looping** — Completion guardrails:
    - Failing: If completion keeps failing to verify, check if the verification command in PLAN.md is actually runnable. If the test command references a missing test file, report it as a planning gap rather than a completion failure.
    - Unsure: Default to the most conservative interpretation — if you cannot verify completion, treat it as incomplete and report why.
    - Contradiction: If the user says "it's done" but automated verification fails, run the verification command in front of the user and discuss the specific failure.
    - Edge case: If a task has no `<automated>` verification at all, flag it as a Nyquist gap and suggest adding verification before claiming done.

    **hm-user-intent-interactive-loop** — User intent probing:
    - Failing: If the user keeps giving unclear answers, switch from open-ended questions to multiple-choice options. If the user seems frustrated with questions, summarize what you know and ask for confirmation.
    - Unsure: If you cannot determine if intent is clear after 3 exchanges, present your best understanding and ask "Is this correct?" rather than asking more questions.
    - Contradiction: If the user contradicts an earlier answer, acknowledge the change explicitly and ask which version is current. Update your understanding and proceed.
    - Edge case: If the user provides requirements in a format the skill doesn't handle (e.g., a document link), read the document and extract intent from it rather than asking the user to restate.
