<decisions>
## Implementation Decisions

### Closure Architecture
- **D-01:** Gap closure uses sub-phases approach — 16.5.1 (Schema resilience + Discovery), 16.5.2 (Framework stacking + Runtime validation), 16.5.3 (Mixed-primitive batch + Eval harness)
- **D-02:** 16.5.1 scope: Both Blocker 3 (schema `.strict()` fix) and Blocker 1 (object discovery/loader) in one phase — tightly coupled because discovery needs resilient schemas to read existing configs

### Schema Resilience (Blocker 3)
- **D-03:** Fix `.strict()` time-bomb — schemas must accept unknown future OpenCode fields without rejecting valid configs. Use `passthrough()` or dual-mode validation (strict for writes, lenient for reads).

### Object Discovery (Blocker 1)
- **D-04:** Hybrid discovery approach — file-system scanner (walk `.opencode/` directories, parse YAML frontmatter from `.md` files) for breadth coverage PLUS SDK-based discovery for authoritative validation
- **D-05:** Cross-reference scanner results against SDK registry to detect orphaned/invalid configs that could crash the terminal TUI — warn user about discrepancies

### Framework Stacking (Blocker 2)
- **D-06:** Rename skill from `hivefiver-agent-config` to framework-agnostic name (e.g., `opencode-agent-config`) — reflect in skill file, command references, and orchestrator routing
- **D-07:** Detect co-existing frameworks by scanning for their config files (`.planning/` for GSD, `bmad.yaml` for BMAD, Speckit configs) and namespace-tag generated configs to avoid name collisions

### Agent's Discretion
- Semantic intent detection depth (beyond keyword matching) — planner decides approach based on phase scope
- Specific framework markers beyond GSD/BMAD/Speckit — researcher discovers during planning
- Exact new skill name — planner proposes, executor implements
</decisions>
