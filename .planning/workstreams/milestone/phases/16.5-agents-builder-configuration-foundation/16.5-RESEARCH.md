# Phase 16.5: Agents Builder Configuration Foundation — Research

**Researched:** 2026-04-25
**Domain:** OpenCode primitive configuration, Zod schema validation, cross-primitive conflict detection, config compilation
**Confidence:** HIGH

## Summary

Phase 16.5 builds the programmatic foundation for configuring OpenCode agents and related primitives (commands, skills, tools, permissions, MCP servers, rules) through structured schemas, validation, and compilation — wired to the hivefiver meta-builder module. Prior to this phase, ALL agent configuration in the project was done through interactive LLM-guided conversations with specialist agents. There was no Zod schema validation, no config-to-compiler tool, no cross-primitive conflict detection, and no batch configuration capability.

Cycle 1 (Zod Schema Foundation) has been completed: 7 schema files totaling 772 LOC (plus 290 LOC of tests) now exist in `src/schema-kernel/`, covering agent frontmatter, command frontmatter, permission rulesets, skill metadata, MCP server configuration, tool definitions, and config precedence. 606 total tests pass with typecheck clean. Cycles 2–6 remain pending progressive execution.

**Primary recommendation:** Continue with Cycle 2 (Cross-Primitive Validator) as the next highest-value cycle, since it creates the conflict detection engine that all subsequent cycles depend on for correctness guarantees.

### Pattern 1: Schema-per-Primitive with Barrel Re-export
**What:** Each OpenCode primitive gets its own schema file with self-contained types and exports, unified through `index.ts`.
**When to use:** All new schema additions.
**Example:**
```typescript
// Source: src/schema-kernel/agent-frontmatter.schema.ts (VERIFIED on disk)
import { z } from "zod"

export const AgentFrontmatterSchema = z.object({
  description: z.string().min(1),
  mode: z.enum(["primary", "subagent", "all"]).optional(),
  model: z.string().optional(),
  temperature: z.number().min(0).max(2).optional(),
  // ... all 14+ fields
}).strict()

export type AgentFrontmatter = z.infer<typeof AgentFrontmatterSchema>
```

### Pattern 2: Strict Schema with Deprecation Tracking
**What:** Use `.strict()` on all schemas to reject unknown fields, but include deprecated fields with informational refines that always pass.
**When to use:** Agent frontmatter (`tools`, `maxSteps` deprecated fields).
**Example:**
```typescript
// Source: src/schema-kernel/agent-frontmatter.schema.ts (VERIFIED)
.refine((_data) => true, { message: "Field 'tools' is deprecated — prefer permission rules", path: ["tools"] })
.refine((_data) => true, { message: "Field 'maxSteps' is deprecated — use 'steps' instead", path: ["maxSteps"] })
```

### Pattern 3: Discriminated Union for MCP Servers
**What:** Use `z.discriminatedUnion()` for the MCP server type (local vs remote).
**When to use:** MCP server configuration where `type` field determines the shape.
**Example:**
```typescript
// Source: src/schema-kernel/mcp-server.schema.ts (VERIFIED)
export const MCPServerConfigSchema = z.discriminatedUnion("type", [
  LocalMCPServerSchema,    // type: "local", command: string[]
  RemoteMCPServerSchema,   // type: "remote", url: string
])
```

### Pattern 4: Plugin Tool Contract
**What:** All tools return string responses through the standard tool-response envelope.
**When to use:** `configure-primitive.ts` tool (Cycle 3).
**Reference:** `src/shared/tool-response.ts`

### Anti-Patterns to Avoid
- **Giant unified schema:** Don't create one file for all primitives. Separate files enable partial validation and independent evolution.
- **Non-strict schemas:** Don't use `.passthrough()` — OpenCode rejects unknown fields in frontmatter.
- **Tool doing validation logic:** Tools route to libraries. Validation belongs in `src/lib/`, not `src/tools/`.
- **Direct filesystem writes in schemas:** Schema files are pure contracts. I/O belongs in the tool or compiler layer.
- **Bypassing the barrel file:** All consumers import from `schema-kernel/index.js`, not individual files.

### Permission System

**Three actions:** `"allow"` | `"ask"` | `"deny"`

**14 permission keys:** `read`, `edit`, `glob`, `grep`, `bash`, `task`, `skill`, `lsp`, `question`, `webfetch`, `websearch`, `codesearch`, `external_directory`, `doom_loop`

**Two formats (union):**
1. **Pattern-based:** `{ "bash": { "*": "ask", "git *": "allow" } }` — wildcard first, last match wins
2. **Rules-based:** `{ "rules": [{ "permission": "write", "action": "allow", "pattern": "src/**" }] }` — ordered list

**Agent-level overrides:** Agent `permission` field overrides global permissions.

## Cross-Primitive Relationships

### Binding Map

| Relationship | How Connected | Conflict Type |
|--------------|---------------|---------------|
| Agent↔Command | Command `agent:` field references agent name | BLOCK if agent doesn't exist |
| Agent↔Skill | Agent loads skills via `skill()` tool at runtime | WARN if referenced skill doesn't exist |
| Agent↔Agent | Multiple agents with `mode: "primary"` may overlap roles | WARN if descriptions suggest similar roles |
| Permission↔Agent | Agent `permission` field overrides global | WARN if agent denies a tool that its commands need |
| Command↔Skill | Command body may load skills | WARN if command body references skills that don't exist |
| MCP↔Tool | Custom tools may depend on MCP servers | WARN if MCP server not configured |
| Rules↔Config | `instructions` array in config references rule files | BLOCK if referenced file doesn't exist |
| Agent↔Model | Agent `model` field references provider/model-id | INFO if model not in provider config |

### Conflict Detection Categories (for C2)

1. **Missing binding:** Command references non-existent agent
2. **Permission deadlock:** Agent denies a tool that its own commands use
3. **Role overlap:** Two primary agents with overlapping descriptions
4. **Missing dependency:** Agent references skills that don't exist in any discovery location
5. **MCP gap:** Tool references MCP server not in config
6. **Rule gap:** Config `instructions[]` references non-existent files

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| YAML parsing | Custom YAML parser | `yaml` npm package | Handles edge cases, nested types, comments |
| Frontmatter extraction | Regex for `---` blocks | `gray-matter` npm package | Handles nested YAML, empty bodies, CRLF |
| Schema validation | Manual field checks | Zod schemas from `schema-kernel/` | Already built and tested (C1 complete) |
| Permission matching | Custom glob matcher | `PatternEntrySchema` + last-match-wins logic | OpenCode uses ordered matching; replicate exactly |
| Config merging | Custom merge logic | Config precedence levels from `config-precedence.schema.ts` | 8-level hierarchy is well-defined; don't invent new rules |

**Key insight:** OpenCode's primitive formats are well-documented and stable. Don't invent new formats or validation rules — match what the platform actually does.

## Common Pitfalls

### Pitfall 1: Strict Mode Rejecting Valid OpenCode Configs
**What goes wrong:** OpenCode adds new frontmatter fields in updates, but our `.strict()` schemas reject them.
**Why it happens:** Zod `.strict()` rejects unknown keys.
**How to avoid:** Version-pin schema knowledge. Add `.refine()` for informational deprecation rather than blocking unknown fields in future updates. Track OpenCode release notes.
**Warning signs:** Schema validation suddenly fails after OpenCode update.

### Pitfall 2: Permission Pattern Matching Mismatch
**What goes wrong:** Our pattern matching differs from OpenCode's actual evaluation.
**Why it happens:** Last-match-wins with glob patterns has subtle edge cases (prefix wildcards, exact matches).
**How to avoid:** Test against known OpenCode permission behaviors. Verify with `PermissionRuleSchema` that requires `pattern` field.
**Warning signs:** Agent can't use a tool that should be allowed.

### Pitfall 3: Config Compiler Producing Invalid Frontmatter
**What goes wrong:** Generated `.md` files have incorrect YAML frontmatter that OpenCode rejects.
**Why it happens:** YAML serialization edge cases (multiline strings, special characters, quotes).
**How to avoid:** Round-trip testing: compile → write → decompile → compare. Use `gray-matter` for both extraction and stringification.
**Warning signs:** OpenCode crashes or ignores generated agent files.

### Pitfall 4: Cross-Primitive False Positives
**What goes wrong:** Validator reports conflicts that aren't real (e.g., warns about overlapping agents that are intentionally complementary).
**Why it happens:** Description-based overlap detection is heuristic, not deterministic.
**How to avoid:** Use severity levels (BLOCK/WARN/INFO). Don't block on heuristic matches.
**Warning signs:** Users ignore validator warnings because they're always false.

### Pitfall 5: Breaking Existing Orchestrator Routing
**What goes wrong:** Adding intent classification to `hivefiver-orchestrator` breaks existing `hf-create`/`hf-audit`/`hf-stack` routing.
**Why it happens:** Routing table modification affects all commands.
**How to avoid:** Test all 6 existing commands after any orchestrator modification.
**Warning signs:** `hf-create` no longer routes to specialist agents.

### Pitfall 6: Module Size Exceeding 500 LOC
**What goes wrong:** `config-compiler.ts` or `cross-primitive-validator.ts` exceeds the 500 LOC limit from Phase 16.4.
**Why it happens:** Compiler and validator have many primitive types to handle.
**How to avoid:** Split compiler into `agent-compiler.ts` + `command-compiler.ts` + `skill-compiler.ts` if needed. Split validator into conflict categories.
**Warning signs:** File approaching 400 LOC during implementation.
