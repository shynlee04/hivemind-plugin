## Blocker Issues

### BK-01: Fire-and-forget `fs.writeFile` — tool reports success before write completes

**File:** `src/tools/configure-primitive.ts:169`
**Issue:** The single-primitive compile path uses `fs.writeFile()` (from `promises as fs`) without `await`. The function returns `renderToolResult(success(...))` immediately, before the write completes. If the write fails (permissions, disk full, path issues), the tool STILL reports success and the user believes the file was written. The Promise rejection is silently swallowed.

```typescript
// Line 168-171 — the write is not awaited
mkdirSync(path.dirname(filePath), { recursive: true })
fs.writeFile(filePath, result.content, "utf-8")  // ← NOT awaited

return renderToolResult(success("Primitive configured", { filePath, primitive: args.primitive }))
```

**What breaks in practice:** If the user compiles an agent and the write fails (e.g., read-only filesystem, permission error), the tool says "Primitive configured" but no file exists. User proceeds thinking everything is fine. On next restart, the primitive is missing.

**Fix:**
```typescript
// Make the function async and await the write
async function handleCompile(...): Promise<string> {
  // ... existing code ...
  mkdirSync(path.dirname(filePath), { recursive: true })
  try {
    await fs.writeFile(filePath, result.content, "utf-8")
  } catch (e) {
    return renderToolResult(error(`Failed to write file: ${e instanceof Error ? e.message : String(e)}`, { filePath }))
  }
  return renderToolResult(success("Primitive configured", { filePath, primitive: args.primitive }))
}
```

Note: `mixedBatchCompile` in `config-compiler.ts` correctly uses `writeFileSync` with try/catch — only the single-primitive path has this bug.

## Major Issues

### MA-01: Test fixtures use wrong property names for `SkillFile` type

**File:** `tests/lib/cross-primitive-validator.test.ts:57-61`
**Issue:** `makeSkillFile()` creates objects with `filePath` and `dirName` properties, but the actual `SkillFile` schema (from `skill-metadata.schema.ts`) expects `skillPath` and `directoryName`. The type assertion `as SkillFile["frontmatter"]` masks this mismatch.

```typescript
// Test creates:
return {
  frontmatter: { ... } as SkillFile["frontmatter"],
  body: body || `### ${name}`,
  filePath: `.opencode/skills/${dirName}/SKILL.md`,  // ← WRONG: should be skillPath
  dirName,                                             // ← WRONG: should be directoryName
}
```

**What breaks in practice:** Any code that accesses `skillPath` or `directoryName` on these test objects gets `undefined`. Currently works because the validator doesn't access those fields, but future code changes will silently break.

**Fix:**
```typescript
return {
  frontmatter: { name, description: `Skill ${name}`, ...frontmatter } as SkillFile["frontmatter"],
  body: body || `### ${name}`,
  skillPath: `.opencode/skills/${dirName}/SKILL.md`,
  directoryName: dirName,
}
```

### MA-02: Test fixtures use wrong property names for `ToolFile` type

**File:** `tests/lib/cross-primitive-validator.test.ts:63-80`
**Issue:** `makeToolFile()` creates objects with `content` and `filePath` properties, but the actual `ToolFile` schema expects only `definition` and `sourcePath`. Extra properties (`content`, `filePath`) are not in the schema.

```typescript
return {
  definition: { ... },
  content: "",                                    // ← NOT in ToolFile schema
  filePath: `.opencode/tools/${name}.ts`,         // ← WRONG: should be sourcePath
}
```

**What breaks in practice:** Same as MA-01 — type assertions mask the mismatch. Any code accessing `sourcePath` gets `undefined`.

**Fix:**
```typescript
return {
  definition: {
    name,
    description: `Tool ${name}`,
    args: {},
    hasExecute: true,
    filePath: `.opencode/tools/${name}.ts`,
    exports: [],
    ...definition,
  },
  sourcePath: `.opencode/tools/${name}.ts`,
}
```

### MA-03: Duplicate `PrimitiveMap` type across two modules

**File:** `src/lib/cross-primitive-validator.ts:43-50` and `src/lib/primitive-loader.ts:44-51`
**Issue:** Both files export an identical `PrimitiveMap` type. If one changes, the other won't, creating a type drift risk. The `primitive-loader.ts` one is used only in its own file and the `cross-primitive-validator.ts` one is the authoritative type used by `runtime-validator.ts` and tools.

**Fix:** Export `PrimitiveMap` from a single location (e.g., `cross-primitive-validator.ts` or a shared `types.ts`) and import it in `primitive-loader.ts`.

### MA-04: `runtime-validator.ts` uses `as any` to access config permissions

**File:** `src/lib/runtime-validator.ts:200`
**Issue:** `(primitives.config as any).permission || {}` bypasses type safety. The `OpenCodeConfig` type may or may not have a `permission` field. Using `as any` means this code won't catch type errors if the config schema changes.

```typescript
const globalPerms = (primitives.config as any).permission || {}
```

**Fix:**
```typescript
const config = primitives.config as Record<string, unknown>
const globalPerms = (config.permission && typeof config.permission === "object")
  ? config.permission as Record<string, unknown>
  : {}
```

### MA-05: `validate-restart.ts` is useless for projects without `opencode.json`

**File:** `src/tools/validate-restart.ts:50-52`
**Issue:** If a project only uses `.opencode/` directory structure without an `opencode.json` file, the tool immediately returns an error: "No opencode.json found — cannot validate restart". Many OpenCode projects use only the directory structure without a config JSON. The tool should still validate primitives found on disk even without `opencode.json`.

**What breaks in practice:** Projects that don't use `opencode.json` get no restart validation at all — the tool just errors out.

**Fix:** Make config optional. Load primitives from `.opencode/` directories even without `opencode.json`. Use an empty config object as fallback for the `PrimitiveMap`.

## Minor Issues

### MI-01: `configure-primitive.ts` uses `as any` for batch frontmatter

**File:** `src/tools/configure-primitive.ts:194`
**Issue:** `frontmatter: frontmatter as any` in batch compilation bypasses type checking.

**Fix:** Use proper type narrowing or a type-safe frontmatter construction pattern.

### MI-02: Empty catch block in `config-compiler.ts` rollback

**File:** `src/lib/config-compiler.ts:358`
**Issue:** `catch { /* ignore rollback errors */ }` silently swallows rollback failures. If rollback fails after a partial write, there's no log or metric.

**Fix:** At minimum, add a console.warn or counter for rollback failures:
```typescript
catch (rollbackErr) {
  // Rollback failed — partial files may remain
}
```

### MI-03: Duplicate role-overlap detection logic

**File:** `src/lib/cross-primitive-validator.ts:132-165` and `src/lib/runtime-validator.ts:233-272`
**Issue:** `detectRoleOverlaps` and `validatePipelinePosition` implement nearly identical keyword-overlap logic with the same 50% threshold. If the threshold or algorithm needs changing, both must be updated in sync.

**Fix:** Extract a shared `computeKeywordOverlap(agentA, agentB)` utility and use it in both places.

### MI-04: `PrimitiveMap.config` is required but `LoadResult.config` is nullable

**File:** `src/lib/primitive-loader.ts:41` vs `src/lib/cross-primitive-validator.ts:49`
**Issue:** `LoadResult.config` is `OpenCodeConfig | null` but `PrimitiveMap.config` is `OpenCodeConfig` (non-null). The `validate-restart.ts` tool handles this with an early return, but the type mismatch means every consumer must remember to null-check. A safer pattern would be `PrimitiveMap.config: OpenCodeConfig | null` with explicit handling.

### MI-05: `SkillFile` schema refine silently passes on mismatch

**File:** `src/schema-kernel/skill-metadata.schema.ts:65-71`
**Issue:** The `.refine()` that checks `data.frontmatter.name === data.directoryName` will reject valid skill files where the directory was renamed but the frontmatter wasn't updated. This is a correctness check that may cause false rejections during refactoring.
