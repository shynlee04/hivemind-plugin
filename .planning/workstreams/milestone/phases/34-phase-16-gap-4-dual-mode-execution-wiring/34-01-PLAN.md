# Plan 34-01: Dual-Mode Execution Wiring — Verification & Closure

## Context

Plan 16-06 (Dual-Mode Execution Wiring) claimed narrowed delivery of only D-07 (single lifecycle owner) and D-11 (runtime-truthful test strategy), leaving 6 requirements formally undelivered:
- D-04A: Dual-Mode Execution Architecture
- D-05: Lazy PTY Loading with Graceful Fallback
- D-12: Concurrency Authority Centralization
- D-13: Thin Spawner Adapters
- D-16: Expand Delegation Recovery Filter
- D-17: Real Message-Count Stability Polling

**Key Finding:** All 6 requirements are already implemented in the current codebase. Subsequent work (Phase 33 gap closures, ongoing development) appears to have closed these gaps without formal tracking.

This plan **verifies closure** and **adds targeted runtime-truthful tests** for any remaining coverage gaps.

## Gap Analysis: Requirement vs. Current Implementation

### D-04A: Dual-Mode Execution Architecture — **CLOSED**

**Requirement:** PTY for command/process surfaces; SDK child-session for agent delegations. Not PTY-for-all.

**Evidence of closure:**
- `delegation-manager.ts:137` — SDK dispatch sets `executionMode: "sdk"`
- `command-delegation.ts:99` — PTY dispatch sets `executionMode: "pty"`
- `command-delegation.ts:181` — Headless fallback sets `executionMode: "headless"`
- `delegation-manager.ts:29-34` — `deriveDelegationSurface()` and `deriveRecoveryGuarantee()` map executionMode to surface/guarantee
- `plugin.ts` — Shared PtyManager wired to both DelegationManager and run-background-command tool
- Verification Truth #10 (PASS by-design): dual-path IS the architecture

**Status:** VERIFIED CLOSED. No code changes needed.

---

### D-05: Lazy PTY Loading with Graceful Fallback — **CLOSED**

**Requirement:** Load `bun-pty` only when PTY features needed. Silent fallback if unavailable.

**Evidence of closure:**
- `pty-manager.ts` — `spawn()` uses dynamic `import()` for bun-pty
- `command-delegation.ts` — PTY attempt wrapped in try/catch, falls back to `spawnHeadlessProcess()`
- `delegation-manager.ts:265-266` — Recovery derives guarantee from executionMode (pty = best-effort, headless = non-resumable)
- `pty-buffer.ts` — Pure TypeScript, no native deps

**Status:** VERIFIED CLOSED. No code changes needed.

---

### D-12: Concurrency Authority Centralization — **CLOSED**

**Requirement:** `concurrency.ts` owns queue-key computation. No other module computes queue keys.

**Evidence of closure:**
- `delegation-manager.ts:3` — imports `buildDelegationQueueKey` and `DelegationConcurrencyQueue` from `concurrency.ts`
- `delegation-manager.ts:10` — imports `resolveDelegationConcurrencyKey` from `spawner/concurrency-key.ts`
- `delegation-manager.ts:177-178` — `buildDelegationQueueKey(queueContext)` then `semaphore.acquire(queueKey)`
- `command-delegation.ts` — receives `queueKey` as parameter (doesn't compute it)
- No other module independently computes queue keys

**Status:** VERIFIED CLOSED. No code changes needed.

---

### D-13: Thin Spawner Adapters — **CLOSED**

**Requirement:** Spawner adapters are thin wrappers, not independent implementations.

**Evidence of closure:**
- `command-delegation.ts` — CommandDelegationHandler is injected via callback to DelegationManager
- `sdk-delegation.ts` — SdkDelegationHandler is injected via callback
- `delegation-manager.ts:177-180` — `dispatchCommand()` delegates to `commandHandler.dispatchCommand(params, queueKey, nestingDepth)`
- Handlers don't duplicate logic; they delegate to canonical functions (PtyManager.spawn(), concurrency acquire)

**Status:** VERIFIED CLOSED. No code changes needed.

---

### D-16: Expand Delegation Recovery Filter — **CLOSED**

**Requirement:** Recovery must find delegations in `dispatched` state, not just `running`.

**Evidence of closure:**
- `delegation-manager.ts:216` — `recoverPending()` checks `status !== "running" && status !== "dispatched"` (skips non-active)
- `delegation-manager.ts:249` — `buildResult()` same filter: `status !== "running" && status !== "dispatched"`
- `delegation-manager.ts:330` — `cancelDelegation()` same filter
- `delegation-manager.ts:350` — `getDelegationStatus()` same filter
- Recovery explicitly covers BOTH dispatched and running states

**Status:** VERIFIED CLOSED. No code changes needed.

---

### D-17: Real Message-Count Stability Polling — **CLOSED**

**Requirement:** Replace blind counter increment with `getSessionMessageCount()` for actual stability detection.

**Evidence of closure:**
- `sdk-delegation.ts` — Uses stability polling with real message count checks
- `delegation-manager.ts` — SDK path delegates to `sdkHandler` which implements stability detection
- No blind counter increments found in current code

**Status:** VERIFIED CLOSED. No code changes needed.
