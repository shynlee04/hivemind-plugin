### CR-03: gracePeriodExpiresAt never populated on delegation object

**File:** `src/lib/delegation-manager.ts:323-334`
**Issue:** The `Delegation` interface defines `gracePeriodExpiresAt?: number` (types.ts:370) and `delegation-status.ts:55` exposes it in the status tool response. However, `scheduleGracePeriodCleanup()` never sets this field on the delegation object:

```typescript
private scheduleGracePeriodCleanup(delegationId: string): void {
  const existingTimer = this.gracePeriodTimers.get(delegationId)
  if (existingTimer) {
    clearTimeout(existingTimer)
  }
  const timer = setTimeout(() => {
    this.gracePeriodTimers.delete(delegationId)
    this.delegations.delete(delegationId)
  }, TASK_CLEANUP_DELAY_MS)
  this.gracePeriodTimers.set(delegationId, timer)
}
```

The field `gracePeriodExpiresAt` is never assigned, so it will always be `undefined` at runtime. Plan Wave 3 Task 3 says "During grace period: `getStatus()` returns delegation normally" and Wave 6 Task 4 says "Include `gracePeriodExpiresAt` when applicable" — but the field is never set, so consumers cannot know when the grace period expires or whether one is active.

**Fix:** In `scheduleGracePeriodCleanup()`, set `delegation.gracePeriodExpiresAt` before scheduling the timer:
```typescript
private scheduleGracePeriodCleanup(delegationId: string): void {
  const delegation = this.delegations.get(delegationId)
  if (!delegation) return

  const existingTimer = this.gracePeriodTimers.get(delegationId)
  if (existingTimer) clearTimeout(existingTimer)

  delegation.gracePeriodExpiresAt = Date.now() + TASK_CLEANUP_DELAY_MS
  this.persistAllDelegations()

  const timer = setTimeout(() => {
    this.gracePeriodTimers.delete(delegationId)
    this.delegations.delete(delegationId)
  }, TASK_CLEANUP_DELAY_MS)
  this.gracePeriodTimers.set(delegationId, timer)
}
```

### WR-01: normalizePersistedDelegation omits gracePeriodExpiresAt and lastMessageCountChangeAt

**File:** `src/lib/delegation-persistence.ts:48-73`
**Issue:** The `normalizePersistedDelegation()` function reconstructs a `Delegation` from persisted JSON but does not include `gracePeriodExpiresAt`. While this field is currently always undefined (see CR-03), once CR-03 is fixed, persisted delegations with grace periods will lose the field on restart. Additionally, `lastMessageCountChangeAt` is only included via a fallback chain (`createdAt` or `Date.now()`) but should explicitly read from the persisted data when available (which it does at line 61-66 — this is actually correct). The `gracePeriodExpiresAt` omission is the real concern.

**Fix:** Add `gracePeriodExpiresAt` to the normalized return:
```typescript
gracePeriodExpiresAt:
  typeof record.gracePeriodExpiresAt === "number"
    ? record.gracePeriodExpiresAt
    : undefined,
```

### WR-02: Notification test does not actually test notification failure

**File:** `tests/lib/delegation-manager.test.ts:1563-1585`
**Issue:** The test "notifyParent failure does not crash finalization" at line 1563 has a comment saying "DelegationManager doesn't have a notifyParent in current implementation" — but it DOES now (Phase 16.2 added `notifyDelegationTerminal`). ### WR-03: dispatch() resolves nesting depth twice — potential TOCTOU

**File:** `src/lib/delegation-manager.ts:92,110`
**Issue:** `checkNestingDepth()` is called at line 92 (before semaphore acquire), and then `resolveNestingDepth()` is called again at line 110 (after semaphore acquire, when creating the delegation). If another delegation is registered between these two calls that changes the parent session's chain, the two depth values could differ. In practice, this is very unlikely because the semaphore serializes dispatches for the same queue key, but the parent session's delegation chain is keyed by `parentSessionId`, not `queueKey`, so concurrent dispatches from different queue keys could interleave.

**Fix:** Either (a) resolve depth once and pass it through, or (b) note in a comment that the double-resolution is intentional for defense-in-depth. The simplest fix is to cache the depth from the check:
```typescript
const nestingDepth = this.resolveNestingDepth(params.parentSessionId)
if (nestingDepth > MAX_DELEGATION_DEPTH) { throw ... }
// ... later use nestingDepth directly instead of calling resolveNestingDepth again
```

### WR-04: Polling interval calculation unreachable branch for active state

**File:** `src/lib/sdk-delegation.ts:53-61`
**Issue:** `calculateAdaptiveInterval()` returns intervals from `POLL_INTERVAL_BASE_MS` (5000ms) up to `POLL_INTERVAL_DEEP_IDLE_MS` (30000ms), but never returns `POLL_INTERVAL_ACTIVE_MS` (2000ms). The active interval is only used when `justChanged=true` is passed to `scheduleStabilityPoll()`. The function's contract is unclear — it calculates "adaptive interval based on idle time" but the most active state (2000ms) is handled externally via a boolean flag. This is functionally correct but the function's name and JSDoc are misleading.

**Fix:** Update the JSDoc to clarify:
```typescript
/**
 * Compute adaptive poll interval based on time since last message count change.
 * Note: POLL_INTERVAL_ACTIVE_MS (2000ms) is only used via the `justChanged` 
 * parameter in scheduleStabilityPoll(), not returned by this method.
 */
```
