# Phase 54 Context: Sidecar & Product-Detox Integration Runway

**Date:** 2026-04-28  
**Workstream:** milestone  
**Phase:** 54 — Sidecar & Product-Detox Integration Runway  
**Status:** research-locked / pre-planning  
**Reader:** researcher, planner, or architecture reviewer preparing a future sidecar/read-model runway without violating Phase 16.4 guardrails.  
**Post-read action:** derive a bounded runway phase for read-only sidecar contracts, product-detox concept migration, and user-configurable primitive UX boundaries.

## Intent

Phase 54 is a runway phase, not an implementation phase. Its purpose is to make later sidecar and product-detox integration work plan-able without re-opening the architecture confusion that Phase 16.4 explicitly closed. The repo already knows the sidecar should be read-only, that product-detox concepts must migrate through gates rather than wholesale code copy, and that user-configurable primitives live in `.opencode/` while canonical runtime state lives in `.hivemind/`.

What is still missing is a research-locked frame that combines those truths into one future-facing integration runway: how a Next.js + `@json-render/react` sidecar should consume read models, how product-detox concepts can enter safely, and how primitive/permission/hook-injection UX can remain user-configurable without broadening runtime authority.

## Scope

- Define the read-only sidecar contract implied by Phase 42, Phase 16.4, and Phase 51 grounding.
- Bound the Next.js / `@json-render/react` integration runway as a read/query/render/request surface rather than a state-writing authority.
- Define product-detox migration as concept/contract extraction only, gated through the Phase 16.4 migration control plane.
- Frame user-configurable primitive, permission, and hook-injection UX concerns around existing `opencode.json`, primitive-loader, configure-primitive, runtime-validator, and tool/hook surfaces.

## Non-Goals

- Do not implement `apps/sidecar/` or `apps/side-car/`; both are currently absent in this worktree.
- Do not copy product-detox source code into the harness.
- Do not change canonical runtime state roots or permit sidecar writes to `.hivemind/` or `.opencode/`.
- Do not conflate sidecar runway work with release proof or lifecycle closure.

## Assumptions

- **The sidecar should remain read-only against canonical runtime state.** If this is wrong, Phase 54 would need to revisit locked architecture decisions rather than build a runway.
- **Product-detox remains concept input, not code input.** If this is wrong, future work risks copying stale code and reintroducing Phase 16.4 failure modes.
- **Primitive/permission UX should wrap current runtime surfaces instead of creating a second configuration authority.** If this is wrong, the runway could over-design UI abstractions disconnected from real tool/plugin behavior.
- **Next.js + `@json-render/react` is a contract question before it is a build question.** If this is wrong, the phase may over-document UI architecture without enough concrete data-model grounding.

## Success Criteria

1. A future planner can describe the sidecar as a read/query/render/request surface with explicit no-write boundaries.
2. Product-detox migration candidates are framed as gated concepts/contracts and not as copy-forward implementation tasks.
3. The primitive/permission/hook-injection UX runway is anchored to the real `opencode.json`, configure-primitive, primitive-loader, runtime-validator, and plugin hook surfaces.
4. The phase produces downstream research questions about read models, proposal flows, config precedence, and permission readability before any UI build begins.

## Downstream Research Questions

1. What exact read models must exist before a sidecar can safely render delegations, journals, active work, and settings proposals?
2. How should `@json-render/react` specs map to those read models without becoming a second source of truth?
3. Which product-detox concepts are runway-safe to translate into schemas/contracts first, and which remain blocked by ownership or risk?
4. How should a user-facing primitive editor expose permissions, hook behavior, and plugin registration without bypassing current validation/runtime boundaries?
5. What no-write guard tests are required before any sidecar or config-proposal surface is implemented?
