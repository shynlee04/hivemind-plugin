# Phase 52 Validation Frame: End-User Harness Workflow Acceptance

## Purpose

This is a pre-planning validation contract. It defines what must be true before detailed Phase 52 planning or execution begins.

## Preconditions Before Planning Starts

| Check | Must Be True | Evidence Source |
|---|---|---|
| Scope lock | Phase 52 is treated as live acceptance proof, not a general runtime refactor | ROADMAP Phase 52 + runtime handoff |
| Upstream truth | Phases 49-51 are treated as local-readiness inputs, not inherited acceptance passes | 49/50/51 verification artifacts + 52 handoff |
| Evidence strategy | Planner commits to L1/L2-first scenario design rather than L4-first regression design | this validation frame + evidence-gate rules |
| Runtime availability | A real OpenCode/harness runtime path is identified for at least one live parent/child workflow | planning prerequisite; currently not yet evidenced |
| Artifact capture | Planned scenarios include capture of continuity, journal, lineage, and user-facing outputs | 52 handoff + `.hivemind` architecture docs |

## Preconditions Before Execution Starts

1. Acceptance scenarios are bounded and mapped to specific workflow claims.
2. Each scenario identifies the required L1/L2/L3 evidence before execution begins.
3. The run environment defines where parent session IDs, child session IDs, delegation IDs, command session IDs, and exported artifacts will be recorded.
4. The phase owner agrees to classify environment blockers honestly instead of downgrading evidence standards.
5. If execution changes code, the follow-up automated verification commands are declared in advance.

## Invalid Starting Conditions

- Starting Phase 52 with only historical `ses_22ee` evidence and no new runtime path.
- Treating `validate-restart` success as enough to skip live workflow acceptance.
- Planning isolated tool checks without a composed user journey.
- Proceeding without a capture plan for `.hivemind/state`, journal, and lineage artifacts.
- Treating blocked provider/runtime access as an implicit pass.

## Go / No-Go Rule

**GO** only if the planner can name at least one real end-to-end workflow, one real runtime path, and the artifacts to capture for L1/L2 proof.

**NO-GO** if the work would begin as another local-regression phase with no live acceptance path.

## Handoff Requirement

The eventual plan must preserve the distinction between:

- user-facing acceptance proof,
- local automated regression proof,
- historical UAT evidence, and
- structural repository inspection.

If that distinction is not explicit in the plan, validation fails before execution begins.
