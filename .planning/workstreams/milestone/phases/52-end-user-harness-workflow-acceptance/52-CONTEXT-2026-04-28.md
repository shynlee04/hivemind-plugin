# Phase 52 Context: End-User Harness Workflow Acceptance

**Date:** 2026-04-28  
**Workstream:** milestone  
**Phase:** 52 — End-User Harness Workflow Acceptance  
**Status:** research-locked / pre-planning  
**Reader:** researcher, planner, verifier, or human reviewer preparing Phase 52 without treating Phases 49-51 as acceptance proof.  
**Post-read action:** derive bounded acceptance scenarios and evidence-first planning for real end-to-end harness workflows.

## Intent

Phase 52 exists to prove that real users can complete production-ready harness workflows through the actual OpenCode-facing surface, not just through isolated tool tests or local structural inspection. The immediate trigger is the Phase 52 runtime evidence handoff, which explicitly says Phases 49-51 provide implementation and local-readiness evidence but do not satisfy release or acceptance truth.

This phase is the first place where orchestrator-led delegation, status polling, background command execution, lineage export, primitive readiness, and interruption recovery must be exercised as a single user-facing workflow. The goal is not “all tools pass individually”; the goal is “a real workflow completes end-to-end with honest evidence and clear pass/partial/fail outcomes.”

## Scope

- Real end-to-end parent → subagent → tool → journal acceptance scenarios through the production harness surface.
- Live acceptance evidence for `delegate-task`, `delegation-status`, `run-background-command`, `session-journal-export`, `configure-primitive`, and `validate-restart` when used as one workflow.
- Recovery/restart behavior using persisted `.hivemind/` state after interruption.
- Honest evidence capture that separates L1/L2 runtime proof from L4 automated regression and L5 planning context.
- One stack/research-guidance scenario to prove that Phase 51 grounding is usable in a real user workflow rather than only discoverable by the loader.

## Non-Goals

- Do not treat focused vitest coverage, typecheck, build, or local file inspection as release acceptance proof.
- Do not redesign tool schemas or primitive loader semantics beyond what is required to run the acceptance scenarios.
- Do not close lifecycle/CQRS architecture debt wholesale; Phase 52 only gathers the user-facing proof needed to expose what still blocks release.
- Do not implement sidecar/UI or product-detox migration work; those remain downstream runway concerns.

## Assumptions

- **Acceptance must be proven through composed workflows, not isolated tools.** If this is wrong, the phase will over-invest in scenario composition instead of release evidence. Current handoff evidence strongly suggests the opposite.
- **A passing `validate-restart` invocation is necessary but insufficient.** The tool itself ran successfully, but the handoff says this is not proof of a real app restart or user workflow continuity.
- **A real OpenCode runtime or equivalently live harness session is available for capture.** If the environment cannot supply L1/L2 evidence, Phase 52 may only be able to classify blockers instead of closing acceptance.
- **Historical `ses_22ee` evidence is useful as regression input, but not as proof of current correctness.** If this is wrong, planners may accidentally inherit stale failures or stale passes.
- **Phase 51 guidance quality must be tested through a user journey, not only skill discovery.** If this is wrong, stack/research grounding could remain technically present but practically unusable.

## Success Criteria

1. A user completes a full orchestrator-led workflow that dispatches a specialist, observes terminal status, executes a background command lifecycle, and exports lineage/journal evidence for the same run.
2. The workflow can be interrupted and resumed from persisted `.hivemind/` state without false completion, lost delegation state, or misleading journal output.
3. The acceptance artifact explicitly distinguishes pass, partial, fail, and externally blocked behaviors for each scenario.
4. At least one scenario exercises Phase 51 stack/research grounding in a real user workflow and records whether the guidance is actionable.
5. The phase produces L1/L2 evidence where available and labels any remaining L4/L5-only claims as insufficient for release acceptance.

## Downstream Research Questions

1. What is the minimum scenario matrix that still proves the whole harness workflow rather than a collection of tool calls?
2. Which acceptance steps require a live provider-backed child session versus a local OpenCode runtime with no external provider dependency?
3. What artifacts should be captured from `.hivemind/state/`, `.hivemind/journal/`, `.hivemind/lineage/`, and event-tracker output for each scenario?
4. Which historical `ses_22ee` cases must be replayed verbatim, and which should be rewritten around the corrected tool contracts from Phase 49?
5. If the workflow partially succeeds, what evidence threshold is enough to proceed to Phase 53 versus requiring Phase 52 repair work first?
