## Summary

Phase 52 should be planned as an evidence-first live acceptance phase, not a feature-building or regression-only phase. The official roadmap goal is a narrow, observable, end-to-end user workflow through production orchestrator/subagent/tool/journal surfaces with a lifecycle transcript. [VERIFIED: `ROADMAP.md`:1086-1105] The runtime handoff states release/acceptance fails until fresh L1/L2 proof exists, even though focused tests, typecheck, and `validate-restart` passed in prior phases. [VERIFIED: `52-RUNTIME-EVIDENCE-HANDOFF-2026-04-28.md`:10-18,68-80]

The planner should design one canonical acceptance journey with variants, not six disconnected tool checks. The canonical path is: user prompt -> orchestrator delegates to specialist via `delegate-task` -> `delegation-status` polls through terminal state -> same parent starts and observes a `run-background-command` PTY lifecycle -> `session-journal-export` exports lineage for the same run -> `configure-primitive` dry-run/read/list/inspect and `validate-restart` prove primitive readiness without state-boundary corruption -> interruption/restart validates persisted `.hivemind/state` recovery. [VERIFIED: `52-VERIFICATION-2026-04-28.md`:34-44; `52-RUNTIME-EVIDENCE-HANDOFF-2026-04-28.md`:97-125]

**Primary recommendation:** Plan Phase 52 around a thin scenario matrix whose first wave captures L1/L2 runtime evidence before any repair work; only if a scenario fails should later waves perform minimal fixes plus fresh L3/L4 regression commands. [VERIFIED: evidence hierarchy refs; `52-VALIDATION-2026-04-28.md`:15-31]

## Architecture Patterns

### System Architecture Diagram

```text
User prompt in live OpenCode parent session
  -> Orchestrator/front-facing agent invokes delegate-task
    -> DelegationManager.dispatch()
      -> OpenCode SDK child session create + promptAsync
      -> .hivemind/state/delegations.json persisted
    -> delegation-status polls in-memory + persisted delegation state
      -> terminal status/result/error surfaced to parent
  -> Parent invokes run-background-command(action=run)
    -> DelegationManager.dispatchCommand()
      -> PTY manager starts process, owns ptySessionId through delegation record
    -> output/list/terminate validate caller-visible ownership
  -> Parent invokes session-journal-export(sessionId/pipelineKey)
    -> continuity + delegations read from .hivemind/state
    -> execution lineage rendered for same run
  -> Parent invokes configure-primitive dry-run/read/list/inspect + validate-restart
    -> .opencode primitives read/validated
    -> .hivemind internal state remains separate
  -> Interruption/restart
    -> DelegationManager.recoverPending() hydrates persisted records
    -> status/export prove no false completion or lost lineage
```

## Common Pitfalls

### Pitfall 1: L4 Regression Evidence Inflated to L1 Acceptance

**What goes wrong:** Focused Vitest, typecheck, build, and structural plugin registration get summarized as end-user acceptance. [VERIFIED: `52-VERIFICATION-2026-04-28.md`:45-52]  
**Why it happens:** Prior phases passed many local gates, but the live parent/child/provider runtime remained partial. [VERIFIED: `52-RUNTIME-EVIDENCE-HANDOFF-2026-04-28.md`:68-80]  
**How to avoid:** Label command outputs by evidence level and require at least one L1 live run plus matching L2 persisted record. [VERIFIED: `52-VERIFICATION-2026-04-28.md`:63-70]  
**Warning signs:** Verification says “tests pass” but lacks parent session ID, delegation ID, child session ID, or `.hivemind` snapshots.

### Pitfall 2: Historical `session-ses_22ee.md` Treated as Current Proof

**What goes wrong:** Historical user-visible failures/partial passes become stale acceptance evidence. [VERIFIED: `52-CONTEXT-2026-04-28.md`:39-47]  
**How to avoid:** Use `ses_22ee` only as regression input; replay current scenarios through corrected Phase 49 tool contracts. [VERIFIED: `52-VERIFICATION-2026-04-28.md`:47-50]

### Pitfall 3: Missing Recovery Proof

**What goes wrong:** A workflow reaches terminal status but restart/interruption recovery is not exercised. [VERIFIED: `52-VALIDATION-2026-04-28.md`:22-31]  
**Why it happens:** Recovery requires deliberate interruption and state inspection, not just happy-path completion.  
**How to avoid:** Include an interruption after delegation/command start, then resume and prove persisted records avoid false completion/lost context. [VERIFIED: `52-RUNTIME-EVIDENCE-HANDOFF-2026-04-28.md`:121-124]

### Pitfall 4: `.opencode` / `.hivemind` Boundary Corruption

**What goes wrong:** Acceptance writes runtime state into `.opencode` or primitive files into `.hivemind`, violating Q6. [VERIFIED: AGENTS.md injected instructions; `continuity.ts`:19-43]  
**How to avoid:** Use `configure-primitive` dry-run/read/list/inspect for acceptance unless a scoped repair explicitly requires write; snapshot both roots before/after. [VERIFIED: `src/tools/configure-primitive.ts`:197-216]

### Pitfall 5: PTY Availability Misread as Feature Failure

**What goes wrong:** If PTY is unavailable, output/input/list/terminate fail and are reported as product failure instead of environment-blocked. [VERIFIED: `src/tools/run-background-command.ts`:179-181]  
**How to avoid:** Preflight `bun-pty`/PTY availability and classify missing PTY as blocked/partial with fallback evidence.
