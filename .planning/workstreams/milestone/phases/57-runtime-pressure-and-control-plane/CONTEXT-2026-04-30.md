# Phase 57 Context: Runtime Pressure & Control Plane
**Date:** 2026-04-30

**Status:** research-locked / pre-planning
**Reader:** researcher, planner, verifier, or human reviewer preparing Phase 57 as a product-detox concept migration phase.
**Post-read action:** derive bounded implementation scope and evidence-first planning for migrating runtime pressure and control-plane gate concepts from product-detox into harness-experiment.

## Intent
Port the runtime pressure contract and control-plane concepts from product-detox. The pressure model gives the harness a 10-tier steady → advisory → gated → blocking scale for runtime health, capacity, and safety signals. The control-plane `detect()` gate turns those signals into explicit decisions before tools, commands, and delegated workflows proceed. This phase creates the decision substrate needed before Phase 39 auto-loop work and before Phase 58 agent work contracts rely on pressure-aware execution.

## Scope
- Runtime pressure contract — 10-tier pressure model from steady to advisory to gated to blocking
- Control-plane `detect()` gate — evaluates runtime pressure, tool authority, and workflow readiness before execution
- Tool catalog authority matrix — maps tools to read/write authority, state surface, and pressure gating behavior
- Pressure event types — structured status, reason, severity, recommended action, and blocking rationale
- Control-plane decision outputs — allow, advise, require approval, defer, or block
- Integration with Phase 55 doc intelligence (control-plane decisions can reference structured doc surfaces)
- Integration with Phase 56 trajectory/session v3 (pressure decisions attach to trajectory evidence and session lineage)

## Non-Goals
- Do not implement Phase 39 auto-loop behavior — this phase provides pressure gates that Phase 39 consumes
- Do not replace existing runtime policy loading — pressure is an additional decision layer, not the policy store itself
- Do not rewrite tool implementations — tool catalog authority records describe tool surfaces before enforcement work is planned
- Do not build a sidecar UI for pressure states — sidecar rendering remains separate from control-plane contracts

## Assumptions
- Runtime pressure augments existing runtime policy rather than replacing it
- The 10-tier model can be represented as a stable enum or discriminated union with explicit escalation bands
- Control-plane `detect()` must be pure or side-effect-light so it can run before write-side tool execution
- Tool catalog authority must align with existing CQRS boundaries and `.hivemind/` state ownership rules
- Phase 56 trajectory provides a place to attach pressure gate evidence for auditability

## Success Criteria
1. A planner can distinguish runtime pressure (health/capacity/safety signal) from runtime policy (configured limits and defaults)
2. The 10 pressure tiers are named and grouped into steady, advisory, gated, and blocking bands
3. The control-plane `detect()` decision contract is defined with allow/advice/approval/defer/block outcomes
4. The tool catalog authority matrix maps each relevant tool to authority level and pressure behavior
5. Downstream dependencies to Phase 39 auto-loop and Phase 58 agent work contracts are documented

## Downstream Research Questions
1. What are the exact 10 tier names and thresholds in product-detox, and do any rely on project-specific runtime metrics?
2. Which existing harness tools require blocking pressure gates versus advisory-only pressure reporting?
3. How should pressure gate decisions be attached to trajectory events without creating circular dependencies?
4. Which runtime signals are available today, and which signals must be explicitly marked unavailable until later phases?
5. How should control-plane `detect()` compose with existing runtime policy and circuit breaker behavior?
