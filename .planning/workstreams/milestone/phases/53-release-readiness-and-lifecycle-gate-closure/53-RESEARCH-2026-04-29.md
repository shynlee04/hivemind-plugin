### Anti-Patterns to Avoid

- **Documentation-as-evidence:** L5 alone never passes any gate. [CITED: gate-evidence-truth/references/anti-patterns.md]
- **Tool invocation as behavior proof:** Successful `session-journal-export` invocation is not lineage proof if it returns zero sessions/delegations. [CITED: 52-04-JOURNAL-BOUNDARY-TRANSCRIPT-2026-04-29.md]
- **Validator/restart confusion:** `validate-restart` validates primitive discoverability; it is not proof of restart/resume recovery. [CITED: 52-ACCEPTANCE-SUMMARY-2026-04-29.md]
- **Unsafe recovery simulation:** Do not edit `.hivemind/state` to fake recovery. [CITED: 52-05-RECOVERY-TRANSCRIPT-2026-04-29.md]
- **Broad cleanup drift:** Phase 53 is not a generic architecture cleanup phase. [CITED: 53-CONTEXT-2026-04-28.md]

## Common Pitfalls

### Pitfall 1: Treating Phase 52 as Passed

**What goes wrong:** Planner assumes Phase 52 satisfied release input because one delegation retry completed. [CITED: 52-VERIFICATION-2026-04-29.md]  
**Why it happens:** The highest-confidence row (E52-01) masks partial/blocked rows. [CITED: 52-EVIDENCE-MATRIX-2026-04-29.md]  
**How to avoid:** Plan from a blocked-input ledger: E52-02/E52-03/E52-04 partial, E52-05/E52-06 blocked. [CITED: 52-EVIDENCE-MATRIX-2026-04-29.md]  
**Warning signs:** Any Phase 53 plan says “after Phase 52 PASS” or “release-ready acceptance.”

### Pitfall 2: Confusing Validator Output with Restart/Recovery Proof

**What goes wrong:** `validate-restart` becomes a substitute for real restart/resume behavior. [CITED: 52-ACCEPTANCE-SUMMARY-2026-04-29.md]  
**Why it happens:** It is live tool output and looks like L1. [CITED: 52-04-JOURNAL-BOUNDARY-TRANSCRIPT-2026-04-29.md]  
**How to avoid:** Classify it as primitive-readiness evidence only; require separate recovery proof. [CITED: 52-04-JOURNAL-BOUNDARY-TRANSCRIPT-2026-04-29.md]

### Pitfall 3: Running Recovery Without Operator Approval

**What goes wrong:** Agent chooses an interruption method that risks destructive state or invalid evidence. [CITED: 52-05-RECOVERY-TRANSCRIPT-2026-04-29.md]  
**Why it happens:** Release gate needs recovery proof, but the safe interruption method was not pre-authorized. [CITED: 52-VERIFICATION-2026-04-29.md]  
**How to avoid:** Phase 53 plan must include an approval checkpoint with named method, rollback, and allowed read/write boundaries before any recovery run. [CITED: hm-production-readiness/references/rollback-plan-template.md]

### Pitfall 4: Fixing Everything Instead of Release Blockers

**What goes wrong:** Phase 53 absorbs Phase 36/37/38/54 work and becomes a token-wasting architecture cleanup loop. [CITED: 53-CONTEXT-2026-04-28.md]  
**Why it happens:** Concerns doc contains many real debts. [CITED: CONCERNS.md]  
**How to avoid:** Only fix source if it blocks a named release evidence target; otherwise downgrade/defer with rationale. [CITED: 53-CONTEXT-2026-04-28.md]

## Release Gate Decision Model

| Decision | Required Conditions | Outcome |
|---|---|---|
| SHIP | All release-critical surfaces have required evidence: delegation L1+L2, PTY output L1 or waived non-critical, journal lineage L1+L2, recovery L1+L2 or explicit operator non-applicability, regression suite green. [CITED: gate-evidence-truth/references/evaluation-checklist.md] | Release readiness can pass. |
| CONDITIONAL SHIP WITH WAIVERS | A gap remains but is non-critical, explicitly waived by operator/release owner, and documented with risk/rollback. [CITED: hm-production-readiness/references/rollback-plan-template.md] | Release may proceed only with named waiver and rollback trigger. |
| NO-SHIP | Any release-critical L1/L2 blocker remains: recovery not approved/proven, same-run lineage remains empty, delegation cannot complete, or CQRS/root boundary violation is found. [CITED: gate-evidence-truth/references/evaluation-checklist.md] | Phase 53 closes as blocked with gap-closure plan. |
| REPLAN | Evidence loop repeats without new runtime facts or produces only L4/L5 artifacts. [CITED: gate-evidence-truth/references/anti-patterns.md] | Stop execution; auto-plan smallest gap-closure task. |

**Can Phase 53 execute while Phase 52 is BLOCKED/PARTIAL?** Yes, but only as a blocked-input release readiness audit that carries Phase 52 blockers forward and cannot produce a SHIP verdict until blockers are closed or explicitly waived.
