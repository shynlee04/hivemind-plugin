# Phase 59 Context: SDK Supervisor & Command Engine
**Date:** 2026-04-30
**Workstream:** milestone
**Phase:** 59 — SDK Supervisor & Command Engine
**Status:** research-locked / pre-planning
**Reader:** researcher, planner, verifier, or human reviewer preparing Phase 59 as a product-detox concept migration phase.
**Post-read action:** derive bounded implementation scope and evidence-first planning for migrating SDK supervisor and slash-command engine concepts from product-detox into harness-experiment.

## Intent
Port the SDK supervisor and slash-command engine concepts from product-detox. The SDK supervisor provides health, heartbeat, and diagnostics around OpenCode SDK interactions. The command engine discovers command bundles, routes slash-command requests, validates command contracts, renders context, and transforms messages for command execution. This phase creates the command-supervision substrate that Agent Synthesis AS-9 depends on.

## Scope
- SDK supervisor — health checks, heartbeat tracking, diagnostics, and SDK surface readiness status
- Slash-command engine — command bundle discovery, routing, and contract validation
- Command contracts — command metadata, arguments, expected context, output shape, and failure states
- Context renderer — prepares bounded context payloads for command execution
- Messages transform — transforms command inputs/outputs into the expected message format
- Integration with Phase 57 pressure/control-plane gates before command routing or SDK operations
- Integration with existing command/tool surfaces without replacing current primitive discovery

## Non-Goals
- Do not replace OpenCode primitive discovery — command bundle discovery supplements harness command routing
- Do not rewrite existing project commands under `.opencode/commands/`
- Do not implement Agent Synthesis AS-9 — this phase provides SDK/command supervision contracts AS-9 consumes
- Do not rebuild general prompt analysis — context renderer prepares command context, not prompt-quality scoring

## Evidence Sources
- product-detox research target: SDK supervisor source and health/heartbeat/diagnostics contracts

- product-detox concept source to inspect: command contract schema and context renderer
- product-detox concept source to inspect: messages transform concepts
- `.planning/workstreams/milestone/phases/57-runtime-pressure-and-control-plane/CONTEXT-2026-04-30.md` — pressure/control-plane dependency
- `src/lib/session-api.ts` — existing typed OpenCode SDK wrappers
- `src/tools/run-background-command.ts` — existing command-like runtime surface
- `src/tools/configure-primitive.ts` and `src/tools/validate-restart.ts` — existing primitive configuration and restart validation surfaces

## Assumptions
- SDK supervisor observes and reports SDK readiness; it does not replace the SDK wrappers directly
- Command bundle discovery must respect `.opencode/` as primitive configuration, not internal runtime state
- Command routing must be pressure-aware and may be blocked or degraded by Phase 57 control-plane decisions
- Messages transform must remain bounded and avoid reintroducing broad system-transform behavior

## Success Criteria
1. A planner can distinguish SDK supervision (health/heartbeat/diagnostics) from SDK wrapper calls
2. Command bundle discovery, routing, contracts, context rendering, and message transformation are separately defined
3. The command engine can be planned without modifying existing `.opencode/commands/` files in this phase
4. Pressure/control-plane dependency from Phase 57 is explicit for SDK and command execution decisions
5. Downstream dependency on Agent Synthesis AS-9 is documented

## Downstream Research Questions
1. What SDK health and heartbeat signals does product-detox supervise, and which are available in current OpenCode SDK surfaces?
2. How does command bundle discovery differ from existing OpenCode command primitive discovery?
3. What command contract fields are required before a slash command can be routed safely?
4. How should context renderer output be bounded to avoid runaway prompt/message expansion?
5. Which message transform behavior should be migrated, and which should remain excluded because of prior system-transform risks?
