## Warnings (Non-Blocking)

### WR-01: `as never` Casts in Test Code Pattern

**Artifacts:** Plan 02 (line 175: `createFakeClient() as never`), Plan 03 (lines 278-279, 302: `fakeManager as never`), Plan 04 (lines 217-218: `fakeClient as never`)

**Issue:** Plans use `as never` type casts when passing fake clients and managers to hook/tool factories. While this is standard vitest practice for partial mock interfaces, it reduces type safety and could allow tests to pass with incorrect shapes that the real SDK would reject.

**Recommendation:** Where possible, type fakes against the actual interfaces (e.g., `OpenCodeClient`, `DelegationManager`) using `Partial<>` or interface subsets. The stateful fake client pattern in Plan 04 (lines 118-159) is a good example of structured fakes.

**Severity:** WARN — does not block execution. Standard test code pattern, not a production concern.

### WR-02: Coverage Tests for Deprecated Module

**Artifact:** Plan 02, Task 4 — `tests/lib/notification-handler.test.ts`

**Issue:** The `src/lib/AGENTS.md` documentation lists `notification-handler.ts` as "DEPRECATED: Dead code. WaiterModel polling replaces push notifications. Retained for potential re-integration." at ~100 LOC. However, Plan 02 lists it at 290 LOC (from Phase 48.4 RESEARCH.md) and creates comprehensive behavioral tests. Testing deprecated code is still valuable (prevents regressions if re-integrated), but the plan does not explicitly acknowledge the deprecated status.

**Recommendation:** Add a note to Plan 02 acknowledging the deprecated status and confirming the 290 LOC figure is correct (vs. the 100 LOC listed in AGENTS.md). If the module is truly dead code, consider whether coverage investment is justified vs. other priorities.

**Severity:** WARN — does not block execution. Low risk: testing dead code wastes effort but causes no harm.

### WR-03: Dependency on Future Phase 48.5 Refactoring

**Artifact:** Plan 03, Task 4 — `tests/lib/event-tracker/writer.test.ts`

**Issue:** The plan notes that `createEventTrackerArtifactsFromHook` "may not exist yet in its final form" and that "Phase 48.5 will split this file." Plan correctly instructs executor to "test the current exported functions." This is self-aware and handled, but the test file may need updates after Phase 48.5 splits `writer.ts`.

**Recommendation:** Add a note to the SUMMARY.md output of Plan 03 documenting which functions were tested and noting that Phase 48.5 may require test updates.

**Severity:** WARN — does not block execution. Mitigated by plan's self-aware instructions.
