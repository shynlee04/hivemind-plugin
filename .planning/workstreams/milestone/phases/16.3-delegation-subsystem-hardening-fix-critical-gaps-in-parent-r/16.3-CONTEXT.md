# Phase 16.3: Delegation Subsystem Hardening — Fix critical gaps in parent resumption, notification delivery, and signal handling - Context

**Gathered:** 2026-04-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Harden the existing delegation subsystem so parent recovery, completion delivery, and terminal-state behavior are reliable across the already-existing background delegation surfaces. This phase is not a redesign of delegation architecture; it is a hardening pass across the real subsystem that makes background delegation work end-to-end: tool surfaces, plugin wiring, hooks, recovery/persistence, notification delivery, and terminal-state handling.

</domain>

<decisions>
## Implementation Decisions

### Tool surface boundaries
- **D-01:** Keep **3 distinct surfaces**: OpenCode built-in `task`, harness `delegate-task`, and PTY/background command execution (`run-background-command`) are separate tools with separate promises.
- **D-02:** `delegate-task` remains the semantic **agent delegation** surface; PTY/background command execution remains a separate **command/process** surface. Do not unify them under a single misleading tool contract.
- **D-03:** Phase 16.3 must treat background delegation as a **subsystem**, not just `delegation-manager` + `spawner`. Hooks, plugin wiring, notification delivery, status retrieval, and persistence/recovery are all in scope because they materially determine whether background delegation works in real use.

### Parent recovery contract
- **D-04:** Use **surface-specific honesty** for recovery guarantees.
- **D-05:** SDK agent delegations are the primary **resumable** surface after restart or parent interruption.
- **D-06:** PTY command sessions are **best-effort resumable** only.
- **D-07:** Headless command fallback is explicitly **non-resumable after restart** and must be reported truthfully rather than presented as equivalent to SDK or PTY recovery.

### Completion delivery
- **D-08:** Completion delivery is **notification-first**. Parent sessions should normally receive an automatic completion/resumption message when background work reaches a terminal state.
- **D-09:** Poll/status surfaces remain secondary control and inspection tools; they are not the primary completion UX.
- **D-10:** Parent completion delivery must include **human-readable summary + structured metadata** in the same contract so both chat UX and automation/tooling are supported.

### Terminal and signal semantics
- **D-11:** Terminal states remain **surface-aware**, not collapsed into an oversimplified shared model. Success, error, cancelled, interrupted, signal-killed, and unrecoverable fallback states must remain distinguishable where they reflect materially different outcomes.
- **D-12:** If a PTY/headless command dies because of a signal rather than a clean exit code, report it by default as **interrupted by signal**, unless the user explicitly requested cancellation.
- **D-13:** Phase 16.3 planning must preserve truthful terminal reporting across SDK aborts, PTY exits, headless exits, and signal-driven termination rather than hiding those differences behind generic failure wording.

### Nested delegation policy
- **D-14:** **No nested delegation by default** inside delegated child sessions. Keep child workers non-delegating unless a future phase explicitly opens bounded nesting as a supported feature.

### the agent's Discretion
- Exact schema and field names for the structured completion metadata.
- Exact human summary template and wording for completion/resumption notifications.
- Exact representation of resumable vs best-effort vs non-resumable states in tool outputs.
- Exact mapping of internal runtime states to user-facing terminal labels, so long as the locked distinctions above are preserved.

</decisions>

## Existing Code Insights

### Reusable Assets
- `DelegationManager` + `SdkDelegationHandler` + `CommandDelegationHandler`: the runtime already separates SDK agent delegation from command/PTTY execution.
- `PtyManager`: shared PTY substrate for spawn/read/write/terminate/list operations.
- `src/lib/spawner/*`: extracted session creation, queue-key resolution, and working-directory helpers already exist.
- `notification-handler.ts`: existing parent notification transport and message formatting seam.
- `delegation-status.ts`: existing retrieval/status surface for non-blocking delegation workflows.
- `run-background-command.ts`: existing PTY/headless command surface that should remain distinct from semantic agent delegation.
- `continuity.ts` + `delegation-persistence.ts`: existing durability/restart-recovery foundation.

### Established Patterns
- Background delegation is already implemented as **multiple coordinated surfaces**, not one tool.
- SDK child-session completion depends on **event-hook delivery** plus adaptive polling, not spawn alone.
- PTY command execution is already **separate and PTY-first**, with headless fallback.
- All terminal paths already converge through a **single terminal transition seam**.
- Child delegated sessions are already **write-capable but non-delegating** by policy.

### Integration Points
- `src/plugin.ts` is the top-level wiring point for tools, PTY manager, recovery, and event observers.
- `src/hooks/create-core-hooks.ts` is the hook seam that makes idle/deletion-driven completion possible.
- `src/lib/notification-handler.ts` and `src/lib/delegation-manager.ts` jointly define the parent completion-delivery path.
- `src/lib/sdk-delegation.ts` and `src/lib/command-delegation.ts` are the two execution-mode behavior seams.
- `src/lib/spawner/session-creator.ts` is the policy seam for parent linkage and child-session tool restrictions.

</code_context>

<specifics>
## Specific Ideas

- The user explicitly wants the phase to consider **real-life use cases** of PTY background task delegation, Phase 14 `delegate-task`, and the built-in `task` tool together — but also explicitly believes they **cannot be treated as the same tool**.
- The user explicitly asked for a deeper search across hooks, plugins, tools, and utilities that make advanced “background” delegation truly work, rather than only inspecting `background-agent` / `spawner`.
- External OMO references named by the user should inform planning, especially `background-agent`, `spawner`, `background-task`, and `background-notification`.

</specifics>

<deferred>
## Deferred Ideas

- Opening **bounded nested delegation** as a supported feature — future phase if the product later wants orchestrated task trees inside delegated workers.
- Any attempt to **unify built-in task, delegate-task, and PTY command execution into one tool** — explicitly rejected for this phase.
- Broader delegation architecture redesign beyond hardening the existing subsystem — out of scope here.

</deferred>
