### Real Work Phase: Session Journal + Execution Lineage Bridge

**Goal:** Build the first `.hivemind/`-aligned projection/audit layer that records session turns, execution lineage, delegation-relevant events, and resumable context without replacing existing runtime truth.

**Why this first:**

- It supports the future delegation manifest without raw system-reminder firehose.
- It gives front-facing agents a turn-aware view before taking action.
- It prepares sidecar/read-model visibility without making the sidecar runtime authority.
- It gives product-detox concepts a safe concept-migration target.
- It exercises the `.opencode/state` → `.hivemind` transition model without moving canonical writers prematurely.

## Minimum Real-Work Scope

The first implementation phase after 16.4 should produce a working, testable slice:

1. **Journal contract**
   - JSON + Markdown compatible output shape.
   - Session ID, turn ID, parent/child session relationship, actor, event type, timestamp, source, and summary fields.
   - Explicit marker for whether each entry is canonical runtime state, audit trail, or projection.

2. **Execution lineage projection**
   - Records plan/task/delegation relationships as rebuildable lineage, not competing status truth.
   - Can represent pipeline grouping later through optional `pipelineKey`.

3. **`.hivemind/` taxonomy starter**
   - Defines the initial state root categories needed for journal and lineage only.
   - Does not migrate all existing state writers yet.

4. **Export/read surface**
   - Supports JSON + Markdown output.
   - Provides quick-read summaries for agents and future sidecar views.

5. **Guarded integration**
   - Existing `.opencode/state/opencode-harness/` remains current implementation input.
   - No second runtime authority.
   - No product-detox code copied as-is.
