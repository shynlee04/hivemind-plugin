## Target Code-Tree Baseline Superseding Phase 11

Phase 16.4 supersedes stale Phase 11 assumptions per D-06. The target tree is not a broad source move; it is a classification baseline. Broad source-tree moves require explicit plan tasks and verification gates per D-09. Tiny validation-supporting artifacts are allowed per D-10.

- Hard Harness remains the npm package source under `src/`: plugin assembly, hooks, tools, lib runtime owners, shared utilities, schema contracts.
- Hiveminder becomes a built-in meta-concept/runtime-integration module when planned: code-first, schema-defined, SDK-backed, configuration-oriented.
- Hivefiver remains the end-user meta-builder module for OpenCode primitives, sharing some substrate but owning user-facing skill/agent/command authoring capabilities.
- Sidecar/read models are projections over canonical state and audit trails, not a new runtime authority.
- Future graph/vector/local memory layers remain optional until schemas, privacy, rebuild, and authority gates exist.

## Hiveminder / Hivefiver / Shared Boundary

Hiveminder is the built-in meta-concept/runtime-integration module for Hivemind Harness. Hivefiver is the end-user meta-builder module for OpenCode primitives. Shared capabilities may include schema helpers, inspection utilities, authoring patterns, and guard conventions, but ownership must remain explicit.

Neither is a soft harness. The exact locked phrase for future reuse is: neither is a soft harness. `.md files are transition-stage artifacts toward code-first, schema-defined, SDK-backed, configuration-oriented runtime integration`.

Hivefiver-exclusive responsibilities include end-user skill/agent/command/tool authoring workflows, meta-builder UX, validation of OpenCode primitive packages, and human-facing creation/audit flows. Hiveminder-exclusive responsibilities include built-in harness-aware runtime integration, collaboration projections, session/journal/lineage surfaces, and hard-harness configuration bridges.

## OpenCode Platform Conformance Rules

- plugins may expose multiple tools.
- custom tools are TS/JS definitions that may call other languages when wrapped safely.
- hooks may mutate documented hook outputs.
- event hooks must not be sole blocking transaction gates.
- permissions are first-class.
- SDK/server APIs own session orchestration surfaces.
- HiveMind-specific conventions must be labeled as conventions, not platform constraints, unless backed by current OpenCode evidence.

## Human / Agent Collaboration Model

The harness is collaborative agent-led workflow infrastructure, not only background task execution. Before action, a front-facing coordinator should inspect or receive projections for:

| Required View | Purpose | Producer | Consumer |
|---|---|---|---|
| active task stems | Show current user intent branches and active work roots. | future journal/task projection | coordinator, sidecar |
| delegated tasks | Show child-session work, status, and handoff evidence. | delegation records/manifest projection | coordinator, human user |
| workflow/task hierarchy | Preserve plan/phase/task dependencies and wave order. | GSD artifacts + future workflow graph | coordinator, verifier |
| blockers | Surface auth gates, human decisions, failed validations. | journal + status projection | human user, coordinator |
| deferred ideas | Keep out-of-scope ideas without contaminating active plans. | journal/notes projection | future planners |
| must inspect before action criteria | Require current state, active delegations, user intent shifts, and stale-claim register before implementation. | collaboration projection | all future agents |

## GUI / Hivefive Sidecar Compatibility Boundaries

The GUI/Hivefive sidecar may `read`, `query`, `render`, and `request`, but must not become runtime authority over canonical state per P16.4-13.

| Sidecar Surface | May Do | Must Not Do | Source of Truth | Request Path |
|---|---|---|---|---|
| read-only projections | read and render active work, delegation status, journals, and coverage | write canonical continuity/delegation records | current runtime state + journal/projections | none |
| config proposals | render proposed config changes and request approval | silently mutate OpenCode/Hivemind settings | config files + future config bridge | approved tool/command gate |
| OpenCode/Hivemind settings | query and explain effective settings | bypass permissions or project config precedence | OpenCode config + harness config bridge | human-approved settings flow |
| interactive planning/steering | request decisions, capture intent shifts, show alternatives | become autonomous lifecycle authority | GSD plan/journal/projection | coordinator-mediated request |
| delegation visibility | render delegated tasks and completion evidence | write terminal delegation status | DelegationManager records/projection | status/manifest query |
| non-breaking user guidance | render help, warnings, and suggested next actions | execute code changes or migrations directly | architecture baseline + guard results | explicit user/coordinator action |
