# Phase 16.4: Harness Architecture Baseline & Migration Control Plane — Specification

**Created:** 2026-04-25
**Ambiguity score:** 0.16 (gate: ≤ 0.20)
**Requirements:** 14 locked

## Goal

Establish an evidence-verified architecture baseline and migration control plane that invalidates stale/hallucinated prior decisions, revises stale Phase 11 architecture where needed, and defines the runtime/state/component boundaries required before further delegation expansion, product-detox concept migration, agent hierarchy hardening, runtime configuration, user settings, or GUI/sidecar implementation.

## Background

Phase 16.3 proved the harness can dispatch and track delegations, but post-UAT analysis found the architecture still lacks a reconciliation layer between delegation lifecycle and human conversation lifecycle. The immediate temptation is to continue with delegation manifest and notification mediation work, but the project has accumulated stale architecture decisions across `.planning/`, `docs/draft/`, product-detox migration drafts, and current `src/` reality.

Current code shows that architecture documents cannot be treated as runtime truth. Examples found during 16.4 scouting include: `notification-handler.ts` being described as dead while current terminal transitions still call it, Phase 02 summaries referencing files that do not exist, `delegation-manager.ts` exceeding documented module-size limits, and old CQRS rules claiming hooks are read-only while current OpenCode hook contracts and project hooks support bounded mutations.

The user explicitly clarified that Phase 16.4 may create tests, scripts, checklists, validation guards, and architecture-control artifacts; it may revise stale Phase 11 architecture; and product-detox should be migrated as concepts and patterns, not copied code. The phase therefore exists to prevent future agents from hallucinating from stale documents and to define the real architecture starters for a long-running, collaborative, agent-led local harness.

## Requirements

1. **Stale-decision validation register**: The phase must create a decision-validation register that classifies prior architecture claims as `valid`, `partial`, `stale`, `rejected`, or `needs-research`.
   - Current: Architecture claims are scattered across `.planning/`, `docs/draft/`, migration plans, and session artifacts; later agents can mistake stale claims for source of truth.
   - Target: A single validation artifact records claim, source path, code/platform evidence, verdict, replacement rule, and downstream impact.
   - Acceptance: A verifier can inspect the artifact and find at least the known stale clusters: global CQRS dogma, hook mutation prohibition, soft/hard harness flattening, stale Phase 11 assumptions, nonexistent file claims, notification-handler dead-code claim, and product-detox copy-forward assumptions.

2. **Soft CQRS architecture rule**: The phase must replace global “tools write, hooks read” dogma with a case-by-case mutation authority model.
   - Current: Some docs state tools are write-side and hooks are read-side, but OpenCode hooks support output mutation and current hooks perform bounded state/context updates.
   - Target: The architecture baseline defines when tools, hooks, plugins, libraries, SDK calls, scripts, and sidecars may read, query, mutate, append, or project state.
   - Acceptance: The baseline includes a mutation-authority matrix with owner, allowed callers, persistence target, idempotency rule, recovery behavior, and examples for tool writes, hook output mutation, hook side effects, library store writes, and script/report-only helpers.

3. **State ownership model**: The phase must define canonical, derived, projected, archived, and future/optional state categories.
   - Current: Current harness state uses `.opencode/state/opencode-harness/`; product-detox uses `.hivemind/**`; future concepts reference manifests, journals, trajectory, task graphs, agent-work-contracts, vector memory, graph memory, and sidecar projections.
   - Target: The architecture baseline specifies which state roots and files are canonical runtime truth, which are human/agent-readable projections, which are migration inputs, and which are forbidden until approved.
   - Acceptance: A verifier can determine the allowed state role for continuity records, delegation records, pending notifications, future delegation manifest, session journal, trajectory/lineage, workflow task graph, agent-work-contract, `.hivemind` categories, vector memory, graph memory, and GUI/sidecar read models.

4. **Runtime classification framework**: The phase must define runtime types by lifecycle phase, delegation boundary, actor/consumer, workflow type, task type, and persistence rule.
   - Current: Runtime classification is shallow and often collapses execution mode, session status, workflow state, and task type into the same concept.
   - Target: The baseline defines at minimum: session initiation runtime, mid-flow intervention runtime, post-completion runtime, delegated subtask runtime, and background command/runtime surfaces.
   - Acceptance: Each runtime type has context boundaries, triggering events/tools/commands, state owner, allowed mutation surfaces, completion/recovery behavior, and primary consumers: human user, front-facing coordinator, subagent, sidecar, or automated hook.

5. **Lifecycle ownership rule**: The phase must decide how lifecycle authority is assigned and how lifecycle transitions are validated.
   - Current: `DelegationManager`, `HarnessLifecycleManager`, hooks, `CompletionDetector`, `SdkDelegationHandler`, `CommandDelegationHandler`, and `notification-handler` all participate in lifecycle behavior; `lifecycle-manager.ts` is currently a minimal facade/stub.
   - Target: The baseline names lifecycle owners and allowable transition responsibilities for SDK delegations, PTY/headless commands, notifications, pending replay, turn-start hydration, and future manifests/journals.
   - Acceptance: The artifact includes a lifecycle ownership diagram or table and a pass/fail rule for identifying forbidden duplicate lifecycle authorities.

6. **Component map and code-tree baseline**: The phase must produce a current component map and a target code-tree architecture that can supersede stale Phase 11 assumptions.
   - Current: Phase 11’s target architecture is stale; current `src/` includes flat `plugin.ts`, `hooks/`, `tools/`, `lib/`, `lib/spawner/`, `lib/pty/`, `shared/`, and `schema-kernel/`, with overloaded modules and mismatched docs.
   - Target: The baseline maps current files to architecture domains and proposes a target organization for hard harness runtime, Hiveminder built-in meta layer, Hivefiver end-user meta-builder layer, sidecar projections, and future optional memory/graph/vector layers.
   - Acceptance: The target architecture explicitly marks which current files stay, move later, split later, retire, or require deeper module design; it includes rationale and does not perform broad runtime moves unless planned and verified.

7. **Hiveminder/Hivefiver boundary clarification**: The phase must formally distinguish Hiveminder from Hivefiver without labeling either as a soft harness.
   - Current: Prior documentation sometimes flattens hard harness, soft meta-concepts, Hivefiver, Hiveminder, and `.md` staging artifacts into one category.
   - Target: The baseline states that Hiveminder is the built-in meta-concept/runtime-integration module for Hivemind Harness, Hivefiver is the end-user meta-builder module for OpenCode primitives, shared capabilities are explicitly listed, and Hivefiver-exclusive responsibilities are explicitly listed.
   - Acceptance: The clarification includes the transition-state history: `.md` files are current drafting/live-test representations, while long-term form is code-first, schema-defined, SDK-backed, configuration-oriented runtime integration.

8. **OpenCode platform conformance**: The phase must validate architecture against current OpenCode SDK, server-client API, plugin, hook, custom tool, command, skill, agent, permission, and config behavior.
   - Current: Some architecture claims are based on assumptions rather than platform evidence; the project uses `@opencode-ai/plugin` and `@opencode-ai/sdk` near current 2026 versions.
   - Target: The baseline identifies absolute platform constraints versus HiveMind-specific conventions.
   - Acceptance: The artifact records evidence-backed rules including: plugins may expose multiple tools; custom tools are TS/JS definitions that may call other languages; hooks may mutate documented hook outputs; event hooks must not be sole blocking transaction gates; permissions are first-class; SDK/server APIs own session orchestration surfaces.

9. **Concept-migration control plane**: The phase must define gates for migrating product-detox concepts without copying product-detox code wholesale.
   - Current: Product-detox contains useful concepts (`event-tracker`, `session-journal`, `trajectory`, `task`, `agent-work-contract`, `runtime-entry`, `session-entry`, `workflow-management`, `.hivemind`, `apps/side-car`) but also bloat, overlap, stubs, and conflicting state roots.
   - Target: Migration starts from concept maps, target contracts, rejected files, ownership classification, and acceptance tests before code movement.
   - Acceptance: The migration gate requires every candidate concept to declare: source paths, purpose, target runtime category, state owner, conflict risk, concept-migration plan, code-copy prohibition exceptions, tests/guards required, and first-big-win scoring.

10. **First-big-win selection**: The phase must select or define criteria for the first architecture-aligned migration/build win across the whole landscape.
    - Current: Candidates include delegation manifest/notification mediation, session journal/execution lineage, doc intelligence, agent-work-contract schema, task/trajectory graph, and sidecar visibility, but no gate decides which unlocks the most architecture value safely.
    - Target: The baseline ranks first-big-win candidates by user value, architectural leverage, reversibility, state-risk, OpenCode-platform fit, and testability.
    - Acceptance: The phase outputs either one selected first-big-win or a scored shortlist with a clear recommendation and reasons for rejecting/defering other candidates.

11. **Human/agent collaboration model**: The phase must frame the harness as collaborative agent-led human workflow infrastructure, not only background task execution.
    - Current: Delegation work has focused on runtime dispatch/completion, while user intent shifts, task stems, active landscape overview, deferred idea capture, and cross-session collaboration remain under-specified.
    - Target: The baseline defines how front-facing agents should see active task stems, dependencies, user intent shifts, deferred work, and behind-the-scenes governance before taking action.
    - Acceptance: The baseline includes required views or projections for active tasks, delegated tasks, workflow/task hierarchy, blockers, deferred ideas, and “must inspect before action” criteria.

12. **Memory and retrieval taxonomy**: The phase must classify on-disk memory and future retrieval systems by purpose and consumer.
    - Current: Product-detox `.hivemind` folders show many categories but overlap; current harness lacks a complete memory taxonomy; vector/graph memory is not implemented but may be necessary later.
    - Target: The baseline defines episodic/session memory, workflow memory, delegation evidence, human-readable journal, architecture decisions, long-term concept memory, graph-query projections, and optional local vector memory as separate categories.
    - Acceptance: The taxonomy includes metadata/indexing requirements that enable fast grep/glob/line-read/tool-read access without forcing large context loads.

13. **GUI/Hivefive sidecar compatibility**: The phase must treat safe-build GUI/sidecar interaction as an architecture driver even if implementation is later.
    - Current: Product-detox has an `apps/side-car` concept, but current harness has no stable read model or sidecar API; old plans often defer GUI until after architecture decisions.
    - Target: The baseline defines what future GUI/sidecar may read, query, render, and request without becoming runtime authority.
    - Acceptance: The baseline states sidecar boundaries for read-only projections, config proposals, OpenCode/Hivemind settings, interactive planning/steering, delegation visibility, and non-breaking user guidance.

14. **Architecture validation guards**: The phase must allow implementation of tests, scripts, checklists, or validation guards that prevent stale architecture regression.
    - Current: Agents can create files into folders without design patterns; docs can drift; stale architecture claims can contaminate future turns.
    - Target: The phase may add non-random, architecture-serving guards that report facts and/or validate constraints before later implementation phases.
    - Acceptance: Any guard added in this phase has a documented purpose, input, output, non-governance or governance classification, failure behavior, and relationship to the decision-validation register.

## Boundaries

**In scope:**
- Create or update Phase 16.4 architecture/spec/research/context/planning artifacts.
- Validate stale/hallucinated architecture decisions against current source code, current OpenCode platform behavior, and comparable harness projects.
- Revise or supersede Phase 11 architecture assumptions when evidence shows they are stale.
- Define architecture rules for mutation authority, state ownership, lifecycle ownership, runtime classifications, code-tree organization, and concept migration.
- Define Hiveminder/Hivefiver/shared capability boundaries and the transition from `.md` staging to code/schema/SDK/config runtime integration.
- Define product-detox concept-migration gates and first-big-win selection criteria.
- Add tests, scripts, checklists, or validation guards if they directly support 16.4 architecture validation and are traceable to requirements.
- Use current OpenCode docs/source/API evidence and comparable project research before locking platform-sensitive claims.

**Out of scope:**
- Copying product-detox code wholesale — this phase migrates concepts and contracts first.
- Implementing the full delegation manifest/notification mediation layer — paused until baseline gates exist.
- Implementing full sidecar/GUI — this phase defines compatibility boundaries and read-model requirements only.
- Implementing vector memory, graph query, or long-term memory stack — this phase classifies future requirements and constraints only.
- Broad source-tree moves without a validated plan — allowed only if the phase plan explicitly scopes and verifies a small architecture guard or baseline-supporting change.
- Treating `.md` agent/skill/command files as final architecture — they are staging/live-test artifacts for later runtime integration.
- Treating old docs as authoritative without code/platform validation.

## Constraints

- User instructions and latest user answers override older planning docs when they conflict.
- The project is a local, single-user harness unless a phase explicitly introduces multi-user/server exposure constraints.
- OpenCode platform behavior must be validated with current local references, current package versions, or current internet/MCP evidence before becoming a hard rule.
- Architecture artifacts must include evidence paths or source citations for material claims.
- Scripts/guards must not silently mutate runtime state unless their mutation authority is explicitly approved and documented.
- Event hooks must not be the sole mechanism for mandatory ordered state transitions unless an idempotent retry/recovery path exists.
- Sidecar/GUI work must not become runtime authority over canonical state.
- Future migration must preserve installability and maintainability of the npm harness and not turn the plugin into a monolithic application.
