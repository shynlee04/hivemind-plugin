**Researched:** 2026-04-24  
**Domain:** delegation surface hardening, recovery truthfulness, parent notification replay, terminal/signal semantics  
**Confidence:** HIGH

## Locked Decisions Carried Into Planning

- **D-01 / D-02 / D-03:** Keep `task`, `delegate-task`, and `run-background-command` as distinct surfaces; treat the subsystem as tools + hooks + plugin wiring + persistence + recovery, not just `DelegationManager`.
- **D-04 / D-05 / D-06 / D-07:** Recovery guarantees must be surface-specific and truthful: SDK agent delegations are resumable, PTY command sessions are best-effort resumable, headless fallback is non-resumable after restart.

- **D-11 / D-12 / D-13:** Terminal reporting must stay surface-aware; signal-killed PTY/headless sessions must default to “interrupted by signal” unless cancellation was explicit.
- **D-14:** Child delegated sessions remain non-delegating by default.

- No unification of `task`, `delegate-task`, and PTY command execution.
- No broader delegation architecture redesign.
- No bounded nested delegation feature work.

## Summary

Phase 16.3 should **harden truthfulness and recovery semantics across the existing delegation subsystem**, not reopen the Phase 16 / 16.2 architectural split. The live code already has the right separation points: `delegate-task` dispatches SDK child sessions, `run-background-command` dispatches PTY/headless command sessions, `DelegationManager` owns terminal transitions, `notification-handler.ts` already has direct parent-prompt delivery, and `create-core-hooks.ts` already has continuity-backed `pendingNotifications` state. The hardening gap is that these pieces do **not yet tell the same truthful story** end-to-end.

The strongest external evidence remains consistent with the user's decisions: oh-my-openagent succeeds by combining **notification-first completion**, **result retrieval tools**, and **resume/recovery logic** around a single background manager, while opencode-background-agents documents the tradeoffs of isolated sessions and timeout-driven background work. For PTY surfaces, opencode-pty reinforces the correct tool model: PTY sessions need explicit list/read/write/kill semantics with offset-based output retrieval, not implicit treatment as generic agent delegations.

## Key Findings

### 1. Recovery truth exists internally but is not surfaced honestly enough

- `recoverPending()` in `src/lib/delegation-manager.ts` already distinguishes SDK, PTY, and headless restart behavior.
- `Delegation`, `DelegationResult`, `delegation-status`, and `run-background-command` do **not yet provide an explicit recovery contract** that callers can rely on.
- `src/lib/delegation-persistence.ts` normalizes legacy records but only restores coarse execution metadata (`sdk` / `pty` / `headless`).

**Planning implication:** first plan should extend the record/result contract so persisted state and user-facing tools report `resumable` vs `best-effort` vs `non-resumable-after-restart` explicitly.

### 2. Notification-first completion is only partially implemented

- `src/lib/notification-handler.ts` already sends direct parent prompts and already emits structured JSON for terminal notifications.
- `transitionToTerminal()` already calls `notifyDelegationTerminal()`.
- `src/hooks/create-core-hooks.ts` reads `pendingNotifications`, but its replay path currently **clears the queue without injecting or re-sending anything**.

**Planning implication:** Phase 16.3 must finish the last mile: queue undelivered notifications durably and replay them on parent resumption with a single contract that contains both readable summary text and machine-parsable metadata.

### 3. Terminal outcomes are still too coarse for the user's signal semantics

- `DelegationStatus` remains `dispatched | running | completed | error | timeout`.
- `command-delegation.ts` maps any non-zero exit to generic error and does not distinguish signal death from explicit cancellation.
- `run-background-command.ts` terminates PTY sessions directly, so explicit user cancellation is not yet represented as a first-class terminal detail.

**Planning implication:** preserve coarse top-level status for compatibility, but add a more specific terminal-detail layer (for example `terminalKind`, `terminationSignal`, `explicitCancellation`, or equivalent) so D-11/D-12/D-13 are honored without a breaking rewrite of all callers.

### 4. D-14 is already implemented and should be preserved, not redesigned

- `src/lib/spawner/session-creator.ts` denies both `delegate-task` and `task` in child sessions.
- `MAX_DELEGATION_DEPTH = 3` from Phase 16.2 is a guardrail, but D-14 is stronger: nested delegation should stay disabled by default in child workers.

**Planning implication:** regression coverage must ensure surface-hardening work does not accidentally re-open child delegation capability.

## External Evidence

- **oh-my-openagent** background-agent stack: notification-first completion, background output/result retrieval, queued pending notifications, resume flow, and stale task handling all live together in the background manager and background-notification hook.
- **opencode-pty**: PTY surfaces should expose explicit list/read/write/kill semantics with offset-based output retrieval rather than pretending PTY is the same thing as agent delegation.
- **opencode-background-agents**: isolated background sessions trade away undo/branching and rely on timeout-heavy recovery; this reinforces D-05/D-06/D-07 and argues for explicit non-equivalence in user-facing contracts.

## Common Pitfalls to Avoid

1. **Do not collapse surfaces into one “background task” abstraction.** Preserve distinct tool promises per D-01/D-02.
2. **Do not clear `pendingNotifications` before successful replay.** Current hook behavior already risks dropping resumable parent delivery.
3. **Do not represent headless fallback as equivalent to SDK/PTY recovery.** After restart it must be surfaced as non-resumable.
4. **Do not over-expand `DelegationStatus` if terminal-detail fields can preserve compatibility.** Use a detail layer for signal/cancel distinctions.
5. **Do not remove the child-session deny rules while touching delegation contracts.** D-14 is locked.

## Recommended Plan Shape

1. **Contract + recovery-truth plan** — add explicit surface/recovery metadata to persisted delegation records and user-facing tool results.
2. **Notification-first replay plan** — finish parent delivery and pending notification replay on parent resumption.
3. **Terminal/signal hardening plan** — distinguish interruption, cancellation, non-resumable fallback, and runtime-specific terminal details with targeted tests.
