# Phase 16.5 Fix Research: What the Architecture Actually Needs

**Researched:** 2026-04-25
**Domain:** OpenCode primitive configuration, workflow orchestration, programmatic control
**Confidence:** HIGH (verified against OpenCode platform reference files on disk)

## Summary

Phase 16.5 built solid Zod schemas (C1), a cross-primitive validator (C2), a config compiler (C3), a workflow skill (C4), and enhanced orchestrator (C5) — 8 plans, 772+ LOC schemas, 680+ tests passing. The CRITICAL-AUDIT identified 14 BLOCKERs. The GAP-ANALYSIS identified 10 critical gaps. But the **fundamental architectural failure** is more specific than either document captures:

**The system has NO PROGRAMMATIC WORKFLOW CONTROL.** The 8-turn workflow in `opencode-config-workflow/SKILL.md` is a TEXT INSTRUCTION to the LLM, not a state machine. It cannot enforce turn order, persist state across context resets, resume after interruption, or prevent the agent from skipping steps. Every "checkpoint" in the skill is a polite request, not a gate. This is the root cause of "theater code" — the system works when the LLM cooperates but has zero resilience when it doesn't.

**Primary recommendation:** Replace the text-instruction SKILL.md workflow with a PROGRAMMATIC state machine backed by `continuity.ts`-style durable persistence. Keep the schemas, compiler, loader, validator, and framework detector — these are salvageable. The fix is adding a workflow orchestration layer that uses OpenCode plugin hooks and tools to enforce step order.

## 2. Runtime Loading Order and Resolution

**Source: [VERIFIED] DeepWiki query against sst/opencode + platform reference docs**

### Exact Discovery Sequence

1. `Config.Service` loads global config via `getGlobal()`
2. Custom config from `OPENCODE_CONFIG` env var
3. Project config discovered by traversing UP to nearest Git directory
4. For each `.opencode` directory found:
   - `ConfigAgent.load()` discovers agents from `agent/` and `agents/`
   - `ConfigCommand.load()` discovers commands from `command/` and `commands/`
   - Loads modes, plugins from respective directories
5. All discovered items merged using `mergeDeep()`

### Permission Cascade

1. **Default permissions** (base layer — permissive: most tools "allow", `.env` files "ask")
2. **Global permission config** (from `opencode.json` → `permission` key)
3. **Agent-specific overrides** (from agent frontmatter → `permission` key)
4. **User session responses** (when user clicks "always" on an "ask" prompt)

### Skills Loading

- `Skill.Service.discoverSkills()` scans all configured directories
- `Skill.available(agent)` filters by agent's permission config
- Skills with `"ask"` action are hidden from agent entirely
- Skills only visible if `skill` tool is not disabled in agent config

### What Happens on Conflicts

- **Two agents with same name:** Later-loaded overrides earlier (project > global)
- **Custom tool with same name as built-in:** Custom tool takes precedence
- **Custom command with same name as built-in:** Custom command overrides built-in
- **Two agents claim same tool:** Both get it — tools are not exclusive resources
- **Command references non-existent agent:** Command fails at invocation time (no compile-time check)

## 3. Workflow Control Architecture — THE MISSING PIECE

### Why SKILL.md Is Insufficient

The current `opencode-config-workflow/SKILL.md` is a **181-line text instruction** that tells the agent:
1. What turns exist (0–7)
2. What to do at each turn
3. Where to checkpoint

**What it CANNOT do:**
- Enforce turn order (agent can skip Turn 3 and go to Turn 5)
- Persist state (if context resets, all turn progress is lost)
- Resume after interruption (user closes laptop, comes back tomorrow)
- Prevent duplicate work (agent re-runs Turn 0 every time)
- Validate completeness (agent claims "done" but skipped validation)

### What PROGRAMMATIC Workflow Control Requires

**Source: [CITED] opencode-plugins.md hook system + continuity.ts pattern**

#### A. State Machine with Durable Persistence

Use the existing `continuity.ts` pattern for persistence:

```
WorkflowState (in continuity.json):
{
  "configWorkflow": {
    "id": "wf-abc123",
    "type": "agent-config",
    "currentTurn": 3,
    "turns": {
      "0": { "status": "complete", "output": {...} },
      "1": { "status": "complete", "output": {...} },
      "2": { "status": "complete", "output": {...} },
      "3": { "status": "in_progress", "output": null }
    },
    "startedAt": 1714032000000,
    "updatedAt": 1714032100000,
    "targetPrimitives": [...],
    "scope": "project",
    "mode": "create"
  }
}
```

#### B. Tool-Based Turn Enforcement

The `configure-primitive` tool should accept a `workflowTurn` parameter:

```typescript
// In configure-primitive.ts
const args = {
  action: z.enum([...]),
  // ... existing args
  workflowTurn: z.number().min(0).max(7).optional(),
  workflowState: z.string().optional(), // JSON workflow state ID
}
```

When `workflowTurn` is provided:
1. Load workflow state from continuity store
2. Verify `currentTurn === workflowTurn` (or `currentTurn + 1 === workflowTurn`)
3. If mismatch: return error with expected turn
4. If match: execute action, update state, persist

#### C. Plugin Hook for Auto-Checkpoint

```typescript
// In plugin.ts
"tool.execute.after": async (input, output) => {
  if (input.tool === "configure-primitive" && output.workflowTurn !== undefined) {
    // Auto-persist workflow state after each tool call
    persistWorkflowState(output.workflowId, {
      currentTurn: output.workflowTurn,
      turns: { [output.workflowTurn]: { status: "complete", output } }
    })
  }
}
```

#### D. Resume Capability

A new tool `workflow-resume` or extend `configure-primitive` with `action: "resume"`:
1. Read persisted workflow state
2. Return: "You left off at Turn X. Here's what was collected. Ready to continue?"

### Implementation Approach

**Don't build a workflow engine from scratch.** Build a minimal turn-tracking state machine:

```
src/lib/config-workflow/
├── workflow-state.ts      # ~80 LOC — state machine + transitions
├── workflow-persistence.ts # ~60 LOC — continuity.ts-backed persistence
└── workflow-guards.ts      # ~40 LOC — turn precondition validators
```

**Total: ~180 LOC** — well under 500 LOC limit.

**State transitions:**
```
Turn 0 (Discovery) → Turn 1 (Investigate) → Turn 2 (Collect) → 
Turn 3 (Proposal) → Turn 4 (Validate) → Turn 5 (Compile) → 
Turn 6 (Test) → Turn 7 (Save/Restart)

Linear progression. No branching. Skip-back allowed (re-collect).
```

## 4. Hybrid Configuration Composition

### Dependency Graph for Multi-Primitive Config

```
agent ──references──→ tools (via permission)
agent ──references──→ skills (via skill tool at runtime)
command ──references──→ agent (via agent field)
command ──references──→ model (via model field)
skill ──no-external-refs── (standalone)
tool ──may-reference──→ MCP servers (at runtime)
```

### Batch Compilation Rules

**Current state:** `mixedBatchCompile()` exists in `config-compiler.ts` but each item is compiled independently. Cross-type validation happens AFTER all compile, not during.

**Required fix:** 
1. Phase 1: Validate all specs against their schemas
2. Phase 2: Build a dependency graph (agent→command→skill)
3. Phase 3: Detect circular deps (A references B references A)
4. Phase 4: Compile in topological order (leaves first)
5. Phase 5: Run cross-primitive validation on the batch
6. Phase 6: Write all files atomically (all-or-nothing)

**Rollback strategy:** Write to temp files first, then rename all to final paths. If any rename fails, reverse the completed ones.

## 7. Architecture Recommendation

### The Fix Approach

**Keep the bottom 5 layers. Add a workflow orchestration layer on top.**

```
Layer 6 (NEW): Workflow State Machine
  src/lib/config-workflow/workflow-state.ts      ~80 LOC
  src/lib/config-workflow/workflow-persistence.ts  ~60 LOC
  src/lib/config-workflow/workflow-guards.ts       ~40 LOC
  src/lib/config-workflow/index.ts                 ~20 LOC
                                              Total: ~200 LOC

Layer 5 (EXISTING): Tools
  src/tools/configure-primitive.ts  (add workflowTurn param)
  src/tools/validate-restart.ts     (add workflow integration)

Layer 4 (EXISTING): Libraries
  src/lib/config-compiler.ts
  src/lib/cross-primitive-validator.ts
  src/lib/runtime-validator.ts
  src/lib/framework-detector.ts
  src/lib/primitive-loader.ts

Layer 3 (EXISTING): Schemas
  src/schema-kernel/*.schema.ts + lenient variants

Layer 2 (EXISTING): Shared
  src/shared/tool-response.ts + tool-helpers.ts

Layer 1 (EXISTING): Plugin
  src/plugin.ts (add workflow hooks)
```

### Key Design Decisions

1. **Don't build XState or a full workflow engine.** A linear 8-turn state machine with durable persistence is sufficient. No branching, no parallelism, no conditions. 200 LOC.

2. **Use continuity.ts pattern for persistence.** The `continuity.json` file already stores session state. Add a `configWorkflows` section. Deep-clone-on-read.

3. **Enforce turns through tool parameters, not hook interception.** The `configure-primitive` tool accepts `workflowTurn`. If the turn doesn't match the persisted state, return an error. Simple, testable.

4. **Resume via a dedicated action.** `configure-primitive` gets `action: "resume"` that reads persisted state and returns context.

5. **SKILL.md stays as GUIDE, not CONTROLLER.** The skill tells the agent WHAT to do at each turn. The tool enforces WHERE the agent is in the workflow. These are complementary.

### Dependency Chain for Fix

```
1. Add configWorkflow to continuity.ts types     (~20 LOC in types.ts)
2. Create src/lib/config-workflow/                (~200 LOC new)
3. Add workflowTurn to configure-primitive.ts     (~40 LOC modification)
4. Add workflow hooks to plugin.ts                (~20 LOC modification)
5. Update SKILL.md to reference workflow tool     (~30 LOC modification)
6. Write tests                                    (~200 LOC new tests)
```

**Total new/modified LOC: ~510** — well within budget.

### What NOT to Build

- **No XState/state machine library.** A linear 8-step progression doesn't need it.
- **No intent classification ML model.** Keyword matching is sufficient for this phase.
- **No new schemas for rules/prompts/workflows/hooks.** These are deferred items.
- **No plugin architecture for extensible schemas.** `*Lenient` variants handle forward compat.
- **No semantic embedding-based routing.** Overkill for configuration workflow.

## Open Questions

1. **Can `tool.execute.after` hooks access the tool's return value?**
   - What we know: Plugin hooks receive `(input, output)` where input has tool name and args
   - What's unclear: Whether `output` contains the tool's return value or just args
   - Recommendation: Test with a simple hook that logs output structure

2. **What happens when two parallel sessions run configure-primitive simultaneously?**
   - What we know: continuity.json is a single file with no locking
   - What's unclear: Whether concurrent writes cause corruption
   - Recommendation: Add simple file-level lock or accept single-session limitation

3. **Does OpenCode support custom tools registering additional actions?**
   - What we know: Tools export a single `execute()` function
   - What's unclear: Whether one tool can handle multiple logical operations via args
   - Recommendation: Current `action` enum in configure-primitive already demonstrates this works
