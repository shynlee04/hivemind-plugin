## Top 3 Highest-Impact Issues

### 1. Schema `.strict()` rejects unknown fields (D7) â€” 6 BLOCKERs
**Impact:** If OpenCode adds even one new frontmatter field, permission key, or config precedence level, the harness will reject valid user configurations across ALL installed projects. This is a compatibility time-bomb.
**Files:** `agent-frontmatter.schema.ts`, `command-frontmatter.schema.ts`, `skill-metadata.schema.ts`, `mcp-server.schema.ts`, `config-precedence.schema.ts`, `permission.schema.ts`
**Fix priority:** P0. Must fix before any user installs the package.

### 2. Hardcoded `.opencode/` output paths in compiler (D1) â€” 4 BLOCKERs
**Impact:** The config compiler assumes all output goes to `.opencode/`, completely ignoring global configs (`~/.config/opencode/`) and custom project layouts. This limits the harness to project-local configurations only, making it useless for users who manage agents globally.
**Files:** `16.5-02-PLAN.md` (compileAgent, compileCommand, compileSkill)
**Fix priority:** P0. Add `basePath` parameter to all compile functions.

### 3. Hivefiver branding in generic workflow skill (D2/D6) â€” 1 BLOCKER
**Impact:** The workflow skill is named `hivefiver-agent-config` and references hivefiver-specific agents (`hivefiver-orchestrator`, `hivefiver-agent-builder`). Users not using the hivefiver framework will be confused and unable to use the skill. The SPEC explicitly claims "Conflict-free stacking with other frameworks" but the deliverable is hivefiver-only.
**Files:** `16.5-04-PLAN.md`
**Fix priority:** P1. Rename and rebrand before C4 execution.
