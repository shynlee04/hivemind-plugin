# Phase 16.5 Fix Plan: Add Programmatic Workflow Control

## Goal

Replace the text-instruction SKILL.md workflow controller with a programmatic 8-turn state machine backed by durable persistence, so that `configure-primitive` tool calls enforce turn order, persist progress across context resets, and support resume after interruption.

## Why This Fix

The 8-turn workflow in `opencode-config-workflow/SKILL.md` is a 181-line text instruction to the LLM — it cannot enforce turn order, persist state across context resets, resume after interruption, or prevent the agent from skipping steps. Every "checkpoint" is a polite request, not a gate. This fix adds a ~200 LOC state machine that makes the workflow enforceable through tool parameters, while keeping the SKILL.md as a descriptive guide (not a controller).

## Pre-Conditions

1. All 772 existing tests pass (`npm test`)
2. TypeScript compiles clean (`npm run typecheck`)
3. The research document `16.5-RESEARCH-FIX.md` exists and has been read
4. `continuity.ts` persistence pattern is understood (atomic write via tmp+rename, deep-clone-on-read)
5. `task-status.ts` transition guard pattern is understood (VALID_TRANSITIONS map + canTransition() guard)
6. `delegation-persistence.ts` file-per-concern pattern is understood (separate JSON file in same state dir)

### Step 2: Create `src/lib/config-workflow/workflow-state.ts` (~80 LOC)

**What:** The core state machine — turn transition logic, state creation, and advancement.

**Pattern to follow:** `src/lib/task-status.ts` — static VALID_TRANSITIONS map + canTransition() guard function.

**Exact exports:**

```typescript
import type { ConfigWorkflowState, WorkflowTurn, WorkflowTurnRecord, WorkflowTurnStatus } from "../types.js"

// Turn ordering: 0-7 linear progression, skip-back allowed (re-do a previous turn)
export const TURN_NAMES: WorkflowTurn[] = [
  "discovery", "investigate", "collect", "proposal",
  "validate", "compile", "test", "save"
]

export const TOTAL_TURNS = 8

/** Create a new workflow state at Turn 0 */
export function createWorkflowState(options: {
  type: ConfigWorkflowState["type"]
  targetPrimitives: ConfigWorkflowState["targetPrimitives"]
  scope: ConfigWorkflowState["scope"]
  mode: ConfigWorkflowState["mode"]
}): ConfigWorkflowState

/** Check if advancing from currentTurn to targetTurn is valid.
 *  Rules: target must be currentTurn + 1 (forward) OR <= currentTurn (re-do). */
export function canAdvanceTurn(state: ConfigWorkflowState, targetTurn: number): boolean

/** Advance to a specific turn. Returns new state (immutable — does not mutate input).
 *  Throws [Harness] error if transition is invalid. */
export function advanceTurn(state: ConfigWorkflowState, targetTurn: number): ConfigWorkflowState

/** Mark the current turn as complete with output data.
 *  Returns new state with turn marked complete and updatedAt refreshed. */
export function completeCurrentTurn(
  state: ConfigWorkflowState,
  output: Record<string, unknown>
): ConfigWorkflowState

/** Get the name of a turn by its numeric index */
export function getTurnName(turnIndex: number): WorkflowTurn

/** Check if a workflow is complete (all 8 turns done or final turn complete) */
export function isWorkflowComplete(state: ConfigWorkflowState): boolean

/** Deep-clone a workflow state for safe return (prevents mutation aliasing) */
export function cloneWorkflowState(state: ConfigWorkflowState): ConfigWorkflowState
```

**Transition rules (MUST be enforced):**
- Forward: `targetTurn === state.currentTurn + 1` → always allowed
- Skip-back: `targetTurn <= state.currentTurn` → always allowed (re-do a turn)
- Skip-forward: `targetTurn > state.currentTurn + 1` → REJECTED (can't skip turns)
- `targetTurn < 0` or `targetTurn > 7` → REJECTED (out of range)

**Error format:** `[Harness] Invalid turn transition: cannot advance from turn ${from} to turn ${to}. Expected turn ${from + 1} or any turn <= ${from}.`

**Acceptance:** Module exports 7 functions. No side effects. No file I/O. Pure functions only. Deep-clone on all return paths.

**Exact exports:**

```typescript
/** Validate that a turn can be executed given the current workflow state.
 *  Returns { valid: true } if guard passes, { valid: false, errors: [...] } if not. */
export function validateTurnPrecondition(
  state: ConfigWorkflowState,
  turn: number
): { valid: true } | { valid: false; errors: string[] }

 *  Requires at least one target primitive defined. */

 *  Requires at least Turn 2 (Collect) output with a spec. */
export function hasCompilePrerequisites(state: ConfigWorkflowState): boolean
```

**Turn-specific guards:**
- Turn 0 (Discovery): Always valid — no prerequisites for starting a workflow
- Turn 1 (Investigate): Requires `state.targetPrimitives.length > 0`
- Turn 2 (Collect): Requires Turn 1 output exists
- Turn 3 (Proposal): Requires Turn 2 output with collected fields
- Turn 4 (Validate): Requires Turn 3 output with assembled spec
- Turn 5 (Compile): Requires Turn 4 output with validation pass
- Turn 6 (Test): Requires Turn 5 output with written file path
- Turn 7 (Save): Requires Turn 6 output with test results

**Acceptance:** 3 exported functions. Pure functions. No side effects. No file I/O. Errors are descriptive strings that tell the agent what's missing.

```typescript
function handleResume(args: z.infer<typeof ConfigurePrimitiveInputSchema>): string {
  if (!args.workflowId) {
    return renderToolResult(error("workflowId is required for resume action"))
  }
  const workflow = readWorkflow(args.workflowId)
  if (!workflow) {
    return renderToolResult(error(`Workflow ${args.workflowId} not found`))
  }
  const result: WorkflowResumeResult = {
    workflowId: workflow.id,
    currentTurn: workflow.currentTurn,
    currentTurnName: getTurnName(workflow.currentTurn),
    completedTurns: Object.values(workflow.turns).filter(t => t.status === "complete").length,
    totalTurns: TOTAL_TURNS,
    lastOutput: workflow.turns[workflow.currentTurn - 1]?.output ?? null,
    canContinue: !isWorkflowComplete(workflow),
  }
  return renderToolResult(success({ resume: result }))
}
```

```typescript
// Workflow turn enforcement
if (args.workflowTurn !== undefined && args.workflowId) {

  if (!workflow) {
```

### Step 7: Add Plugin Hook in `src/plugin.ts` (~20 LOC added)

**What:** Add a `tool.execute.after` hook that auto-persists workflow state after `configure-primitive` tool calls with `workflowTurn`.

**Pattern:** The plugin already uses hook factories from `src/hooks/`. However, for a simple inline hook, we add it directly in the plugin composition root.

**Exact change:** After the tool registration block (line 97), add a hook:

```typescript
// In the return object, add hooks section if not present, or add to existing:
"tool.execute.after": async (input: { tool: string; args?: Record<string, unknown> }, output?: unknown) => {
  if (input.tool !== "configure-primitive") return
  const args = input.args
  if (!args || typeof args.workflowId !== "string" || typeof args.workflowTurn !== "number") return
  
  // Auto-persist workflow state after each configure-primitive call with workflow params
  const { readWorkflow, persistWorkflow, advanceTurn, completeCurrentTurn } = await import("./lib/config-workflow/index.js")
  const workflow = readWorkflow(args.workflowId)
  if (!workflow) return
  
  const advanced = advanceTurn(workflow, args.workflowTurn)
  const completed = completeCurrentTurn(advanced, { toolOutput: typeof output === "string" ? output.substring(0, 500) : "completed" })
  persistWorkflow(completed)
}
```

**Important:** Use dynamic `await import()` for the config-workflow module — the plugin should not eagerly load the workflow module at startup. The hook only fires when `configure-primitive` is called with workflow params.

**Acceptance:** Hook fires only for `configure-primitive` tool. Hook only acts when both `workflowId` and `workflowTurn` are present in args. Does not throw on errors (failures are silently ignored — persistence is best-effort). Existing tool behavior unchanged when workflow params are absent.
