## Critical Findings

### 1. The "Score Summary" Tests Are Self-Affirming Tautologies

**File:** `eval/correctness.test.ts:104-114`, `eval/stability.test.ts:73-86`

```typescript
// correctness.test.ts line 111-113
expect(metrics.roundTripFidelity).toBe(3)   // hardcoded
expect(metrics.requiredFieldsPresent).toBe(3) // hardcoded
expect(metrics.totalChecks).toBe(6)           // hardcoded
```

These tests assert that hardcoded constants equal themselves. They produce impressive-looking numbers (6/6 checks!) without measuring anything about the actual system. If every other test in the suite failed, these "metrics" would still show 6/6.

**Verdict:** Theater. Remove these or make them derive from actual test outcomes.

### 2. The Coherence Tests Can Never Fail

**File:** `eval/coherence.test.ts:33,88,99`

```typescript
expect(overlaps.length).toBeGreaterThanOrEqual(0)    // line 33 — ALWAYS passes
expect(unresolved.length).toBeGreaterThanOrEqual(0)   // line 88 — ALWAYS passes
expect(orphans.length).toBeGreaterThanOrEqual(0)      // line 99 — ALWAYS passes
```

These assertions are mathematical tautologies (any integer ≥ 0). The tests found real issues (92 orphans, 1 broken cross-reference) but the eval CANNOT fail on them. The logging is useful, but calling this "eval" is generous — it's instrumentation without enforcement.

### 3. The "Concurrent Compilation" Test Is Not Concurrent

**File:** `eval/stability.test.ts:29-44`

```typescript
const promises = Array.from({ length: 10 }, (_, i) =>
  Promise.resolve().then(() => compileAgent(...))
)
const results = await Promise.all(promises)
```

`Promise.resolve().then()` creates microtasks that execute sequentially on Node.js's single thread. This proves reentrancy (the compiler doesn't mutate shared state), but it does NOT test concurrent access, race conditions, or thread safety. The test name claims "concurrent compilations" but it's sequential microtask execution.

### 4. Would NOT Catch the Fire-and-Forget writeFile BLOCKER

**The BLOCKER:** `configure-primitive.ts:169`:
```typescript
fs.writeFile(filePath, result.content, "utf-8")  // fire-and-forget — no await
```

This is a Promise returned from `fs.writeFile()` that is never awaited. The function returns `renderToolResult(success(...))` immediately, before the write completes. If the write fails, the error is silently swallowed.

**Would the eval catch this?** No. The stability test uses `mixedBatchCompile(dryRun: true)` — it never writes to disk. The correctness test never writes to disk either. No eval test actually verifies that files are written to disk and their content matches. The atomic write path in `mixedBatchCompile` uses synchronous `writeFileSync`, so it's correct — but the single-primitive path in `handleCompile` has the fire-and-forget bug, and no eval exercises that code path with actual disk writes.

### 5. Would NOT Catch Path Traversal or Security Issues

The compiler has `validateOutputPath()` checking for `..` in paths. But:
- No eval test sends `../etc/passwd` as a name
- No eval test sends a name with path separators (`foo/bar`)
- No eval test validates the scope parameter isn't used for directory escape
- The `validateOutputPath` only checks the relative path, not the resolved absolute path

### 6. Fixtures Are Toy Examples, Not Realistic

All test specs are 2-3 field objects with trivial bodies (`"# Body"`, `"## Cmd body"`). Real agent configs have:
- Multi-paragraph descriptions with special characters
- Nested permission objects
- Multiple frontmatter fields (tools, temperature, model, category, subagent flags)
- Cross-references to other primitives
- References/ subdirectory structures (skills)

The eval would miss bugs that only manifest with realistic configs.

## Critical Gaps

1. **CRITICAL: No test exercises disk write path** — The fire-and-forget `fs.writeFile` bug in single-primitive compile is invisible to evals
2. **CRITICAL: Tautological assertions** — "Score summary" tests assert hardcoded numbers, not measured outcomes
3. **CRITICAL: Coherence tests cannot fail** — `toBeGreaterThanOrEqual(0)` is always true

## Remediation Plan

### Must fix before production:

1. **Add disk-write verification test**
   - Test that `handleCompile` with `dryRun:false` actually writes file to disk
   - Test that the written content matches `result.content`
   - Test that the fire-and-forget `fs.writeFile` is awaited (fix the bug AND add the test)
   - Priority: CRITICAL — this is a data-loss bug

2. **Replace tautological assertions with real gates**
   - `eval/correctness.test.ts:104-114`: Remove hardcoded metric assertions, replace with actual pass/fail aggregation
   - `eval/stability.test.ts:73-86`: Same treatment
   - Use vitest's `context` or a shared accumulator pattern

3. **Make coherence tests enforce thresholds**
   - `eval/coherence.test.ts:33`: Change `toBeGreaterThanOrEqual(0)` to `toBeLessThan(MAX_OVERLAPS)` where threshold is configurable
   - `eval/coherence.test.ts:88`: Change to `toBe(0)` for cross-references or use known-exception list
   - `eval/coherence.test.ts:99`: Change to `toBeLessThan(MAX_ORPHANS)` with configurable threshold

### Should fix soon:

4. **Add edge case tests**
   - Empty spec (all fields missing)
   - Malformed YAML input to `parseSpec`
   - Unicode in names and body (emoji, CJK characters)
   - Names with path separators and shell metacharacters
   - Null/undefined frontmatter values

5. **Add cross-primitive conflict test**
   - Test that `mixedBatchCompile` rejects when two primitives share a name
   - Test that conflict message includes both types
   - Verify fail-fast behavior (no partial writes)

6. **Replace fake concurrency test with real parallelism**
   - Use `worker_threads` or at minimum `setImmediate`-based interleaving
   - Or remove the "concurrent" claim and rename to "sequential reentrancy"

7. **Add realistic fixture corpus**
   - Extract 10-20 real agent/command/skill configs from `.opencode/`
   - Use them as the test corpus instead of inline toy examples
   - Parameterize tests to iterate over corpus

### Nice to have:

8. **Add CI integration**
   - Add `npm run eval` script to `package.json`
   - Add eval step to CI workflow
   - Make eval results machine-readable (JSON output)

9. **Add security-focused eval**
   - Path traversal attempts
   - Name injection with `../`, `/`, `\`
   - Spec injection with YAML bomb or prototype pollution

10. **Add observability**
    - Add timing metrics to all eval tests
    - Generate a summary report file (not just console.log)
    - Track eval scores over time (regression detection)
