    Rewrite `src/lib/delegation-manager.ts` completely. The new implementation follows the WaiterModel + dual-signal architecture:

    ## Architecture:
    1. **dispatch()** — Always background. Creates child session via SDK, persists delegation state, sends prompt, returns immediately with `{ delegationId, status: "dispatched" }`. The foreground continues. No awaiting completion.
    
    2. **Dual-signal completion** — Uses existing `CompletionDetector` from `completion-detector.ts`:
       - `handleSessionIdle()` receives the idle event
       - Starts stability polling: every `STABILITY_POLL_INTERVAL_MS` (3s), check message count
       - If message count unchanged for `STABILITY_THRESHOLD` (3) consecutive polls → mark completed
       - Extract result from messages, persist, clean up
    
    3. **Safety ceiling** — Optional `safetyCeilingMs` parameter (default 30 min). This is a MAX ceiling, not a deadline. If dual-signal hasn't confirmed completion by this time, abort child session and mark as timeout.
    
    4. **Hybrid persistence** — 
       - Persist to `delegations.json` (same path as current, in continuity store dir) after EVERY state transition
       - `recoverPending()` reads from disk on plugin load
       - For sessions that completed while down: finalize immediately
       - For sessions still running: re-register in Maps, re-attach to CompletionDetector
    
    5. **Integration with existing modules**:
       - `DelegationConcurrencyQueue` from `concurrency.ts` for per-key concurrency control
       - `unwrapData` from `helpers.ts` for SDK response handling
       - `getContinuityStoragePath` from `continuity.ts` for persistence path
       - `buildDelegationQueueKey` from `concurrency.ts` for queue key generation
