# Phase 54 Validation Frame: Sidecar & Product-Detox Integration Runway

## Purpose

This validation frame defines what must be true before detailed planning for the sidecar/product-detox runway can start.

## Preconditions Before Planning Starts

| Check | Must Be True |
|---|---|
| Architecture baseline respected | Phase 16.4 sidecar and migration-control rules are treated as binding inputs |
| No-write rule accepted | The future sidecar is framed as read/query/render/request only |
| Config/runtime grounding accepted | User-configurable primitive UX must map to current `opencode.json`, configure-primitive, primitive loader, runtime validator, and hook/plugin surfaces |
| Product-detox discipline accepted | Concept/contract extraction comes before any code movement |

## Preconditions Before Execution Starts

1. The planner can name the read models needed for the future sidecar.
2. The planner can name the migration candidates and the gate each must pass.
3. The planner can show how config proposals reach approved write paths without creating a second authority.
4. The planner can show how hook-related UX remains request-mediated rather than directly mutating runtime authority.

## No-Go Conditions

- Planning starts from UI implementation details before read models are defined.
- Product-detox code movement is proposed without concept-gate evidence.
- Sidecar or UX flows are allowed to write canonical state directly.
- Primitive UX is designed independently of the real validator/compiler/runtime pipeline.

## Go Rule

**GO** only when the runway can be described as an evidence-backed contract surface for later implementation, not as a speculative UI concept.

If the plan cannot preserve read-only sidecar boundaries and gated concept migration, validation fails before execution begins.
