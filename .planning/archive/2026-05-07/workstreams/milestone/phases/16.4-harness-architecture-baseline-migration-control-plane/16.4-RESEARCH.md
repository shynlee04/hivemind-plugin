## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Validation artifact design

- **D-01:** Phase 16.4 will produce a decision-validation register plus guard checklist/scripts, not only prose architecture notes.
- **D-02:** The decision-validation register classifies prior claims as `valid`, `partial`, `stale`, `rejected`, or `needs-research`.
- **D-03:** Every decision-register entry must include: claim, source path, current code/platform evidence, verdict, replacement rule, downstream impact, and whether the claim is safe for future agents to reuse.
- **D-04:** Guard artifacts may be either fact-reporting or enforcement-oriented, but each guard must explicitly declare its classification, purpose, input, output, failure behavior, and relationship to a SPEC requirement.
- **D-05:** The validation system must prevent stale docs from poisoning later turns; docs are evidence inputs, not runtime truth.

#### Target code-tree authority

- **D-06:** Phase 16.4 has authority to supersede Phase 11 because Phase 11 is stale.
- **D-07:** Phase 16.4 should produce a new target code-tree architecture plus a migration move map.
- **D-08:** The target tree must classify current files as: keep, split, move later, retire, validate first, or future/optional.
- **D-09:** Broad source-tree moves are not performed during discuss/context capture and should not happen without explicit plan-phase tasks and verification gates.
- **D-10:** Tiny validation-supporting artifacts are allowed if they directly support the architecture baseline, stale-decision register, or migration control plane.

#### State and memory ownership

- **D-11:** `.hivemind/` is the preferred canonical Hivemind harness state root going forward because `.opencode/` can be deleted accidentally by users and should not be trusted as durable harness memory.
- **D-12:** Current `.opencode/state/opencode-harness/` state must be treated as the existing implementation reality and migration input, not as the desired long-term canonical root.
- **D-13:** The plan must define a safe transition from current `.opencode/state/opencode-harness/` persistence to `.hivemind/` canonical state, including compatibility, migration, fallback, and recovery behavior.
- **D-14:** `.hivemind/` must not become an unstructured dump. It needs purpose-based categories, metadata, indexes, quick-read surfaces, and clear canonical-vs-projection markers.
- **D-15:** Sidecar views, journals, trajectory/lineage, graph/vector layers, and human-readable projections may live under or read from `.hivemind/`, but each must declare whether it is canonical runtime state, derived projection, audit trail, or optional future memory.
- **D-16:** No second runtime authority is allowed without an explicit architecture decision/gate; if `.opencode/state` and `.hivemind` coexist during migration, their authority and sync direction must be explicit.

#### Runtime classification model

- **D-17:** Runtime classification must use an actor/consumer/task/workflow matrix, not lifecycle states alone.
- **D-18:** Minimum runtime classes are: session initiation runtime, mid-flow intervention runtime, post-completion runtime, delegated subtask runtime, and background command/runtime surface.
- **D-19:** Each runtime class must define: triggering surface, actor, consumer, task/workflow type, state owner, mutation authority, context boundary, recovery behavior, and completion/handoff behavior.
- **D-20:** The runtime model must distinguish human-visible conversation flow from actual LLM/API session/subsession flows so agents do not confuse parent session, subagent session, tool output, system reminders, and workflow context.

#### First-big-win selection

- **D-21:** The first recommended concept migration after the architecture baseline is a session journal + execution lineage bridge.
- **D-22:** The session journal + execution lineage bridge is preferred because it supports delegation awareness, human context, future sidecar read models, product-detox concept migration, and cross-session continuity without replacing runtime truth prematurely.
- **D-23:** Delegation manifest/notification mediation remains important but should resume after state ownership, runtime classification, and journal/lineage decisions are locked.
- **D-24:** Product-detox concepts are migrated as concepts/contracts first; no product-detox code is copied as-is.

### Anti-Patterns to Avoid

- **Treating old architecture docs as runtime truth:** Phase 16.4 exists because old docs contain stale/nonexistent claims. [VERIFIED: 16.4-SPEC.md lines 11-18]
- **Global CQRS dogma:** Hooks are not globally read-only because platform hooks and current code mutate documented hook outputs. [CITED: https://opencode.ai/docs/plugins; VERIFIED: src/hooks/create-core-hooks.ts lines 113-124]
- **Second runtime authority:** Do not let `.opencode/state` and `.hivemind` both write canonical truth without sync direction and ownership. [VERIFIED: 16.4-CONTEXT.md lines 60-68]
- **Product-detox wholesale copy:** Locked decisions require concept/contracts first and no as-is copy. [VERIFIED: 16.4-CONTEXT.md lines 76-82]
- **Sidecar as authority:** GUI/sidecar must read/project/request, not own canonical state. [VERIFIED: 16.4-SPEC.md lines 81-85 and 119-121]
- **Broad tree moves in 16.4 without gates:** Broad moves are out of scope unless explicitly planned and verified. [VERIFIED: 16.4-SPEC.md lines 103-110]

## Common Pitfalls

### Pitfall 1: Stale Docs as Architecture Poison
**What goes wrong:** A planner reuses Phase 11 or migration-plan statements as truth even when source code and platform docs contradict them. [VERIFIED: 16.4-SPEC.md lines 11-18]
**Why it happens:** Docs contain authoritative language and older verification summaries, but current source changed. [VERIFIED: STATE.md lines 26-29 and 166-170]
**How to avoid:** Require every decision-register row to include current source/platform evidence and a reuse-safe field. [VERIFIED: 16.4-CONTEXT.md lines 44-50]
**Warning signs:** Claims mention nonexistent paths, dead modules that are imported, or global rules like “hooks never mutate.” [VERIFIED: src/lib/notification-handler.ts lines 1-8; VERIFIED: opencode-plugins.md lines 63-70]

### Pitfall 2: Duplicate Lifecycle Authority
**What goes wrong:** Multiple modules can appear to own terminal transitions, event observations, retry, notification, and recovery. [VERIFIED: src/lib/delegation-manager.ts; VERIFIED: src/lib/sdk-delegation.ts; VERIFIED: src/lib/command-delegation.ts; VERIFIED: src/lib/lifecycle-manager.ts]
**Why it happens:** `DelegationManager` delegates SDK/command behavior via callbacks while hooks and lifecycle facade also observe events. [VERIFIED: src/lib/delegation-manager.ts lines 72-88; VERIFIED: src/hooks/create-core-hooks.ts lines 81-96]
**How to avoid:** Assign primary owner per transition and mark helpers as handlers/observers/facades. [VERIFIED: 16.4-SPEC.md lines 41-45]
**Warning signs:** A new feature calls `client.session.prompt`, writes delegation state, and notifies parent without going through the same terminal transition path. [VERIFIED: src/lib/delegation-manager.ts lines 335-379]

### Pitfall 3: `.hivemind/` Becomes an Unstructured Dump
**What goes wrong:** Journal, lineage, workflow, graph, vector, sidecar, and projections all write arbitrary files under `.hivemind/`. [VERIFIED: 16.4-CONTEXT.md lines 60-68]
**Why it happens:** Product-detox and current `.hivemind` examples show many categories, but Phase 16.4 has not yet locked taxonomy. [VERIFIED: Glob .hivemind/**; VERIFIED: docs/plans/migration-plan-detox-to-harness-2026-04-24.md lines 441-459]
**How to avoid:** Define canonical/runtime, derived, projection, audit, archived, and future/optional categories before writer changes. [VERIFIED: 16.4-SPEC.md lines 31-35]
**Warning signs:** A plan adds `.hivemind/state/*.json` without owner, schema, index, retention, recovery, and projection status. [ASSUMED]

### Pitfall 4: Confusing Human Conversation Lifecycle with Delegation Lifecycle
**What goes wrong:** Raw background completion reminders interrupt unrelated human turns and the next agent lacks delegation context. [VERIFIED: session-ses-2413-delegation-design-incidents-2026-04-24.md lines 41-75 and 99-138]
**Why it happens:** Durable delegation records exist, but no turn-scoped manifest/briefing/read model connects them to user-visible conversation cadence. [VERIFIED: session-ses-2413-delegation-design-incidents-2026-04-24.md lines 120-138]
**How to avoid:** Phase 16.4 should define runtime classes and projections; implementation of manifest/notification mediation should wait until after ownership rules are locked. [VERIFIED: 16.4-CONTEXT.md lines 76-82]
**Warning signs:** Plans propose “just add system reminders” or “hold stream until all tasks finish” without actor/consumer/task/workflow classification. [VERIFIED: session-ses-2413-delegation-design-incidents-2026-04-24.md lines 190-217]

### Pitfall 5: Product-Detox Copy-Forward
**What goes wrong:** Useful concepts bring along bloat, stubs, blocking scripts, or conflicting state roots. [VERIFIED: docs/plans/migration-plan-detox-to-harness-2026-04-24.md lines 38-42 and 610-635]
**Why it happens:** Draft migration matrix labels some features READY, but Phase 16.4 locked concept/contracts-first migration. [VERIFIED: docs/plans/migration-plan-detox-to-harness-2026-04-24.md lines 386-430; VERIFIED: 16.4-CONTEXT.md lines 76-82]
**How to avoid:** Gate every candidate with source paths, purpose, target category, state owner, risk, copy exception, tests/guards, and first-big-win score. [VERIFIED: 16.4-SPEC.md lines 61-69]
**Warning signs:** A plan says “bring wholesale” without current code inspection and target contract tests. [VERIFIED: docs/plans/migration-plan-detox-to-harness-2026-04-24.md lines 410-430]
