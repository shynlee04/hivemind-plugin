## Summary

Phase 48.3 should be planned as an integration-alignment refactor. The harness has a canonical session SDK wrapper in `src/lib/session-api.ts`, and delegated session creation already uses it through `spawnDelegatedSession()`. [VERIFIED: `src/lib/session-api.ts:36-187`; `src/lib/spawner/session-creator.ts:16-23`] Several dispatch, recovery, lifecycle, finalization, abort, and notification paths still call the SDK directly and bypass `session-api.ts`. The main CQRS gap is hook-side mutation. [VERIFIED: `src/hooks/create-session-hooks.ts:124-128`; `src/hooks/create-session-hooks.ts:172-185`; `src/hooks/create-session-hooks.ts:187-203`; `src/hooks/create-core-hooks.ts:52-78`; `src/hooks/create-tool-guard-hooks.ts:79-108`; `src/hooks/create-tool-guard-hooks.ts:117-150`; `src/lib/lifecycle-manager.ts:101-112`] A CQRS-compliant plan should move writes behind tools or explicit write-side runtime services and keep hooks observational/projection-only. [ASSUMED]

`src/plugin.ts` is not thin by its own header contract. It wires dependencies, but it also contains helper logic, PTY probing, event routing, event-tracker writes, tool-after multiplexing, and configure-primitive workflow persistence.
