# Phase 16.2 SPEC: PTY Execution Wiring + OMO Safety Patterns

**Purpose:** Close two critical gaps identified in Phase 16 verification: (1) verify PTY execution wiring for command delegations is complete and add unified terminal lifecycle, and (2) backfill OMO-proven safety patterns — grace periods for memory cleanup, parent notifications, adaptive polling, and nesting depth limits.

**Baseline:** Phase 16 post-Wave 6, 462 tests passing, build clean. `DelegationManager` (348 LOC) delegates to `CommandDelegationHandler` (285 LOC) and `SdkDelegationHandler` (161 LOC). PTY infrastructure is wired for command delegations (PTY-first with headless fallback). Delegations accumulate in memory indefinitely (no cleanup). No parent notifications. Fixed-interval polling. No nesting limits.

## Section 1: PTY Execution Wiring (Verification + Gap Refinement)

This section verifies that the PTY runtime path in `command-delegation.ts` is correctly wired: `CommandDelegationHandler.dispatchCommand()` → `ptyManager.spawn()` → `schedulePtyExitPoll()` → `finalizeCommandDelegation()`. The PTY path for command delegations is functional. The remaining gap is ensuring all PTY terminal paths converge through `transitionToTerminal()` (R-LC-05) for unified lifecycle management.

> **Note on D-04A:** Per locked decision D-04A, SDK delegations intentionally continue using `client.session.prompt()` (not PTY). This is the verified dual-path architecture. PTY is the default for command/process execution surfaces only.

### R-PTY-01: Command delegations execute through PTY as primary path

**Statement:** The system SHALL route all command delegations through PTY as the primary execution surface, falling back to headless `child_process.spawn()` only when `PtyManager` is null or `isSupported()` returns false.

**Acceptance Criterion:** Given a `CommandDelegationParams` with a valid command, when `dispatchCommand()` is called and PTY is available, then the delegation record SHALL have `executionMode: "pty"` and `ptySessionId` set to the PTY session identifier. When PTY is unavailable, then `executionMode` SHALL be `"headless"` and `fallbackReason` SHALL contain a descriptive message.

**Priority:** P0

**Affected Files:** `src/lib/command-delegation.ts`, `src/tools/delegate-task.ts`

**OMO Reference:** OMO `background-agent/spawner` — PTY-first dispatch with headless fallback

---

### R-PTY-02: PTY process output is captured and returned as delegation result

**Statement:** The system SHALL capture the full PTY process stdout + stderr output and set it as the `result` field on the `Delegation` record when the PTY process exits.

**Acceptance Criterion:** Given a PTY delegation that completes successfully (exit code 0), when `finalizeCommandDelegation()` runs, then `delegation.result` SHALL contain the full captured output from `ptyManager.read(sessionId, 0).content`. An integration test SHALL verify that a PTY delegation for `echo "hello"` produces a result containing "hello".

**Priority:** P0

**Affected Files:** `src/lib/command-delegation.ts`

**OMO Reference:** OMO `background-agent/result-handler` — output capture from background processes

---

### R-PTY-03: PTY process lifecycle is tied to delegation status transitions

**Statement:** The system SHALL transition delegation status through `running` → `completed`|`error` based on PTY process lifecycle events: active → exit. (Note: PTY delegations start directly at `"running"` — unlike SDK delegations which start at `"dispatched"` and transition to `"running"` on prompt resolution.)

**Acceptance Criterion:** Given a PTY delegation, when the PTY process is spawned, status SHALL be `"running"`. When the PTY process exits with code 0, status SHALL transition to `"completed"`. When the PTY process exits with non-zero code, status SHALL transition to `"error"` with `delegation.error` containing the exit code. When the PTY session disappears before exit, status SHALL transition to `"error"` with message `[Harness] PTY session disappeared before completion`.

**Priority:** P0

**Affected Files:** `src/lib/command-delegation.ts`

**OMO Reference:** OMO `background-agent/background-manager` — lifecycle state machine

---

### R-PTY-04: SDK delegations continue using client.session.prompt()

**Statement:** The system SHALL execute SDK (agent) delegations through `client.session.prompt()` as the canonical execution path. PTY SHALL NOT be used for SDK delegations.

**Acceptance Criterion:** Given a `DelegateParams` with a valid agent name, when `dispatch()` is called, then the delegation record SHALL have `executionMode: "sdk"` and the child session SHALL receive the prompt via `client.session.prompt()`. The PTY path SHALL NOT be invoked for this delegation type.

**Priority:** P0 (by-design, per D-04A — locked)

**Affected Files:** `src/lib/delegation-manager.ts`, `src/lib/sdk-delegation.ts`

**OMO Reference:** N/A — diverges from OMO's PTY-for-all model; SDK path is harness-specific

---

### R-PTY-05: PTY execution failures propagate to delegation result with exit code and stderr

**Statement:** The system SHALL propagate PTY execution failures to the delegation result, including the process exit code and any captured stderr output in the `error` field.

**Acceptance Criterion:** Given a PTY delegation that exits with code 1 and stderr output "permission denied", when `finalizeCommandDelegation()` runs, then `delegation.status` SHALL be `"error"`, `delegation.error` SHALL contain `"[Harness] Command exited with code 1"`, and `delegation.result` SHALL contain the combined stdout + stderr output including "permission denied".

**Priority:** P1

## Section 2: Terminal Lifecycle with Grace Period (Bug #2292 Equivalent)

This section addresses the delegation memory management gap in `handleSessionDeleted()` (line 154–167 of `delegation-manager.ts`). While OMO Bug #2292 had aggressive cleanup that made results unretrievable, the harness has the **inverse problem**: delegations are **never cleaned** from the `delegations` Map after session deletion. `cleanupTracking()` only removes the `delegationsBySession` lookup entry — the delegation record persists in memory indefinitely until process restart. This is a memory leak that grows with every delegation. The grace period adds **scheduled cleanup** with a retrieval window.

### R-LC-01: handleSessionDeleted() defers cleanup by a configurable grace period

**Statement:** The system SHALL schedule deferred in-memory cleanup of terminal delegations using TWO complementary mechanisms:

1. **Per-delegation grace timer** (primary): A configurable grace period (default: 10 minutes, controlled by constant `TASK_CLEANUP_DELAY_MS`). When a delegation enters terminal state, a timer is set. After the timer fires, the delegation record is removed from the `delegations` Map.
2. **Threshold-based batch pruning** (secondary safety net): When the `delegations` Map exceeds `MAX_DELEGATIONS_BEFORE_PRUNE` (default: 50) entries, `pruneCompletedDelegations()` is called to batch-remove delegations older than `DEFAULT_PRUNE_MAX_AGE_MS` (default: 30 minutes). This prevents unbounded growth even if individual grace timers fail to fire (e.g., process crash before timer fires).

Both mechanisms clean from the `delegations` Map only. Neither touches the persistence file. During the grace period, the delegation SHALL remain retrievable via `getStatus()` and `delegation-status` tool.

**Acceptance Criterion:** Given a delegation that enters terminal state (completed, error, timeout, or deleted), when `transitionToTerminal()` schedules grace-period cleanup, then (1) a timer SHALL be set for `TASK_CLEANUP_DELAY_MS` milliseconds, (2) the delegation record SHALL remain in the `delegations` Map during this period, (3) `getStatus(delegationId)` SHALL return the delegation during the grace period, and (4) after the timer fires, the delegation SHALL be removed from `this.delegations` Map. Additionally, when the `delegations` Map exceeds `MAX_DELEGATIONS_BEFORE_PRUNE` entries during `persistAllDelegations()`, `pruneCompletedDelegations()` SHALL remove delegations older than `DEFAULT_PRUNE_MAX_AGE_MS`. An integration test SHALL verify retrieval at t=0 and verify the record is gone after the grace period expires (using fake timers). A separate test SHALL verify that batch pruning fires when the Map exceeds the threshold.

**Priority:** P0

**Affected Files:** `src/lib/delegation-manager.ts`, `src/lib/types.ts`

**OMO Reference:** OMO Bug #2292 fix — `TASK_CLEANUP_DELAY_MS` deferred cleanup; `background-agent/background-manager.ts` grace period pattern

---

### R-LC-02: During grace period, delegation results are retrievable via delegation-status tool

**Statement:** The system SHALL allow the `delegation-status` tool to return delegation results for delegations that are in a terminal state but still within the grace period.

**Acceptance Criterion:** Given a delegation that entered terminal state via `handleSessionDeleted()` within the last `TASK_CLEANUP_DELAY_MS` milliseconds, when the `delegation-status` tool is called with the delegation ID, then it SHALL return the full delegation record including `status`, `result` (if any), and `error`. The response SHALL include a `gracePeriodExpiresAt` timestamp.

**Priority:** P0

**Affected Files:** `src/lib/delegation-manager.ts`, `src/tools/delegation-status.ts`, `src/lib/types.ts`

**OMO Reference:** OMO `background-agent/background-manager.ts` — result retrieval during grace window

---

### R-LC-03: After grace period expires, delegation record is cleaned from in-memory tracking

**Statement:** The system SHALL remove the delegation record from the `delegations` Map after the grace period expires. This happens via two paths: (1) the per-delegation grace timer (R-LC-01) removes individual delegations after `TASK_CLEANUP_DELAY_MS`, and (2) the threshold-based batch pruner (`pruneCompletedDelegations()`) removes delegations older than `DEFAULT_PRUNE_MAX_AGE_MS` when the Map exceeds `MAX_DELEGATIONS_BEFORE_PRUNE`. Both paths clean from the `delegations` Map only. The persisted delegation file SHALL NOT be modified by either cleanup mechanism — it retains the terminal-state record for audit.

**Acceptance Criterion:** Given a delegation that entered terminal state and whose grace period has expired, when the grace timer fires, then (1) the delegation SHALL be removed from `this.delegations` Map, (2) all timers (safety, stability, command poll) SHALL be cleared, and (3) `persistAllDelegations()` SHALL NOT be called during grace-period cleanup (the persisted file already has the terminal record from the original state transition). Additionally, `pruneCompletedDelegations()` SHALL remove delegations in terminal state that are older than `DEFAULT_PRUNE_MAX_AGE_MS` when the Map size exceeds `MAX_DELEGATIONS_BEFORE_PRUNE`, without calling `persistAllDelegations()`.

**Priority:** P1

**Affected Files:** `src/lib/delegation-manager.ts`

**OMO Reference:** OMO `background-agent/background-manager.ts` — post-grace cleanup

---

### R-LC-04: Delegation persistence file is NOT deleted on session deletion

**Statement:** The system SHALL NOT delete or truncate the delegation persistence file when a session is deleted. Only in-memory lookup Maps (`delegations`, `delegationsBySession`) are subject to cleanup.

**Acceptance Criterion:** Given a persisted delegation file at the standard path, when `handleSessionDeleted()` runs for any delegation, then the file SHALL still exist and contain all delegation records (including the newly terminal one). A test SHALL verify the file contents before and after `handleSessionDeleted()`.

**Priority:** P1

**Affected Files:** `src/lib/delegation-manager.ts`, `src/lib/delegation-persistence.ts`

**OMO Reference:** OMO disk persistence model — file retains records independently of in-memory state

---

### R-LC-05: All terminal paths converge to a single transitionToTerminal() method

**Statement:** The system SHALL route all delegation terminal-state transitions (completed, error, timeout, deleted) through a single private `transitionToTerminal()` method that: sets status, records completedAt, clears all timers, persists, triggers parent notification (R-NOTIF-01), and schedules grace-period cleanup (R-LC-01).

> **Architectural Note — Callback Reference Required:** Currently, `command-delegation.ts` (`finalizeCommandDelegation()`) and `sdk-delegation.ts` (`finalizeSdkDelegation()`) handle terminal transitions locally — they set status, persist, and clean up independently. To route these through `transitionToTerminal()` on `DelegationManager`, both handlers will need a **callback reference** (e.g., passed via constructor or method parameter) to `DelegationManager.transitionToTerminal()`. This is a significant architectural change: the handlers currently have no reference back to their parent `DelegationManager`. The implementation must introduce this callback parameter without creating circular dependencies. The handler constructors (or their factory methods in `DelegationManager`) must receive the callback at wiring time.

**Acceptance Criterion:** Given any of the following events: (1) SDK stability poll confirms completion, (2) PTY process exits, (3) headless process exits, (4) safety ceiling reached, (5) session deleted, (6) prompt dispatch fails — then all SHALL call `transitionToTerminal(delegationId, newState, error?)` which handles the full terminal lifecycle. No terminal transition SHALL bypass this method. A test SHALL instrument `transitionToTerminal()` and verify it is called for all 6 event types.

**Priority:** P0

**Affected Files:** `src/lib/delegation-manager.ts`, `src/lib/command-delegation.ts`, `src/lib/sdk-delegation.ts`

**OMO Reference:** OMO `background-agent/background-manager.ts` — unified terminal path

## Section 3: Parent Notification on All Terminal Paths

Currently, when a delegation reaches a terminal state, the parent session is never notified. The parent only discovers the result by actively polling via `delegation-status`. This section adds proactive notification.

### R-NOTIF-01: Parent session is notified when a delegation reaches any terminal state

**Statement:** The system SHALL notify the parent session when a delegation reaches any terminal state (`completed`, `error`, `timeout`). The notification SHALL be delivered through `transitionToTerminal()` (R-LC-05) so that no terminal path can bypass it.

**Acceptance Criterion:** Given a delegation that transitions to terminal state, when `transitionToTerminal()` executes, then the parent session SHALL receive a notification containing the delegation ID and terminal state. For SDK delegations, this SHALL use `client.session.prompt()` on the parent session with a structured notification message. For command delegations, the notification SHALL be delivered if the parent session ID is available. A test SHALL verify that `transitionToTerminal()` calls the notification function for states: completed, error, timeout.

**Priority:** P1

**Affected Files:** `src/lib/delegation-manager.ts`, `src/lib/notification-handler.ts`

**OMO Reference:** OMO `hooks/background-notification` — proactive parent notification on completion

---

### R-NOTIF-02: Notification includes taskId, terminal state, and result summary

**Statement:** The system SHALL include the following fields in every parent notification: `taskId` (delegation ID), `terminalState` (completed/error/timeout), `resultSummary` (first 500 chars of result or error message), and `duration` (ms since creation).

**Acceptance Criterion:** Given a completed delegation with a 2000-character result, when the parent notification is delivered, then it SHALL contain the delegation ID, the string "completed", the first 500 characters of the result, and a duration in milliseconds. The notification payload SHALL be a JSON string parseable by the receiving agent.

**Priority:** P1

**Affected Files:** `src/lib/delegation-manager.ts`, `src/lib/notification-handler.ts`

**OMO Reference:** OMO `background-notification` — structured notification payload

---

### R-NOTIF-03: Notification delivery failure does not block delegation state transition

**Statement:** The system SHALL treat notification delivery as best-effort. If notification fails (session not found, network error, SDK error), the failure SHALL be logged with `[Harness]` prefix but SHALL NOT prevent the delegation from transitioning to terminal state or prevent grace-period scheduling.

**Acceptance Criterion:** Given a delegation that completes successfully, when the notification call throws an error, then (1) the error SHALL be caught and logged, (2) `transitionToTerminal()` SHALL complete normally, (3) the delegation state SHALL be persisted as `"completed"`, and (4) grace-period cleanup SHALL be scheduled. A test SHALL mock the notification to throw and verify the delegation still reaches terminal state.

**Priority:** P0

**Affected Files:** `src/lib/delegation-manager.ts`, `src/lib/notification-handler.ts`

**OMO Reference:** OMO `background-notification` — fire-and-forget notification semantics

---

### R-NOTIF-04: Notification is delivered via direct SDK call with fire-and-forget semantics

**Statement:** The system SHALL deliver parent notifications by calling `client.session.prompt()` directly on the parent session, wrapped in try/catch with fire-and-forget semantics. This approach is required because OpenCode's plugin API allows **subscribing** to hooks (e.g., `session.created`, `session.idle`, `session.deleted`) but does **NOT support emitting custom events** from within a plugin. The existing deprecated `notification-handler.ts` module already implements this pattern (direct `client.session.prompt()` call) — it will be re-activated as the canonical notification path.

> **Platform Constraint Note:** OpenCode's hook system is event-receiving only (`plugin.hook(event, handler)`). Plugins cannot emit custom events into the pipeline. The only viable notification path is direct SDK calls. This diverges from OMO's hook-based notification architecture, which assumes a full event bus.

**Acceptance Criterion:** Given a terminal delegation, when notification is triggered from `transitionToTerminal()`, then the system SHALL call `client.session.prompt()` on the parent session with a structured notification message, wrapped in try/catch. The notification function SHALL NOT hold a long-lived reference to the parent session's client beyond the immediate call. A test SHALL verify that the notification mechanism uses `client.session.prompt()` (not event emission) and that the try/catch pattern prevents notification failures from propagating.

**Priority:** P2

**Affected Files:** `src/lib/delegation-manager.ts`, `src/lib/notification-handler.ts`

**OMO Reference:** OMO `hooks/background-notification` — hook-based notification architecture (diverges: harness uses direct SDK call due to platform constraints); parallel research finding on callback-chain leaks (mitigated by fire-and-forget pattern)

## Section 4: Stability and Polling Improvements

The current SDK stability polling uses a fixed 3-second interval (`STABILITY_POLL_INTERVAL_MS`) with a fixed threshold of 3 polls (`STABILITY_THRESHOLD`). OMO uses adaptive intervals that slow down as the child becomes less active, reducing polling overhead for long-running delegations.

### R-POLL-01: Completion polling uses adaptive intervals based on child activity

**Statement:** The system SHALL use adaptive polling intervals for SDK delegation stability checks, following a tiered model:
- **Active** (message count changing): 2 seconds
- **Base** (message count stable < 30s): 5 seconds
- **Idle** (message count stable 30s–5min): 10 seconds
- **Deep idle** (message count stable > 5min): 30 seconds

The interval SHALL be calculated based on the time since the last observed message count change.

**Acceptance Criterion:** Given an SDK delegation, when the stability poll runs:
- If `delegation.lastMessageCount` changed in the last poll, the next poll SHALL be scheduled in 2000ms.
- If `delegation.lastMessageCount` has been unchanged for < 30 seconds, the next poll SHALL be scheduled in 5000ms.
- If unchanged for 30–300 seconds, next poll in 10000ms.
- If unchanged for > 300 seconds, next poll in 30000ms.
A test SHALL verify that 5 consecutive polls with unchanged message count produce increasing intervals.

**Priority:** P1

**Affected Files:** `src/lib/sdk-delegation.ts`, `src/lib/types.ts`

**OMO Reference:** OMO `background-agent/task-poller` — adaptive polling with activity-based intervals

---

### R-POLL-02: Fast completions are deferred before declaring stable

**Statement:** The system SHALL defer declaring a delegation stable if the delegation has been running for less than `MIN_IDLE_TIME_MS` (default: 5000ms). This prevents false-positive completion for tasks that emit a quick response and then go idle before producing more output.

**Acceptance Criterion:** Given an SDK delegation that has been running for 3 seconds and has a stable message count for `STABILITY_THRESHOLD` consecutive polls, when the stability check runs, then the system SHALL NOT finalize the delegation because `Date.now() - delegation.createdAt < MIN_IDLE_TIME_MS`. The system SHALL continue polling. Once the delegation age exceeds `MIN_IDLE_TIME_MS` and stability threshold is met, finalization SHALL proceed.

**Priority:** P1

**Affected Files:** `src/lib/sdk-delegation.ts`, `src/lib/types.ts`

**OMO Reference:** OMO `background-agent/task-poller` — MIN_IDLE_TIME_MS deferral pattern

---

### R-POLL-03: Stale session detection uses configurable timeout

**Statement:** The system SHALL detect stale delegations (running but no activity detected) using a configurable timeout `DEFAULT_STALE_TIMEOUT_MS` (default: 45 minutes). When a delegation has been running for longer than this timeout with no observed activity (no message count changes in any poll), the system SHALL transition it to `"timeout"` state.

**Acceptance Criterion:** Given an SDK delegation that has been running for 46 minutes with no message count change in any stability poll, when the next stability poll runs, then the delegation SHALL transition to `"timeout"` with error message containing `"stale session"`. The stale timeout SHALL be independent of `safetyCeilingMs` (which is a hard ceiling, not activity-based).

**Priority:** P2

**Affected Files:** `src/lib/sdk-delegation.ts`, `src/lib/types.ts`

**OMO Reference:** OMO `background-agent/background-manager` — DEFAULT_STALE_TIMEOUT_MS

---

### R-POLL-04: Stability requires MIN_STABILITY_TIME_MS AND STABLE_POLLS_REQUIRED consecutive polls

**Statement:** The system SHALL require both a minimum elapsed time since last message change (`MIN_STABILITY_TIME_MS`, default: 10000ms) AND a minimum number of consecutive stable polls (`STABLE_POLLS_REQUIRED`, default: 3) with unchanged message count before declaring completion.

**Acceptance Criterion:** Given an SDK delegation with `MIN_STABILITY_TIME_MS = 10000` and `STABLE_POLLS_REQUIRED = 3`, when the stability poll runs:
- If only 2 consecutive stable polls have occurred (even if > 10s since last change), finalization SHALL NOT proceed.
- If 3 consecutive stable polls have occurred but < 10s since last message change, finalization SHALL NOT proceed.
- If 3 consecutive stable polls have occurred AND > 10s since last message change, finalization SHALL proceed.

**Priority:** P1

**Affected Files:** `src/lib/sdk-delegation.ts`, `src/lib/types.ts`

**OMO Reference:** OMO `background-agent/task-poller` — dual stability gate (time + poll count)

## Section 5: Nesting Depth Limits

Currently `MAX_DELEGATION_DEPTH = 1` is defined in `types.ts` (line 405) but there is no enforcement of this limit anywhere in the delegation dispatch path. A child session could itself delegate, creating recursive delegation chains.

### R-NEST-01: Delegation nesting depth is tracked and enforced

**Statement:** The system SHALL track the delegation nesting depth for each delegation and enforce a maximum depth. The depth SHALL be computed from the delegation chain: a top-level delegation has depth 1, a delegation spawned by a depth-1 child has depth 2, etc.

**Acceptance Criterion:** Given a parent session that creates a delegation, when `dispatch()` is called, then the system SHALL check the current nesting depth of the calling context. The nesting depth SHALL be stored on the `Delegation` record as `nestingDepth: number`. A test SHALL verify that depth 1 delegations are allowed, depth 2 delegations are allowed (up to max), and depth N+1 delegations (where N = max depth) are rejected.

**Priority:** P1

**Affected Files:** `src/lib/delegation-manager.ts`, `src/lib/types.ts`, `src/tools/delegate-task.ts`

**OMO Reference:** OMO `background-agent/background-manager` — nesting depth tracking

---

### R-NEST-02: Default max nesting depth is 3, configurable via RuntimePolicy

**Statement:** The system SHALL enforce a default maximum nesting depth of 3. This limit SHALL be overridable via `RuntimePolicy` when loaded through the existing `runtime-policy.ts` module.

**Acceptance Criterion:** Given the default configuration, when a depth-4 delegation is attempted, then it SHALL fail with error `[Harness] Maximum delegation nesting depth (3) exceeded`. Given a RuntimePolicy that sets `maxDelegationDepth: 5`, when a depth-5 delegation is attempted, then it SHALL succeed. The constant `MAX_DELEGATION_DEPTH` in `types.ts` SHALL be updated from 1 to 3.

**Priority:** P1

**Affected Files:** `src/lib/types.ts`, `src/lib/delegation-manager.ts`, `src/lib/runtime-policy.ts`

**OMO Reference:** OMO `background-agent/spawner` — configurable depth limits

---

### R-NEST-03: Delegation attempts exceeding max depth fail fast with descriptive error

**Statement:** The system SHALL reject delegation attempts that would exceed the maximum nesting depth immediately (fail fast) before acquiring the concurrency semaphore or creating any child session.

**Acceptance Criterion:** Given a session at the maximum nesting depth, when `dispatch()` or `dispatchCommand()` is called, then (1) the method SHALL throw an error before calling `semaphore.acquire()`, (2) no child session SHALL be created, (3) no delegation record SHALL be registered, and (4) the error message SHALL include the current depth, the maximum, and suggest using the result retrieval pattern instead. A test SHALL verify zero side effects on the delegation state when over-depth dispatch is attempted.

**Priority:** P0

**Affected Files:** `src/lib/delegation-manager.ts`, `src/tools/delegate-task.ts`

**OMO Reference:** OMO `background-agent/background-manager` — early rejection pattern

## Section 6: Error Surface and Observability
