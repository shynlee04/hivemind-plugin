# Phase 16.2 Plan 01: PTY Execution Wiring + OMO Safety Patterns

**Goal:** Close two critical gaps identified in Phase 16 verification: (1) verify PTY execution wiring for command delegations is complete and add unified terminal lifecycle, and (2) backfill OMO-proven safety patterns — grace periods for memory cleanup, parent notifications, adaptive polling, and nesting depth limits.

**Baseline:** Phase 16 post-Wave 6, 462 tests passing, build clean. DelegationManager (348 LOC) delegates to CommandDelegationHandler (285 LOC) and SdkDelegationHandler (161 LOC). PTY infrastructure is wired for command delegations. Delegations accumulate in memory indefinitely. No parent notifications. Fixed-interval polling. No nesting limits.

**Prerequisites (blocking):**
- Fix 3 typecheck errors from partial merge: `MAX_DELEGATIONS_BEFORE_PRUNE`, `DEFAULT_PRUNE_MAX_AGE_MS`, `DelegationStatus` import.

## Wave 3: Safety — Nesting + Grace Period

**Duration estimate:** 25 min
**Files:** `src/lib/delegation-manager.ts`, `src/tools/delegate-task.ts`, `src/lib/types.ts`

### Tasks

1. **Compute and store `nestingDepth` on each delegation**
   - Derive depth from parent session's delegation chain
   - Store `nestingDepth: number` on `Delegation` record at creation

2. **Enforce max nesting depth (default 3, RuntimePolicy override)**
   - Check depth BEFORE `semaphore.acquire()` in both `dispatch()` and `dispatchCommand()`
   - Fail fast with descriptive error: `[Harness] Maximum delegation nesting depth (3) exceeded`
   - Read override from `RuntimePolicy.maxDelegationDepth` if available

3. **Implement per-delegation grace period timer**
   - In `transitionToTerminal()`: set timer for `TASK_CLEANUP_DELAY_MS`
   - After timer fires: remove from `this.delegations` Map (only)
   - During grace period: `getStatus()` returns delegation normally

4. **Implement threshold-based batch pruning**
   - In `persistAllDelegations()`: if `this.delegations.size > MAX_DELEGATIONS_BEFORE_PRUNE`
   - Call `pruneCompletedDelegations()` to remove terminal delegations older than `DEFAULT_PRUNE_MAX_AGE_MS`
   - Does NOT call `persistAllDelegations()` during pruning

5. **Verify persistence file is never deleted**
   - `handleSessionDeleted()` persists terminal state before cleanup
   - Grace-period cleanup removes from Map only, never touches file

**Verification:**
- Integration test: retrieval at t=0, record gone after grace period (fake timers).
- Integration test: batch pruning fires when Map exceeds threshold.
- Test: file contents unchanged after `handleSessionDeleted()`.

## Wave 5: Polling Improvements

**Duration estimate:** 25 min
**Files:** `src/lib/sdk-delegation.ts`, `src/lib/types.ts`

### Tasks

1. **Replace fixed interval with adaptive polling**
   - Calculate interval based on time since last message count change:
     - Active (changed in last poll): 2000ms
     - Base (stable < 30s): 5000ms
     - Idle (stable 30s–5min): 10000ms
     - Deep idle (stable > 5min): 30000ms

2. **Add fast-completion deferral (R-POLL-02)**
   - Before declaring stable, check `Date.now() - delegation.createdAt >= MIN_IDLE_TIME_MS`
   - If running < 5s, continue polling even if stability threshold met

3. **Add stale session detection (R-POLL-03)**
   - Track time since last message count change
   - If running > `DEFAULT_STALE_TIMEOUT_MS` (45 min) with no activity → transition to "timeout"

4. **Add dual stability gate (R-POLL-04)**
   - Require BOTH `MIN_STABILITY_TIME_MS` (10s) AND `STABLE_POLLS_REQUIRED` (3) consecutive polls
   - Neither gate alone is sufficient

5. **Track `lastMessageCountChangeAt` timestamp**
   - Update on each poll when message count changes
   - Used for interval calculation and stale detection

**Verification:**
- Test: 5 consecutive polls with unchanged count produce increasing intervals.
- Test: delegation at 3s with stable polls is NOT finalized (MIN_IDLE_TIME_MS).
- Test: 46-minute idle delegation transitions to timeout.
- Test: 3 stable polls but < 10s does NOT finalize; 3 stable polls + > 10s DOES finalize.
