## Summary

Phase 48.2 should prioritize four directly exploitable boundaries before broader policy cleanup: headless command environment, PTY ownership, delegation-status ownership, and persistence redaction. [VERIFIED: codebase reads listed in Sources] The highest-risk issue is the headless fallback in `CommandDelegationHandler`, which intentionally has a minimal env helper but bypasses it by merging the full parent `process.env` into child processes. [VERIFIED: `src/lib/command-delegation.ts:166-169`, `src/lib/command-delegation.ts:360-371`]

The current runtime has no caller-scoped authorization path for reading or controlling PTY sessions and no caller-scoped authorization path for delegation-status reads. [VERIFIED: `src/tools/run-background-command.ts:102-120`, `src/tools/delegation-status.ts:79-115`] Both tools have access to caller session context but either only use it for spawning (`run-background-command`) or ignore it entirely (`delegation-status`). [VERIFIED: `src/tools/run-background-command.ts:48-51`, `src/tools/run-background-command.ts:81-97`, `src/tools/delegation-status.ts:17`, `src/tools/delegation-status.ts:79`]

The category system currently exists mostly as metadata and queue-key input, not as an enforceable authorization boundary. [VERIFIED: `src/lib/types.ts:28-40`, `src/lib/delegation-manager.ts:130-138`, `src/lib/delegation-manager.ts:515-530`] Runtime policy loading handles concurrency, budgets, and a trusted-runtime boolean; it does not define or enforce category/tool/agent restrictions. [VERIFIED: `src/lib/runtime-policy.ts:28-41`, `src/lib/runtime-policy.ts:103-138`, `src/lib/runtime-policy.ts:157-205`]

**Primary recommendation:** Implement a shared security-boundary layer in small steps: (1) env allowlist for all command execution, (2) lineage-scoped ownership checks for PTY and delegation records, (3) redaction helpers at all persistence/output seams, and (4) category gate policy with auditable ask records. ## Gap 1 — CRITICAL Headless Env Leak

### Evidence

```typescript

const child = spawnHeadlessProcess(params.command, params.args ?? [], {
  cwd: workingDirectory,
  env: { ...process.env, ...this.buildMinimalEnv(params.env) },
  stdio: ["pipe", "pipe", "pipe"],
})
```

The PTY path passes only `this.buildMinimalEnv(params.env)` to `ptyManager.spawn`. [VERIFIED: `src/lib/command-delegation.ts:75-88`] The headless fallback passes `{ ...process.env, ...this.buildMinimalEnv(params.env) }`, which exposes all environment variables from the parent process to the child. [VERIFIED: `src/lib/command-delegation.ts:166-169`] The helper is already designed to allow only `PATH`, `HOME`, `TERM`, `LANG`, and `PWD`, plus caller-provided `params.env`. [VERIFIED: `src/lib/command-delegation.ts:360-371`]

- Headless fallback inherits full parent process environment, including likely secrets such as tokens, API keys, CI credentials, and cloud credentials when present. [VERIFIED: `src/lib/command-delegation.ts:166-169`; secret names are ASSUMED examples]
- The code has no redaction or asklist pass at spawn time. [VERIFIED: no redaction helper found by codebase grep for `secret|redact|sanitize|mask|token|password|api.key|API_KEY|credential` in `src` except scalar sanitizers and schema text]

### Minimal Fix Direction

Replace line 168 with `env: this.buildMinimalEnv(params.env)`. [ASSUMED] This aligns headless fallback with the PTY path and preserves explicit caller-provided `params.env`. [VERIFIED: `src/lib/command-delegation.ts:81`, `src/lib/command-delegation.ts:168`, `src/lib/command-delegation.ts:360-371`]

## Common Pitfalls

### Pitfall 1: Fixing PTY but leaving headless fallback unsafe

**What goes wrong:** PTY path remains safe while fallback leaks full env. [VERIFIED: `src/lib/command-delegation.ts:75-88`, `src/lib/command-delegation.ts:166-169`]  
**How to avoid:** Test the fallback path explicitly with PTY unavailable. [ASSUMED]

### Pitfall 2: Scoping in-memory delegations only

**What goes wrong:** Unauthorized users can still read persisted delegations after in-memory cleanup. [VERIFIED: `src/tools/delegation-status.ts:83-87`, `src/tools/delegation-status.ts:101-115`]  
**How to avoid:** Apply ownership filtering to both `delegationManager.getAllDelegations()` and `readPersistedDelegations()`. [ASSUMED]

### Pitfall 3: Treating truncation as redaction

**What goes wrong:** Secrets in the first 240 or 2,000 characters still persist. [VERIFIED: `src/plugin.ts:37-42`, `src/lib/event-tracker/writer.ts:132-142`, `src/lib/event-tracker/writer.ts:346-348`]  
**How to avoid:** Redact before truncation/serialization. [ASSUMED]

### Pitfall 4: Category gate that broadens primitive permissions

**What goes wrong:** Harness gates accidentally allow tools denied by OpenCode primitive metadata. [VERIFIED: primitive metadata is parsed in `src/lib/spawner/agent-primitive-policy.ts:14-28` and used in `src/lib/spawner/spawn-request-builder.ts:70-95`]  
**How to avoid:** Category gates must only narrow the resolved permission profile. [ASSUMED]
