## Intent
Port the doc intelligence engine from product-detox. The engine provides remark-based markdown AST parsing with outline extraction, frontmatter reading, heading hierarchy traversal, token-aware chunking, and a doc surface router. The `hivemind_doc` tool exposes 5 actions: skim, skim_directory, read, chunk, search. This provides structured document reading capabilities that downstream phases (Phase 56 trajectory evidence, Phase 57 control-plane decisions) depend on.

## Scope
- `lib/doc-intelligence/parser.ts` — remark markdown AST parser with outline, frontmatter, and heading hierarchy extraction
- `lib/doc-intelligence/chunker.ts` — token-aware chunking for structured document segmentation
- `lib/doc-intelligence/router.ts` — doc surface router for dispatching read/skim/search operations
- `lib/doc-intelligence/types.ts` — doc intelligence types: outline nodes, chunk schemas, search results
- Tools: `hivemind_doc` — 5 actions: `skim`, `skim_directory`, `read`, `chunk`, `search`
- Dependencies: remark, unist-util-visit, fast-glob, ignore
- Integration with Phase 56 trajectory (trajectory events reference doc intelligence for evidence)
- Integration with Phase 57 control-plane (control-plane detect() may query doc surfaces)

## Non-Goals
- Do not replace existing `prompt-skim` or `prompt-analyze` tools — doc intelligence is a document-level reader, not a prompt analysis tool
- Do not build a full-text search engine — search action uses heading/outline matching, not inverted index
- Do not implement the sidecar document viewer (Phase 42 sidecar reads .hivemind/ artifacts; doc intelligence provides parsing, not rendering)
- Do not modify existing session-journal or continuity modules — doc intelligence is a standalone read-layer

## Assumptions
- Doc intelligence is a read-only tool layer — it reads files but never writes to harness state
- Remark AST parsing is synchronous for single files; async for directory skims with fast-glob
- Token-aware chunking uses a configurable token limit (default matching model context windows)
- The doc surface router dispatches to the correct parser based on file extension (.md, .mdx, .txt)
- Dependencies (remark, unist-util-visit, fast-glob, ignore) are already available or can be added without version conflicts

## Success Criteria
1. A planner can distinguish doc intelligence (structured document reading) from prompt-skim (prompt metadata extraction) and prompt-analyze (prompt quality assessment)
2. The `hivemind_doc` tool's 5 actions are contract-defined with input/output schemas
3. The remark-based parser extracts outline, frontmatter, heading hierarchy from markdown files
4. Token-aware chunking produces segments within configurable token limits
5. Downstream dependency on Phase 56 (trajectory evidence references) is explicitly documented

## Downstream Research Questions
1. What remark plugin chain does product-detox use, and are all plugins available on npm?
2. How does token-aware chunking handle code blocks, tables, and nested list structures?
3. What file extensions should the doc surface router support beyond .md and .mdx?
4. How should doc intelligence integrate with the existing tool response envelope?
5. What is the performance budget for directory skim operations (fast-glob + remark parse per file)?
