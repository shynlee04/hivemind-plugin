# Phase 48.4: Production Evidence & Coverage Recovery — Research

**Researched:** 2026-04-27
**Domain:** Test evidence quality, coverage gap analysis, mock-to-behavioral migration patterns for TypeScript OpenCode plugin
**Confidence:** HIGH

## Summary

Phase 48.4 addresses the remaining evidence debt from the Phase 48 runtime integration: two key requirements (REM-RUNTIME-04: dynamic tool execution, REM-RUNTIME-05: successful real child delegation) remain unproven, and the test suite contains 982 passing tests, but many are mock-heavy existence assertions rather than behavioral proofs. This research identifies exactly which tests are mock-based, which modules lack coverage entirely, and provides a prescriptive upgrade path from mock-only assertions to behavioral integration tests that exercise real code paths without requiring a live OpenCode runtime.

**Primary recommendation:** Add dedicated behavioral tests for 8 untested modules (1,601 lines of untested runtime code), upgrade the 7 most mock-dependent test files with fake-client patterns that exercise real code paths, and harden coverage with vitest thresholds. Classify the two remaining live-OpenCode requirements (REM-RUNTIME-04/05) as deferred external-environment limitations with documented manual verification procedures.

### Pattern 1: Fake Client Pattern (Recommended Upgrade)
**What:** Instead of `vi.fn()` mocks that return hardcoded values, create a `createFakeClient()` factory that maintains internal state and responds realistically.

**When to use:** For any test currently using `createMockClient()` with `vi.fn()`. The fake maintains a `Map<string, SessionState>` internally so SDK calls like `session.create()`, `session.status()`, and `session.messages()` produce consistent, stateful responses.

**Example:**
```typescript
// Source: [PROJECT PATTERN — derived from existing continuity.test.ts patterns]
function createFakeClient() {
  const sessions = new Map<string, { status: string; messages: unknown[] }>()

  return {
    session: {
      create: async (params: unknown) => {
        const id = `fake-${sessions.size + 1}`
        sessions.set(id, { status: "running", messages: [] })
        return { data: { id } }
      },
      messages: async (params: { sessionID: string }) => {
        const s = sessions.get(params.sessionID)
        return { data: s?.messages ?? [] }
      },
      status: async (params: { sessionID: string }) => {
        const s = sessions.get(params.sessionID)
        return { data: s ? { status: s.status } : {} }
      },
      // ... etc
    },
  }
}
```

### Pattern 2: Event Simulator Pattern (for Hook Tests)
**What:** Simulate OpenCode lifecycle events by calling hook functions directly with fake event payloads, then asserting state changes.

**When to use:** For testing `createCoreHooks`, `createSessionHooks`, and `createToolGuardHooks`.

**Example:**
```typescript
// Source: [PROJECT PATTERN — derived from plugin-lifecycle.test.ts]
const hooks = createCoreHooks({ client: fakeClient, lifecycleManager, stateManager })

// Simulate session creation
await hooks.event({ event: { type: "session.created", sessionID: "ses_test" } })

// Assert lifecycle manager received the event
expect(lifecycleManager.handleEvent).toHaveBeenCalledWith(
  expect.objectContaining({ eventType: "session.created", sessionID: "ses_test" })
)

// Assert shell.env hook sets CI=true
// (already tested in hook-cqrs-boundary but test the actual factory output)
```

### Pattern 3: Contract-Based Tool Verification
**What:** Test tool factories by calling them with valid and invalid input shapes, verifying Zod validation and return shape contracts.

**When to use:** For any tool factory (delegate-task, delegation-status, etc.) — verify Zod schemas reject invalid inputs and return standard response envelopes.

**Example:**
```typescript
// Source: [PROJECT PATTERN]
const tool = createDelegateTaskTool({ client: fakeClient, delegationManager: fakeManager })

// Valid: should succeed
const result = await tool.execute({ agent: "builder", prompt: "do work" })
expect(result.status).toBe("ok")

// Invalid: missing required field
const badResult = await tool.execute({ agent: "", prompt: "" })
expect(result.status).toBe("error")
```

### Anti-Patterns to Avoid
- **Mock verification theater:** `expect(mockFn).toHaveBeenCalled()` without checking arguments or return values. Always verify what was passed and what happened to state.
- **Mocking what you own:** Don't `vi.mock()` your own modules. Test real integration between them. Only mock external SDKs and native modules.
- **Coverage-only tests:** Don't write tests solely to hit uncovered lines. Each test must assert a behavioral invariant.
- **Unstructured fake clients:** Don't use `vi.fn().mockResolvedValue({})` without internal state. The fake must maintain session state internally.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Coverage reporting | Custom coverage scripts | vitest `coverage.provider: 'v8'` | Native V8 coverage, thresholds, reporters |
| Timer-based test control | `setTimeout` + `await` in tests | `vitest.useFakeTimers()` + `vi.advanceTimersByTime()` | Already used; deterministic, instant |
| Temp file I/O for persistence tests | Mock `fs` module | Real `mkdtempSync` + `rmSync` with temp dirs | Already used in continuity/lifecycle tests; proves real I/O |
| Fake SDK client | Ad-hoc `vi.fn()` per test | Stateful `createFakeClient()` factory | Consistent behavior across tests; catches real integration bugs |
| Event simulation | Multiple `vi.spyOn` chains | Direct hook invocation with typed payloads | Hooks are plain functions; test them as such |

**Key insight:** The existing test infrastructure already has the right patterns (temp dirs for real I/O, fake timers for time-dependent logic). The gap is that these patterns are inconsistently applied — some tests use them, others fall back to mock-only. The upgrade path is standardization, not new tooling.

## Tool & Hook Evidence Strategy

### Approach: Fake Client + Event Simulator + Real State

The core insight: both tools and hooks are **plain functions** that accept dependencies (client, manager, state). The OpenCode SDK `client` is the only external dependency. By replacing it with a stateful fake, we can prove:

1. **Tools execute with correct contracts** — Call tool factories with valid/invalid inputs, verify Zod validation, verify return envelope shape, verify state mutations in delegation manager.
2. **Hooks fire on simulated events** — Call hook factories' `event()`, `shell.env()`, and `tool.execute.before/after` methods directly with typed event payloads. Verify lifecycle manager state changes, verify event tracker artifact creation, verify CQRS boundary enforcement.

### Why This Works Without Live SDK
- **Tools** are thin wrappers: `delegateTask()` → `delegationManager.dispatchDelegation()` → `delegateTask()` returns standard envelope. The fake client ensures the inner delegation flow works.
- **Hooks** are event routers: `createCoreHooks()` returns `{ event: (input) => {...} }`. The function body calls `lifecycleManager.handleEvent()`. With a real lifecycle manager and fake events, we test the router.
- **Editorial judgment:** REM-RUNTIME-04 and REM-RUNTIME-05 (dynamic tool execution, successful real child delegation) require a **live OpenCode runtime**. These are deferred to manual verification with documented procedures.

## Common Pitfalls

### Pitfall 1: Writing Tests That Only Hit Lines Without Asserting Behavior
**What goes wrong:** Adding tests purely for coverage numbers, e.g., calling `inferContinuityStatusFromEvent()` with every status signal but only checking return type, not correctness.
**Why it happens:** Pressure to close coverage gaps fast.
**How to avoid:** Each test must assert a specific behavioral invariant. For `runtime.ts`, each branch must verify the returned status string exactly.
**Warning signs:** Tests that call functions without assertions, or with only `toBeDefined()`/`toBeTypeOf()`.

### Pitfall 2: Fake Client Drift From Real SDK
**What goes wrong:** The `createFakeClient()` diverges from real `OpenCodeClient` types, so tests pass but real integration breaks.
**Why it happens:** Fake client is maintained separately from SDK types.
**How to avoid:** Type the fake as `OpenCodeClient` (or a subset). When SDK types change, compile errors will show.
**Warning signs:** `as never` casts in fake client creation.

### Pitfall 3: Over-Mocking What Should Be Real
**What goes wrong:** Mocking `vi.useFakeTimers()` when real timers would work, or mocking `node:fs` when temp dirs work.
**Why it happens:** Habit — once you start mocking, it's tempting to mock everything.
**How to avoid:** Default to real. Mock only external SDKs and native modules. Test real module integration.

### Pitfall 4: Missing Integration-Level Assertions
**What goes wrong:** Testing `delegationManager.dispatchDelegation()` only checks the return value, not what happened to the internal `delegations` Map, the persistence file, or the continuity store.
**Why it happens:** Focusing on the direct return value rather than side effects.
**How to avoid:** After every mutation call, verify side effects. Check the stateful fake client's internal sessions map. Read the persistence file. Check the continuity store.

### Behavioral Test: runtime.ts event→status mapping
```typescript
// Source: [PROJECT PATTERN — new test]
import { inferContinuityStatusFromEvent } from "../../src/lib/runtime.js"

describe("inferContinuityStatusFromEvent", () => {
  it("returns pending for session.created events", () => {
    const result = inferContinuityStatusFromEvent({
      event: { type: "session.created" },
      eventType: "session.created",
    })
    expect(result).toBe("pending")
  })

  it("returns failed for error status signals", () => {
    const result = inferContinuityStatusFromEvent({
      event: { type: "session.updated", properties: { status: { type: "failed" } } },
      eventType: "session.updated",
    })
    expect(result).toBe("failed")
  })

  it("preserves terminal failure even when idle signal arrives", () => {
    const result = inferContinuityStatusFromEvent({
      event: { type: "session.idle" },
      eventType: "session.idle",
      currentStatus: "failed",
```
