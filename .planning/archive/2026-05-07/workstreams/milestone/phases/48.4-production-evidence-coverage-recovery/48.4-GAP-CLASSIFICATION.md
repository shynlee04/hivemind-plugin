## Deferred Gaps (External-Environment Limitations)

### DFR-01: REM-RUNTIME-04 — Dynamic Tool Execution Proof

**Classification:** External-environment limitation
**Reason:** Proving a tool executes end-to-end with correct contracts requires a running OpenCode instance with provider connectivity. The tool factories can be tested with fake clients (Plans 02-04 verify input validation, return envelope shape, and side effects), but the final SDK→OpenCode boundary cannot be tested without a live runtime.

**What IS proven (with fake client):**
- Tool Zod schemas reject invalid inputs (Plan 04 — delegate-task contract tests)
- Tool return envelopes are correctly shaped (Plan 04)
- Tool side effects (delegation creation, status updates) are verified (Plans 03-04)
- Tool guard hooks enforce budget limits and circuit breaker (Plan 03 — 9 tests)
- _harness metadata is injected on tool.execute.after (Plan 03 + Plan 04)

**What is NOT proven (requires live OpenCode):**
- A tool invocation in a real session produces the documented behavior
- Tool output appears in the actual session message stream
- Tool execution respects the OpenCode permission model at runtime

**Manual Verification Procedure:**
1. Start OpenCode in a project with the harness plugin enabled
2. Open a session and invoke a harness tool (e.g., via delegate-task)
3. Verify the tool returns the expected response format
4. Check session messages for the tool call and result
5. Verify the delegation manager recorded the interaction correctly

**Expected Result:** Tool executes, returns structured output, and leaves audit trail in session messages and delegation records.

**Partial Confidence:** The fake client pattern provides ~80% confidence. Only the final SDK integration boundary remains unverified.

### DFR-02: REM-RUNTIME-05 — Successful Real Child Delegation

**Classification:** External-environment limitation
**Reason:** Proving a real child session is spawned with non-empty provider completion requires a running OpenCode instance with an active AI provider (Claude, GPT, etc.). The delegation manager can be tested with stateful fake clients (Plan 04 proves dispatch, stability polling, grace period cleanup, and recovery), but the actual child session lifecycle with real AI output cannot be reproduced in CI.

**What IS proven (with stateful fake / behavioral tests):**
- dispatchDelegation creates session and records delegation (Plan 04)
- Stability polling detects idle→completed transitions (Plan 04)
- Grace period cleanup removes stale delegations (Plan 04)
- Recovery flow hydrates and resumes persisted delegations (Plan 04)
- Notification handler builds correct messages for terminal states (Plan 02 — 18 tests)
- Concurrent dispatch with different queue keys (Plan 04)

**What is NOT proven (requires live OpenCode + provider):**
- A real child session completes with non-empty AI-generated content
- The provider delivers messages that pass through the message-count wrapper
- The completion detector receives a real `<promise>DONE</promise>` signal
- Parent session receives notification with actual child output

**Manual Verification Procedure:**
1. Start OpenCode with provider access configured
2. Trigger a delegated task (e.g., delegate-task with agent "builder")
3. Wait for child session to complete (monitor via delegation-status)
4. Verify child session has messages with non-empty content
5. Verify parent session receives terminal notification with child output
6. Check continuity store for delegation record with completed status

**Expected Result:** Child session spawns, receives AI-generated responses, completes with content, and parent is notified.

**Partial Confidence:** The stateful fake client provides ~75% confidence. Stability polling, grace period cleanup, recovery, and notification building are proven. Only the provider integration boundary remains unverified.
