## Assumptions

- **The tool contract must be end-user discoverable, not only internally correct.** UAT failed on `action: "start"` before discovering `action: "run"`; even if `run` is technically valid, the exposed schema/UX misled the tester. If this assumption is wrong, Phase 49 may over-focus on naming/documentation and miss a pure runtime defect.
- **`run-background-command` should support reliable command execution through explicit executable plus args, and should not imply shell-string support unless that contract is deliberate.** The UAT showed command strings and `bash -c` strings were treated as single executable paths, while `command` plus `args` activated PTY mode. If this assumption is wrong, downstream plans may accidentally require shell parsing that the harness intentionally avoids for safety.
- **PTY output capture must preserve early output from process start.** The long-running command captured lines 2-5 and `COMPLETE`, but lost `Line 1`; end users cannot trust terminal evidence if first output is race-prone. If this assumption is wrong, Phase 49 may target output buffering when the issue is actually read timing or display semantics.
- **Prompt analysis results must align with user-facing clarity expectations, not just heuristic matches.** The UAT found that vague input received a perfect clarity score, and a technical prompt was flagged as contradictory despite complementary requirements. If this assumption is wrong, Phase 49 may change expected scoring behavior that earlier tests implicitly rely on.
- **Journal export filters should filter records rather than mutate returned lineage metadata.** The `pipelineKey` UAT used a nonexistent value and all records were returned with that value stamped into lineage rows. If this assumption is wrong, Phase 49 may reclassify a write/intention parameter as a read filter incorrectly.

## Success Criteria

1. A user can start, observe, send input to, and terminate a background command using documented action names and schema-visible parameters.
2. Command execution either clearly rejects shell strings with actionable guidance or supports them through an explicit, safe contract; it must not report a successful start when the command immediately fails as a malformed executable path.
3. PTY output reads include process output from the beginning of execution, including fast first-line output, with stable offset behavior.
4. Prompt-skim returns risk signals that distinguish simple prompts from scope-bloated/flooding prompts in a way an end-user tester can trust.
5. Prompt-analyze flags vague, underspecified, contradictory, and well-structured prompts with calibrated clarity scores and avoids obvious false positives on complementary technical requirements.
6. Session journal export filters narrow the returned records and do not silently overwrite or stamp lineage metadata in read responses.
7. Phase 49 verification uses real harness tool calls from an OpenCode-like runtime surface where possible, with mock-only results clearly labeled if any remain.

## Downstream Research Questions

1. What is the current intended public contract for `run-background-command` actions and parameters: `run/output/list/input/terminate`, aliases, or a separate accessibility terminal view?
2. Should command strings be intentionally unsupported for safety, or should the tool provide a safe shell-wrapper mode with explicit opt-in semantics?
3. Where is PTY output buffering initialized relative to process spawn, and what contract should guarantee first-output capture for fast commands?
4. How should `sessionId`, internal `ptySessionId`, and delegation ownership be exposed so users can read output without guessing internal IDs?
5. Which prompt-analysis heuristics should count as vagueness, contradiction, flooding, and complementary requirements for user-facing clarity scores?
6. Is `pipelineKey` intended as an export filter, a labeling override, or both? If both, what parameter names separate read filters from metadata stamping?
7. What UAT fixture set should become the Phase 49 regression suite so these end-user tool contract failures do not recur?
