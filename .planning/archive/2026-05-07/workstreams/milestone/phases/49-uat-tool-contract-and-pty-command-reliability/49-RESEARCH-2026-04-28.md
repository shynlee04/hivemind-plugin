**Researched:** 2026-04-28  
**Domain:** OpenCode plugin tools, PTY command execution, prompt-quality heuristics, session lineage export  
**Confidence:** HIGH for repo-local contracts; MEDIUM for PTY timing remediation until live runtime proof is rerun

## User Constraints

### Locked Phase Boundary

Phase 49 converts the `ses_22ee` end-user UAT findings into stable contracts for harness tools that directly affect real OpenCode engine usage. The phase is limited to command execution reliability, prompt-analysis expectations, prompt-skimming risk signals, and journal export filter semantics. [VERIFIED: `.planning/workstreams/milestone/phases/49-uat-tool-contract-and-pty-command-reliability/49-CONTEXT-2026-04-28.md:10-13`]

The roadmap places Phase 49 after Phase 48.5 and before primitive restart readiness. It is a downstream production-usability phase, not a general runtime rewrite. [VERIFIED: `.planning/workstreams/milestone/phases/49-uat-tool-contract-and-pty-command-reliability/49-CONTEXT-2026-04-28.md:14-15`; `.planning/workstreams/milestone/ROADMAP.md:1219-1238`]

### Non-Goals

- Do not implement Phase 50 primitive restart readiness issues such as circular command dependencies, missing agent references, permission inheritance serialization, or invalid primitive frontmatter. [VERIFIED: `49-CONTEXT-2026-04-28.md:37-40`]
- Do not rework delegation lifecycle, queue persistence, SDK/CQRS alignment, or production evidence coverage that belongs to Phase 48.x. [VERIFIED: `49-CONTEXT-2026-04-28.md:37-43`]
- Do not expand natural-language routing beyond its intentionally exposed test-command surface unless downstream research proves it is part of the Phase 49 contract. [VERIFIED: `49-CONTEXT-2026-04-28.md:37-43`]
- Do not change session patch security behavior; UAT evidence showed path validation and traversal rejection behaved correctly. [VERIFIED: `49-CONTEXT-2026-04-28.md:37-43`]
- Do not treat this phase as a general documentation cleanup or broad tool redesign. [VERIFIED: `49-CONTEXT-2026-04-28.md:37-43`]

### Required Success Criteria

1. A user can start, observe, send input to, and terminate a background command using documented action names and schema-visible parameters. [VERIFIED: `49-CONTEXT-2026-04-28.md:45-53`]
2. Command execution either clearly rejects shell strings with actionable guidance or supports them through an explicit, safe contract; it must not report a successful start when the command immediately fails as a malformed executable path. [VERIFIED: `49-CONTEXT-2026-04-28.md:45-53`]
3. PTY output reads include process output from the beginning of execution, including fast first-line output, with stable offset behavior. [VERIFIED: `49-CONTEXT-2026-04-28.md:45-53`]
4. Prompt-skim returns risk signals that distinguish simple prompts from scope-bloated/flooding prompts in a way an end-user tester can trust. [VERIFIED: `49-CONTEXT-2026-04-28.md:45-53`]
5. Prompt-analyze flags vague, underspecified, contradictory, and well-structured prompts with calibrated clarity scores and avoids obvious false positives on complementary technical requirements. [VERIFIED: `49-CONTEXT-2026-04-28.md:45-53`]
6. Session journal export filters narrow the returned records and do not silently overwrite or stamp lineage metadata in read responses. [VERIFIED: `49-CONTEXT-2026-04-28.md:45-53`]
7. Phase 49 verification uses real harness tool calls from an OpenCode-like runtime surface where possible, with mock-only results clearly labeled if any remain. [VERIFIED: `49-CONTEXT-2026-04-28.md:45-53`]

## Summary

Phase 49 should be planned as a user-facing contract stabilization phase across four tool surfaces: `run-background-command`, `prompt-skim`, `prompt-analyze`, and `session-journal-export`. [VERIFIED: `.planning/workstreams/milestone/ROADMAP.md:1219-1235`] The highest-risk area is `run-background-command`: UAT found invalid user expectations around `start`/`read`, successful-looking launch responses for malformed command strings, and a PTY first-line output loss. [VERIFIED: `session-ses_22ee.md:1663-2089`; `session-ses_22ee.md:4788-4805`]

The standard approach should preserve the existing OpenCode plugin tool surface, Zod-validated input contracts, `DelegationManager.dispatchCommand()` orchestration, and shared `PtyManager` session ownership checks. [VERIFIED: `src/plugin.ts:92-100`; `src/tools/run-background-command.ts:39-45`; `src/lib/delegation-manager.ts:224-257`; `src/lib/pty/pty-manager.ts:50-102`] Do not bypass the delegation manager or create a separate terminal tool unless planning explicitly documents why the existing tool contract cannot be made discoverable. [VERIFIED: `src/tools/run-background-command.ts:116-129`; `49-CONTEXT-2026-04-28.md:57-63`]

**Primary recommendation:** plan Phase 49 around contract-first regression tests: make accepted actions discoverable, choose an explicit command-string policy, prove first-output capture with real PTY/headless commands, calibrate prompt heuristics with UAT fixtures, and split journal export filtering from projection labeling. [VERIFIED: `session-ses_22ee.md:4752-4867`; `src/tools/session-journal-export.ts:53-74`]

### Pattern 1: Contract-First Tool Surface

**What:** Keep public OpenCode tool args explicit and Zod-validated, then produce actionable error envelopes when users call unsupported actions or unsafe command shapes. [VERIFIED: `src/tools/run-background-command.ts:10-45`; `src/shared/tool-response.ts:27-39`]  
**When to use:** Every Phase 49 change should start with the user-visible tool contract before internal runtime edits. [VERIFIED: `session-ses_22ee.md:1663-1699`; `session-ses_22ee.md:4788-4805`]  
**Example contract direction:** accepted actions should remain `run`, `output`, `input`, `list`, `terminate`, with either documented aliases (`start` -> `run`, `read` -> `output`) or explicit, helpful rejection messages. [VERIFIED: `src/tools/run-background-command.ts:10-45`; `session-ses_22ee.md:1669-1716`; `session-ses_22ee.md:1893-1940`]

### Pattern 2: Executable + Args, Not Accidental Shell Parsing

**What:** Treat `command` as the executable and `args` as arguments unless an explicit shell mode is designed. [VERIFIED: `src/lib/command-delegation.ts:77-81`; `src/lib/command-delegation.ts:166-170`]  
**When to use:** Plan command reliability around preventing successful-looking launches for malformed command strings. [VERIFIED: `session-ses_22ee.md:1724-1780`; `session-ses_22ee.md:1782-1844`]  
**Example:** `command: "bash", args: ["-lc", "echo one && echo two"]` should be the documented shell-wrapper form if shell strings remain unsupported. [ASSUMED]

### Pattern 3: Subscribe Before User-Visible Success

**What:** PTY output capture should be initialized before the tool reports a command as started, and tests should prove `offset: 0` includes early output. [VERIFIED: `src/lib/pty/pty-manager.ts:50-91`; UAT missing first line in `session-ses_22ee.md:2016-2089`]  
**Risk:** Current code creates the process, creates the buffer, then registers `onData`; very fast output could occur before the listener is attached. [VERIFIED: `src/lib/pty/pty-manager.ts:50-71`; MEDIUM confidence root-cause because runtime timing must be proven]

### Pattern 4: Filter Parameters Must Not Mutate Projection Metadata

**What:** Export filters should narrow inputs or output records; metadata stamping should use separate names if still needed. [VERIFIED: `src/tools/session-journal-export.ts:61-74`; `src/lib/execution-lineage.ts:74-84`]  
**When to use:** Phase 49 should decide whether `pipelineKey` is a filter, a projection label, or split into `pipelineKeyFilter` and `pipelineKeyLabel`. [VERIFIED: `session-ses_22ee.md:3113-3232`; `session-ses_22ee.md:4815-4823`]

## Common Pitfalls

### Pitfall 1: Treating UAT “start” as only user error

**What goes wrong:** The Zod discriminator rejects `action: "start"` with low-level invalid-union detail. [VERIFIED: `session-ses_22ee.md:1669-1699`]  
**Why it happens:** The implementation accepts only `run`, `output`, `input`, `list`, and `terminate`; the exposed `args.action` description is a plain string, not an enum-like contract. [VERIFIED: `src/tools/run-background-command.ts:10-45`; `src/tools/run-background-command.ts:96-108`]  
**How to avoid:** Plan either alias support or actionable error normalization that lists valid actions and maps likely user intent. [ASSUMED]  
**Warning signs:** Tests assert only happy-path `run` and do not cover invalid action guidance. [VERIFIED: `tests/tools/run-background-command.test.ts:100-166`]

### Pitfall 2: Reporting “started” for command strings that immediately fail

**What goes wrong:** A free-form shell string is passed as the executable path and returns a success envelope before the child process fails. [VERIFIED: `session-ses_22ee.md:1724-1780`; `session-ses_22ee.md:1782-1844`]  
**Why it happens:** `child_process.spawn(params.command, params.args ?? [])` and `ptyManager.spawn({ command, args })` both treat `command` as the executable. [VERIFIED: `src/lib/command-delegation.ts:77-81`; `src/lib/command-delegation.ts:166-170`]  
**How to avoid:** Validate command shape before dispatch, return guidance for shell metacharacters when `args` is missing, or add an explicit shell opt-in. [ASSUMED]  
**Warning signs:** Fallback reason says PTY spawn failed while the actual issue is command contract mismatch. [VERIFIED: `session-ses_22ee.md:1735-1751`; `session-ses_22ee.md:1795-1811`]

### Pitfall 3: Misidentifying public session IDs vs internal PTY IDs

**What goes wrong:** Users pass a chosen session ID and receive access denied because the tool expects the returned `ptySessionId`. [VERIFIED: `session-ses_22ee.md:1846-1956`]  
**Why it happens:** The tool does not use user-supplied `sessionId` during `run`; output/input/terminate require the internal PTY session ID and validate delegation ownership. [VERIFIED: `src/tools/run-background-command.ts:10-16`; `src/tools/run-background-command.ts:136-157`]  
**How to avoid:** Make `ptySessionId` prominent in the success payload and documentation; if a friendly alias is accepted, store and resolve it through delegation metadata. [ASSUMED]

### Pitfall 4: Losing the first PTY output chunk

**What goes wrong:** UAT long-running output captured `Line 2` through `COMPLETE` but not `Line 1`. [VERIFIED: `session-ses_22ee.md:2016-2089`]  
**Why it happens:** Likely race window around process spawn/listener registration or process output before buffer availability. [VERIFIED: `src/lib/pty/pty-manager.ts:50-71`; MEDIUM confidence root-cause]  
**How to avoid:** Create regression tests using fast first-line output and inspect whether listener registration can be moved earlier or whether spawn requires a shell/session-start handshake. [ASSUMED]

### Pitfall 5: Prompt heuristics matching words rather than intent

**What goes wrong:** Vague prompts can receive `clarity_score: 100`, while complementary event sourcing/CQRS requirements can be flagged as contradictory. [VERIFIED: `session-ses_22ee.md:735-768`; `session-ses_22ee.md:774-820`]  
**Why it happens:** `VAGUE_RE`, `MISSING_SCOPE_RE`, and `CONTRADICTION_PAIRS` are small regex sets with no concept grouping. [VERIFIED: `src/tools/prompt-analyze/tools.ts:12-20`; `src/tools/prompt-analyze/tools.ts:86-129`]  
**How to avoid:** Add UAT fixtures for known vague, clear, contradictory, and complementary prompts before changing scoring. [VERIFIED: `tests/tools/prompt-analyze.test.ts:23-129`]

### Pitfall 6: Confusing pipeline labels with filters

**What goes wrong:** `pipelineKey: "nonexistent-pipeline"` returns all records stamped with that key instead of zero filtered records. [VERIFIED: `session-ses_22ee.md:3113-3232`]  
**Why it happens:** `pipelineKey` is passed as an option into `buildExecutionLineage()` and conditionally added to every projection record. [VERIFIED: `src/tools/session-journal-export.ts:71-74`; `src/lib/execution-lineage.ts:74-84`]  
**How to avoid:** Split read filtering from projection labeling and test both semantics explicitly. [ASSUMED]
