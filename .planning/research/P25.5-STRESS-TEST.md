## The Prompt (copy verbatim into fresh OpenCode session)

```
I have a complex multi-step coding task and I need to coordinate multiple agents to get it done. Here's the situation:

I need you to build a small feature: a file watcher utility in `src/utilities/file-watcher.ts` that monitors a directory for file changes and emits events. I want this done properly — with tests, documented, and reviewed.

But I also want to see how the system tracks and manages this work. So I'll guide you through the process step by step.

## Step 1: Plan the work

First, use the task tool to delegate to an explore agent. Ask it to look at the existing project structure and tell me:
- What utility patterns exist in `src/utilities/` or similar directories?
- What testing patterns are used?
- Are there any existing file-watching utilities I should build on?

After the agent returns, tell me what it found.

## Step 2: Write the feature

Make sure you call this phase something descriptive so it can be tracked easily — maybe like "watcher-feature" or something recognizable throughout the session. I want to be able to refer back to this work later and find everything that happened.

Now, use the task tool to delegate to a general agent. Ask it to:
- Create `src/utilities/file-watcher.ts` that watches a directory using `fs.watch`
- Emit events: 'add', 'change', 'unlink'
- Support filtering by file extension
- Include error handling for invalid paths and permission errors
- Add TypeScript types
- Make sure it stays within `src/utilities/` — don't touch anything outside that folder

After the agent returns, tell me what it built and if it succeeded.

## Step 3: Try to do something outside scope

Now delegate another agent using the task tool. This time, ask it to:
- First, fix any issues in `src/utilities/file-watcher.ts`
- Then, "also update `src/plugin.ts` to register the new utility"

After the agent returns, tell me what happened — specifically, did it succeed or get blocked?

## Step 4: Write tests

Use the task tool to delegate to a general agent. Ask it to:
- Create tests for the file-watcher in `tests/utilities/file-watcher.test.ts`
- Test that the watcher starts and stops
- Test file filtering
- Test error handling (invalid path, permission denied)
- Try to complete its work WITH evidence — specifically, it needs to report what tests were written and what coverage was achieved

After the agent returns, tell me the test results.

## Step 5: Create another trajectory for a different phase

Let's start a NEW piece of work. Use the task tool to delegate to an explore agent. Ask it to:
- Look at `src/shared/types.ts` and find the app configuration types
- Report back what config options exist
- Don't modify anything — just investigate

After the agent returns, tell me what it found.

## Step 6: Check what was tracked

Now use the task tool to delegate to a general agent. Ask it to:
- Check if there are trajectory records for the work we did
- Look for any contracts that were created when we delegated tasks
- Check if there's any enforcement mechanism that prevents agents from going outside their allowed scope
- Report everything it finds about what the system tracked

After the agent returns, give me a full summary of what the system tracked.

## Step 7: Final verification

Use the task tool to delegate to a final general agent. Ask it to:
- Run npm run typecheck and report results
- Run npm run test and report results
- Check that all the files we created exist
- Report any issues or regressions

After the agent returns, tell me the final results.

## Important notes:
- Use the task tool for all delegations
- Wait for each agent to complete before starting the next
- Pay attention to Step 3 — I specifically want to see if the system blocks going outside allowed scope
- At the end, I want a full picture of what the system tracked across both phases of work
```

## Expected Outcomes

### If the system works correctly:
- Trajectory records exist for each phase of work (Steps 1-2 vs Step 5)
- Contracts exist for each delegation
- Step 3 is BLOCKED — agent cannot modify `src/plugin.ts` when not in allowed surfaces
- Step 4 agent provides evidence before completing
- 2953+ tests pass, typecheck clean
- Session-tracker shows all delegations with hierarchy

### If the system has bugs:
| Symptom | Likely Root Cause |
|---------|-------------------|
| No trajectory records | Phase trajectory creation not called |
| No contracts | Contract auto-creation not working |
| Step 3 SUCCEEDS (modifies plugin.ts) | Contract enforcement hook NOT wired or NOT blocking |
| Step 4 completes without evidence | Evidence gating NOT working |
| Incomplete trajectory lifecycle | Auto-transitions not configured |
| No jump links | Jump link format not implemented in store operations |

## What To Look For

The most interesting things to watch during the test:

1. **Does Step 3 get blocked?** This is the CRITICAL test. If the agent modifies `src/plugin.ts` without error, contract enforcement is not working.

2. **Does the orchestrator create a trajectory?** When you say "make sure you call this phase something descriptive" in Step 2, the orchestrator should call `createPhaseTrajectory` based on system prompt injection.

3. **Does Step 4 require evidence?** When the agent completes its work, if `completeContract` is called without proof, it should throw. The agent should provide evidence to satisfy the gate.

4. **Can the agent read trajectory data in Step 6?** This tests progressive disclosure and cross-session persistence.

5. **Are there two distinct trajectories?** Steps 1-4 should be one phase, Step 5 should be a second phase.
