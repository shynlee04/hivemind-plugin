### Circular Dependency Analysis

**No circular dependencies detected.** The dependency graph is strictly acyclic:
- Leaf modules (types, helpers, task-status) have zero or type-only internal imports
- Each level only imports from lower levels
- `types.ts` → `task-status.ts` (type re-export) does NOT create a cycle because `task-status.ts` has no imports
- `types.ts` → `pending-notifications.ts` (type re-export) — `pending-notifications.ts` imports from `continuity.js` and `notification-handler.js`, neither of which imports from `types.ts`

## Lifecycle State Machine

### Phases and Valid Transitions

```
created ──────────────────────────────────────┐
  │                                           │
  ▼                                           │
queued ──────────────────────────────────────►│
  │                                           │
  ▼                                           │
dispatching ─────────────────────────────────►│
  │                                           │
  ├─────► running ──┬──► completed            │
  │                 │                          │
  │                 └──► failed ◄──────────────┘
  │
  (completed and failed are terminal)
```

**Transition table** (`lib/lifecycle-state.ts`):

| From | Valid To |
|------|----------|
| `created` | `queued`, `dispatching` |
| `queued` | `dispatching` |
| `dispatching` | `running` |
| `running` | `completed`, `failed` |
| `completed` | *(terminal)* |
| `failed` | *(terminal)* |

### Event-Driven Phase Mapping

The lifecycle manager maps SDK events to phases via `inferContinuityStatusFromEvent()` in `lib/runtime.ts`:

| SDK Event | Status | Phase |
|-----------|--------|-------|
| `session.created` | `running` | `created`/`queued`/`dispatching` (preserves previous) |
| `session.updated` | derived | derived from status |
| `session.idle` | `completed` | `completed` |
| `session.deleted` | *(cleanup)* | forget session + delete continuity |
| `session.error` | `error` | `failed` |
| `session.interrupt` | `interrupt` | `running` or preserve previous |

### Execution Modes (Three Submodes)

The `classifyExecutionMode()` function in `lib/execution-mode.ts` produces an `ExecutionModeResult`:

| Submode | When | Runner |
|---------|------|--------|
| `tmux-pane` | tmux available, interactive task, not research | `runLifecycleTmuxTask()` |
| `builtin-process` | non-interactive, headless, OS-level work | `runLifecycleProcessTask()` |
| `builtin-subsession` | default — OpenCode child session | `runLifecycleSubsessionTask()` |

### Lifecycle Observation Chain

```
SDK event → handleEvent() → buildLifecycleState() → patchSessionContinuity() → syncDelegationPacketStatus()
                                                                    │
                                                                    ▼
                                                    patchSessionDelegationPacket()
```

## Delegation Flow

### Parent → Child → Notification

```
1. PARENT SESSION calls delegate-task tool
   │
   ├── resolveSpecialistRoute() — category/agent/model resolution
   ├── walkParentChain() — build parent chain, check depth (max 3)
   ├── reserveSubagentSpawn() — budget reservation
   ├── classifyExecutionMode() — tmux-pane / builtin-process / subsession
   └── lifecycleManager.launchDelegatedSession()
       │
       ├── createSession() — OpenCode SDK call
       ├── commitDescendant() — budget commit
       ├── setDelegationMeta() — in-memory
       ├── recordSessionContinuity() — durable JSON
       ├── enqueueWaitingLifecycle() → acquireLifecycleQueue()
       │
       └── Execute (async or sync):
           ├── tmux-pane: runLifecycleTmuxTask()
           ├── builtin-process: runLifecycleProcessTask()
           └── subsession (default): runLifecycleSubsessionTask()
               │
               ├── sendPrompt() / sendPromptAsync()
               ├── CompletionDetector.poll() — two-signal detection
               ├── observeBackgroundCompletion() — lifecycle-background-observer
               └── notifyParentSession() → persistPendingNotification()
                   │
                   ▼
2. PARENT receives notification via:
   - system.transform hook (injects pending notifications into system prompt)
   - shell.env hook (env var injection)
   - create-core-hooks replay on session.created/updated events
   - Toast via client.tui.showToast
```

## Key Architectural Observations

### Strengths
1. **Clean CQRS separation** — tools (write) vs hooks (read) is well-maintained
2. **Factory pattern** — all hooks and tools use factory functions, enabling testability
3. **Thin plugin** — 57 LOC composition root with zero business logic
4. **Type authority** — `types.ts` is the single source for domain types
5. **No circular dependencies** — strictly acyclic import graph
6. **Comprehensive test coverage** — 42 test files with dedicated crash/malformed edge-case tests

### Concerns
1. **`lifecycle-manager.ts` at 735 LOC** — 2x over the 350 target. This is the primary refactoring candidate.
2. **`continuity-normalizers.ts` at 706 LOC** — Bloated with type imports. Could be split into focused normalizer modules.
3. **Empty directories** — `cli/`, `harness/`, `kernel/`, `plugins/` are placeholders consuming namespace. Either implement or remove.
4. **`agent-registry.ts` missing** — Referenced in documentation but doesn't exist on disk. Either create or update docs.
5. **`lifecycle-process-runner.ts` at 456 LOC** — 30% over target. Extract notification logic.
6. **Duplicate `formatRuntimeInjectionBlock()`** — Exists in both `create-core-hooks.ts` and `create-session-hooks.ts`. Should be extracted to `lib/injection-engine.ts`.
