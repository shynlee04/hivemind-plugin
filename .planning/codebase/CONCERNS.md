## Tech Debt

### Lifecycle Manager Stub

- Issue: `lifecycle-manager.ts` is an explicit stub — `launchDelegatedSession()` throws, `noteObservedActivity()` is a no-op, `isValidTransition()` always returns `true`, `handleEvent()` does minimal event routing only
- Files: `src/lib/lifecycle-manager.ts` (lines 37-41, 72-74, 122-124)
- Impact: Core session lifecycle is incomplete. The `DelegationManager` class in `src/lib/delegation-manager.ts` handles delegation independently, but `HarnessLifecycleManager` is still wired into `plugin.ts` and hooks. Two competing lifecycle management paths exist.
- Fix approach: Complete Plan 14-02 — merge lifecycle management into `DelegationManager` or restore full `HarnessLifecycleManager`. The stub's existence creates confusion about which system owns what.

### Continuity.ts Mixed Responsibilities (401 LOC)

- Issue: `continuity.ts` combines three concerns: deep-clone helpers (~60 LOC), normalization/parsing (~80 LOC), and CRUD store operations (~140 LOC). The inline clone functions replace deleted `continuity-clone.ts` and `continuity-normalizers.ts` modules but are now embedded without separation.
- Files: `src/lib/continuity.ts` (lines 62-122 clone helpers, lines 128-165 normalizer, lines 209-401 store I/O + public API)
- Impact: Hard to test clone and normalize logic independently. Changes to persistence format require editing the same file as normalization logic.
- Fix approach: Extract into `src/lib/continuity-clone.ts`, `src/lib/continuity-normalizers.ts`, and `src/lib/continuity.ts` (CRUD + public API only). Each under 200 LOC.

### Duplicate `extractAssistantText`

- Issue: Two separate implementations of assistant-text extraction exist — one in `delegation-manager.ts` and one in `create-session-hooks.ts`. They use slightly different approaches (delegation-manager iterates forward and joins; session-hooks iterates backward and returns first match).
- Files: `src/lib/delegation-manager.ts` (lines 435-442), `src/hooks/create-session-hooks.ts` (lines 74-98)
- Impact: Behavioral divergence risk. The delegation-manager version uses local `MessageLike` type; the session-hooks version uses `getNestedValue` helper. Different edge-case handling.
- Fix approach: Consolidate into `src/lib/helpers.ts` as a single shared function `extractAssistantText(messages: unknown[]): string` with the backward-search-last-assistant pattern.

## Architecture Concerns

### DelegationManager Exceeds 500 LOC Target (450 LOC)

- Issue: `delegation-manager.ts` is at 450 LOC — within limits but dense. It manages dispatch, dual-signal completion, safety ceiling, persistence, agent validation, timer management, and text extraction in a single class.
- Files: `src/lib/delegation-manager.ts`
- Impact: Any change to delegation dispatch, completion detection, or persistence touches the same class. The class has 8 private method groups.
- Fix approach: Extract persistence into `DelegationPersistence` helper (lines 359-406). Extract timer management into `DelegationTimers` helper (lines 411-428). Target <350 LOC for the manager class itself.

### Dual Lifecycle Management Confusion

- Issue: Both `HarnessLifecycleManager` and `DelegationManager` are instantiated in `plugin.ts`. They have overlapping responsibilities: both track session status, both interact with continuity persistence, both handle session events. The lifecycle manager is a stub; the delegation manager is fully functional.
- Files: `src/plugin.ts` (lines 29-37), `src/lib/lifecycle-manager.ts`, `src/lib/delegation-manager.ts`
- Impact: Hook factories (`create-core-hooks.ts`, `create-session-hooks.ts`, `create-tool-guard-hooks.ts`) receive `lifecycleManager` as a dependency but it's mostly inert. `DelegationManager` events are handled via a separate inline observer in `plugin.ts`.
- Fix approach: Either (a) remove `HarnessLifecycleManager` and route everything through `DelegationManager`, or (b) make `HarnessLifecycleManager` a thin facade that delegates to `DelegationManager`. Current state is transitional and confusing.

### Module-Level Singleton in continuity.ts

- Issue: `storeCache` at `src/lib/continuity.ts:19` is a module-level mutable singleton. It prevents isolated unit testing — any test that calls continuity functions pollutes the cache for subsequent tests.
- Files: `src/lib/continuity.ts` (line 19)
- Impact: Tests must either reload the module or carefully reset state. No `resetStore()` or `invalidateCache()` export exists.
- Fix approach: Export a `_test_resetStore()` function gated behind `process.env.NODE_ENV === "test"`, or wrap store operations in a class that can be instantiated per-test.

### Module-Level Singleton in state.ts

- Issue: `taskState` at `src/lib/state.ts:188` is a process-wide singleton. The file also exports 14 backward-compatible wrapper functions that delegate to the singleton, adding boilerplate without encapsulation benefit.
- Files: `src/lib/state.ts` (line 188, lines 195-251)
- Impact: Same testing isolation issue as `continuity.ts`. The wrapper functions add ~55 LOC of indirection.
- Fix approach: Consumers should import the `TaskStateManager` class or the `taskState` instance directly. Remove wrapper functions once all callers are updated.

## Type Safety Issues

### Unsafe Type Assertions in Normalizer

- Issue: `continuity.ts` normalizer uses multiple `as` casts on untrusted JSON data: `rec.metadata as Record<string, unknown>`, `meta.constraints as string[]`, `meta.status as SessionContinuityMetadata["status"]`. These bypass runtime validation.
- Files: `src/lib/continuity.ts` (lines 133-163)
- Impact: Malformed persistence data could produce incorrectly typed objects that pass TypeScript checks but fail at runtime with confusing errors.
- Fix approach: Use Zod schema validation for persistence data (zod is already a dependency). Define `ContinuityStoreFileSchema` and parse instead of cast.

### Session-API Double Cast

- Issue: `getSessionStatusMap()` in `session-api.ts` performs a double-cast `(data as Record<string, unknown>) as Record<string, { type: string }>`. This provides no runtime validation.
- Files: `src/lib/session-api.ts` (line 64)
- Impact: If SDK response shape changes, this silently produces incorrectly typed data.
- Fix approach: Add runtime validation or use a Zod schema for the status map response.

### Local Type Aliases Bypass Shared Types

- Issue: `delegation-manager.ts` defines local `TextPart` and `MessageLike` types instead of importing from shared types. These could diverge from the actual SDK message shape.
- Files: `src/lib/delegation-manager.ts` (lines 25-26)
- Impact: If SDK message structure changes, only this file's local types would need updating — easy to miss.
- Fix approach: Define shared `MessageLike` and `TextPart` types in `src/lib/types.ts` and import everywhere.

## Error Handling Gaps

### Silent Failure in Prompt Dispatch

- Issue: In `delegation-manager.ts:103`, the `.catch(() => { ... })` on the fire-and-forget prompt dispatch silently catches prompt failures and only transitions to "error" via a `setTimeout(0)` macrotask. If the setTimeout itself fails or the delegation was already cleaned up, the error is silently lost.
- Files: `src/lib/delegation-manager.ts` (lines 103-115)
- Impact: Prompt dispatch failures may not be observable if the delegation tracking has already been cleaned up by the time the macrotask fires.
- Fix approach: Log prompt dispatch failures to a diagnostic channel or add a warning to the session stats.

### Silent Failure in Notification

- Issue: `notifyParentSession()` in `notification-handler.ts:154` catches all errors and returns `false` but never logs why. The caller has no diagnostic information.
- Files: `src/lib/notification-handler.ts` (lines 149-156)
- Impact: Notification delivery failures are invisible. If the parent session was deleted or the SDK is unreachable, the notification is silently dropped.
- Fix approach: Add optional logging or return the error reason alongside the `boolean` result.

### JSON Parse Failures Treated as Empty Store

- Issue: Both `continuity.ts:256` and `delegation-manager.ts:389` catch JSON parse errors and return empty arrays/stores. This means corrupted persistence data is silently replaced with empty state.
- Files: `src/lib/continuity.ts` (line 256), `src/lib/delegation-manager.ts` (line 389)
- Impact: Data loss on corruption. If the JSON file is partially written (crash during persist), the next load silently discards all session state.
- Fix approach: Attempt recovery from backup before falling back to empty. At minimum, log a warning with the parse error and file path.

### Lifecycle Manager handleEvent Uses Untyped Event

- Issue: `lifecycle-manager.ts:79` casts `args.event` through multiple `as Record<string, unknown>` assertions to extract a status signal. No runtime validation.
- Files: `src/lib/lifecycle-manager.ts` (lines 79-81)
- Impact: If event structure changes, this silently produces an empty string, and idle detection fails without error.
- Fix approach: Use the same `getNestedValue` helper pattern used elsewhere in the codebase.

## Security Concerns

### Path Traversal in Session-Patch Tool

- Issue: `session-patch/tools.ts` accepts an absolute file path (`sessionFilePath`) directly from tool arguments. No validation that the path is within a safe directory or that it's a markdown file. An attacker with tool access could read/patch arbitrary files.
- Files: `src/tools/session-patch/tools.ts` (lines 40-81)
- Impact: Arbitrary file read/write via the tool. The backup is written before validation, meaning even a failed patch attempt creates a backup copy of the target file.
- Fix approach: Validate that `sessionFilePath` ends with `.md` and is within the workspace root or `.opencode/state/` directory. Reject paths containing `..`.

### Synchronous File I/O Throughout

- Issue: All persistence uses synchronous `readFileSync`/`writeFileSync`. This blocks the Node.js event loop during every continuity read/write operation.
- Files: `src/lib/continuity.ts` (lines 225, 266), `src/lib/delegation-manager.ts` (lines 368, 382), `src/tools/session-patch/tools.ts` (lines 49-50, 71, 81)
- Impact: Under high delegation volume, synchronous writes could block the event loop, causing timeout detection delays and stability poll misses.
- Fix approach: Migrate to async `fs.promises.readFile`/`fs.promises.writeFile`. This is low urgency but becomes critical at scale.

### Environment Variable Parsing Without Validation

- Issue: `lifecycle-manager.ts:50` parses `OPENCODE_HARNESS_CONCURRENCY_LIMIT` with `parseInt()` and only checks for NaN and < 1. Negative floats (e.g., `-0.5`) pass the check because `parseInt("-0.5")` returns `0`, which fails the < 1 check, but `parseInt("3abc")` returns `3` which succeeds.
- Files: `src/lib/lifecycle-manager.ts` (line 50)
- Impact: Low — the fallback to 3 handles most cases. But partial parsing (`parseInt("3abc")` → 3) could be surprising.
- Fix approach: Use `Number()` instead of `parseInt()` and validate `Number.isFinite()`.

## Complexity Hotspots

### DelegationManager.dispatch (lines 56-124)

- Issue: The `dispatch()` method is ~70 LOC with nested closures, fire-and-forget promise chains, and setTimeout(0) macrotask scheduling. The control flow is: validate → acquire semaphore → create session → register delegation → persist → fire-and-forget prompt → setTimeout(0) status transition.
- Files: `src/lib/delegation-manager.ts` (lines 56-124)
- Impact: Hard to reason about error propagation. The finally block releases the semaphore, but the fire-and-forget prompt continues asynchronously after release.
- Fix approach: Extract the fire-and-forget logic into a named private method `dispatchPrompt(delegation)` with clear JSDoc about its async lifecycle.

### patchSessionContinuity (lines 308-349)

- Issue: The patch function in continuity.ts handles 7 different optional fields, each with a conditional clone-or-preserve pattern. The resulting code is deeply nested conditional spreading.
- Files: `src/lib/continuity.ts` (lines 308-349)
- Impact: Adding new metadata fields requires adding another conditional clone line in exactly the right pattern. Easy to forget to deep-clone a new field.
- Fix approach: Use a merge utility that automatically deep-clones any changed field. Or define a `cloneMap` that maps field names to clone functions.

## Dependency Risks

### @opencode-ai/sdk Caret Range (^1.4.2)

- Issue: The SDK is pinned at `^1.4.2` — minor version updates could change API shapes. The codebase makes extensive use of `unwrapData()` helper specifically because SDK response shapes have been unstable.
- Files: `package.json` (line 29)
- Impact: SDK minor bumps could break at runtime without compile-time errors if response types change but `unwrapData` still compiles.
- Fix approach: Pin to exact version (`1.4.2`) for production, use caret only in dev. Add integration tests that validate actual SDK response shapes.

### @opencode-ai/plugin as Peer Dependency

- Issue: `@opencode-ai/plugin >= 1.1.0` is a peer dependency. The `tool` import from `@opencode-ai/plugin/tool` is used in 4 tool files. If the peer dependency version doesn't match, tool registration fails at runtime.
- Files: `package.json` (line 33), `src/tools/delegate-task.ts`, `src/tools/delegation-status.ts`, `src/tools/prompt-skim/tools.ts`, `src/tools/prompt-analyze/tools.ts`, `src/tools/session-patch/tools.ts`
- Impact: Version mismatch between host OpenCode and harness plugin causes silent tool registration failure.
- Fix approach: Add runtime version check at plugin startup with a clear error message.

### Zod v4 Dependency

- Issue: `zod: ^4.0.0` — Zod v4 is a major version with breaking API changes from v3. If any consumer expects v3 patterns, schema validation will fail.
- Files: `package.json` (line 30)
- Impact: Ecosystem tools or plugins expecting Zod v3 patterns may break when interacting with harness schemas.
- Fix approach: Verify all Zod usage conforms to v4 patterns. Document the Zod version requirement.

## State Management Risks

### Race Condition in Delegation Dispatch

- Issue: In `delegation-manager.ts`, the dispatch flow creates a child session, registers the delegation, persists to disk, then fires a prompt. The `persistAllDelegations()` call at line 83 happens BEFORE the prompt is sent. If the process crashes between persist and prompt, the delegation is persisted as "dispatched" but the child session has an empty prompt.
- Files: `src/lib/delegation-manager.ts` (lines 82-115)
- Impact: On recovery, the delegation would remain in "dispatched" status with no prompt sent. The safety ceiling timer would eventually time it out, but the child session would be orphaned.
- Fix approach: Persist after prompt send confirmation, or add a recovery step that re-sends prompts for "dispatched" delegations with no child messages.

### In-Memory State Not Recovered for DelegationManager

- Issue: `DelegationManager.recoverPending()` reads delegations from disk and re-registers them in memory. However, the `DelegationConcurrencyQueue` (semaphore) state is NOT recovered — all lanes start fresh. This means the concurrency limits could be temporarily exceeded after a restart if delegations were mid-acquire when the process died.
- Files: `src/lib/delegation-manager.ts` (lines 181-221), `src/lib/concurrency.ts`
- Impact: After a crash restart, more delegations could run concurrently than the policy allows.
- Fix approach: Recover semaphore state from persisted delegations — count "running" delegations per key and set `lane.active` accordingly.

### Auto-Loop State Not Persisted

- Issue: `autoLoopStates` in `create-session-hooks.ts:150` is an in-memory `Map` that is never persisted. If the process restarts, auto-loop iteration counts reset to zero, potentially causing excessive retries.
- Files: `src/hooks/create-session-hooks.ts` (line 150)
- Impact: After restart, a session that was at iteration 4/5 would restart at 0/5, getting 5 more retries instead of 1.
- Fix approach: Persist auto-loop state in the continuity record's `lifecycle` field.

## Missing Safeguards

### No File Size Limit on Persistence Files

- Issue: Neither `continuity.ts` nor `delegation-manager.ts` limits the size of their JSON persistence files. In a long-running session with many delegations, `delegations.json` could grow unbounded.
- Files: `src/lib/continuity.ts`, `src/lib/delegation-manager.ts`
- Impact: Very large persistence files slow down load/parse on every read operation.
- Fix approach: Add a max-file-size check before reading. Add a cleanup step for completed delegations older than N hours.

### No Max Delegation Depth Enforcement in DelegationManager

- Issue: While `MAX_DELEGATION_DEPTH = 1` is defined in `types.ts:378`, `DelegationManager.dispatch()` does not check the delegation depth. The depth limit exists in the type system but is not enforced at dispatch time.
- Files: `src/lib/types.ts` (line 378), `src/lib/delegation-manager.ts` (lines 56-124)
- Impact: Nothing prevents a child session from creating its own delegation, creating arbitrary nesting depth.
- Fix approach: Check `delegation.depth` in `dispatch()` and reject if `depth >= MAX_DELEGATION_DEPTH`. This requires passing depth information from the parent session's delegation metadata.

### No Backup Before Persistence Write

- Issue: `persistAllDelegations()` in `delegation-manager.ts` and `persistStore()` in `continuity.ts` directly overwrite the JSON file. If the process crashes mid-write, the file is corrupted.
- Files: `src/lib/delegation-manager.ts` (lines 365-373), `src/lib/continuity.ts` (lines 261-267)
- Impact: Crash during write = data loss for ALL delegations or ALL sessions.
- Fix approach: Write to a temporary file first, then atomically rename (`fs.renameSync`) to the target path. This is a standard write-ahead safety pattern.
