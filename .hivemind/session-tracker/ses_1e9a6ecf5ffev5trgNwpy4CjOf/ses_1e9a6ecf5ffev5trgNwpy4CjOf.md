222: ## Ingestion Pipeline
223: 
224: The pipeline has five phases. Complete each in order:
225: 
226: ```
227: [1] DETECT → [2] DISCOVER → [3] INGEST → [4] ORGANIZE → [5] INDEX
228: ```
229: 
230: ### Phase 1: DETECT — Identify What to Ingest
231: 
232: **Input:** Project manifest files or explicit dependency list.
233: 
234: **Procedure:**
235: 
236: 1. Read the project manifest files:
237:    ```bash
238:    # Detect Node.js/TypeScript stack
239:    ls package.json && cat package.json | grep -E '"dependencies"|"devDependencies"|"peerDependencies"' -A 50
240: 
241:    # Detect Python stack
242:    ls requirements.txt && cat requirements.txt
243: 
244:    # Detect Go stack
245:    ls go.mod && cat go.mod
246: 
247:    # Detect Rust stack
248:    ls Cargo.toml && cat Cargo.toml
249:    ```
250: 
251: 2. For each detected dependency, extract:
252:    - **Package name** (npm: `"@scope/name"`, pip: `name`, go: `module/path`)
253:    - **Version constraint** (npm: `"^1.2.3"`, pip: `>=1.2,<2.0`, go: `v1.2.3`)
254:    - **Type** (framework, SDK, utility, testing, build-tool)
255:    - **Priority** (critical/foundational vs. utility)
256: 
257: 3. Prioritize: Ingest foundational dependencies first (SDKs, frameworks, type systems). These are the ones quality gates will validate against. Utilities and build tools can be ingested lazily.
258: 
259: 4. Build the ingest list:
260:    ```markdown
261:    ## Ingest Queue
262:    | Priority | Package | Version | Type | Reason |
263:    |----------|---------|---------|------|--------|
264:    | HIGH | [your-sdk-package] | ^X.Y.Z | SDK | Quality gate validation |
265:    | HIGH | [your-schema-lib] | ^X.Y.Z | Schema lib | API surface verification |
266:    | MEDIUM | [your-test-framework] | ^X.Y.Z | Testing | Test pattern reference |
267:    | LOW | [your-utility] | ^X.Y.Z | Utility | Lazy-load if needed |
268:    ```
269: 
270: ### Phase 2: DISCOVER — Find the Canonical Repos
271: 
272: For each package in the ingest queue, discover where its canonical source lives.
273: 
274: **Procedure:**
275: 
276: 1. **Resolve the canonical repository URL:**
277:    ```bash
278:    # For npm packages: extract repository from registry metadata
279:    npm view <package-name> repository.url 2>/dev/null
280:    npm view <package-name> homepage 2>/dev/null
281: 
282:    # For Go modules: the module path IS typically the repo
283:    # go.mod line: "github.com/user/repo v1.2.3" → repo is the module path
284:    ```
285: 
286: 2. **Discover documentation sources** using MCP tools (see `references/mcp-tool-cheatsheet.md` for full mapping):
287:    - `repomix_pack_remote_repository` — download entire repo as structured pack
288:    - `deepwiki_read_wiki_contents` / `deepwiki_ask_question` — get structured documentation
289:    - `gitmcp_search_github_com_code` — find specific files/code
290:    - `gitmcp_fetch_github_com_documentation` — fetch repo documentation
291:    - `context7_resolve_library_id` + `context7_query_docs` — API docs with examples
292:    - `tavily_tavily_search` / `tavily_tavily_crawl` — web documentation
293:    - `exa_web_search_exa` / `exa_web_fetch_exa` — web research
294: 
295: 3. **Select the ingestion strategy** based on what's available:
296: 
297: | Source Available | Strategy | MCP Tool |
298: |-----------------|----------|----------|
299: | Full GitHub repo (public) | Download entire repo | `repomix_pack_remote_repository` |
300: | GitHub repo with docs | Fetch structured docs | `deepwiki_read_wiki_contents` |
301: | Specific API docs needed | Query API surface | `context7_query_docs` |
302: | No GitHub repo (private/pkg) | Web crawl + extract | `tavily_tavily_crawl` + `exa_web_fetch_exa` |
303: | Multiple sources | Combine: repomix for code + context7 for API + deepwiki for docs | All three |
304: 
305: ### Phase 3: INGEST — Download and Persist
306: 
307: Execute the ingestion according to the selected strategy.
308: 
309: **Procedure for repomix (full repo):**
310: 
311: ```bash
312: # Ingest a remote repository as a structured pack
313: # Store the output in a temporary location first
314: repomix --remote "github.com/user/repo" --style xml --compress false
315: ```
316: 
317: Then move the output to `references/tech-stacks/<stack-name>/raw/`.
318: 
319: **Procedure for context7 (API docs):**
320: 
321: 1. Resolve the library: `context7_resolve_library_id("zod", "zod")`
322: 2. Query the API surface: `context7_query_docs("/colinhacks/zod", "all exported types and functions")`
323: 3. Store the response in `references/tech-stacks/<stack-name>/api/`.
324: 
325: **Procedure for deepwiki (structured docs):**
326: 
327: 1. Read wiki structure: `deepwiki_read_wiki_structure("colinhacks/zod")`
328: 2. Read full contents: `deepwiki_read_wiki_contents("colinhacks/zod")`
329: 3. Store in `references/tech-stacks/<stack-name>/docs/`.
330: 
331: **Procedure for gitmcp (targeted code search):**
332: 
333: 1. Search for specific files: `gitmcp_search_github_com_code("repo:colinhacks/zod path:src types")`
334: 2. Fetch results: `gitmcp_fetch_github_com_documentation()`
335: 3. Store in `references/tech-stacks/<stack-name>/api/`.

551: ## Update Mechanism
552: 
553: When the project's installed version differs from the cached version, the cached assets are stale and must be re-ingested.
554: 
555: ### Check Staleness
556: 
557: ```bash
558: # Compare installed version vs cached version
559: # For npm:
560: npm ls <package-name> --depth=0 2>/dev/null
561: cat references/tech-stacks/<package-name>/metadata.json | grep version
562: 
563: # For pip:
564: pip show <package-name> | grep Version
565: cat references/tech-stacks/<package-name>/metadata.json | grep version
566: ```
567: 
568: ### Trigger Re-Ingestion
569: 
570: | Condition | Action |
571: |-----------|--------|
572: | Installed version > cached version | Re-ingest the NEW version. Archive old version under `archive/<version>/`. |
573: | Installed version < cached version | Flag as `NEEDS_INVESTIGATION` — downgrade may be intentional or a misalignment. |
574: | Cached version missing | Run full ingestion pipeline. |
575: | Cache > 90 days old | Schedule re-ingestion to capture doc updates even if version unchanged. |
576: 
577: ### Re-Ingestion Procedure
578: 
579: 1. Run Phase 2 (DISCOVER) to confirm the canonical source still exists.
580: 2. Run Phase 3 (INGEST) to download the current version.
581: 3. Run Phase 4 (ORGANIZE) to update the directory structure.
582: 4. Run Phase 5 (INDEX) to rebuild search references.
583: 5. Update `metadata.json` with new `ingest_date` and `version`.
584: 6. Append to `changelog.md` with the re-ingestion record.
585: 
586: See `references/version-tracking.md` for the full version lifecycle.

410: 
411: ```

414:   - Also check lockfiles for transitive dependency versions
