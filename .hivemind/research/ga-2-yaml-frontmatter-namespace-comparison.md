## GA-2: YAML Frontmatter Schema — Namespace/Origin Metadata Storage Strategy

| Option | Pros | Cons | Complexity | Recommendation |
|--------|------|------|------------|----------------|
| **Inline Frontmatter Extension** — add `namespace: gsd \| hf \| test \| core` field to each command's YAML frontmatter | All metadata in single file; native OpenCode YAML parsing; no dual-file sync or drift risk; schema validation auto-enforced via existing Zod pipeline | Migrate 86 files (trivial YAML key add per file); strict-mode schema v1.0.0 rejects unknown fields — requires version bump to v1.1.0; each new command must remember the field | 86 files, 1 schema (+ v bump) — Risk: low (field is optional string, default = "core") | **Rec if schema-first governance is priority** — field validated at parse time, no drift, minimal migration cost per file |
| **Name-based Convention Only** — derive namespace from filename prefix (`gsd-*`, `hf-*`, `test-*`, unprefixed = `core`) | Zero migration; zero new files; already implicitly in use by all 86 commands works today via name; no schema change needed | Brittle: 8 unprefixed commands ambiguous (is `sync-agents-md` hf or core?); multi-word namespaces impossible; no formal whitelist validation; breaks if rename + namespace diverge | 0 files — Risk: medium (silently wrong namespace on hyphenation edge cases) | **Rec if zero-cost migration is priority** — works today, but breaks for any command outside the 4 established prefixes; not forward-compatible |
| **Registry Manifest** — single centralized `.opencode/commands/manifest.yaml` mapping command name → namespace + metadata | Single source of truth; easy to audit all 86 commands in one view; no change to individual `.md` files; centralized overrides for routing exceptions | Registry drift: new commands need both creation + registry update; extra lookup hop during routing (O(n) or O(1) with map); no frontmatter validation feedback at file-edit time; not how OpenCode reads commands | 1 new file, 1 new schema — Risk: medium (discovery-time drift between manifest and filesystem) | **Rec if centralized auditability is priority** — useful for large teams or cross-namespace routing rules, but adds an always-sync obligation |
| **Hybrid: Frontmatter + Registry** — `namespace` in frontmatter for primary grouping, registry manifest for routing overrides (agent fallback chains, model pinning) | Covers both primary assignment and secondary override needs; namespace still validated per-file; overrides centralized without bloating every command | Two systems to maintain: 86 frontmatter changes + 1 registry; cognitive load for what-goes-where decision; over-engineered for current scale (single-developer project) | 86 files + 1 new file — Risk: medium-high (overhead exceeds value at current project scale) | **Rec if multi-team multi-namespace with override needs** — premature for single-developer harness; revisit at 200+ commands or 3+ namespace owners |

**Rationale:** For this project's current scale (86 commands, single-developer, 4 clear namespace categories with 8 unprefixed outliers), the **Inline Frontmatter Extension** offers the best balance of explicitness, validation, and migration cost. It adds one `namespace` field (enum: `gsd | hf | test | core`) to each command's YAML frontmatter, bumping `CommandFrontmatterSchema` from strict v1.0.0 to v1.1.0 with the new optional field. The schema change is minimal (add `z.enum(["gsd", "hf", "test", "core"]).optional()`), the migration is a straightforward 86-file batch YAML key insertion, and the field integrates with the existing framework-detector `namespacePrefix` pattern. The name-based convention is cheaper (zero migration) but fails for the 8 unprefixed commands and any future namespace that doesn't map cleanly to a filename prefix — this brittleness is unacceptable for a harness meant to support plugin authors. The registry manifest and hybrid approaches add sync obligations and lookup overhead that provide no benefit at the current single-developer scale.
