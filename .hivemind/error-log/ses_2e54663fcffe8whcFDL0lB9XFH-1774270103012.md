## Unified Architecture

### Directory Structure

```
.hivemind/
├── event-log/                          ← Replaces error-log/ + session-inspection/
│   ├── index.md                        ← Master session index (flat table, links)
│   └── {sessionId}/                    ← One folder per session
│       ├── events.md                   ← Semantic event log (append-only, .md)
│       ├── diagnostics.log             ← Verbose log (append-only, one-line-per-event)
│       ├── injection.md                ← Injection payload history (append-only)
│       └── delegation.md               ← Delegation create/close events (per-session)
├── config/                             ← Existing (runtime-attachment.json)
├── state/                              ← Existing (trajectory-ledger.json)
└── ...
```

**Files per session: 4** (down from N+1 per turn in error-log, plus 2 per session in session-inspection)

### Why 4 Files, Not More

| File | Purpose | Why Separate |
|------|---------|-------------|
| `events.md` | Human-readable turn-by-turn journal | Primary forensic surface |
| `diagnostics.log` | Machine-grepable one-line-per-event | `grep "T002"`, `grep "error:"` |
| `injection.md` | What HiveMind injected per turn | Audit trail for context governance |
| `delegation.md` | Sub-session create/close lifecycle | Cross-session delegation chain |

Keeping them separate means each file has ONE concern. Merging them (as time-machine proposed with a single journal.md) creates grep noise and file-locking contention.

### Why NOT Nested Sub-Sessions

The time-machine design proposed `sub-sessions/` directories nested inside parent sessions. We reject this because:

1. **Sub-sessions can be resumed across main sessions** — a sub-session started by session A could be resumed by session B. Nesting it under A's directory is misleading.
2. **Flat is greppable** — `event-log/index.md` lists all sessions with parent/child columns. No need for filesystem nesting.
3. **Each session owns its own delegation.md** — this captures its delegation lifecycle regardless of which parent created it.
