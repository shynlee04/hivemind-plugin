## 2. Path Resolution Pattern Analysis

### 2.1 Correct Pattern (Session-Specific Files)

```
safeSessionPath(projectRoot, sessionId, "session-continuity.json")
→ .hivemind/session-tracker/{sessionId}/session-continuity.json
```

Used correctly by:
- `session-tracker.ts`: export-session, get-status, get-summary (all via `resolvePaths`)
- `session-hierarchy.ts`: readContinuity, handleGetManifest
- `session-context.ts`: readContinuity, synthesize-context
- `hivemind-session-view.ts`: readSessionData

### 2.2 Incorrect Pattern — `project-continuity.json` Reference ⛔ CRITICAL

**Written by** `project-index-writer.ts` line 75:
```
resolve(sessionTrackerRoot(this.projectRoot), "project-continuity.json")
→ .hivemind/session-tracker/project-continuity.json   [FLAT FILE]
```

**Read by** `session-tracker.ts:180` and `session-context.ts:66`:
```
safeSessionPath(projectRoot, "project-continuity", "project-continuity.json")
→ .hivemind/session-tracker/project-continuity/project-continuity.json   [SUBDIRECTORY]
```

**Impact:** The reader constructs a path to a subdirectory (`project-continuity/`) containing a file named `project-continuity.json`. But the writer places the file directly at `.hivemind/session-tracker/project-continuity.json` (no subdirectory). **The read operations always fail unless the file happens to exist at both locations.**

**Remediation:** Replace with `resolve(sessionTrackerRoot(projectRoot), "project-continuity.json")` or equivalently use `safeSessionPath(projectRoot, ".", "project-continuity.json")` — but the latter abuses the API. Better to import `sessionTrackerRoot` and use `resolve()` directly.

### 2.3 Incorrect Pattern — `hierarchy-manifest.json` Global Reference ⛔ CRITICAL

**Written by** `hierarchy-manifest.ts:166`:
```
safeSessionPath(projectRoot, rootMainSessionID, "hierarchy-manifest.json")
→ .hivemind/session-tracker/{rootMainSessionID}/hierarchy-manifest.json   [PER-SESSION]
```

**Read by** `session-tracker.ts:291`:
```
safeSessionPath(projectRoot, "project-continuity", "hierarchy-manifest.json")
→ .hivemind/session-tracker/project-continuity/hierarchy-manifest.json   [WRONG LOCATION]
```

**Impact:** The `filter-sessions` action always hits the `catch` block and returns `"Hierarchy manifest not found. Enable hierarchy tracking first."` **This action is a dead code path.**

**Remediation:** There is no single global `hierarchy-manifest.json`. The filter logic must either:
1. Read the project-continuity.json index (which it already tries but gets wrong), then iterate per-session hierarchy manifests, OR
2. Use an aggregated approach that reads all session directories' hierarchy-manifest.json files

### 2.4 Hardcoded Paths (Acceptable but Brittle)

Used by `hivemind-session-view.ts` lines 63, 76:
```
resolve(projectRoot, ".hivemind", "state", "delegations.json")
resolve(projectRoot, ".hivemind", "state", "trajectory-ledger.json")
```

**Assessment:** These paths are not under session-tracker control, so `safeSessionPath` is not the right tool. However, they lack traversal validation. If `projectRoot` is ever user-supplied or untrusted, these paths could be exploited.

## 9. Root Cause Analysis

The path resolution bugs share a common root cause: **the tools were written assuming `safeSessionPath()` is the universal path builder, but `project-continuity.json` lives at the tracker root (not inside a session subdirectory).** The `safeSessionPath()` API wraps the sessionID into a subdirectory path `<trackerRoot>/<sessionID>/<filename>`, which is correct for session-specific files but wrong for project-level files.

The `hierarchy-manifest.json` bug is a separate design issue: there is no global hierarchy index. The writer creates per-session manifest files; the filter-sessions action assumes a global one exists.

Both bugs would be caught by **integration tests** that write data through the persistence layer and read it back through the tool layer without mocking paths.
