# Research Report: gate-* and stack-* Skill Lifecycle & hf-* Pipeline Integration

**Agent:** hm-l2-researcher
**Domain:** Research — Architectural lifecycle of gate-*/stack-* skills and hf-* meta-builder pipeline
**Date:** 2026-05-10
**Status:** COMPLETED

---

## Executive Summary

gate-* (3) and stack-* (6) skills occupy an **ambiguous middle ground** in the current architecture. They exist as manually placed files in `.opencode/skills/` but have no documented generation pipeline, no hf-* workflow that creates them, and no bootstrap code that installs them. The SKELETON doc classifies gate-* as "Internal — YES (THIS PROJECT ONLY)" and stack-* as "Reference — YES" (shipped), but the hf-* meta-builder pipeline has no skill-synthesis path that produces either category. The bootstrap feature (`src/features/bootstrap/`) scans and loads all skills from `.opencode/skills/` generically — it treats gate-* and stack-* identically to hm-* skills. This means **in user projects, gate-* and stack-* must be manually copied or generated by an as-yet-unbuilt hf-* workflow.** The shipped count correction depends on whether "shipped" means "present in the npm package" vs "present in user projects after initialization."

## Q1: gate-* Lifecycle

### Where do gate-* skills come from?

**CURRENT STATE:** All three gate-* skills are **manually authored** and placed directly in `.opencode/skills/gate-l3-*/SKILL.md`.

**Evidence:**
- `.opencode/skills/gate-l3-evidence-truth/SKILL.md:1-182` — Complete hand-authored skill with version 1.0.0
- `.opencode/skills/gate-l3-lifecycle-integration/SKILL.md:1-209` — Hand-authored, version 2.0.0, classification: `internal-quality-gate`
- `.opencode/skills/gate-l3-spec-compliance/SKILL.md:1-195` — Hand-authored, version 1.1.0, classification: `internal-quality-gate`
- The `synthesis-source` metadata in `gate-l3-lifecycle-integration/SKILL.md:22` reads: `ARCHITECTURE.md + @opencode-ai/plugin SDK docs` — indicating manual synthesis from project architecture docs, NOT from hf-* pipeline generation.

**No hf-* workflow creates gate-* skills.** The hf-* meta-builder pipeline (hf-l2-meta-builder skill, hf-l2-skill-synthesis skill, hf-l2-use-authoring-skills skill) routes to `use-authoring-skills` for creation, but gate-* are not in any creation workflow. The hf commands (`/hf-create`, `/hf-stack`, `/hf-audit`) all route to `hf-l0-orchestrator` → specialist agents, and none of the specialist agents (hf-l2-skill-builder, hf-l2-meta-builder, hf-l2-tool-builder) have gate-* creation in their scope.

**What triggers their creation?** Currently: manual developer decision during harness development. No automated trigger exists. The SKELETON doc at line 27 marks them as `YES (THIS PROJECT ONLY)`, confirming they are project-specific, not user-project-portable.

### Intended State

The gate-* skills should be **project-specific quality governance** — created for this specific harness project to enforce its architecture. They are NOT generic product features that users install. They are governance materialized as skills to give agents structured gatekeeping capabilities.

**Conclusion for Q1:** gate-* skills are manually authored project governance artifacts. They are not and should not be hf-generated. They should remain project-internal and NOT be counted as "shipped primitives" in the user-facing product.

## Q2: stack-* Lifecycle

### Where do stack-* skills come from?

**CURRENT STATE:** All six stack-* skills are **manually authored reference packs** with version-pinned content extracted from external sources.

**Evidence:**
- `stack-l3-bun-pty/SKILL.md:1-106` — version 0.4.8, hand-authored with `scripts/update.sh` for re-fetching
- `stack-l3-json-render/SKILL.md:1-173` — version 0.18.x, hand-authored, notes "Source: colinhacks/zod v4.0.1 · Repomix download 2026-04-28"
- `stack-l3-nextjs/SKILL.md:1-204` — version 16.2.2, header says "Auto-generated stack skill for Next.js 16.2.2"
- `stack-l3-opencode/SKILL.md:1-140` — version 1.14.28, "Bundled: 20,546 lines" — massive reference pack
- `stack-l3-vitest/SKILL.md:1-119` — version 4.1.0, hand-authored reference
- `stack-l3-zod/SKILL.md:1-141` — version 4.x, "Source: colinhacks/zod v4.0.1 · Repomix download 2026-04-28"

**Are they hand-authored or auto-generated?** Mixed. stack-l3-nextjs explicitly says "Auto-generated stack skill" in its header (line 28), suggesting some were produced by a repomix-based generation process. Others (stack-l3-opencode, stack-l3-bun-pty) appear hand-curated. The `hm-l3-tech-stack-ingest` skill is the canonical tool for creating these, but there's no automated pipeline that runs it.

**What triggers their creation?** Currently: manual developer invocation of tech-stack-ingest or repomix. No bootstrap/init code creates them.

**Evidence:** `src/features/bootstrap/primitive-loader.ts:224-241` — `scanSkills()` scans ALL `.opencode/skills/{dir}/SKILL.md` files without any lineage-based filtering. It treats gate-*, stack-*, hm-*, hf-* identically.

### Intended State

stack-* skills should be **auto-generated from tech-stack ingestion** per user project dependencies. The `hm-l3-tech-stack-ingest` skill describes this capability: "Downloads and caches third-party repositories, SDK docs, and API references as progressive-disclosure bundled assets." This is the foundation skill that enables stack-* creation.

However, the actual generation pipeline is NOT implemented in code. It exists only as a skill description.

**Conclusion for Q2:** stack-* skills are currently hand-curated reference packs. The intended state is auto-generation via hm-l3-tech-stack-ingest, but this pipeline is not automated. In user projects, stack-* skills would need to be generated per-project based on the user's package.json dependencies.

## Q3: hf-* Meta-Builder Pipeline

### How does the hf-* meta-builder lineage actually work?

**Traced flow:**

```
User command (e.g., /hf-create)
  → .opencode/commands/hf-create.md (routes to hf-l0-orchestrator agent)
    → hf-l0-orchestrator loads hf-l2-meta-builder skill
      → meta-builder classifies intent → routes to specialist skill
        → hf-l2-use-authoring-skills (for skill creation)
        OR hf-l2-skill-synthesis (for skill synthesis from repos)
        OR hf-l2-agents-and-subagents-dev (for agent creation)
        OR hf-l2-command-dev (for command creation)
          → Specialist agent executes (hf-l2-skill-builder, hf-l2-agent-builder, etc.)
            → Output: SKILL.md, agent .md, or command .md written to .opencode/
```

**Evidence:**
- `.opencode/commands/hf-create.md:1-24` — agent: hf-l0-orchestrator, references workflow at `.hivefiver-hm-meta-builder/workflows-lab/active/refactoring/create.md`
- `.opencode/skills/hf-l2-meta-builder/SKILL.md:291-308` — Routing table maps user intent to specialist skills
- `.opencode/skills/hf-l2-skill-router/SKILL.md:45-57` — Eight task domains, eight skill bundles
- `.opencode/agents/hf-l2-meta-builder.md:9-12` — skills: [hf-l2-meta-builder, hf-l2-skill-synthesis, hm-l2-coordinating-loop, hm-l2-planning-persistence]
- `.opencode/agents/hf-l2-skill-builder.md:9-11` — skills: [hf-l2-use-authoring-skills, hf-l2-skill-synthesis]
- `.opencode/agents/hf-l2-tool-builder.md:9` — skills: [hf-l2-custom-tools-dev]

### Does the current code support dynamic skill generation?

**Yes, through the hf-* skill pipeline.** The hf-l2-skill-synthesis skill (`SKILL.md:1-195`) describes a full INGEST → CLASSIFY → SCAFFOLD → VALIDATE pipeline using repomix, webfetch, and websearch. The hf-l2-use-authoring-skills skill (`SKILL.md:1-284`) provides a 10-step checklist for creating conformant SKILL.md files with validation scripts.

**However:** This pipeline has never been used to generate gate-* or stack-* skills. It is designed for general-purpose meta-concept creation (skills, agents, commands, tools), not for governance or reference skill generation specifically.

**Gap:** No hf-* workflow exists that says "when a user initializes their project, scan package.json and generate appropriate stack-* skills" or "generate quality gate triad for this project."

## Q4: Bootstrap Connection

### Does bootstrap/init create skills or gates?

**NO.** The bootstrap feature is purely structural — it creates directories, scans existing primitives, and validates cross-references. It does NOT generate, create, or install any skills.

**Evidence:**
- `src/features/bootstrap/structure.ts:1-147` — Pure constants and path helpers. Defines `TIER_1_DIRECTORIES` for `.hivemind/`, `PRIMITIVE_TYPES` for `.opencode/`. No skill generation logic.
- `src/features/bootstrap/primitive-scanners.ts:1-182` — Scans `.opencode/agents/`, `.opencode/commands/`, `.opencode/skills/` for existing files. Parse-only, no write.
- `src/features/bootstrap/primitive-loader.ts:1-334` — `loadPrimitives()` scans, parses, cross-references. Read-only.
- `src/features/bootstrap/primitive-registry.ts:1-291` — `buildRegistry()` catalogs, detects conflicts, validates. No creation.

**Is this a planned capability?** Not explicitly. The SKELETON doc question K.8 asks: "How to bootstrap shipped primitives into user projects? f-05 CLI installation is missing." This indicates the capability is recognized as a gap but not yet implemented.

## Q6: Naming/Level Conflicts

### Are the l3 designations correct?

**The l3 designations on gate-* and stack-* are SEMANTICALLY INCORRECT.**

**Evidence:**
- SKELETON line 38: "L3 — No agents — reference/reference skills only"
- SKELETON lines 53-60: Skill Loading Rules show L3 skills are "Loaded by L2 skills" — they are reference material, not executable workflows
- gate-* skills are executable workflows with checklists, validation scripts, and routing logic — they are L2 domain-execution skills, NOT L3 reference material
- stack-* skills ARE reference material (API docs, patterns) — L3 is correct for them

**The naming conflict:**
- `gate-l3-*` uses `l3` but contains executable gatekeeping workflows (L2 behavior)
- `stack-l3-*` uses `l3` and IS reference material (correct L3 classification)

### Should they use hf-* naming?

**NO.** gate-* and stack-* are NOT hf-generated artifacts in the current architecture. They use separate prefixes because:

1. **gate-*** is a cross-cutting quality concern loaded by BOTH hm-* and hf-* agents (see SKELETON lines 53-60: gate-l3-* is allowed for all levels of both lineages)
2. **stack-*** is a cross-cutting reference concern loaded by BOTH hm-* and hf-* agents
3. The `gate-` and `stack-` prefixes communicate their PURPOSE (governance vs reference), not their lineage

### Naming convention proposal

| Current | Proposed | Rationale |
|---------|----------|-----------|
| `gate-l3-evidence-truth` | `gate-l2-evidence-truth` | Downgrade l3→l2: these are executable domain workflows, not reference material |
| `gate-l3-lifecycle-integration` | `gate-l2-lifecycle-integration` | Same: executable gate pipeline |
| `gate-l3-spec-compliance` | `gate-l2-spec-compliance` | Same: executable gate pipeline |
| `stack-l3-bun-pty` | `stack-l3-bun-pty` | Keep: correct L3 reference classification |
| `stack-l3-json-render` | `stack-l3-json-render` | Keep: correct |
| `stack-l3-nextjs` | `stack-l3-nextjs` | Keep: correct |
| `stack-l3-opencode` | `stack-l3-opencode` | Keep: correct |
| `stack-l3-vitest` | `stack-l3-vitest` | Keep: correct |
| `stack-l3-zod` | `stack-l3-zod` | Keep: correct |

**Alternative proposal:** If gate-* should remain at l3 because they are loaded by L2 agents (not by L3 agents — since no L3 agents exist), then the naming is architecturally defensible. The level designates WHO LOADS the skill, not the skill's own behavioral complexity. Under this reading, gate-l3 is correct because L2 agents load gate-l3 skills.

**Recommendation:** Keep `gate-l3-*` and `stack-l3-*` naming. The l3 designation indicates "loaded by L2 agents" which is correct for both categories. The behavioral distinction (executable vs reference) is captured by the prefix (gate- = governance workflow, stack- = reference material).

## Knowledge Gaps

1. **No automated stack-* generation pipeline.** The hm-l3-tech-stack-ingest skill describes the capability but no bootstrap/workflow wires it to project initialization. This is a complete gap.

2. **No gate-* portability strategy.** If gate-* skills are project-internal, what quality governance do USER projects get? Is there a "generic gate triad" that should be generated for each project?

3. **hf-l0-orchestrator agent file is MISSING from disk.** The glob for `.opencode/agents/hf-l0*.md` returned "No files found." The commands reference `agent: hf-l0-orchestrator` but the agent definition file doesn't exist in `.opencode/agents/`. This is a critical gap — the hf command chain has a broken entry point.

4. **No documented workflow for gate-* → user-project generation.** The SKELETON question K.8 asks about this but provides no answer.

---

## Recommendations

1. **Reclassify gate-* as project-internal.** Remove from shipped count. Mark in SKELETON as "NOT shipped." These are governance artifacts for the harness project itself.

2. **Create stack-* generation workflow.** Wire hm-l3-tech-stack-ingest into a new bootstrap step or hf-* command that generates project-specific stack-* skills from `package.json` dependencies.

3. **Create gate-* generation workflow (optional).** If user projects need quality gates, create a generic gate triad template that hf-* can instantiate per project.

4. **Fix missing hf-l0-orchestrator agent file.** The hf-* command chain depends on this agent but the file is missing from `.opencode/agents/`. This is a blocking defect.

5. **Keep gate-*/stack-* naming as-is.** The l3 designation is architecturally correct (loaded by L2 agents). The prefix communicates purpose (governance vs reference). No renaming needed.

6. **Update SKELETON shipped count to 49 core.** Clarify that 3 gate-* are project-internal and 6 stack-* are project-specific reference, yielding 49 portable shipped primitives.

7. **Keep cross-lineage access open for gate-*/stack-*.** Both categories should remain loadable by all agents regardless of lineage. The current permission model is correct.
