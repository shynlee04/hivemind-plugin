### Pattern 1: Two-Stage Sequential Routing

**What:** Command resolution (Stage 1) → Agent selection (Stage 2), sequential execution.

**When to use:** Khi user nhập `/command args` hoặc natural language như "gsd command".

**Example:**
```typescript
// Từ CONTEXT.md §42
// Stage 1: resolve command
const resolvedCommand = await resolveCommand({
  input: "gsd command",  // Natural language
  commands: [/* all commands */]
})
if (!resolvedCommand) {
  // Command not found → ask user and STOP (no Stage 2)
  return "Command not found. Available commands: /gsd-stats, /plan, ..."
}

// Stage 2: resolve agent (independent, can fallback)
const resolvedAgent = await resolveAgent({
  intent: "gsd",  // Extracted from Stage 1
  agents: [/* all agents */]
})
// If no agent matches → use default agent (continue execution)
```

**Source:** CONTEXT.md §42-48, SPEC.md §41-50

### Anti-Patterns to Avoid

- **❌ Parallel routing without error handling:** Command và agent resolution independent nhưng nếu command fail → không chạy agent selection. **Fix:** Sequential strict, command fail → ask user and stop.
- **❌ Hardcoded namespace values:** `namespace?: string` optional, no enum, no hardcoded values (gsd/hf/test/core). **Fix:** Defer namespace routing logic to P24.3.3.2.
- **❌ LLM fallback for semantic matching:** Zero LLM calls, instant response, no hallucination risk. **Fix:** Pure keyword + fuzzy matching only.
- **❌ Validate SDK-driven fields (agent, subtask, model, tools):** SDK handles inheritance automatically. **Fix:** Only validate structural fields (name, description, triggers).

## Common Pitfalls

### Pitfall 1: Semantic Matching Over-Engineering
**What goes wrong:** Implement LLM fallback hoặc fuzzy library (fuzzy-search) khi keyword matching đủ.
**Why it happens:** Training data shows "semantic matching = LLM" pattern.
**How to avoid:** Pure keyword + fuzzy matching only. Agents already have LLM access + all metadata available.
**Warning signs:** Adding LLM calls, importing fuzzy libraries, complex NLP logic.

### Pitfall 2: Breaking Backward Compatibility
**What goes wrong:** Existing commands fail vì validation logic quá strict.
**Why it happens:** Validator validate SDK-driven fields (agent, subtask) thay vì chỉ structural fields.
**How to avoid:** Chỉ validate structural fields (name, description, triggers). SDK-driven fields → inherit from context, not validate.
**Warning signs:** Tests fail for existing commands, validation rejects valid frontmatter.

### Pitfall 3: Agent Validation Race Condition
**What goes wrong:** Agent validation gọi `getAppAgents()` nhưng registry chưa load xong.
**Why it happens:** Không chờ app-agents registry ready trước khi validate.
**How to avoid:** Use `client.app.agents()` (SDK method) — handles async loading, hoặc fallback gracefully.
**Warning signs:** Intermittent agent not found errors, validation fails then succeeds.

### Pitfall 4: Two-Stage Routing Logic Error
**What goes wrong:** Stage 1 (command) và Stage 2 (agent) chạy parallel thay vì sequential.
**Why it happens:** Không hiểu requirement "Sequential execution (STRICT): Stage 1 → Stage 2".
**How to avoid:** Hard code sequence: `if (!resolvedCommand) → ask user and STOP`. No Stage 2 nếu command fail.
**Warning signs:** Agent selected but command not found, or vice versa.

### 1. Command Discovery with Fuzzy Matching (Existing Pattern)

```typescript
// Từ execute-slash-command.ts:479-509
async function findCommandBundle(projectRoot: string, commandName: string): Promise<CommandBundle | undefined> {
  // Cache discovery kết quả (5s TTL)
  const cached = discoveryCache.get(projectRoot)
  let commands: CommandBundle[]
  if (cached && now - cached.timestamp < CACHE_TTL_MS) {
    commands = cached.commands
  } else {
    const discovery = await discoverCommandBundles({ projectRoot })
    commands = discovery.commands
    discoveryCache.set(projectRoot, { commands, timestamp: now })
  }

  // Exact match first
  const exact = commands.find((command) => command.name === commandName)
  if (exact) return exact

  // Case-insensitive, hyphen/underscore normalized comparison
  const normalizedSearch = commandName.toLowerCase().replace(/[-_]/g, "")
  const fuzzy = commands.find((command) => {
    const normalizedCandidate = command.name.toLowerCase().replace(/[-_]/g, "")
    return normalizedCandidate === normalizedSearch
  })
  if (fuzzy) return fuzzy

  // Substring prefix/suffix matching
  return commands.find((command) => {
    const candidate = command.name.toLowerCase()
    const search = commandName.toLowerCase()
    return candidate.includes(search) || search.includes(candidate)
  })
}
```

### 3. Contract Validation with ValidationResult (Target Pattern)

```typescript
// Target: validate-command.ts
import { z } from "zod"
import { ValidationResult } from "../../shared/types.js"

export function validateCommandContract(args: any): ValidationResult {
  const errors: string[] = []

  // Structural fields (required)
  if (!args.command || typeof args.command !== "string" || args.command.length === 0) {
    errors.push("command: required, must be a non-empty string")
  }
  if (!args.description || typeof args.description !== "string") {
    errors.push("description: required, must be a non-empty string")
  }
  if (!args.triggers || !Array.isArray(args.triggers) || args.triggers.length === 0) {
    errors.push("triggers: required, must be a non-empty array")
  }

  // SDK-driven fields (optional, NOT validated here)
  // agent, subtask, model, tools → inherit from context, not validate

  return {
    valid: errors.length === 0,
    errors,
  }
}
```

## Open Questions

1. **Q1: Semantic matching algorithm complexity**
   - What we know: Keyword + fuzzy matching (case-insensitive, hyphen/underscore normalized)
   - What's unclear: Should we add cosine similarity hoặc Levenshtein distance cho fuzzy ranking?
   - Recommendation: Start simple (keyword + substring match), add Levenshtein only if needed (user feedback).

2. **Q2: Agent selection fallback behavior**
   - What we know: If no agent matches → use default agent (from TUI or `commandBundle?.agent`)
   - What's unclear: What is "default agent"? How to determine from TUI?
   - Recommendation: Default to `commandBundle?.agent` if available, else first agent from registry.

3. **Q3: Two-stage routing error messages**
   - What we know: Command fail → ask user "Command not found. Available: ..."
   - What's unclear: Should we list ALL commands or filtered by intent?
   - Recommendation: List top 5 commands matching user intent keywords.
