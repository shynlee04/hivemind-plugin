# Audit: hivemind-delegation-protocol

**Date:** 2026-03-22
**Auditor:** OpenCode skill-review
**Skill version:** refactored-skills pack (post-L1 cutover)
**Files reviewed:** SKILL.md (113 lines), 4 templates, 4 references, 1 test, pack-level AGENTS.md

---

## 1. Executive Summary

**Overall Grade: B+**

The hivemind-delegation-protocol skill is structurally sound and conceptually well-bounded. Its SKILL.md is concise (113 lines), the progressive disclosure model loads references on demand, and the delegation packet/handoff-brief templates are clean and usable. The core protocol is strong on orchestrator protection, sequential-first discipline, and structured return contracts.

However, the skill has significant gaps in three areas: (1) the SKILL.md is a **skeleton with section headers but no bodies** — six sections reference content or say "Read X" but several sections (Shared Return Contract, Delegation Modes) are bare stubs that repeat what references already cover; (2) test coverage is minimal — a single scenario with no validation criteria; (3) failure handling, recovery, and error-escalation procedures are largely absent. These gaps are addressable without restructuring the pack.

## 3. Top 5 Priority Improvements

### Priority 1: Add failure handling and recovery protocol (W-1, M-2)
**Impact:** Critical — without this, agents encountering failed/partial delegations have no guidance, leading to silent stalls or incorrect synthesis.
**Effort:** Medium — add one section (~30 lines) to SKILL.md and one reference file (~60 lines).
**Fix:** Add "Failure and Recovery" section covering: partial return handling, timeout protocol, blocked-route escalation ladder, parallel-slice failure isolation.

### Priority 2: Specify delegation decision criteria (D-1, D-4)
**Impact:** High — agents cannot reliably decide when to delegate vs execute inline without explicit criteria.
**Effort:** Low — expand SKILL.md step 1 with a decision heuristic (~10 lines) and add "When Not to Delegate" sub-section (~5 lines).
**Fix:** Add concrete criteria (file count, session freshness, concern type) and anti-criteria (single-file, fresh context, user asked to do it directly).

### Priority 3: Expand task decomposition guidance (W-4)
**Impact:** High — "Write the slice boundary" is the core action but has no decomposition heuristics.
**Effort:** Low — add decomposition rules to SKILL.md (~15 lines).
**Fix:** Decompose by authority surface, concern type, file clusters. Max 5 files per slice. Split read-only from write-capable.

### Priority 4: Add 3+ test scenarios with validation criteria (X-1, X-2)
**Impact:** High — the single unvalidated test scenario cannot prove the skill works for any agent.
**Effort:** Medium — create 3 test files (~20 lines each) with concrete pass/fail checks.
**Fix:** Add tests for: simple execution delegation, codescan multi-pass, failure with blocked_routes.

### Priority 5: Expand session resume and memory retrieval (W-2)
**Impact:** Medium — the resume protocol references `task_id` and `pass_id` but doesn't explain how to detect prior completion or diff returns.
**Effort:** Low — expand SKILL.md lines 92-98 (~10 lines).
**Fix:** Add field-check protocol (status values, re-delegate conditions, resume-from-action logic).

## 4. Strengths to Preserve

1. **SKILL.md conciseness (113 lines).** The file is under 500 lines and uses progressive disclosure correctly — references are loaded on demand rather than inlined. This is the right structural pattern.

2. **Orchestrator protection rules (lines 27-34).** Four tight, imperative rules that prevent the most common failure mode (orchestrator doing deep work instead of delegating). The "stale context is a delegation trigger" rule is particularly well-framed.

3. **Sequential-first discipline.** Both the SKILL.md (line 40) and references/delegation-modes.md enforce sequential by default with a clear parallel gate. This prevents the race-condition and merge-conflict class of bugs.

4. **Structured return contract.** The return contract (routing, identity, scope tracking, evidence, control) is well-decomposed and covers the essential fields. The instruction "if a child cannot produce this shape, return a bounded partial result rather than pretending the slice is complete" is excellent pragmatic guidance.

5. **Template quality.** All four templates are clean, well-structured, and ready to fill. The delegation-packet template (20 fields) covers packet identity, scope, constraints, evidence, and return contract. The loop-checkpoint template includes iteration and bead-entry schemas for fine-grained tracking.

6. **Codescan delegation is the strongest sub-protocol.** The 6-step scan pass structure, agent selection table, resumable scan protocol, and multi-pass chaining are all concrete and actionable. This section has the most complete operational guidance in the entire skill.

7. **Anti-patterns are listed (iterative-loop-control.md:123-129).** Five anti-patterns covering the most common loop-control failures. While they lack WHY explanations, the list itself is correct and complete.

8. **Pack-level integration is consistent.** The activity folder path (`{project}/.hivemind/activity/delegation/`), `activity_type`/`phase_type` enums, and deterministic pathing all align with the pack-level AGENTS.md convention. No conflicts detected.
